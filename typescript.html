TypeScript: Beginner to Master

Complete 15-Week Curriculum — From Static Types to Production-Grade TypeScript Applications

About This Curriculum

This is a 15-week, phase-based TypeScript curriculum designed for JavaScript developers and absolute beginners who want to reach professional-level TypeScript mastery. The roadmap moves from TypeScript setup, core types, and type inference through functions, interfaces, generics, and modules, into OOP, error handling, utility types, and real-world libraries, all the way to decorators, advanced type patterns, concurrency, and three complete capstone projects.

The curriculum has four phases: Phase 1 covers absolute basics and TypeScript's type system, Phase 2 covers control flow, functions, interfaces, generics, and modules, Phase 3 covers intermediate and advanced core skills, and Phase 4 covers master-level patterns and three complete production projects.

By Week 15, you should be able to architect large TypeScript codebases, write advanced type utilities, integrate TypeScript with real libraries, and build production-quality applications with full type safety end to end.

Level: Beginner → Master  |  Duration: 15 weeks  |  Study Time: ~3–5 hours/week  |  Audience: JavaScript developers and beginners

Phase 1 — Absolute Basics

Weeks 1–3: Setup, core types, type inference, and type narrowing.

Week 1 — Introduction to TypeScript & Environment Setup

Understand what TypeScript is, why it exists, and get a working development environment running.

Core Focus

Understanding what TypeScript is, why it exists, and getting a working development environment running.

Topics & Subtopics

  1. What is TypeScript & Why Use It? — TypeScript is a statically typed superset of JavaScript that compiles to plain JavaScript, catching errors at build time rather than runtime. It adds optional static types, interfaces, and modern language features that make large codebases dramatically safer and easier to maintain.
  2. Installing TypeScript & Configuring tsconfig.json — TypeScript is installed via npm and configured through tsconfig.json, which controls compiler behavior, target JavaScript version, and strictness. Running tsc --init generates a sensible default config.
  3. Running TypeScript (tsc, ts-node, tsx) — tsc compiles .ts files to .js; ts-node runs TypeScript directly without a separate compile step; tsx is a faster modern alternative. For development, ts-node or tsx paired with nodemon provides a smooth watch-and-reload workflow.
  4. Type Annotations vs Type Inference — TypeScript can infer types from assigned values, so you do not always need to annotate explicitly. Explicit annotations are best for function signatures, API boundaries, and anywhere the inferred type might be ambiguous.

Why This Week Matters: TypeScript's entire value proposition is catching errors before they reach production. Setting up with strict: true from the start establishes the discipline and safety that makes TypeScript worth using at all.

Code Example

// TypeScript catches this at compile time — JavaScript would not
function greet(name: string): string {
    return `Hello, ${name}!`;
}
greet("Alice");  // valid
// greet(42);   // Error: Argument of type 'number' is not assignable to 'string'

// Type inference — no annotation needed
let score = 100;        // inferred: number
let username = "Alice"; // inferred: string

// Explicit annotation — required when no initial value
let result: number;

🛠 Practical Task

Install TypeScript and create a new project with tsconfig.json set to strict: true. Write a greetUser(name: string, age: number): string function, run it with ts-node, intentionally pass wrong types, and observe the compiler errors.

Week 2 — Primitive Types, Arrays & Tuples

Learn TypeScript's core built-in types and how to type collections of data.

Core Focus

Learning TypeScript's core built-in types and understanding how to type collections of data.

Topics & Subtopics

  1. Primitive Types — TypeScript's fundamental value types: string, number, boolean, null, undefined, symbol, and bigint. With strict: true, null and undefined are not assignable to other types unless explicitly declared.
  2. Arrays & Readonly Arrays — Arrays are typed as Type[] or Array<Type>. Use readonly arrays when the array should never be mutated after creation — ideal for configuration data and function parameters.
  3. Tuples — Tuples are fixed-length arrays where each position has a specific type. They are ideal for structured pairs or triples like coordinates, key-value pairs, or multi-value function returns.
  4. The any, unknown, and never Types — any disables all type checking and should be avoided. unknown is the type-safe alternative requiring narrowing before use. never represents values that can never occur and is a powerful correctness tool.

Why This Week Matters: unknown versus any is one of TypeScript's most important distinctions. Using unknown forces you to handle data safely, while any silently defeats the entire purpose of TypeScript.

Code Example

let name: string    = "Alice";
let age: number     = 30;
let active: boolean = true;
let id: bigint      = 9007199254740993n;

const scores: number[]        = [95, 87, 100];
const tags: readonly string[] = ["ts", "types"];
// tags.push("new"); // Error: readonly array

const user: [string, number, boolean] = ["Alice", 30, true];
const point: [x: number, y: number]   = [10, 20];

function parseData(input: unknown): string {
    if (typeof input === "string") return input.toUpperCase();
    if (typeof input === "number") return input.toFixed(2);
    throw new Error("Unsupported type");
}

🛠 Practical Task

Model a simple CSV row as a tuple with named elements [id: number, name: string, score: number]. Write a function that accepts unknown data, narrows it, and extracts the row. Use readonly arrays to prevent mutation of loaded data.

Week 3 — Union Types, Intersection Types & Type Narrowing

Combine types flexibly and learn how TypeScript narrows types through control flow.

Core Focus

Combining types flexibly with union and intersection, and learning how TypeScript narrows types at runtime.

Topics & Subtopics

  1. Union Types (|) — A union type allows a value to be one of several types. They are common in API responses, form inputs, and situations where multiple types are genuinely valid. TypeScript forces you to handle each case.
  2. Intersection Types (&) — Intersection types combine multiple types into one — the result must satisfy all of them. Most useful for merging object shapes like composing a base type with role-specific properties.
  3. Type Narrowing — TypeScript tracks type information through control flow. After a typeof, instanceof, or equality check, TypeScript knows the narrowed type inside that block and unlocks its specific methods.
  4. Discriminated Unions & Exhaustiveness Checking — A discriminated union is a union of object types sharing a literal tag property. Combined with never checks, TypeScript errors at compile time if you add a new union member but forget to handle it.

Why This Week Matters: Discriminated unions with exhaustiveness checking are one of TypeScript's most powerful correctness guarantees. They make it impossible to add a new state or action without handling it everywhere — exactly what large codebases need.

Code Example

type ID = string | number;
function formatId(id: ID): string {
    if (typeof id === "number") return `#${id.toFixed(0).padStart(6, "0")}`;
    return id.trim().toUpperCase();
}

type Timestamped = { createdAt: Date; updatedAt: Date };
type Named = { name: string };
type Entity = Named & Timestamped;

type Shape =
    | { kind: "circle"; radius: number }
    | { kind: "square"; side: number };

function area(shape: Shape): number {
    if (shape.kind === "circle") return Math.PI * shape.radius ** 2;
    return shape.side ** 2;
}

type Result<T> =
    | { status: "ok";    data: T }
    | { status: "error"; message: string };

🛠 Practical Task

Model an API response using a discriminated union with three states: loading, success<T>, and error. Write a renderUI(state) function that handles all three, and add an exhaustiveness check with never so TypeScript errors when you add a fourth state without updating the function.

Phase 2 — Control Flow & Logic

Weeks 4–7: Functions, interfaces, generics, and modules.

Week 4 — Functions & Advanced Function Types

Master TypeScript's rich function typing system — overloads, callbacks, and higher-order functions.

Core Focus

Mastering TypeScript's rich function typing system including overloads, callbacks, and higher-order functions.

Topics & Subtopics

  1. Typed Function Signatures — Every function parameter and return type should be explicitly annotated at API boundaries. void signals the return value should not be used; never signals the function never returns.
  2. Optional & Default Parameters, Rest Parameters — Optional parameters (suffix ?) and default values make function APIs flexible without overloads. Rest parameters (...args: T[]) collect a variable number of arguments into a typed array.
  3. Function Overloads — Overloads let you declare multiple call signatures for a single function with different input/output type combinations. The implementation signature must be compatible with all overloads but is not visible to callers.
  4. Higher-Order Functions & Function Type Aliases — Functions that accept or return other functions need proper typing. Use type aliases for callback shapes to keep signatures readable and reusable across the codebase.

Why This Week Matters: Function overloads and typed higher-order functions are essential in any real TypeScript codebase. They let you write APIs that are both flexible for callers and completely type-safe internally.

Code Example

function divide(a: number, b: number): number {
    if (b === 0) throw new Error("Division by zero");
    return a / b;
}

function createUser(
    name: string,
    role: string = "viewer",
    age?: number
): string {
    return `${name} (${role})${age ? `, age ${age}` : ""}`;
}

function parse(input: string): number;
function parse(input: number): string;
function parse(input: string | number): string | number {
    return typeof input === "string" ? parseFloat(input) : String(input);
}

type Predicate<T>       = (value: T) => boolean;
type Transformer<T, U> = (value: T) => U;

function filter<T>(arr: T[], pred: Predicate<T>): T[] {
    return arr.filter(pred);
}

🛠 Practical Task

Write a typed pipe(...fns) utility that chains transformations so that pipe(double, addOne, toString)(5) produces "11". Use function type aliases, rest parameters, and generics. Add overloads so pipe with 1, 2, and 3 functions returns correctly typed results.

Week 5 — Interfaces & Object Types

Design structured object shapes with interfaces and type aliases, and understand when to use each.

Core Focus

Designing structured object shapes with interfaces and type aliases, and understanding when to use each.

Topics & Subtopics

  1. Interfaces — Defining Object Shapes — Interfaces define the contract an object must fulfill. They support optional properties, readonly fields, and method signatures, and can be extended across multiple declarations via declaration merging.
  2. Extending Interfaces & Type Aliases — Interfaces extend other interfaces with extends, creating inheritance chains. Type aliases achieve composition with & intersection. Use extends for classical hierarchies and & for ad-hoc composition.
  3. Index Signatures & Record Types — Index signatures describe objects with dynamic keys of a consistent type. Record<K, V> is a cleaner shorthand for the same pattern and is preferred in modern TypeScript.
  4. interface vs type — When to Use Each — interface is best for public API shapes, class contracts, and extensible object types. type is best for unions, intersections, mapped types, and conditional types that interfaces cannot express.

Why This Week Matters: The interface vs type decision comes up constantly in TypeScript. Understanding their distinct capabilities helps you make the right structural choices from the start.

Code Example

interface User {
    readonly id: number;
    name:        string;
    email?:      string;
    greet():     string;
}

interface Animal { name: string; }
interface Pet    extends Animal { owner: string; }
interface Dog    extends Pet    { breed: string; }

type Scores = Record<string, number>;
const scores: Scores = { Alice: 95, Bob: 87 };

type Status  = "active" | "inactive" | "pending";
type Handler = (event: MouseEvent) => void;
interface Repository<T> { findById(id: number): Promise<T>; }

🛠 Practical Task

Design an interface hierarchy for a blogging platform: Author extends User, Post has an Author and an array of Comment. Add an index signature for Post metadata. Build a PostRepository interface with findById, findAll, and save methods.

Week 6 — Generics

Write reusable, type-safe code that works across many types using TypeScript's generics system.

Core Focus

Writing reusable, type-safe code that works across many types using TypeScript's generics system.

Topics & Subtopics

  1. Generic Functions — Generics add a type parameter to a function so it can work with any type while preserving type information. The caller decides the type; the function body remains agnostic, giving safety without sacrificing flexibility.
  2. Generic Interfaces & Type Aliases — Generics applied to interfaces and type aliases create reusable data structure definitions. A single ApiResponse<T> interface describes every endpoint's response shape.
  3. Generic Constraints (extends) — Constraints limit which types a generic can accept. T extends SomeType ensures the generic has at least the properties of SomeType, letting you safely access those properties in the function body.
  4. Generic Defaults & Multiple Type Parameters — Generic parameters can have default types (<T = string>) making them optional at call sites. Multiple type parameters model relationships between types such as key-value structures or transformation pairs.

Why This Week Matters: Generics are what transform TypeScript from a "JavaScript with types" language into a truly powerful type-safe system. Every major library and framework you will use is built on generics.

Code Example

function identity<T>(value: T): T { return value; }
function first<T>(arr: T[]): T | undefined { return arr[0]; }

const name = first(["Alice", "Bob"]); // inferred: string | undefined

interface ApiResponse<T> {
    data:      T;
    status:    number;
    message:   string;
    timestamp: Date;
}

interface HasId { id: number; }
function findById<T extends HasId>(items: T[], id: number): T | undefined {
    return items.find(item => item.id === id);
}

function mapObject<K extends string, V, U>(
    obj: Record<K, V>,
    transform: (value: V) => U
): Record<K, U> {
    return Object.fromEntries(
        Object.entries(obj).map(([k, v]) => [k, transform(v as V)])
    ) as Record<K, U>;
}

🛠 Practical Task

Build a generic Stack<T> class with push, pop, peek, and isEmpty methods. Then write a generic groupBy<T, K extends string>(arr: T[], key: (item: T) => K): Record<K, T[]> utility function using constraint types throughout.

Week 7 — Modules, Namespaces & Declaration Files

Organise TypeScript code into modules, import third-party libraries safely, and understand .d.ts declaration files.

Core Focus

Organizing TypeScript code into modules, importing third-party libraries safely, and understanding declaration files.

Topics & Subtopics

  1. ES Modules in TypeScript — TypeScript uses ES module syntax natively. Named exports share multiple values; default exports share one primary value. Named exports are preferred for better refactoring support and tree-shaking in bundlers.
  2. Re-exports & Barrel Files (index.ts) — A barrel file re-exports from multiple modules through a single index.ts, creating a clean public API for a folder. Consumers import from the folder, not individual files, making internal refactoring invisible to callers.
  3. Installing & Using @types Packages — Many JavaScript libraries ship without TypeScript types. The @types scope on npm provides type declarations for thousands of packages. Install them as dev dependencies.
  4. Writing .d.ts Declaration Files — When TypeScript cannot infer types for a plain JavaScript file or global library, you write a .d.ts declaration file describing its types. These are also what you produce when publishing a typed library.

Why This Week Matters: Professional TypeScript projects are always multi-file, multi-module codebases. Barrel files, @types packages, and declaration files are what make large TypeScript projects maintainable and safe at scale.

Code Example

// math.ts
export function add(a: number, b: number): number { return a + b; }
export const PI = 3.14159;

// src/utils/index.ts — barrel file
export { formatDate, parseDate } from "./date";
export { formatCurrency }        from "./currency";
export type { FormatOptions }    from "./types";

// npm install --save-dev @types/lodash
import _ from "lodash";
const grouped = _.groupBy([1, 2, 3, 4], n => n % 2 === 0 ? "even" : "odd");

// global.d.ts
declare global {
    interface Window { analytics: { track: (event: string) => void }; }
}
declare module "my-untyped-lib" {
    export function compute(input: number): number;
}

🛠 Practical Task

Refactor a single-file TypeScript project into a module structure with separate files for types, utilities, and feature modules unified by a barrel index.ts. Install lodash and @types/lodash, write a typed wrapper, and write a .d.ts for a hypothetical untyped legacy JS utility.

Phase 3 — Intermediate & Advanced Core

Weeks 8–11: OOP, error handling, utility types, and real-world libraries.

Week 8 — Object-Oriented TypeScript (Classes)

Use TypeScript's class system including access modifiers, abstract classes, and interface contracts.

Core Focus

Using TypeScript's class system including access modifiers, abstract classes, and implementing interfaces.

Topics & Subtopics

  1. Classes, Constructors & Access Modifiers — TypeScript adds public, private, protected, and readonly modifiers to class members. Constructor parameter shorthand simultaneously declares and assigns properties in one step.
  2. Inheritance & Method Overriding — TypeScript classes support single inheritance via extends. Use super() for the parent constructor and super.method() for parent methods. The override keyword (TS 4.3+) makes overriding explicit and compiler-checked.
  3. Abstract Classes & Interfaces as Contracts — Abstract classes define a base with some implementation and some abstract methods subclasses must provide. When a class implements an interface, TypeScript verifies the full contract at compile time.
  4. Static Members & Private Class Fields — static members belong to the class itself, useful for factories, counters, and utilities. Private class fields (#field) using native JavaScript syntax are truly private at runtime.

Why This Week Matters: TypeScript's access modifiers and abstract classes enforce design contracts at compile time that JavaScript simply cannot. This is where TypeScript's OOP capabilities genuinely exceed JavaScript's.

Code Example

class BankAccount {
    private balance: number;
    constructor(public readonly owner: string, initial: number) {
        this.balance = initial;
    }
    deposit(amount: number): void { this.balance += amount; }
    getBalance(): number          { return this.balance; }
}

class SavingsAccount extends BankAccount {
    constructor(owner: string, initial: number, private rate: number) {
        super(owner, initial);
    }
    override deposit(amount: number): void {
        super.deposit(amount * (1 + this.rate));
    }
}

abstract class Shape {
    abstract area(): number;
    describe(): string { return `Area: ${this.area().toFixed(2)}`; }
}

class IdGenerator {
    static #count = 0;
    static next(): number  { return ++IdGenerator.#count; }
    static reset(): void   { IdGenerator.#count = 0; }
}

🛠 Practical Task

Design a class hierarchy for a payment system: abstract PaymentMethod with abstract charge(amount: number) and refund(amount: number) methods, then concrete CreditCard and Wallet subclasses. Implement a PaymentProcessor interface and verify the contract at compile time.

Week 9 — Error Handling & Type-Safe Async Code

Handle errors idiomatically in TypeScript, type async operations correctly, and model failures explicitly.

Core Focus

Handling errors idiomatically, typing async operations correctly, and modeling failures explicitly.

Topics & Subtopics

  1. Typed Error Handling — In strict mode, catch blocks receive unknown typed errors, requiring narrowing before accessing properties. This prevents the common mistake of assuming every caught value is an Error instance.
  2. Custom Error Classes — Create typed error subclasses to carry domain-specific context. instanceof checks on custom errors enable precise, typed error handling that distinguishes between ValidationError, NetworkError, and NotFoundError.
  3. Promises & Async/Await with Correct Types — Promise<T> is the return type for async operations resolving to T. async functions implicitly return Promise<T>. Typing async operations correctly ensures callers know what to expect.
  4. Result Types — Explicit Error Modelling — Instead of throwing, model operations as Result<T, E> — a union of success and failure. This forces callers to acknowledge potential failures and eliminates invisible error paths.

Why This Week Matters: TypeScript's unknown error type in catch blocks and the Result pattern together represent a major advancement over runtime error handling. They make failure paths as type-safe as success paths.

Code Example

class AppError extends Error {
    constructor(message: string, public readonly code: string) {
        super(message);
        this.name = "AppError";
        Object.setPrototypeOf(this, new.target.prototype);
    }
}
class NotFoundError extends AppError {
    constructor(entity: string, id: number) {
        super(`${entity} with id ${id} not found`, "NOT_FOUND");
    }
}

async function fetchPost(id: number): Promise<Post> {
    const res = await fetch(`https://jsonplaceholder.typicode.com/posts/${id}`);
    if (!res.ok) throw new NotFoundError("Post", id);
    return res.json() as Promise<Post>;
}

type Ok<T>  = { success: true;  value: T };
type Err<E> = { success: false; error: E };
type Result<T, E = Error> = Ok<T> | Err<E>;

function divide(a: number, b: number): Result<number, string> {
    if (b === 0) return { success: false, error: "Division by zero" };
    return { success: true, value: a / b };
}

🛠 Practical Task

Week 10 — Utility Types & Advanced Type Manipulation

Use TypeScript's powerful built-in utility types and begin writing your own type-level transformations.

Core Focus

Using TypeScript's powerful built-in utility types and beginning to write custom type-level transformations.

Topics & Subtopics

  1. Core Utility Types — TypeScript ships with utility types that transform existing types: Partial<T> makes all fields optional, Required<T> makes them all required, Pick<T, K> selects fields, and Omit<T, K> removes fields.
  2. Conditional Types (T extends U ? X : Y) — Conditional types select between types based on a type-level condition. They are the foundation of many advanced utility types and enable powerful type inference patterns.
  3. Mapped Types — Mapped types transform every property in an existing type by iterating over its keys. Combined with modifiers you can add or remove optionality and readonly constraints programmatically.
  4. Template Literal Types & infer — Template literal types compose string types at the type level — useful for typed event names, CSS property strings, and API routes. infer extracts a type from within a conditional type for reuse.

Why This Week Matters: Utility types and mapped types are what enable TypeScript to be genuinely expressive rather than just annotative. They allow you to derive types from other types, keeping your code DRY even at the type level.

Code Example

interface User { id: number; name: string; email: string; age?: number; }

type CreateUserDTO = Omit<User, "id">;
type UserPreview   = Pick<User, "id" | "name">;
type PartialUser   = Partial<User>;

type IsArray<T> = T extends any[] ? true : false;
type Unwrap<T>  = T extends Array<infer U> ? U : T;

type Nullable<T> = { [K in keyof T]: T[K] | null };
type Mutable<T>  = { -readonly [K in keyof T]: T[K] };

type EventName<T extends string> = `on${Capitalize<T>}`;
type ReturnTypeOf<T> = T extends (...args: any[]) => infer R ? R : never;

🛠 Practical Task

Week 11 — Working with Real-World Libraries (Node.js, Zod, Prisma)

Apply TypeScript in real-world scenarios with typed libraries for validation, database access, and server development.

Core Focus

Applying TypeScript in real-world scenarios with typed libraries for runtime validation, database access, and server development.

Topics & Subtopics

  1. TypeScript with Node.js & Express — Use @types/node and @types/express for full type coverage in a Node.js server. Type Request, Response, and middleware signatures correctly to prevent runtime errors in route handlers.
  2. Runtime Validation with Zod — TypeScript types are erased at runtime. Zod fills that gap with runtime-safe schema validation whose schemas also infer static TypeScript types. Define once, validate everywhere.
  3. Type-Safe Database Access with Prisma — Prisma generates TypeScript types from your database schema, making every query fully typed. You get autocomplete on model fields, relationships, and filters — and TypeScript errors if a query does not match the schema.
  4. Environment Variables with Type Safety (dotenv + Zod) — Environment variables are untyped strings by default. Validate and type them at startup using Zod so every reference to process.env.DATABASE_URL is guaranteed non-empty, failing fast with clear errors if misconfigured.

Why This Week Matters: Zod and Prisma together with TypeScript create genuine end-to-end type safety — from the database schema through the API layer to the response. This is the architecture behind most modern production TypeScript applications.

Code Example

import express, { Request, Response } from "express";
const app = express();
app.get("/users/:id", async (req: Request, res: Response): Promise<void> => {
    const id = Number(req.params.id);
    res.json({ id, name: "Alice" });
});

import { z } from "zod";
const UserSchema = z.object({
    name:  z.string().min(1).max(100),
    email: z.string().email(),
    age:   z.number().int().min(0).optional(),
});
type User = z.infer<typeof UserSchema>;
const user = UserSchema.parse(req.body);

const EnvSchema = z.object({
    DATABASE_URL: z.string().url(),
    PORT:         z.coerce.number().default(3000),
    NODE_ENV:     z.enum(["development", "production", "test"]),
});
export const env = EnvSchema.parse(process.env);

Phase 4 — Master Level Concepts

Weeks 12–15: Decorators, advanced type patterns, concurrency, and three complete production projects.

Week 12 — Decorators & Metaprogramming

Master TypeScript decorators to add cross-cutting concerns like logging, validation, and caching declaratively.

Core Focus

Mastering TypeScript decorators to add cross-cutting concerns declaratively.

Topics & Subtopics

  1. Class Decorators — A class decorator is a function applied to a class constructor, allowing modification or wrapping of the class. Used by frameworks like NestJS and Angular for dependency injection and metadata registration.
  2. Method & Property Decorators — Method decorators wrap a class method, enabling patterns like @Log, @Cache, @Validate, or @Throttle without modifying the method body. They receive the target, property key, and property descriptor.
  3. Parameter Decorators & Reflect Metadata — Parameter decorators mark constructor or method parameters with metadata. Combined with the reflect-metadata library, frameworks can inspect injected types at runtime for dependency injection containers.
  4. Decorator Factories & Composition — Decorator factories are functions that return a decorator, allowing configuration via parameters. Stacking multiple decorators composes their effects, applied bottom-up (innermost first).

Why This Week Matters: Decorators are what enable the clean, declarative API of frameworks like NestJS, Angular, and TypeORM. Understanding how they work under the hood makes you a far more effective user of these frameworks.

Code Example

function Log(target: any, key: string, descriptor: PropertyDescriptor) {
    const original = descriptor.value;
    descriptor.value = function (...args: any[]) {
        console.log(`Calling ${key}(${JSON.stringify(args)})`);
        const result = original.apply(this, args);
        console.log(`${key} returned ${JSON.stringify(result)}`);
        return result;
    };
}

function Throttle(ms: number) {
    return function (target: any, key: string, desc: PropertyDescriptor) {
        const original = desc.value;
        let last = 0;
        desc.value = function (...args: any[]) {
            if (Date.now() - last >= ms) {
                last = Date.now();
                return original.apply(this, args);
            }
        };
    };
}

🛠 Practical Task

Build a mini NestJS-style DI container using decorators: @Injectable() marks a class as injectable, @Inject(token) marks constructor parameters, and a Container class resolves the dependency graph. Test with a 3-layer service tree: UserController → UserService → UserRepository.

Week 13 — Advanced Type Patterns & TypeScript Internals

Write expert-level type utilities, understand variance, and learn patterns used in large TypeScript codebases.

Core Focus

Writing expert-level type utilities, understanding variance, and learning patterns used in large TypeScript codebases.

Topics & Subtopics

  1. Recursive Types — TypeScript supports recursive type aliases, enabling deeply nested structures like trees, linked lists, and nested JSON. Use the interface form or type with lazy evaluation to model unlimited depth.
  2. Variance (Covariance & Contravariance) — TypeScript uses structural typing, but function parameter types are contravariant and return types are covariant. Understanding this prevents subtle type bugs in callbacks and generic containers.
  3. The satisfies Operator (TS 4.9+) — satisfies validates that a value conforms to a type without widening the inferred type. This keeps literal type information while still catching structural mismatches.
  4. Advanced infer Patterns & Distributive Conditional Types — Distributive conditional types apply over union members individually when T is a bare type parameter. Combining this with infer allows extracting deeply nested types — the technique behind ReturnType, Parameters, Awaited, and more.

Why This Week Matters: This is the week that separates TypeScript users from TypeScript experts. Recursive types, variance, and the satisfies operator are the tools used in TypeScript's own standard library and in top-tier open source projects.

Code Example

type Json =
    | string | number | boolean | null
    | Json[]
    | { [key: string]: Json };

Week 14 — Concurrency, Streams & Performance Patterns

Master async patterns — concurrent requests, AbortController, async iterators, and typed worker communication.

Core Focus

Mastering async patterns including concurrent requests, AbortController, async iterators, and typed worker communication.

Topics & Subtopics

  1. Concurrent Async Patterns — TypeScript preserves tuple types through Promise.all, giving typed destructuring of concurrent results. Understanding when to use allSettled, all, race, and any is critical for robust async code.
  2. Async Iterators & for await...of — Async iterators process streams of data lazily without loading everything into memory. TypeScript types the yielded values through AsyncIterable<T> and AsyncGenerator<T>.
  3. AbortController & Cancellable Operations — AbortController provides a standard way to cancel async operations. TypeScript types the signal parameter and abort() call, making cancellation intent explicit in function signatures.
  4. Worker Threads with Typed Message Passing — Node.js worker_threads run CPU-intensive work off the main thread. Use discriminated unions to type messages between threads, ensuring both ends of the postMessage channel agree on the protocol.

Why This Week Matters: Typed async iterators and typed worker thread messages are genuinely advanced TypeScript skills that very few developers master. They are the difference between a codebase that handles concurrency correctly and one that silently drops errors or corrupts state.

Code Example

async function fetchAll(ids: number[]): Promise<[User[], Post[]]> {
    const [users, posts] = await Promise.all([fetchUsers(ids), fetchPosts(ids)]);
    return [users, posts];
}

async function* paginate<T>(
    fetchPage: (page: number) => Promise<T[]>
): AsyncGenerator<T> {
    let page = 0;
    while (true) {
        const results = await fetchPage(page++);
        if (results.length === 0) break;
        for (const item of results) yield item;
    }
}

async function fetchWithTimeout<T>(url: string, ms: number): Promise<T> {
    const controller = new AbortController();
    const timeout = setTimeout(() => controller.abort(), ms);
    try {
        const res = await fetch(url, { signal: controller.signal });
        return res.json() as Promise<T>;
    } finally {
        clearTimeout(timeout);
    }
}

type WorkerMessage =
    | { type: "compute"; payload: number[] }
    | { type: "result";  value: number }
    | { type: "error";   message: string };
