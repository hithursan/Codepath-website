About This Roadmap

This roadmap takes a complete beginner to a confident, portfolio-ready React Native developer in 20 weeks of full-time study (~35–40 hours/week). It is structured in five phases.

React Native is an open-source framework created by Meta that lets you write one codebase in JavaScript (or TypeScript) and deploy it to both iOS and Android. It is used by major apps including Facebook, Instagram, Shopify, Discord, and Walmart.

TypeScript is a typed superset of JavaScript — virtually all professional React Native projects use it. This roadmap teaches TypeScript gradually alongside JavaScript rather than treating it as a separate hurdle.

Stack: React Native + TypeScript  |  Duration: 20 Weeks (Full-Time)  |  Level: Beginner → Job-Ready  |  Platforms: iOS & Android (Cross-Platform)

5 Phases Overview

  • Phase 1 (Weeks 1–3): Foundations — How mobile apps work, JavaScript ES6+, and TypeScript essentials.
  • Phase 2 (Weeks 4–7): React fundamentals and React Native core — components, styling, navigation, and user input.
  • Phase 3 (Weeks 8–12): Intermediate skills — APIs/networking, local data persistence, animations, device hardware access, and push notifications.
  • Phase 4 (Weeks 13–17): Advanced topics — global state management (Redux Toolkit/Zustand), testing, performance optimization, and publishing to the App Store and Play Store.
  • Phase 5 (Weeks 18–20): Capstone — Firebase backend integration, full capstone app build, portfolio assembly, and interview/viva prep.

By Week 20, you will have built ~14 mini-projects, one major mid-point project, and a complete, published-ready capstone app — "TaskMate," a cross-platform task and productivity manager.

Week 1 — How Mobile Apps Work, Dev Environment & JavaScript Essentials

Understand what React Native apps are, set up your full development environment, and become confident with core JavaScript needed for React Native.

Topics & Subtopics

  1. Mobile app landscape — Native vs hybrid vs cross-platform development; how React Native bridges JS to native components; iOS vs Android differences; app lifecycle concept.
  2. Development environment setup — Node.js, Expo CLI, Expo Go installation; VS Code setup with extensions; Git basics.
  3. JavaScript fundamentals — Variables (var/let/const), data types, operators.
  4. Functions — Traditional and arrow functions.
  5. Conditionals & Loops — if/else, ternary, for, while, forEach.
  6. Arrays — Create, access, iterate, .map(), .filter().
  7. Objects — Create, access, spread operator.

Why This Week Matters: Every line of React Native code is JavaScript. Starting without a solid JS foundation causes compounding confusion from Week 4 onward. This week is intentionally designed so that you write real, runnable JavaScript from Day 1.

Daily Study Plan

Day Focus

Day 2 JS variables, data types, operators; solve 10 small problems in the Node REPL.
Day 3 Functions (traditional and arrow); convert 5 of yesterday's solutions to use functions.
Day 4 Conditionals and loops; build a simple grade calculator function.
Day 5 Arrays — create/access/.map()/.filter(); practice 8 array manipulation problems.
Day 6 Objects — create/access/spread; build a simple "user profile" object and manipulate it.
Day 7 Review + Git commit of all JS practice files + explore the default Expo app on your phone.

Practice Tasks

  • Solve 15 small JavaScript logic problems (FizzBuzz, palindrome checker, array sum, object key counter).
  • Create a GitHub repository and push all practice code with meaningful commit messages.

🛠 Mini Project

A "Personal Info App" — your very first Expo app showing your name, city, a short bio, and a list of 3 skills displayed as text components. No interactivity yet — just getting comfortable creating an Expo project, editing App.tsx, and seeing changes live on your phone via Expo Go.

Common Mistakes

  • Spending Days 1–3 fighting with environment setup and not getting to JavaScript at all.
  • Copy-pasting code examples without typing them out.
  • Installing a different version of Node.js than the LTS version, causing compatibility issues with Expo.

How to Avoid Them

  • Follow the Expo setup guide line by line; if stuck, the Expo Discord is the fastest help source.
  • Always type every code example. Your fingers need to learn the syntax, not just your eyes.
  • Always install the Node.js LTS (Long Term Support) version, not "Current."

Week 2 — JavaScript ES6+ Deep Dive (The React Native-Critical Parts)

Master the specific JavaScript features that React Native and React use constantly — destructuring, spread/rest, modules, Promises, and async/await.

Topics & Subtopics

  1. ES6+ syntax essentials — Destructuring (arrays and objects), spread (...) and rest operators, template literals, default parameter values, short-circuit evaluation (&&, ||, ??).
  2. Modules — export / export default, import (named and default), module-based code organization.
  3. Asynchronous JavaScript — The event loop concept, callbacks → callback hell, Promises (.then(), .catch(), .finally(), Promise.all()), async/await, error handling with try/catch.
  4. Array methods (advanced) — .find(), .findIndex(), .some(), .every(), .reduce().

Why This Week Matters: Destructuring appears in virtually every React component. async/await appears in virtually every API call. The short-circuit operators appear constantly in conditional rendering. Skipping this week makes every week from Week 4 onward harder.

Daily Study Plan

Day Focus
Day 1 Destructuring (arrays and objects); practice 10 destructuring transformations on sample data.
Day 2 Spread/rest, template literals, default params, short-circuit evaluation.
Day 3 ES6 modules — import/export; refactor Day 1–2 practice into a multi-file project.
Day 4 Event loop concept + Promises + .then()/.catch().
Day 5 async/await + try/catch; rewrite Day 4's Promise chains as async functions.
Day 6 Advanced array methods — .find(), .reduce(), .some(); solve 8 problems.
Day 7 Review + mini project + GitHub commit.

Practice Tasks

  • Rewrite 5 promise chains as async/await functions.
  • Write an async function that fetches data from a free public API and logs the result.
  • Use .reduce() to calculate the total price of a sample shopping cart array.

🛠 Mini Project

A Node.js "Weather Info Fetcher" script — a command-line async function that fetches current weather from a free weather API (e.g., Open-Meteo, no API key needed), displays formatted results using template literals, and handles errors gracefully with try/catch — pushed to GitHub.

Common Mistakes

  • Forgetting await inside an async function (logs Promise {} instead of actual data).
  • Not understanding destructuring deeply enough — this bites hard in React component code.
  • Using .then() and async/await inconsistently in the same file.

How to Avoid Them

  • Every time you see Promise {} in a log, check if you forgot await.
  • Practice destructuring daily as default syntax — override muscle memory.
  • Decide once per function: .then() chains or async/await — never mix.

Free Resources

  • freeCodeCamp — ES6 section

Week 3 — TypeScript Fundamentals for React Native

Understand and apply TypeScript's core type system to write safer, more maintainable React Native code.

Topics & Subtopics

  1. Why TypeScript? — Problems with untyped JS at scale; TypeScript's relationship to JavaScript; TypeScript in Expo (on by default in the TypeScript template).
  2. TypeScript type basics — Primitive types (string, number, boolean, null, undefined); arrays (string[] / Array<string>); any (avoid); union types; type inference.
  3. Objects and interfaces — type aliases, interface declarations, optional properties (?), readonly properties.
  4. Functions with types — Typing parameters and return values; typing arrow functions; void return type.
  5. Generics (introductory) — What generics are; common cases: Array<T>, Promise<T>.
  6. TypeScript in React Native context — Annotating component props with an interface; .tsx file extension; React.FC type.

Why This Week Matters: Every component you write from Week 4 onward will use TypeScript to type its props. Every API response in Week 8–9 will need an interface. Getting comfortable with TypeScript now means you code correctly by default from the start.

Daily Study Plan

Day Focus
Day 1 TypeScript setup in Expo + primitive types + type inference; convert a JS file to .ts.
Day 2 Arrays, union types, any-vs-unknown distinction; solve 5 typed problems.
Day 3 type aliases and interface — define interfaces for 3 real-world shapes (User, Product, Task).
Day 4 Function types — add parameter and return-value types to all Week 1–2 functions.
Day 5 Generics conceptual overview + Array<T> and Promise<T> in context.
Day 6 TypeScript in TSX — annotate a component's props with an interface; create 2–3 typed components.
Day 7 Review + mini project + GitHub commit.

Practice Tasks

  • Define TypeScript interfaces for: a User, a Product, a Task, and an ApiResponse<T> generic wrapper.
  • Add TypeScript types to all functions written in Weeks 1–2.
  • Convert a small JS Expo component to a .tsx file with fully typed props.

🛠 Mini Project

A "Typed Product List" Expo app — a screen displaying a list of products (name, price, category, inStock boolean) where all data is typed using TypeScript interfaces, components are typed with prop interfaces, and no any is used anywhere — pushed to GitHub.

Common Mistakes

  • Using any everywhere to silence TypeScript errors (defeats the purpose).
  • Confusing interface and type — paralysis over which to use.
  • Not restarting the TypeScript language server after config changes.

How to Avoid Them

  • Ban any from your vocabulary this week. Use TypeScript Playground to figure out the real type.
  • Use interface for object shapes, type for unions — a clear rule eliminates choice paralysis.
  • Restart VS Code's TypeScript server (Cmd+Shift+P → "TypeScript: Restart TS Server") when types behave unexpectedly.

Free Resources

Week 4 — React Fundamentals (The Foundation of React Native)

Understand core React concepts — components, props, state, and hooks — deeply enough that React Native is immediately intuitive in Week 5.

Topics & Subtopics

  1. React mental model — UI-as-a-function-of-state; components as functions returning UI; virtual DOM (note: React Native does NOT use the DOM).
  2. Components and JSX/TSX — Functional components; JSX syntax rules; component composition.
  3. Props — Passing data parent → child; typing props with TS interfaces; children prop.
  4. State with useState — Reading and updating state; re-render triggers; lifting state up.
  5. useEffect hook — Side effects, dependency array, cleanup functions.
  6. Conditional rendering — &&, ternary, early returns.
  7. Lists — .map() to render lists; the key prop and why it's required.

Why This Week Matters: React Native is React — just with different component primitives (View instead of <div>, Text instead of <p>). Mastering React patterns now means Week 5's React Native intro feels like a small extension.

Daily Study Plan

Day 2 Props — passing data + TypeScript prop interfaces; build a UserCard with typed props.
Day 3 useState — counters, toggles, form inputs; build 3 interactive mini examples.
Day 4 Conditional rendering — show/hide UI, toggle styles, empty state messages; 5 examples.
Day 5 Lists with .map() + key prop; render contacts, products, todos.
Day 6 useEffect — fetch on mount, run on state change, cleanup; 3 examples.
Day 7 Review + mini project + GitHub commit.

Practice Tasks

  • Build a Counter component with increment/decrement/reset buttons.
  • Build a SearchableProductList that filters items as the user types.
  • Build a component that fetches and displays 5 users from jsonplaceholder.typicode.com/users.

🛠 Mini Project

A "GitHub Profile Viewer" React component (web sandbox) — takes a GitHub username as a prop, fetches user data from the GitHub API on mount with useEffect, displays avatar/name/bio/repo count, shows a loading spinner, handles errors gracefully, and uses TypeScript interfaces for the API response shape — pushed to GitHub.

Common Mistakes

  • Mutating state directly (e.g., myArray.push(item)) instead of using setState.
  • Forgetting the key prop on list items.
  • Missing dependencies in useEffect's dependency array (stale closures).

How to Avoid Them

  • Never mutate state. Always spread: setItems(prev => [...prev, newItem]).
  • key prop is non-negotiable on list items.
  • For dependency arrays: "what variables from outside this effect are used inside?" — every one belongs.

Week 5 — React Native Core: Components, Styling & Flexbox

Translate your React knowledge into React Native, master core components, and build correctly-styled mobile UIs using Flexbox.

Topics & Subtopics

  1. React Native vs React (web) — No DOM, no HTML; View, Text, Image, ScrollView, TextInput, TouchableOpacity, Pressable; no CSS files (StyleSheet.create); platform differences iOS vs Android.
  2. Core RN components — View, Text, Image (local + network), TextInput, ScrollView, TouchableOpacity/Pressable, StatusBar.
  3. StyleSheet & styling — StyleSheet.create(); styling differences from CSS (no px, camelCase, numbers); Platform.OS and Platform.select().
  4. Flexbox for mobile — flexDirection (column default in RN!), justifyContent, alignItems, flex, flexWrap.
  5. Safe Area and screen dimensions — SafeAreaView (notches/status bars); Dimensions.get('window'); useWindowDimensions hook.

Why This Week Matters: This is the moment JavaScript knowledge becomes a real mobile app. Getting Flexbox solid now means you stop fighting layouts for the rest of the roadmap — poor layout knowledge is the single most common source of confusion in React Native beginners.

Daily Study Plan

Day Focus

Day 2 Flexbox — flexDirection, justifyContent, alignItems; build header, centered content, footer.
Day 3 flex property + flexWrap + a two-column card grid layout.
Day 4 Image (local + network) + TextInput (controlled input) + TouchableOpacity.
Day 5 ScrollView + SafeAreaView + Dimensions/useWindowDimensions.
Day 6 Platform API — write one component that renders differently on iOS vs Android.
Day 7 Review + mini project + GitHub commit + push to Expo Go on phone.

Practice Tasks

  • Rebuild 3 real app screen layouts from screenshots (Instagram post card, news article card, settings list item).
  • Build a controlled TextInput with character limit counter and clear button.
  • Build a 2-column responsive card grid using flexWrap.

🛠 Mini Project

A "Social Media Profile Screen" — a static mobile screen showing profile photo, username, follower/following counts, short bio, a grid of post thumbnail placeholders, and a styled follow button — fully responsive using Flexbox, using SafeAreaView correctly, looks good on both Android and iOS — pushed to GitHub.

Common Mistakes

  • Forgetting flexDirection: 'column' is the default in RN (opposite of CSS).
  • Wrapping plain strings in View without Text ("Text strings must be rendered within a <Text>").
  • Using height: '100%' when flex: 1 is correct.

How to Avoid Them

  • In RN, default to thinking "column layout" unless deliberately building a row.
  • Every piece of visible text must be inside <Text> — non-negotiable rule.
  • Replace all percentage-based height attempts with flex: 1.

Week 7 — State Management: Hooks, Context API & Local State Patterns

Master the full set of React hooks used in React Native and implement app-wide state sharing with Context API.

Topics & Subtopics

  1. useReducer — Alternative to useState for complex state; actions and reducers pattern.
  2. useCallback and useMemo — Memoizing functions/values to avoid re-renders; avoid premature optimization.
  3. useRef — Referencing elements (focus TextInput programmatically); persisting mutable values across renders.
  4. Custom hooks — Extracting reusable stateful logic; examples: useFetch, useToggle, useDebounce.
  5. Context API — createContext, Provider, useContext; real use case: ThemeContext, AuthContext.
  6. When NOT to use Context — Re-render behavior; intro to why global state libraries exist (sets up Week 13).

Why This Week Matters: useRef, useCallback, useMemo, custom hooks, and Context API are all used constantly in production. Custom hooks transform repeated stateful logic from copy-paste spaghetti into clean reusable code. Context handles "prop drilling" without needing a global state library yet.

Daily Study Plan

Day Focus
Day 1 useReducer — build a Todo manager with add/complete/delete actions.
Day 2 useRef — focus a TextInput on mount; persist a timer value without re-rendering.
Day 3 Custom hooks — build useFetch, useToggle, and useLocalStorage.
Day 4 useCallback/useMemo — one clear example each; use sparingly.
Day 5 Context API — set up a ThemeContext (light/dark mode across all screens).
Day 6 AuthContext — tracks login state; conditionally shows different navigation stacks.
Day 7 Review + mini project + GitHub commit.

Practice Tasks

  • Extract Week 4's fetch logic into a reusable useFetch<T>(url: string) custom hook.
  • Build a ThemeContext with a toggle, connect to at least 3 screens.
  • Build an AuthContext that conditionally renders Auth stack vs Main app stack.

🛠 Mini Project

A "Theme-Aware Notes App" — multi-screen notes app using useReducer for note state (add/delete/edit), ThemeContext for light/dark mode accessible from any screen, custom useNotes hook abstracting reducer logic, and AuthContext showing a simple login screen before the main app — pushed to GitHub.

Common Mistakes

  • Using Context for every state, even local component state (causes unnecessary re-renders).
  • Building custom hooks that aren't actually reusable.
  • Not providing a Context before consuming it (null/undefined errors).

How to Avoid Them

  • Context = truly global state only (theme, auth, language). Per-screen state = component state.
  • A custom hook earns its existence when 2+ components need the same logic.
  • Always wrap all consuming components with the Provider, typically at app root.

Week 10 — Local Data Persistence (AsyncStorage, SQLite & SecureStore)

Persist data on the user's device so app state survives app restarts — critical for any real app.

Topics & Subtopics

  1. Why local persistence? — Offline-first UX, user preferences, caching, auth tokens.
  2. AsyncStorage — @react-native-async-storage/async-storage key-value storage; getItem, setItem, removeItem; limitations (string-only, not secure).
  3. SecureStore (Expo) — expo-secure-store encrypted key-value for sensitive data; auth tokens go HERE, not AsyncStorage.
  4. MMKV (intro) — High-performance alternative; faster synchronous reads.
  5. SQLite (Expo SQLite) — expo-sqlite local relational database; CREATE/INSERT/SELECT/UPDATE/DELETE; when to use vs AsyncStorage.
  6. Persisting Redux/Zustand state — redux-persist with AsyncStorage; Zustand's persist middleware.

Why This Week Matters: Without persistence, every app restart clears user data — immediately frustrating. Proper persistence (auth tokens → SecureStore, preferences → AsyncStorage, structured data → SQLite) is the difference between a real app and a demo.

Daily Study Plan

Day Focus
Day 1 AsyncStorage — save/retrieve user preferences (theme, language); persist on restart.
Day 2 SecureStore — save a JWT-style auth token; retrieve on launch to decide which stack.
Day 3 SQLite — create a tasks table; implement full CRUD via SQL.
Day 4 More SQLite — add filtering, ordering, search queries to tasks DB.
Day 5 Zustand's persist middleware — persist cart/favorites state with AsyncStorage.
Day 6 Integrate persistence into Week 7 Notes app — notes survive app restarts.
Day 7 Review + mini project + GitHub commit.

Practice Tasks

  • Build settings screen persisting 3 preferences (theme, font size, notifications) with AsyncStorage.
  • Store a mock auth token in SecureStore and write app-startup check (Login vs Home).
  • Build a SQLite-backed expense tracker with add/view/delete.

🛠 Mini Project

A "Habit Tracker" app — users add habits (daily goals), check off each day; all data in SQLite (habit names, completion dates); preferences (theme, start day of week) in AsyncStorage; mock auth token in SecureStore with startup check — all data persists across restarts — pushed to GitHub.

Common Mistakes

  • Storing sensitive tokens in AsyncStorage (not encrypted).
  • Not awaiting AsyncStorage calls (always async — forgetting await causes stale reads).
  • Using SQLite for simple key-value preferences when AsyncStorage suffices.

How to Avoid Them

  • Rule: auth tokens/passwords → SecureStore. Preferences → AsyncStorage. Structured/relational → SQLite.
  • AsyncStorage is always async — always await or .then().
  • If you'd query it with WHERE → SQLite; if it's simple key-value → AsyncStorage.
