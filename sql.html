Complete 16-Week Curriculum — From SELECT * FROM to Recursive CTEs, Triggers & Transaction Concurrency Control

About This Curriculum

This is a 16-week, project-driven SQL curriculum designed to take a complete beginner to a professional-level database engineer. The roadmap moves from SELECT, data types, filtering, and sorting through joins, aggregations, subqueries, CASE logic, and data modification, into relational schema design, data import/export, transactions, stored procedures, and indexing, all the way to triggers, recursive CTEs, window functions, transaction isolation levels, and three complete capstone projects.

SQL is a declarative query language with its own powerful equivalents for common programming concepts: OOP → schema design & relational modeling, File Handling → data import/export, Decorators → triggers, Generators → recursive CTEs & window functions, Concurrency → transactions & locking. All examples use standard ANSI SQL (PostgreSQL-flavored).

By Week 16, you will have built three complete, portfolio-ready database systems and be confident designing and querying production-grade relational databases.

Level: Beginner → Master  |  Duration: 16 weeks  |  Stack: PostgreSQL (ANSI SQL)  |  Prerequisites: Basic computer literacy

Phase 1 — Absolute Basics

Weeks 1–4: Fundamentals, data types, filtering, and sorting.

Week 1 — Introduction to Databases & SELECT

Understanding relational databases and retrieving your first data.

Core Focus

Understanding relational databases and writing your first data retrieval queries with SELECT.

Topics & Subtopics

  1. Relational Database Concepts — A relational database organizes data into tables made of rows and columns, where each row represents one record. Tables can be related to one another through shared key values, forming the foundation of all structured data storage.
  2. The SELECT Statement — SELECT is the foundational command for retrieving data from one or more columns in a table. It is the most frequently used statement in all of SQL — virtually every interaction with a database involves SELECT in some form.
  3. FROM and Column Selection — FROM specifies which table to query, while listing specific column names (instead of *) controls exactly what data is returned. Selecting only needed columns improves performance and readability in production queries.

Why This Week Matters: Every SQL query — no matter how complex — is built on the SELECT...FROM foundation. Understanding that SQL is declarative (you describe what you want, not how to get it) from Week 1 establishes the correct mental model for all subsequent weeks. The habit of selecting specific columns instead of SELECT * is a professional best practice that prevents performance issues in large tables.

Code Example

-- Retrieve only the name and email columns from the customers table
SELECT customer_name, email
FROM customers;

-- Select all columns (use sparingly in production)
SELECT *
FROM employees;

-- Add an alias for clarity
SELECT first_name AS "First Name", salary AS "Annual Salary"
FROM employees;

Week 2 — Data Types & Table Creation

Defining the structure and types of data a table can hold.

Core Focus

Defining the structure and types of data a table can hold using CREATE TABLE and understanding SQL's core data types.

Topics & Subtopics

  1. Common Data Types (INT, VARCHAR, DATE, BOOLEAN) — SQL data types define what kind of value a column can store, such as whole numbers, text, dates, or true/false flags. Choosing the right type ensures data integrity and efficient storage — using VARCHAR(255) for a phone number wastes space; using INT for a name causes errors.
  2. CREATE TABLE — CREATE TABLE defines a new table's structure, listing each column's name and data type. This is the first step in designing any database — the schema definition that all future data must conform to.
  3. Primary Keys — A PRIMARY KEY uniquely identifies each row in a table and cannot contain duplicate or NULL values. It is essential for maintaining data integrity and enabling relationships between tables in subsequent weeks.

Why This Week Matters: A poorly designed table structure — wrong data types, missing primary keys, columns that are too wide or too narrow — causes problems that are expensive to fix after data has been loaded. The choices made in CREATE TABLE ripple through every query, index, and foreign key constraint that references that table. Getting schema design right from the start is the most cost-effective database decision you can make.

Code Example

-- Create a table to store book records with a unique ID
CREATE TABLE books (
    book_id     INT PRIMARY KEY,
    title       VARCHAR(100) NOT NULL,
    published_date DATE,
    price       DECIMAL(10, 2),
    in_stock    BOOLEAN DEFAULT TRUE
);

-- Create with auto-incrementing primary key (PostgreSQL)
CREATE TABLE customers (
    customer_id SERIAL PRIMARY KEY,
    email       VARCHAR(255) UNIQUE NOT NULL,
    created_at  TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

Week 3 — Filtering Data with WHERE

Narrowing down results based on conditions.

Core Focus

Narrowing down query results based on conditions using the WHERE clause and logical operators.

Topics & Subtopics

  1. The WHERE Clause — WHERE filters rows so only those matching a specified condition are returned. It is applied before grouping or sorting occurs — understanding its position in the SQL execution order prevents common mistakes.
  2. Comparison Operators (=, >, <, BETWEEN) — These operators compare column values against given criteria, such as finding salaries above a threshold or dates within a range. BETWEEN is a convenient shorthand for inclusive range checks that improves query readability.
  3. Logical Operators (AND, OR, NOT) — Logical operators combine multiple conditions in a WHERE clause to build more precise filters. NOT inverts a condition's result — and operator precedence (AND evaluates before OR) must be understood to avoid logical bugs.

Why This Week Matters: The WHERE clause is the primary mechanism for query selectivity — a query without WHERE reads every row in the table (a full table scan). In a table with millions of rows, a properly filtered WHERE clause is the difference between a query that runs in milliseconds and one that times out. Understanding operator precedence (AND before OR) prevents silent logic bugs that return wrong data without any error.

Code Example

-- Find all active employees earning more than 50000
SELECT first_name, salary
FROM employees
WHERE salary > 50000 AND is_active = TRUE;

-- Use BETWEEN for inclusive range checks
SELECT *
FROM orders
WHERE order_date BETWEEN '2024-01-01' AND '2024-12-31';

-- Combine conditions — parentheses clarify precedence
SELECT *
FROM products
WHERE (category = 'Electronics' OR category = 'Books')
  AND price < 100;

-- Use IN as shorthand for multiple OR conditions
SELECT *
FROM employees
WHERE department IN ('Sales', 'Marketing', 'HR');

Week 4 — Sorting, Limiting & Null Handling

Controlling result order, size, and handling missing data.

Core Focus

Controlling result ordering, limiting result set size for pagination, and correctly handling NULL values.

Topics & Subtopics

  1. ORDER BY — ORDER BY sorts query results by one or more columns, either ascending (ASC, default) or descending (DESC). It can sort by multiple columns to break ties — the second column in ORDER BY only applies when the first column's values are equal.
  2. LIMIT and OFFSET — LIMIT restricts the number of rows returned, while OFFSET skips a number of rows before starting to return results. Together they enable pagination of large result sets — a fundamental requirement for any user-facing application.
  3. NULL and IS NULL — NULL represents missing or unknown data and cannot be compared using =; instead, IS NULL/IS NOT NULL must be used. Understanding NULL behavior — that NULL = NULL evaluates to NULL, not TRUE — prevents subtle filtering bugs that silently exclude data.

Why This Week Matters: NULL is SQL's most confusing feature — any comparison with NULL returns NULL (not true or false), which means rows with NULL values are silently excluded from WHERE clauses that use = or !=. This is a real production bug that affects reporting accuracy. The LIMIT/OFFSET pagination pattern is used in virtually every web application that displays lists of data to users.

Code Example

-- Get the 5 most recently published books, skipping the first 2
SELECT title, published_date
FROM books
ORDER BY published_date DESC
LIMIT 5 OFFSET 2;

-- Sort by multiple columns — department first, then salary descending
SELECT first_name, department, salary
FROM employees
ORDER BY department ASC, salary DESC;

-- Find rows with NULL values — must use IS NULL, not = NULL
SELECT *
FROM students
WHERE birth_date IS NULL;

-- Handle NULLs in output with COALESCE
SELECT full_name, COALESCE(email, 'No email on file') AS contact
FROM students;

Phase 2 — Control Flow & Logic

Weeks 5–8: Functions, modules, and data structures.

Week 5 — Aggregate Functions & GROUP BY

Summarizing data across many rows.

Core Focus

Summarizing data across many rows using aggregate functions, grouping results with GROUP BY, and filtering groups with HAVING.

Topics & Subtopics

  1. Aggregate Functions (COUNT, SUM, AVG, MAX, MIN) — Aggregate functions compute a single summary value (like a total or average) from a set of rows. They are essential for reporting and analytics — the foundation of every dashboard, report, and business intelligence query.
  2. GROUP BY — GROUP BY splits rows into groups sharing the same column value, allowing aggregate functions to be calculated per group. It is commonly paired with aggregates to answer questions like "total sales per region" or "average salary per department."
  3. HAVING vs WHERE — HAVING filters groups after aggregation has occurred, while WHERE filters individual rows before grouping. This distinction is critical: you cannot use an aggregate function like COUNT(*) in a WHERE clause — it must go in HAVING.

Why This Week Matters: The WHERE vs HAVING distinction — that HAVING filters after aggregation and WHERE filters before — is one of the most commonly tested SQL concepts in technical interviews. Understanding the SQL execution order (FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY) explains why aggregate functions appear in HAVING but not WHERE, resolving one of the most confusing aspects of SQL for beginners.

Code Example

-- Find departments with more than 10 employees
SELECT department, COUNT(*) AS employee_count
FROM employees
GROUP BY department
HAVING COUNT(*) > 10;

-- Average salary per department, highest first
SELECT department, AVG(salary) AS avg_salary
FROM employees
GROUP BY department
ORDER BY avg_salary DESC;

-- Multiple aggregates in one query
SELECT
    department,
    COUNT(*)        AS headcount,
    MIN(salary)     AS min_salary,
    MAX(salary)     AS max_salary,
    ROUND(AVG(salary), 2) AS avg_salary
FROM employees
GROUP BY department;

Week 6 — Joins: Combining Tables

Querying related data spread across multiple tables.

Core Focus

Querying related data spread across multiple tables using all types of joins — the most essential skill in relational database querying.

Topics & Subtopics

  1. INNER JOIN — INNER JOIN returns only rows that have matching values in both joined tables. It is the most common type of join for combining related data — but silently excludes rows with no match in either table, which can cause data to disappear from reports.
  2. LEFT JOIN / RIGHT JOIN — LEFT JOIN returns all rows from the left table plus matched rows from the right (with NULLs where unmatched), while RIGHT JOIN does the reverse. These are vital for finding unmatched or optional relationships — like employees with no department, or products with no orders.
  3. FULL OUTER JOIN & Self Joins — FULL OUTER JOIN returns all rows from both tables, matched where possible. A self join joins a table to itself, useful for hierarchical data like employee-manager relationships where both the employee and manager are in the same table.

Why This Week Matters: Joins are the defining capability of relational databases — the reason why data is split across multiple tables instead of stored in one giant flat table. Choosing the wrong join type (INNER when you need LEFT) silently drops data from your results without any error. Understanding when each join type applies is the most important practical SQL skill for day-to-day data work.

Code Example

-- INNER JOIN: only orders with matching customer records
SELECT orders.order_id, customers.customer_name
FROM orders
INNER JOIN customers ON orders.customer_id = customers.customer_id;

-- LEFT JOIN: all employees, even those with no department
SELECT e.first_name, d.department_name
FROM employees e
LEFT JOIN departments d ON e.department_id = d.department_id;

-- Self JOIN: employee with their manager's name
SELECT e.first_name AS employee,
       m.first_name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.employee_id;

-- FULL OUTER JOIN: all employees and all departments, matched where possible
SELECT e.first_name, d.department_name
FROM employees e
FULL OUTER JOIN departments d ON e.department_id = d.department_id;

Week 7 — Functions, Subqueries & CASE Logic

Computing derived values and conditional logic within queries.

Core Focus

Computing derived values with scalar functions, nesting queries as subqueries, and implementing conditional logic with CASE expressions.

Topics & Subtopics

  1. Scalar & String Functions — Functions like UPPER, CONCAT, ROUND, and LENGTH transform individual values during a query. They allow data formatting and computation without changing the underlying stored data — essential for presenting clean output.
  2. Subqueries — A subquery is a query nested inside another query, used to filter or compute values dynamically. They can appear in WHERE (correlated subqueries), FROM (inline views), or SELECT clauses — each with different performance implications.
  3. CASE Expressions — CASE provides if-else style conditional logic directly within a SQL query, returning different values based on conditions. It's SQL's equivalent of a switch statement and is used constantly in reporting to categorize data on the fly.

Why This Week Matters: Subqueries and CASE expressions are what make SQL truly powerful for analytics. The ability to write a query whose WHERE condition is itself the result of another query — or to dynamically categorize rows into labels based on their values — eliminates the need for complex application-layer post-processing. These patterns appear in virtually every real-world reporting query.

Code Example

-- Label employees as 'Senior' or 'Junior' based on salary
SELECT first_name,
       CASE
           WHEN salary >= 70000 THEN 'Senior'
           WHEN salary >= 50000 THEN 'Mid-level'
           ELSE 'Junior'
       END AS level
FROM employees
WHERE salary > (SELECT AVG(salary) FROM employees);

-- Subquery in FROM clause (inline view)
SELECT dept_averages.department, dept_averages.avg_sal
FROM (
    SELECT department, AVG(salary) AS avg_sal
    FROM employees
    GROUP BY department
) AS dept_averages
WHERE dept_averages.avg_sal > 60000;

-- String functions
SELECT UPPER(first_name), CONCAT(first_name, ' ', last_name) AS full_name,
       LENGTH(email) AS email_length
FROM employees;

Week 8 — Modifying Data: INSERT, UPDATE, DELETE

Changing the data stored in tables, not just reading it.

Core Focus

Safely changing the data stored in tables using INSERT, UPDATE, and DELETE — the write side of SQL.

Topics & Subtopics

  1. INSERT INTO — INSERT INTO adds new rows of data into a table, either with explicit values or from the result of another query (INSERT INTO ... SELECT). The SELECT-based form is extremely powerful for copying or transforming data between tables.
  2. UPDATE — UPDATE modifies existing rows in a table, and must almost always be paired with a WHERE clause to avoid accidentally updating every row. The professional practice is to always test the WHERE condition with a SELECT query first before running the UPDATE.
  3. DELETE — DELETE removes rows from a table that match a given condition, and like UPDATE, requires careful use of WHERE. A DELETE without a WHERE clause removes all rows — unlike TRUNCATE, it's logged and can be rolled back inside a transaction.

Why This Week Matters: UPDATE and DELETE without a WHERE clause are among the most dangerous operations in SQL — they silently affect every row in the table. The professional practice of always running a SELECT with the same WHERE clause first, then wrapping UPDATE/DELETE in a transaction so it can be rolled back if wrong, is what prevents production data loss. This habit distinguishes professional database developers from beginners.

Code Example

-- INSERT with explicit values
INSERT INTO students (student_id, full_name, birth_date)
VALUES (101, 'Asha Perera', '2001-03-15');

-- INSERT from a SELECT query — copy rows between tables
INSERT INTO archived_orders
SELECT * FROM orders WHERE order_date < '2020-01-01';

-- UPDATE — always check WHERE with SELECT first!
-- Step 1: SELECT first to verify the target rows
SELECT * FROM students WHERE student_id = 101;
-- Step 2: Then UPDATE
UPDATE students
SET grade_points = grade_points * 1.1,
    is_active = FALSE
WHERE student_id = 101;

-- DELETE — wrapped in a transaction for safety
BEGIN;
DELETE FROM students WHERE student_id = 101 AND is_active = FALSE;
-- ROLLBACK; -- undo if wrong
COMMIT;

Week 9 — Relational Modeling: SQL's "OOP" Equivalent

Designing well-structured, related table schemas — SQL's version of object modeling.

Core Focus

Designing well-structured, normalized, related table schemas — SQL's version of object modeling and encapsulation.

Topics & Subtopics

  1. Foreign Keys & Relationships — A FOREIGN KEY links a column in one table to the primary key of another, enforcing referential integrity between related entities. This is how SQL models real-world "has-a" and "belongs-to" relationships — the relational equivalent of object composition and association.
  2. Normalization (1NF–3NF) — Normalization is the process of organizing tables to reduce data redundancy and avoid update anomalies, following a series of normal forms (1NF, 2NF, 3NF). It's the relational-database parallel to clean, non-duplicated class design — each fact lives in exactly one place.
  3. Views — Reusable "Abstractions" — A VIEW is a saved, virtual table defined by a query, letting you encapsulate complex joins and logic behind a simple, reusable interface. This mirrors how an OOP class encapsulates implementation details behind a clean public interface — callers see the view, not the complexity behind it.

Why This Week Matters: Poor database normalization is the most common cause of long-term database maintenance problems — data stored in multiple places drifts out of sync, UPDATE anomalies corrupt data silently, and INSERT anomalies prevent valid data from being entered. Understanding 1NF, 2NF, and 3NF is the database equivalent of SOLID principles in software engineering — the foundation of maintainable, scalable systems.

Code Example

-- Foreign key linking orders to customers
CREATE TABLE orders (
    order_id    INT PRIMARY KEY,
    customer_id INT NOT NULL REFERENCES customers(customer_id)
                ON DELETE CASCADE,      -- delete orders when customer deleted
    order_date  DATE DEFAULT CURRENT_DATE
);

-- Many-to-many junction table: books and authors
CREATE TABLE book_authors (
    book_id   INT REFERENCES books(book_id),
    author_id INT REFERENCES authors(author_id),
    PRIMARY KEY (book_id, author_id)    -- composite primary key
);

-- View encapsulating a complex join
CREATE VIEW active_customer_orders AS
    SELECT o.order_id, c.customer_name, o.order_date
    FROM orders o
    JOIN customers c ON o.customer_id = c.customer_id
    WHERE c.is_active = TRUE;

-- Query the view as if it were a simple table
SELECT * FROM active_customer_orders WHERE order_date > '2024-01-01';

Week 10 — Data Import, Export & "File Handling"

Moving data into and out of the database — SQL's equivalent of file I/O.

Core Focus

Moving data into and out of the database efficiently — SQL's equivalent of file I/O operations.

Topics & Subtopics

  1. COPY / BULK INSERT — COPY (PostgreSQL) or BULK INSERT (SQL Server) efficiently loads data from an external file like a CSV directly into a table. This is far faster than inserting rows one at a time — loading millions of rows that would take hours with INSERT takes seconds with COPY.
  2. Exporting Query Results — Query results can be exported to a CSV or file using COPY ... TO or client tool export features, letting you share or archive data outside the database. This is the reverse operation of importing — the standard way to extract data for reporting tools, spreadsheets, and data pipelines.
  3. Backup & Restore (pg_dump / mysqldump) — Database backup tools export an entire database's schema and data into a portable file, which can later be restored. This is critical for disaster recovery and migrating between environments — a skill required in any production database role.

Why This Week Matters: In the real world, data rarely starts in a database — it comes from CSV exports, legacy systems, external APIs, and third-party tools. The ability to efficiently import and export data using COPY/BULK INSERT is what makes SQL the universal data interchange layer. A data engineer who cannot load CSV data into a database or extract query results to a file is limited to only greenfield data work.

Code Example

-- Import data from a CSV file into the students table (PostgreSQL)
COPY students(student_id, full_name, birth_date)
FROM '/data/students.csv'
DELIMITER ',' CSV HEADER;

-- Export query results to a CSV file
COPY (
    SELECT student_id, full_name, birth_date
    FROM students
    WHERE is_active = TRUE
    ORDER BY full_name
)
TO '/exports/active_students.csv'
DELIMITER ',' CSV HEADER;

-- Backup a database from the terminal
-- pg_dump mydb > mydb_backup.sql
-- pg_dump -Fc mydb > mydb_backup.dump  (compressed format)

-- Restore from backup
-- psql mydb < mydb_backup.sql
-- pg_restore -d mydb mydb_backup.dump

Week 11 — Error Handling & Transactions

Writing safe, atomic, and fault-tolerant database operations.

Core Focus

Writing safe, atomic, and fault-tolerant database operations using transactions and constraint-based validation.

Topics & Subtopics

  1. Transactions (BEGIN, COMMIT, ROLLBACK) — A transaction groups multiple statements into a single all-or-nothing unit of work; COMMIT saves the changes, while ROLLBACK undoes them. This guarantees data consistency even if an error occurs mid-operation — the classic example is a bank transfer that must debit one account and credit another atomically.
  2. TRY/CATCH or Exception Blocks — Many SQL dialects (like T-SQL's TRY/CATCH or PL/pgSQL's EXCEPTION) let you catch and handle runtime errors gracefully inside stored procedures. This is SQL's direct equivalent of error handling in general-purpose languages — catching division-by-zero or constraint violations without crashing the entire script.
  3. Constraints as Validation — CHECK, UNIQUE, and NOT NULL constraints proactively prevent invalid data from ever being inserted, acting as a first line of defense against bad data. This shifts error prevention to the schema level itself — the data is validated by the database, not just by application code that might be bypassed.

Why This Week Matters: ACID (Atomicity, Consistency, Isolation, Durability) — the properties that transactions provide — is what makes relational databases trustworthy for financial and critical systems. Without transactions, a bank transfer that crashes halfway through would debit one account without crediting the other, losing money permanently. Database constraints as the last line of data validation prevents bad data from entering even when application-layer validation is bypassed.

Code Example

-- Safely transfer funds between two accounts as one atomic transaction
BEGIN;
UPDATE accounts SET balance = balance - 100 WHERE account_id = 1;
UPDATE accounts SET balance = balance + 100 WHERE account_id = 2;
-- If anything fails between BEGIN and COMMIT, ROLLBACK undoes both
COMMIT;

-- Add constraints to prevent invalid data at the schema level
ALTER TABLE products
ADD CONSTRAINT chk_positive_price CHECK (price > 0),
ADD CONSTRAINT chk_valid_category CHECK (category IN ('Electronics', 'Books', 'Clothing'));

-- PL/pgSQL exception handling inside a function
DO $$
BEGIN
    UPDATE accounts SET balance = balance - 500 WHERE account_id = 1;
    IF NOT FOUND THEN
        RAISE EXCEPTION 'Account % not found', 1;
    END IF;
EXCEPTION
    WHEN OTHERS THEN
        RAISE NOTICE 'Error: %', SQLERRM;
        ROLLBACK;
END;
$$;

Week 12 — Stored Procedures, Indexes & "Libraries"

Reusable server-side logic and performance optimization.

Core Focus

Building reusable server-side logic with stored procedures and functions, and dramatically improving query performance with indexes.

Topics & Subtopics

  1. Stored Procedures & Functions — A stored procedure is a saved, reusable block of SQL logic (with parameters) that can be invoked by name, similar to calling a function in a programming library. This reduces repeated code, centralizes business logic in the database, and allows permission-controlled access to data without exposing the underlying tables.
  2. Indexes — An INDEX creates a fast lookup structure for a column, dramatically speeding up searches and joins at the cost of extra storage and slightly slower writes. Choosing the right indexes — on columns used in WHERE, JOIN ON, and ORDER BY — is one of the highest-impact performance skills in all of SQL.
  3. Extensions & Built-in Function Libraries — Databases ship with rich built-in function "libraries" (date/time, JSON, full-text search, mathematical functions) and support extensions (like PostgreSQL's pgcrypto or PostGIS) for specialized capabilities. Knowing what's available prevents reinventing functionality in application code.

Why This Week Matters: A missing index on a foreign key column can turn a 1-millisecond join into a 10-second full table scan as the table grows. Adding a single index to the right column has saved more queries from performance problems than any amount of query rewriting. Understanding the trade-off — indexes speed reads but slow writes — is essential for making correct indexing decisions in both OLTP and analytical workloads.

Code Example

-- Create an index on a frequently filtered column
CREATE INDEX idx_orders_customer ON orders(customer_id);
CREATE INDEX idx_employees_dept_salary ON employees(department, salary);

-- Create a stored procedure (PostgreSQL)
CREATE OR REPLACE PROCEDURE get_customer_orders(IN cust_id INT)
LANGUAGE SQL
AS $$
    SELECT o.order_id, o.order_date, p.product_name
    FROM orders o
    JOIN order_items oi ON o.order_id = oi.order_id
    JOIN products p ON oi.product_id = p.product_id
    WHERE o.customer_id = cust_id
    ORDER BY o.order_date DESC;
$$;

-- Call the stored procedure
CALL get_customer_orders(42);

-- Create a reusable function returning a value
CREATE OR REPLACE FUNCTION calculate_discount(price DECIMAL, pct INT)
RETURNS DECIMAL AS $$
    SELECT price * (1 - pct::DECIMAL / 100);
$$ LANGUAGE SQL IMMUTABLE;

Week 13 — Triggers: SQL's "Decorator" Pattern

Automatically wrapping extra behavior around data changes.

Core Focus

Automatically executing extra behavior before or after data changes using triggers — SQL's equivalent of the decorator pattern.

Topics & Subtopics

  1. Triggers (BEFORE/AFTER INSERT/UPDATE/DELETE) — A trigger is a block of logic that automatically executes before or after a row is inserted, updated, or deleted, "decorating" the base operation with extra behavior. This is the closest SQL equivalent to a Python decorator wrapping a function call — the original operation proceeds, but with extra behavior automatically added around it.
  2. Audit Logging with Triggers — A common trigger pattern records every change to a table into a separate audit/history table, automatically and transparently. This provides full traceability — who changed what, when, and what the old value was — without modifying application code at all.
  3. NEW and OLD Row References — Inside a trigger, NEW refers to the row's incoming values (after the change) and OLD to its previous values (before the change), letting you compare, log, or validate changes. This is what makes triggers able to detect what specifically changed about a row.

Why This Week Matters: Triggers implement cross-cutting concerns at the database level — audit logging, enforcing complex business rules, maintaining derived columns — without requiring every application that touches the database to implement the same logic. A trigger on the employees table that logs salary changes fires regardless of whether the change came from the web app, a mobile app, a data migration script, or a manual admin query — guaranteed, transparent enforcement.

Code Example

-- Create an audit table to store salary change history
CREATE TABLE employee_salary_audit (
    audit_id     SERIAL PRIMARY KEY,
    employee_id  INT,
    old_salary   DECIMAL,
    new_salary   DECIMAL,
    changed_at   TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    changed_by   VARCHAR(100) DEFAULT CURRENT_USER
);

-- Trigger function that records the change
CREATE OR REPLACE FUNCTION record_salary_audit()
RETURNS TRIGGER AS $$
BEGIN
    INSERT INTO employee_salary_audit(employee_id, old_salary, new_salary)
    VALUES (OLD.employee_id, OLD.salary, NEW.salary);
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

-- Attach the trigger to the employees table
CREATE TRIGGER log_salary_change
AFTER UPDATE OF salary ON employees
FOR EACH ROW
WHEN (OLD.salary IS DISTINCT FROM NEW.salary)   -- only fire when salary actually changes
EXECUTE FUNCTION record_salary_audit();

Week 14 — Recursive CTEs & Window Functions: SQL's "Generators"

Lazily computing sequences and performing row-by-row sequential analysis.

Core Focus

Computing sequences and hierarchical data with recursive CTEs, and performing advanced row-level analytics with window functions.

Topics & Subtopics

  1. Recursive CTEs (WITH RECURSIVE) — A recursive Common Table Expression repeatedly references itself to build up a result set step by step, ideal for hierarchical data like org charts, file system trees, or generating number sequences. This mirrors how a generator yields a sequence of values one step at a time — each iteration builds on the previous.
  2. Window Functions (ROW_NUMBER, RANK, LAG/LEAD) — Window functions compute a value across a "window" of related rows without collapsing them into a single result, unlike GROUP BY. They enable row-by-row sequential analysis — ranking, comparing to the previous row, computing running totals — that is impossible with simple aggregates.
  3. Cursors — A cursor allows row-by-row, imperative-style iteration through a query result inside a stored procedure, similar to manually pulling values from a generator. They're powerful for complex row-by-row logic but should be used sparingly in favor of set-based operations for performance.

Why This Week Matters: Window functions are the most powerful analytical tool in modern SQL — they enable calculations like "what was this month's revenue compared to last month," "rank each product within its category," and "compute a running total" that would otherwise require complex self-joins or application-layer post-processing. Recursive CTEs unlock tree and graph traversal directly in SQL, making hierarchical data (org charts, bill of materials, category trees) queryable without writing recursive application code.

Code Example

-- Recursive CTE: generate numbers 1 through 10
WITH RECURSIVE counter(n) AS (
    SELECT 1                             -- anchor: starting value
    UNION ALL
    SELECT n + 1 FROM counter WHERE n < 10  -- recursive step
)
SELECT n FROM counter;

-- Recursive CTE: traverse an employee management hierarchy
WITH RECURSIVE management_chain AS (
    SELECT employee_id, first_name, manager_id, 0 AS depth
    FROM employees WHERE employee_id = 42   -- start from one employee
    UNION ALL
    SELECT e.employee_id, e.first_name, e.manager_id, mc.depth + 1
    FROM employees e
    JOIN management_chain mc ON e.employee_id = mc.manager_id
)
SELECT * FROM management_chain;

-- Window functions: rank, running total, compare to previous row
SELECT
    first_name, department, salary,
    RANK() OVER (PARTITION BY department ORDER BY salary DESC) AS dept_rank,
    SUM(salary) OVER (ORDER BY salary) AS running_total,
    LAG(salary) OVER (ORDER BY salary) AS prev_salary
FROM employees;

Week 15 — Transactions, Isolation Levels & Concurrency

Safely handling many simultaneous database users.

Core Focus

Safely handling many simultaneous database users using isolation levels, explicit locking, and understanding concurrency anomalies.

Topics & Subtopics

  1. Concurrency Problems (Dirty Reads, Lost Updates) — When multiple transactions run simultaneously without coordination, problems like dirty reads (reading another transaction's uncommitted data) or lost updates (one transaction overwriting another's changes without knowing) can occur. Understanding these failure modes is essential before tuning concurrency settings.
  2. Isolation Levels — Isolation levels (READ COMMITTED, REPEATABLE READ, SERIALIZABLE) control how strictly transactions are isolated from each other's in-progress changes, trading off consistency against performance. Choosing the right level depends on the application's tolerance for anomalies — most applications use READ COMMITTED, while financial systems require SERIALIZABLE.
  3. Locking (FOR UPDATE) — Explicit row-level locking, such as SELECT ... FOR UPDATE, prevents other transactions from modifying a row until the current transaction completes. This is SQL's direct mechanism for safe concurrent writes, analogous to a mutex in application code — and it's required whenever you read-then-write a value based on its current state.

Why This Week Matters: The "lost update" problem — two concurrent transactions both reading a value, both computing a new value based on it, and both writing back — means one update silently overwrites the other without any error. This is the most common concurrency bug in web applications, causing incorrect inventory counts, double-bookings, and financial discrepancies. SELECT ... FOR UPDATE is the correct fix, and understanding why requires the foundational knowledge of isolation levels covered this week.

Code Example

-- Set the isolation level for a session
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- The "lost update" pattern — UNSAFE without locking:
-- Session 1: SELECT quantity FROM inventory WHERE product_id = 5;  → 10
-- Session 2: SELECT quantity FROM inventory WHERE product_id = 5;  → 10
-- Session 1: UPDATE inventory SET quantity = 9 WHERE product_id = 5;
-- Session 2: UPDATE inventory SET quantity = 9 WHERE product_id = 5;
-- Result: quantity = 9, but TWO items were sold — should be 8!

-- SAFE version: lock the row before reading to compute the update
BEGIN;
SELECT quantity FROM inventory
WHERE product_id = 5
FOR UPDATE;                          -- Row is now locked — other transactions wait

UPDATE inventory
SET quantity = quantity - 1
WHERE product_id = 5;
COMMIT;                               -- Lock released — next transaction can proceed

-- Check current isolation level
SHOW TRANSACTION ISOLATION LEVEL;
