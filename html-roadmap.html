About This Roadmap

This is a 15-week, full-time, beginner-to-advanced HTML roadmap built for a college project website. It covers HTML fundamentals (Weeks 1–6), intermediate/semantic and forms-heavy HTML (Weeks 7–10), advanced topics including accessibility, SEO, multimedia, and light HTML-adjacent CSS/JS integration needed to make HTML "feel complete" (Weeks 11–13), and a final project + portfolio polish phase (Weeks 14–15).

Every week builds directly on the previous one. By Week 15 you will have 1 capstone project and 12+ smaller projects to show in a portfolio, plus the conceptual depth to answer technical interview questions confidently.

Level: Beginner to Advanced  |  Duration: 15 weeks  |  Study mode: Full-time (~30–35 hrs/week)  |  Language: HTML (with light CSS/JS from Week 11)

Introduction to HTML

HTML (HyperText Markup Language) is the standard markup language used to create the structure and content of webpages. It tells the browser what each piece of content is — a heading, a paragraph, a link, an image, a table — rather than how it should look (that's CSS's job) or how it should behave (that's JavaScript's job). Every website on the internet, no matter how complex, has HTML as its skeleton.

Career Relevance

  • Frontend Developer, Full-Stack Developer, UI/UX Designer, Web Designer
  • WordPress Developer, QA/Automation Tester, Technical Writer
  • Digital Marketer / SEO Specialist
  • HTML is a mandatory foundation for CSS, JavaScript, React, Angular, and almost every web framework.

Prerequisites

  • No coding background required.
  • Basic computer literacy (typing, file/folder management, using a browser).
  • Ability to install free software.

Tools Needed

  • Code Editor: Visual Studio Code (VS Code) — recommended, free.
  • Browser: Google Chrome or Mozilla Firefox (with Developer Tools).
  • VS Code Extensions: "Live Server," "HTML CSS Support," "Prettier."
  • Optional: GitHub account (for hosting via GitHub Pages).

Setup Steps

  1. Download and install VS Code from code.visualstudio.com.
  2. Install Google Chrome or Firefox if not already installed.
  3. Open VS Code → Extensions panel (Ctrl+Shift+X) → install "Live Server" by Ritwick Dey.
  4. Create a project folder, e.g., html-roadmap-project.
  5. Inside it, create index.html — right-click → "Open with Live Server."
  6. (Optional, Week 14+) Create a free GitHub account for hosting your final project.

Basic Glossary

  • Tag: Keyword in angle brackets, e.g., <p>.
  • Element: A complete tag structure including opening tag, content, and closing tag.
  • Attribute: Extra info added inside a tag, e.g., href in <a href="...">.
  • Void/Self-closing element: An element with no closing tag, e.g., <img>, <br>.
  • DOM: The browser's in-memory tree representation of your HTML.
  • Semantic HTML: Tags that describe their meaning instead of generic <div>.
  • Block-level element: Takes full width, starts on a new line (e.g., <div>, <p>).
  • Inline element: Takes only as much width as needed (e.g., <span>, <a>).
  • Nesting: Placing one element inside another.
  • Validation: Checking HTML for syntax correctness (e.g., via W3C Validator).

Week 1 — HTML Foundations & Document Structure

Understand what HTML is, set up your environment, and write your first valid HTML documents.

Topics & Subtopics

  1. What is HTML, how the web works — Client-server basics, browser rendering, HTML vs CSS vs JS roles.
  2. Document structure — <!DOCTYPE html>, <html>, <head>, <body>, <title>, <meta charset>.
  3. Basic tags — <h1>–<h6>, <p>, <br>, <hr>, comments.
  4. Text formatting tags — <b>, <strong>, <i>, <em>, <u>, <mark>, <small>, <sub>, <sup>.

Why This Week Matters: Everything else builds on document structure. Mistakes here (like missing DOCTYPE) cause browser quirks-mode issues later. This is the non-skippable foundation of all web development.

Step-by-Step Learning Order

How the web works → Editor setup → Boilerplate structure → Headings/paragraphs → Text formatting → Comments.

Daily Study Plan

Day Focus
Day 1 How the web/browsers work; install VS Code, Chrome, Live Server.
Day 2 HTML boilerplate; <head> vs <body>; create first index.html.
Day 3 Headings and paragraphs; practice 10 small snippets.
Day 4 Text formatting tags (bold, italic, mark, etc.).
Day 5 Comments, <br>, <hr>; review all tags learned.
Day 6 Practice tasks + mini project.
Day 7 Revise, quiz yourself, rest/buffer.

Practice Tasks

  • Build a "My Bio" page using only headings, paragraphs, and text formatting.
  • Recreate a simple Wikipedia-style article structure (title, intro paragraph, bold key terms).

🛠 Mini Project

A one-page "About Me" HTML document with a heading, 3 paragraphs, and at least 5 different formatting tags used correctly — no CSS, pure HTML structure only.

Common Mistakes

  • Forgetting <!DOCTYPE html> at the top.
  • Not closing tags properly.
  • Using <br> for spacing instead of proper structure.
  • Confusing <b>/<strong> (visual vs semantic).

How to Avoid Them

  • Always start from a saved boilerplate template.
  • Use VS Code auto-closing tags to prevent unclosed tags.
  • Remember <strong>/<em> carry meaning; <b>/<i> are purely visual.
  • Type every example yourself — never copy-paste code when learning.

Free Resources

  • MDN Web Docs — "Getting started with HTML"
  • W3Schools HTML Tutorial

Week 2 — Links, Images, Lists & Paths

Learn how to connect pages, embed images, and structure list-based content.

Topics & Subtopics

  1. Hyperlinks — <a>, href, target="_blank", internal vs external links, anchor links (#id).
  2. Images — <img>, src, alt, width/height, image formats (JPG, PNG, SVG, WebP).
  3. File paths — Absolute vs relative paths, folder structuring.
  4. Lists — <ul>, <ol>, <li>, nested lists, <dl>, <dt>, <dd>.

Why This Week Matters: Navigation and media are core to any real website. Path errors are the #1 beginner bug — mastering relative/absolute paths now prevents endless broken-link frustration on every future project.

Step-by-Step Learning Order

Paths concept → Links → Images → Unordered/Ordered lists → Nested lists → Description lists.

Daily Study Plan

Day Focus
Day 1 Hyperlinks basics and attributes.
Day 2 Anchor/jump links within a page.
Day 3 Images — tags, attributes, formats, alt text importance.
Day 4 Absolute vs relative paths; practice folder organization.
Day 5 Ordered/unordered lists.
Day 6 Nested & description lists + practice tasks.
Day 7 Mini project + revision.

Practice Tasks

  • Create 3 linked pages (Home, About, Contact) navigating to each other.
  • Build an image gallery section using relative paths.
  • Create a nested list representing a table of contents.

🛠 Mini Project

A 3-page mini-site (Home/About/Contact) with working navigation links, at least 2 images with proper alt text, and one nested list (e.g., a site menu or FAQ list).

Common Mistakes

  • Wrong relative paths (/images vs images/).
  • Missing alt attributes on images.
  • Forgetting target="_blank" opens in same tab causing confusion.
  • Broken anchor link IDs (mismatched href="#id" and actual element id).

How to Avoid Them

  • Always test links by clicking, not just reading code.
  • Keep a consistent folder structure (/images, /pages).
  • Use lowercase, hyphenated filenames — no spaces (e.g., about-me.html).

Week 3 — Tables & Basic Forms

Structure tabular data and build basic interactive input forms.

Topics & Subtopics

  1. Tables — <table>, <tr>, <td>, <th>, <thead>, <tbody>, <tfoot>, colspan, rowspan, <caption>.
  2. Forms intro — <form>, action, method (GET vs POST).
  3. Input basics — <input> types: text, password, email, number, date, checkbox, radio.
  4. Form structure — <label>, <fieldset>, <legend>, <select>, <option>, <textarea>, <button>.

Why This Week Matters: Tables and forms are the most commonly tested beginner HTML skills and are used in almost every real site — contact forms, login forms, registration forms, and data display tables appear on nearly every production website.

Step-by-Step Learning Order

Tables basics → Table structuring (thead/tbody) → Form container → Input types → Labels/fieldsets → Select/textarea/button.

Daily Study Plan

Day Focus
Day 1 Basic table structure.
Day 2 Advanced table — colspan, rowspan, thead/tbody/tfoot.
Day 3 Form container, action/method, basic text inputs.
Day 4 Input types — checkbox, radio, email, date, number.
Day 5 Labels, fieldset, legend (accessibility-linked).
Day 6 Select, textarea, button + practice.
Day 7 Mini project + revision.

Practice Tasks

  • Build a timetable using colspan/rowspan.
  • Build a student registration form with at least 6 different input types, all linked to <label>.

🛠 Mini Project

A "Student Registration Form" page with a data table (showing sample registered students) and a working form below it — name, email, DOB, gender (radio), course (select), message (textarea), submit button.

Common Mistakes

  • Using tables for page layout (outdated, incorrect practice).
  • Inputs without <label> — hurts accessibility.
  • Missing name attribute on inputs (breaks form submission).
  • Not grouping related radio buttons under the same name.

How to Avoid Them

  • Use tables only for real tabular data — never for layout.
  • Always pair every input with a <label for="id">.
  • Double-check every input has a unique id and a name attribute.

Week 4 — Advanced Forms & Form Validation

Master advanced form inputs and built-in HTML validation attributes.

Topics & Subtopics

  1. Advanced input types — url, tel, range, color, file, search, hidden.
  2. Validation attributes — required, pattern, min, max, minlength, maxlength, step.
  3. Datalist & autocomplete — <datalist>, autocomplete.
  4. Form usability — placeholder, autofocus, disabled, readonly, grouping with <fieldset>.

Why This Week Matters: Real-world forms (sign-ups, payments, surveys) depend heavily on native validation — reduces the need for JavaScript for basic checks. Understanding these attributes is directly tested in interviews and viva evaluations.

Step-by-Step Learning Order

Advanced input types → Validation attributes → Datalist/autocomplete → Usability attributes → Combine into one complex form.

Daily Study Plan

Day Focus
Day 1 Advanced input types — url, tel, range, color, file.
Day 2 required, pattern, min/max.
Day 3 minlength/maxlength/step; test validation live in browser.
Day 4 <datalist>, autocomplete.
Day 5 placeholder, autofocus, disabled, readonly.
Day 6 Build a complete validated form + practice.
Day 7 Mini project + revision/quiz.

Practice Tasks

  • Build a "Job Application Form" with file upload, range slider (experience years), and pattern-validated phone number.
  • Create a search bar using <datalist> with city suggestions.

🛠 Mini Project

A fully validated "Event Registration Form" — required fields, email pattern check, age min/max, phone pattern, and a <datalist> for country selection.

Common Mistakes

  • Relying only on required and ignoring pattern for format checks.
  • Using disabled when readonly was meant — disabled fields don't submit data.
  • Incorrect regex in pattern attribute.

How to Avoid Them

  • Test every validation rule manually by submitting wrong data before moving on.
  • Remember: disabled = not submitted; readonly = visible, submitted, but not editable.
  • Keep pattern regex simple and test it on regex101.com before using.

Week 5 — Semantic HTML & Page Layout Structure

Structure full pages using meaningful, accessible semantic tags instead of generic div soup.

Topics & Subtopics

  1. Semantic elements — <header>, <nav>, <main>, <section>, <article>, <aside>, <footer>.
  2. Generic containers — <div>, <span> and when to use them vs semantic tags.
  3. Document outline — Heading hierarchy best practices (h1 once per page, logical nesting).
  4. Figures & captions — <figure>, <figcaption>.

Why This Week Matters: Semantic HTML improves SEO, accessibility, and code readability — a core "best practice" topic that interviewers and evaluators specifically check for. A page built with proper semantics is easier to maintain, style, and understand.

Step-by-Step Learning Order

Why semantics matter → header/nav/footer → main/section/article → aside → div/span usage rules → figure/figcaption → full page skeleton.

Daily Study Plan

Day Focus
Day 1 Why semantics matter; header, footer.
Day 2 nav, main.
Day 3 section vs article vs div — the trickiest distinction.
Day 4 aside; heading hierarchy rules.
Day 5 figure/figcaption.
Day 6 Build a full semantic page skeleton + practice.
Day 7 Mini project + revision.

Practice Tasks

  • Convert a previous "div-only" page (from Week 1–3) into full semantic HTML.
  • Inspect 2 real websites' structure using DevTools and note which semantic tags they use.

🛠 Mini Project

A semantic "Blog Homepage" layout: header with nav, main with 2–3 <article> blog posts inside a <section>, an <aside> with related links, and a <footer>.

Common Mistakes

  • Using <section> for purely visual grouping (should have a heading and thematic meaning).
  • Using multiple <h1> tags per page.
  • Overusing <div> where semantic tags clearly exist.

How to Avoid Them

  • If a <div> represents a real region (navigation, article, sidebar), use the semantic tag instead.
  • One <h1> per page — always.
  • Give every <section> a heading to justify its existence.

Week 6 — Multimedia: Audio, Video & Embeds

  4. SVG & Canvas basics — Inline <svg> simple shapes, intro to <canvas> (awareness level).

Why This Week Matters: Modern websites are media-rich. Correct embedding avoids performance and security issues. Knowing how to add captions to video is also an accessibility requirement in professional work.

Step-by-Step Learning Order

Audio basics → Video basics → Captions/track → iframe usage & security attributes → SVG intro → Canvas awareness.

Daily Study Plan

Day Focus
Day 1 Audio tag and attributes.
Day 2 Video tag, poster image, multiple sources.
Day 3 Captions with <track>; accessibility importance.
Day 4 iframe basics; embedding YouTube/Google Maps.

Practice Tasks

  • Embed a local video with custom poster image and controls.
  • Embed a YouTube video and a Google Map using iframes with lazy loading.
  • Draw a simple shape (circle/rectangle) using inline SVG.

🛠 Mini Project

A "Media Gallery" page with one audio player, one video player (with poster + captions track), and one embedded map/YouTube video — all responsive and lazy-loaded.

Common Mistakes

  • Forgetting fallback text for unsupported browsers inside <audio>/<video> tags.
  • Embedding iframes without loading="lazy" — hurts page performance.
  • Missing controls attribute — users can't interact with media.
  • Using non-royalty-free media — copyright issues.

How to Avoid Them

  • Always provide fallback text inside <audio>/<video> tags.
  • Always add loading="lazy" on non-critical iframes.
  • Use only royalty-free assets (Pixabay, Pexels) for practice projects.

Week 7 — Accessibility (a11y) in HTML

Write HTML that is usable by everyone, including people using screen readers and assistive tech.

Topics & Subtopics

  1. Why accessibility matters — Legal/ethical/SEO reasons, WCAG basics (awareness level).
  2. ARIA basics — role, aria-label, aria-hidden, aria-describedby (used only when native HTML isn't enough).
  3. Accessible forms & navigation — Label associations, tabindex, skip links, focus states.
  4. Color contrast & text alternatives — alt text quality, contrast awareness.

Why This Week Matters: Accessibility is a major professional and ethical skill. Many real companies and exam evaluators specifically check for it. Accessible sites are also better for SEO and usability for all users, not just those with disabilities.

Step-by-Step Learning Order

Why accessibility → Native HTML accessibility (semantic tags, labels) → ARIA only as a supplement → Keyboard navigation/focus → Alt text & contrast awareness.

Daily Study Plan

Day Focus
Day 1 Why accessibility matters; WCAG overview (conceptual).
Day 2 Native HTML accessibility recap (semantics, labels).
Day 3 ARIA roles/attributes — when and when not to use them.
Day 4 Keyboard navigation, tabindex, skip links.
Day 5 Writing good alt text; contrast awareness.
Day 6 Audit previous projects for accessibility + fix them.
Day 7 Mini project + revision.

Practice Tasks

  • Run your Week 5 mini project through a screen reader and note 3 issues.
  • Add a "skip to main content" link to a multi-section page.

🛠 Mini Project

Take your Week 5 "Blog Homepage" and refactor it into an Accessible Blog Homepage: proper landmark roles, skip link, accessible form, quality alt text, and logical tab order.

Common Mistakes

  • Overusing ARIA when semantic HTML already solves it ("First rule of ARIA: don't use ARIA if you don't have to").
  • Empty alt="" on meaningful images.
  • Removing focus outlines in CSS without providing a visible replacement.

How to Avoid Them

  • Always prefer native semantic HTML first; only add ARIA for genuinely custom widgets.
  • Never set alt="" unless the image is purely decorative.
  • Never remove focus styles without providing a visible alternative.

  • MDN — "Accessibility" guide

Week 8 — SEO-Friendly HTML & Meta Information

Learn how HTML structure affects search engine visibility and write SEO-friendly markup.

Topics & Subtopics

  1. Meta tags — <meta name="description">, <meta name="viewport">, <meta charset>.
  2. Open Graph & social meta — og:title, og:description, og:image (for social sharing previews).
  3. SEO-friendly structure — Proper heading hierarchy, descriptive alt text, descriptive link text (avoid "click here"), <title> best practices.
  4. robots.txt / sitemap.xml awareness — Conceptual overview (separate files, not HTML tags, but commonly grouped with on-page SEO).

Why This Week Matters: Demonstrates real-world relevance of HTML beyond just rendering. Important for college evaluators who ask "why does this tag matter?" — and critical for any career touching web or digital marketing.

Step-by-Step Learning Order

Core meta tags → Viewport (mobile-friendliness) → Open Graph tags → SEO-friendly content structure → robots.txt/sitemap awareness.

Daily Study Plan

Day Focus
Day 1 Core meta tags — charset, description, viewport.
Day 2 Title tag best practices; heading hierarchy for SEO.
Day 3 Open Graph tags for social sharing previews.
Day 4 Descriptive alt text & link text for SEO.
Day 5 robots.txt and sitemap.xml — conceptual overview.
Day 6 Apply all SEO practices to an existing project.
Day 7 Mini project + revision.

Practice Tasks

  • View-source 3 real websites and list all <meta> tags found.
  • Add full SEO meta tags (including Open Graph) to your Week 5 blog project.

🛠 Mini Project

An "SEO-Optimized Landing Page" — complete <head> with charset, viewport, description, title best practices, Open Graph tags, and a body with proper heading hierarchy and descriptive alt/link text throughout.

Common Mistakes

  • Missing viewport meta tag — breaks mobile responsiveness entirely.
  • Duplicate or missing <title> tags.
  • Vague alt text like "image1.jpg".
  • Multiple <h1> tags confusing search engines.

How to Avoid Them

  • Always include the viewport meta tag as a default in every project from now on.
  • Write a unique, descriptive <title> for every page.
  • Write alt text that describes the image's purpose/content — not its filename.

🛠 Mini Project

An "Interactive FAQ + Skills" page combining a <details>-based accordion FAQ section and a <meter>/<progress>-based skills showcase — fully functional with zero or minimal JS.

Common Mistakes

  • Confusing <progress> (task completion) with <meter> (a static measurement like disk usage).
  • Forgetting <dialog> needs JS (.showModal()) to open properly in most cases.
  • Not testing browser support for newer elements.

How to Avoid Them

  • Remember: progress = "how much is done," meter = "how much exists/value."
  • Always check caniuse.com for <dialog> support before relying on it heavily.
  • Clearly separate "pure HTML" sections from "HTML + tiny JS" sections in your code comments.

Week 10 — Responsive HTML Foundations

Week 12 — HTML + Minimal JavaScript Integration

Understand how HTML connects to JavaScript and add light interactivity to your pages.

⚠️ Note: Like Week 11, this week is a deliberate, clearly-flagged crossover into basic JavaScript — only enough to make forms/buttons interactive, not full JS mastery. Your project remains an HTML project with a light JS layer.

Topics & Subtopics

  1. Connecting JS — <script> tag placement (defer, end-of-body), inline onclick (why to avoid it).
  2. DOM basics — document.getElementById, document.querySelector (conceptual link to HTML structure).
  3. Simple event handling — Button clicks, form submit prevention (preventDefault).
  4. HTML5 data attributes — data-* attributes and why they're useful for JS hooks.

Daily Study Plan

Day Focus
Day 1 <script> placement and defer; why inline onclick is discouraged.
Day 2 DOM selection — getElementById, querySelector.
Day 3 Simple click event — show/hide a section.
Day 4 Form submit event with preventDefault + basic validation message.
Day 5 data-* attributes — practical example (tabs).
Day 6 Build a small tabbed interface using data attributes + practice.
Day 7 Mini project + revision.

Practice Tasks

  • Add a "Show more / Show less" toggle button to a text section using minimal JS.
  • Add a custom validation message on form submit (in addition to native HTML validation).

🛠 Mini Project

A "Mini Interactive Dashboard" section: a tabbed interface using data-* attributes + a few lines of JS showing different content panels — Skills, Projects, and Contact tabs.

Common Mistakes

  • Writing too much JS and losing focus on the HTML project goal.
  • Using onclick="..." inline everywhere — hard to maintain, mixes concerns.
  • Forgetting defer and getting "element not found" errors (JS runs before HTML loads).

How to Avoid Them

  • Keep a clear boundary — this is an HTML project with a light JS layer, not a JS project.
  • Always add defer to script tags in <head>, or place <script> at end of <body>.
  • Use external .js files instead of inline handlers.

Week 13 — Code Quality, Validation, Debugging & Best Practices

Write clean, valid, well-organized, and maintainable HTML — essential before building the final project.

Topics & Subtopics

  1. HTML validation — W3C Markup Validator, common validation errors and fixes.
  2. Code organization — File/folder structure conventions, naming conventions, indentation/formatting (Prettier).
  3. Browser DevTools mastery — Elements tab, Console tab (basic errors), Network tab (basic awareness).
  4. Cross-browser & performance basics — Testing in 2+ browsers, image optimization basics, minimizing unnecessary nesting.

Why This Week Matters: Clean, valid, well-structured code is what separates a "passing" project from a "polished, professional" one. Directly affects grading and viva impressions. Evaluators who open DevTools will immediately notice validation errors and poor structure.

Step-by-Step Learning Order

Validation → Folder/naming conventions → Formatting tools → DevTools deep-dive → Cross-browser testing → Performance basics.

Daily Study Plan

Day Focus

Practice Tasks

  • Validate all previous mini-projects through W3C Validator; fix every reported error.
  • Reorganize all project folders into one consistent structure (/css, /js, /images, /pages).

🛠 Mini Project

A "Project Cleanup Audit" — a simple HTML page itself listing each previous mini-project, validation errors found and fixed, and folder/naming improvements made. This documents your quality process.

Common Mistakes

  • Ignoring validator warnings as "not important."
  • Inconsistent file naming (Page1.html, about_us.HTML).
  • Not testing in more than one browser.

How to Avoid Them

  • Treat 0 validation errors as a non-negotiable goal before final submission.
  • Adopt lowercase-hyphenated naming permanently (about-us.html).
  • Always test in at least Chrome and Firefox before marking anything as done.

Week 14 — Final Capstone Project: Planning & Build (Part 1)

Plan and build the structural foundation of your final capstone project — the actual college project website.

Topics & Subtopics

  1. Project planning — Sitemap, page list, content outline, wireframing (simple paper/Figma sketch).
  2. Information architecture — Deciding navigation structure, consistent header/footer across pages.
  3. Building core pages — Home, About, and 2–3 content pages using everything learned.
  4. Reusable structure — Consistent <head> template, consistent navbar/footer across all pages.

Why This Week Matters: This is where all 13 weeks of learning converge into one real, complete, presentable deliverable. Planning now prevents major rework during Week 15. Rushing this week is the single biggest mistake students make on final projects.

Step-by-Step Learning Order

🛠 Week 14 Deliverable

The skeleton (structure + content, lightly styled) of your final capstone website with at least 4 working, linked pages — consistent navigation, semantic HTML, and proper meta tags on every page.

Common Mistakes

  • Skipping planning and jumping straight to coding.
  • Inconsistent navigation across pages (different on every page).
  • Forgetting accessibility/SEO practices under final-project time pressure.

How to Avoid Them

  • Force yourself to write the sitemap before opening the code editor.
  • Copy the exact same header/footer code across pages — don't retype it.
  • Keep a personal checklist: alt text, labels, meta tags, semantic tags — verify on every page.
