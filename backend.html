Web Backend Development

Complete 20-Week Learning Roadmap — Node.js, Express.js, PostgreSQL & Prisma

About This Roadmap

This roadmap presents a complete, structured, beginner-to-advanced learning path for Web Backend Development using Node.js, Express.js, and PostgreSQL — built and documented as a college project.

The roadmap takes a complete beginner with no prior backend or database experience and progresses step by step through internet fundamentals, JavaScript, Node.js core, Express, databases, authentication, security, testing, real-time features, deployment, and a complete capstone project.

Duration: 20 weeks  |  Pace: Full-time (~35 hrs/week)  |  Stack: Node.js · Express.js · PostgreSQL · Prisma · JWT · Docker

Hands-On: 19 mini-projects + 1 full capstone project (Library Management System API)  |  Cost: 100% free tools and resources

5 Phases Overview

  • Phase 1 (Weeks 1–4): Foundations — Web, JavaScript, and Node.js
  • Phase 2 (Weeks 5–8): Backend Basics — Express and Databases
  • Phase 3 (Weeks 9–12): Core Backend Skills — Auth, Validation, Testing
  • Phase 4 (Weeks 13–16): Advanced Concepts — Databases, Security, Performance
  • Phase 5 (Weeks 17–20): Production Readiness & Capstone

Week 1 — How the Web Works & Developer Environment Setup

Build a correct mental model of how the internet and web work, and set up a fully working development environment.

Topics & Subtopics

  1. Client-server model — what a client is, what a server is, how they communicate.
  2. DNS — how domain names resolve to IP addresses.
  3. HTTP/HTTPS — request line, headers, body, common methods (GET, POST, PUT, DELETE).
  4. Ports — what a port is, common ports (80, 443, 3000).
  5. Frontend vs. backend — roles and responsibilities.
  6. Developer environment setup — Node.js, npm, VS Code.
  7. Git basics — init, add, commit, status, log, branches, .gitignore.
  8. GitHub basics — creating a repo, push, pull, remote.

Why This Week Matters: Every concept in this roadmap (APIs, requests, status codes) builds on a correct understanding of HTTP and the client-server model. Skipping this leads to memorizing code without understanding why it works.

Daily Study Plan

Day Focus
Day 1 How the internet works (client-server, IP, DNS) — read + notes.
Day 2 HTTP deep dive (methods, headers, status codes).
Day 3 Install Node.js, npm, VS Code; explore terminal basics (cd, ls/dir, mkdir).
Day 4 Git basics — init, add, commit, log, branches.
Day 5 GitHub — create account, create repo, push/pull, .gitignore.
Day 6 Practice tasks + write Week 1 notes.
Day 7 Review all concepts; complete mini project.

Practice Tasks

  • Run node -v and npm -v to confirm installation.
  • Create a GitHub repository named backend-roadmap-journal.
  • Make at least 5 separate commits practicing add/commit/push.
  • List 10 common HTTP status codes and what they mean, from memory.

🛠 Mini Project

Create the backend-roadmap-journal GitHub repository. Add a week-01.md file containing: (a) a hand-drawn-style diagram of the HTTP request/response cycle, and (b) a written explanation, in your own words, of the difference between frontend and backend.

Common Mistakes

  • Jumping straight into coding tutorials without understanding HTTP basics.
  • Ignoring Git until "later" — leads to messy, hard-to-recover project history.
  • Confusing a domain name with a server.

How to Avoid Them

  • Spend real time on Day 1–2 conceptual learning before touching any code.
  • Commit to GitHub from Day 1, even for small notes.
  • Use simple analogies (e.g., DNS is like a phone book) to lock in concepts.

Free Resources

Week 2 — JavaScript Fundamentals (ES6+)

Become confident writing modern JavaScript — the language that powers your entire backend.

Topics & Subtopics

  1. Variables — let, const, hoisting (conceptual), primitive vs. reference types.
  2. Operators & control flow — if/else, switch.
  3. Functions — parameters, default values, return values, arrow function syntax.
  4. Arrays — map, filter, reduce, forEach, find, includes.
  5. Objects — property access, shorthand syntax, destructuring, nested objects.
  6. Spread/Rest — copying arrays/objects, combining, function rest parameters.
  7. Template literals — string interpolation.
  8. Scope & closures — global vs. function vs. block scope; closures (conceptual introduction).

Why This Week Matters: Node.js and Express are JavaScript. Weak JavaScript fundamentals are the single biggest cause of beginners getting stuck later when learning Express and async code.

Daily Study Plan

Day Focus

Day 2 Functions — declarations, expressions, arrow functions, default parameters.
Day 3 Arrays and array methods (map, filter, reduce, forEach).
Day 4 Objects, destructuring, nested data.
Day 5 Template literals, spread/rest operators.
Day 6 Scope and closures + practice tasks.
Day 7 Review + mini project.

Practice Tasks

  • Write 10 small functions using arrow function syntax.
  • Given an array of objects (e.g., students with grades), use filter and map to extract specific data.
  • Practice destructuring 5 different objects/arrays.
  • Use the spread operator to merge two objects and two arrays.

🛠 Mini Project

Build a Node.js CLI script (taskManager.js, run via node taskManager.js) that manages a list of tasks stored in an array of objects. It should support adding a task, removing a task by ID, marking a task complete, and listing all tasks — all using functions and array methods (no Express yet).

Common Mistakes

  • Confusing var/let/const scoping rules.
  • Avoiding array methods and using manual for loops everywhere instead.
  • Misunderstanding this (light awareness is enough at this stage).

How to Avoid Them

  • Default to const, use let only when reassignment is needed, avoid var entirely.
  • Deliberately practice map/filter/reduce even when a for loop feels easier.
  • Revisit this later once you reach Express middleware in Week 5.

Week 3 — Asynchronous JavaScript & the Node.js Event Loop

Understand and confidently write asynchronous JavaScript — the foundation of how Node.js handles requests, files, and databases.

Topics & Subtopics

  1. Callbacks — passing functions as arguments, callback hell example and why it's a problem.
  2. Promises — .then(), .catch(), .finally(), Promise states (pending/fulfilled/rejected).
  3. async/await — syntax, how it relates to Promises, sequential vs. parallel async calls (Promise.all).
  4. Error handling — try/catch with async/await, unhandled promise rejections.
  5. Event loop — why Node.js is "non-blocking," call stack vs. callback queue (conceptual overview).

Why This Week Matters: Almost everything in backend development is asynchronous — reading files, querying databases, calling external APIs. Misunderstanding async code is the #1 source of bugs for backend beginners.

Daily Study Plan

Day Focus
Day 1 Sync vs async, introduction to callbacks.
Day 2 Callback hell example + Promises introduction.
Day 3 Promises deep dive (.then, .catch, chaining).
Day 4 async/await syntax and rewriting Promise code.
Day 5 Error handling with try/catch, Promise.all for parallel calls.
Day 6 Event loop concept + practice tasks.
Day 7 Review + mini project.

Practice Tasks

  • Rewrite a callback-based function using Promises, then again using async/await.
  • Write a function using setTimeout wrapped in a Promise to simulate an API delay.
  • Use Promise.all to run 3 simulated async calls in parallel and combine results.
  • Deliberately remove a try/catch and observe the unhandled rejection error.

🛠 Mini Project

Build a Node.js script that simulates fetching data from 3 different "fake APIs" (each implemented as a function returning a Promise via setTimeout). Use async/await with proper try/catch to fetch all 3, combine their results into one object, and log it. Include at least one deliberately failing call handled gracefully.

Common Mistakes

  • Forgetting the await keyword, causing a Promise object to be logged instead of the resolved value.
  • Not handling rejected Promises, causing silent failures or crashes.
  • Writing nested .then() chains instead of using async/await for readability.

How to Avoid Them

  • Always pair async/await with try/catch.
  • Test what happens when an async function fails — don't only test the success path.
  • Prefer async/await over .then() chains for anything beyond a single step.

  • Node.js Event Loop Docs
  • MDN — Using Promises

Week 4 — Node.js Core Concepts

Understand Node.js itself — outside of any framework — including its module system, built-in modules, and package management.

Topics & Subtopics

  1. Modules — CommonJS (require/module.exports) and ES modules (import/export).
  2. fs module — reading/writing files, sync vs async file methods.
  3. path module — joining/resolving file paths cross-platform.
  4. http module — creating a server, handling requests manually, sending responses.
  5. npm & package.json — npm init, installing packages, dependencies vs devDependencies.
  6. Environment variables — using a .env file with the dotenv package.

Why This Week Matters: Express is built on top of Node's http module. Understanding Node's core mechanics first prevents Express from feeling like "magic" and makes debugging much easier later.

Daily Study Plan

Day Focus
Day 1 What is Node.js + CommonJS modules (require/exports).
Day 2 fs module — reading and writing files.
Day 3 path and os modules.
Day 4 Raw http module — build a basic server.
Day 5 Manual routing with the http module (if/else on req.url).
Day 6 npm, package.json, dotenv + practice tasks.
Day 7 Review + mini project.

Practice Tasks

  • Write a script that reads a local JSON file and logs its parsed content.
  • Build a server using only the http module that responds differently to /, /about, and any other path (404).
  • Create a .env file with a PORT variable and use it in your server instead of hardcoding the port.
  • Run npm init -y and manually inspect the generated package.json.

🛠 Mini Project

Build a raw Node.js HTTP server (no Express) with simple manual routing: GET / returns a welcome message, GET /about returns project info, GET /books reads and returns the contents of a local books.json file, and any other route returns a 404 JSON response. Use a .env file to configure the port.

Common Mistakes

  • Mixing require and import syntax without configuring "type": "module" in package.json.
  • Committing the node_modules folder to GitHub.
  • Hardcoding the port number or secrets directly in code.

How to Avoid Them

  • Stick to CommonJS (require) throughout this roadmap for consistency.
  • Always create a .gitignore file with node_modules and .env listed.
  • Use dotenv and commit a .env.example showing variable names only.

Week 5 — Introduction to Express.js

Learn Express.js — the framework you'll use for the rest of the roadmap — and understand routing and middleware.

Topics & Subtopics

  1. Setup — npm install express, creating an app, app.listen().
  2. Routing — app.get(), app.post(), route parameters (/users/:id) vs query strings (?search=term).
  3. Request/Response objects — req.params, req.query, req.body, res.json(), res.status().
  4. Middleware — app.use(), built-in middleware (express.json()), the order middleware runs in.
  5. Dev tools — using nodemon for auto-restart during development.
  6. Project structure — separating routes into their own files.

Why This Week Matters: Express is the industry-standard Node.js framework. Nearly every concept for the rest of the roadmap (auth, validation, error handling) is implemented as Express middleware or routes.

Daily Study Plan

Day Focus
Day 1 Install Express, build a minimal "Hello World" server, set up nodemon.
Day 2 Basic routing — GET, POST, PUT, DELETE routes.
Day 3 Route parameters and query strings.
Day 4 Request and response object deep dive.
Day 5 Middleware concept, express.json(), custom middleware example.
Day 6 Organizing routes into separate files + practice tasks.
Day 7 Review + mini project.

Practice Tasks

  • Build 5+ routes: home, about, a dynamic route with a parameter, a route using query strings, and a POST route.
  • Write a simple custom middleware function that logs the request method and URL for every request.
  • Test all routes using Postman or Thunder Client.

🛠 Mini Project

Build an Express server with at least 5 routes: GET / (welcome message), GET /about, GET /users/:id (returns the ID), GET /search?term=... (returns the query value), and POST /echo (returns the JSON body sent). Add a custom logging middleware that prints every incoming request.

Common Mistakes

  • Forgetting express.json(), so req.body is undefined on POST requests.
  • Writing all routes and logic in a single massive file.
  • Not restarting the server after changes (forgetting nodemon).

How to Avoid Them

  • Always add app.use(express.json()) near the top of your app setup.
  • Start separating routes into files from this week, even if the project is small.
  • Use nodemon from Day 1 instead of manually restarting.

Week 6 — REST API Design & CRUD (In-Memory)

Learn REST API design principles and build a complete CRUD API using in-memory data.

Topics & Subtopics

  1. REST principles — statelessness, resource-based URLs, uniform interface.
  2. CRUD mapping — POST=Create, GET=Read, PUT/PATCH=Update, DELETE=Delete.
  3. Status codes — 200, 201, 204, 400, 401, 403, 404, 500 — when to use each.
  4. URL design — /books vs /books/:id, avoiding verbs in URLs.
  5. Postman — creating requests, collections, environments.
  6. Project structure — separating routes from controller logic.

Why This Week Matters: REST API design is the single most important practical skill in backend development — it's how almost every web and mobile app communicates with its server, and it's heavily tested in interviews.

Daily Study Plan

Day Focus
Day 1 REST principles + CRUD mapping + status codes.
Day 2 Design the Books API URL structure on paper; set up Postman.
Day 3 Implement GET all and GET one (by ID) routes.
Day 4 Implement POST (create) and PUT (update) routes.
Day 5 Implement DELETE route; handle "not found" cases properly.
Day 6 Refactor into routes/controllers structure + practice tasks.
Day 7 Review + mini project + export Postman collection.

Practice Tasks

  • Write down the correct status code for 8 different scenarios.
  • Build all 5 CRUD endpoints for a "Books" resource using an in-memory array.
  • Test every endpoint in Postman, including error cases.

🛠 Mini Project

Build a complete in-memory CRUD REST API for "Library Books" with routes: GET /books, GET /books/:id, POST /books, PUT /books/:id, DELETE /books/:id. Organize the code into routes/ and controllers/ folders. Export a Postman collection covering all endpoints.

Common Mistakes

  • Returning 200 OK for every response, even errors.
  • Using verbs in URLs (e.g., /getAllBooks instead of GET /books).
  • Putting all logic directly inside route handlers instead of controllers.

How to Avoid Them

  • Memorize and consistently apply the core status codes (200, 201, 400, 404, 500).
  • Always design URLs around resources (nouns), with HTTP verbs indicating the action.
  • Practice the routes/controllers separation now — it scales much better later.

Week 7 — SQL & PostgreSQL Fundamentals

Learn relational database concepts and SQL well enough to design and query a real database confidently.

Topics & Subtopics

  1. Setup — installing PostgreSQL, using psql or pgAdmin.
  2. DDL — CREATE TABLE, data types (SERIAL, VARCHAR, INTEGER, BOOLEAN, TIMESTAMP).
  3. DML — INSERT INTO, SELECT, UPDATE, DELETE.
  4. Filtering/Sorting — WHERE, ORDER BY, LIMIT.
  5. Keys — primary key concept, foreign key concept, referential integrity.
  6. Joins — INNER JOIN (focus here), brief mention of LEFT JOIN.
  7. Constraints — NOT NULL, UNIQUE, DEFAULT.
  8. Normalization — conceptual overview of 1NF–3NF.

Why This Week Matters: SQL is a universal skill required across nearly all backend roles, regardless of programming language. Poor schema design is one of the most common causes of bugs and performance problems in real applications.

Daily Study Plan

Day Focus
Day 1 Install PostgreSQL + pgAdmin; relational database concepts.
Day 2 CREATE TABLE, data types, constraints.
Day 3 INSERT, SELECT, UPDATE, DELETE.
Day 4 WHERE, ORDER BY, LIMIT, filtering practice.
Day 5 Primary keys, foreign keys, relationships.
Day 6 INNER JOIN + normalization concepts + practice tasks.
Day 7 Review + mini project.

Practice Tasks

  • Create 3 related tables by hand using CREATE TABLE.
  • Write and run at least 15 different SQL queries covering CRUD and filtering.
  • Write 3 INNER JOIN queries combining data from 2 tables.
  • Practice designing a schema on paper before creating it in SQL.

🛠 Mini Project

Design and create a PostgreSQL database for a "Library Management System" with at least 3 related tables: books, members, and borrow_records (with proper foreign keys). Write and execute 15+ SQL queries demonstrating CRUD operations, filtering, sorting, and at least 3 joins.

Common Mistakes

  • Creating tables without planning relationships first.
  • Forgetting primary keys or using the wrong data types.
  • Writing overly complex queries before mastering basic SELECT/WHERE.

How to Avoid Them

  • Always sketch an ER (entity-relationship) diagram before creating tables.
  • Use SERIAL/auto-incrementing IDs as primary keys by default for beginners.
  • Master basic queries thoroughly before attempting joins or subqueries.

Connect your Express application to a real PostgreSQL database using Prisma, replacing in-memory data with persistent storage.

  • Run npx prisma migrate dev and confirm the tables exist in pgAdmin.
  • Write a standalone script using Prisma Client to create, read, update, and delete records.
  • Use npx prisma studio to visually browse your database.

🛠 Mini Project

Rebuild the "Library Books" API from Week 6 so it's fully backed by PostgreSQL via Prisma. Implement a Book-Author relationship (one author has many books). All 5 CRUD endpoints should now read/write real database records instead of an in-memory array.

Common Mistakes

  • Editing schema.prisma but forgetting to run a migration afterward.
  • Hardcoding database credentials instead of using .env.
  • Not understanding that the Prisma schema and the live database are kept in sync only through migrations.

How to Avoid Them

  • Make "edit schema → migrate → test" a strict habit, every time.
  • Always use process.env.DATABASE_URL via .env, never commit .env to GitHub.
  • Use npx prisma studio regularly to visually confirm your database matches expectations.

Week 9 — Authentication & Authorization

Implement secure user registration, login, and route protection using industry-standard practices.

Topics & Subtopics

  1. Authentication — registration flow, login flow, verifying credentials.
  2. bcrypt — hashing passwords before storage, comparing hashed passwords on login.
  3. JWT — structure (header.payload.signature), signing and verifying tokens, setting expiration.
  4. Middleware — writing an authenticate middleware that checks for a valid token.
  5. RBAC — storing a role field (e.g., member/admin), writing an authorize(role) middleware.
  6. Sessions vs. tokens — brief conceptual comparison, why this roadmap uses JWT.

Why This Week Matters: Nearly every real backend application needs user accounts and protected resources. Authentication is also one of the most frequently tested topics in technical interviews.

Daily Study Plan

Day Focus
Day 1 Authentication vs authorization concepts; install bcrypt.
Day 2 Build the registration endpoint with password hashing.
Day 3 Learn JWT structure; install and configure jsonwebtoken.
Day 4 Build the login endpoint that verifies password and issues a JWT.
Day 5 Write authentication middleware to protect routes.
Day 6 Write role-based authorization middleware (admin vs member) + practice tasks.
Day 7 Review + mini project.

Practice Tasks

  • Hash a password with bcrypt and verify it correctly compares on login.
  • Decode a sample JWT manually at jwt.io and identify its 3 parts.
  • Build middleware that rejects requests without a valid token (401 response).
  • Build middleware that rejects non-admin users from an admin-only route (403 response).

🛠 Mini Project

Add full authentication to your Library API: a User model with a role field, POST /auth/register, POST /auth/login (returns a JWT), authentication middleware protecting all write routes, and authorization middleware restricting book creation/deletion to admins only. Ensure password hashes are never returned in any API response.

Common Mistakes

  • Storing plain-text passwords.
  • Returning the password hash field in API responses.
  • Using a weak, guessable JWT secret.
  • Not checking token expiration.

How to Avoid Them

  • Always hash passwords with bcrypt before saving — never store raw passwords.
  • Explicitly exclude sensitive fields (like password hashes) from API responses.
  • Use a long, random JWT secret stored in .env, never hardcoded.
  • Set a reasonable token expiration (e.g., 1 day) and test expired-token behavior.

Week 10 — Validation, Error Handling & Middleware Patterns

Make your API production-quality by validating input properly and handling errors gracefully and consistently.

Topics & Subtopics

  1. Validation — using express-validator or zod to validate request bodies.
  2. Error handling — Express's error-handling middleware signature (err, req, res, next).

  4. Async error handling — wrapping async route handlers to catch errors automatically.
  5. Logging — using morgan for request logging.
  6. Structure — organizing into routes/, controllers/, services/, middleware/.

Why This Week Matters: This is the week where your code starts looking like a professional, production-grade application. Consistent error handling and validation are major signals of code quality in interviews and code reviews.

Daily Study Plan

Day Focus
Day 1 Install and learn express-validator/zod; validate the "create book" route.
Day 2 Apply validation to all remaining write routes.
Day 3 Build a custom AppError class and centralized error-handling middleware.
Day 4 Wrap async route handlers to catch and forward errors properly.
Day 5 Add morgan logging; test various error scenarios.
Day 6 Refactor folder structure (routes/controllers/services/middleware) + practice tasks.
Day 7 Review + mini project.

Practice Tasks

  • Add validation rules (required fields, correct types, length limits) to every write endpoint.
  • Trigger 5 different error scenarios and confirm each returns a consistent JSON error shape.
  • Add morgan and observe the log output for different requests.

🛠 Mini Project

Refactor your Library API to add: input validation on every POST/PUT route, a centralized error-handling middleware returning a consistent { success: false, message, statusCode } format, async error wrapping so no unhandled promise rejections crash the server, and morgan request logging.

Common Mistakes

  • Validating data only on the frontend and trusting backend input blindly.
  • Not catching errors in async route handlers, causing the server to crash.
  • Returning differently-shaped error responses from different routes.

How to Avoid Them

  • Always validate on the backend — never trust client-sent data.
  • Wrap every async controller function so errors are passed to next(err).
  • Standardize one single error response shape used everywhere in the API.

Write a complete Jest + Supertest test suite for your Library API covering at least 10 test cases: CRUD success cases, validation failure cases, and authentication/authorization failure cases (e.g., a member trying to delete a book should return 403).

Common Mistakes

  • Only testing the "happy path" and never testing failures.
  • Tests that depend on each other's execution order.
  • Not resetting test data between test runs, causing flaky tests.

How to Avoid Them

  • For every endpoint, write at least one success test and one failure test.
  • Write each test so it can run independently, in any order.
  • Use a dedicated test database (or reset/seed data in beforeEach/beforeAll hooks).

Week 13 — Advanced Database Concepts

Common Mistakes

  • Performing multi-step updates without transactions, risking inconsistent data.
  • Ignoring the N+1 query problem, causing slow API responses as data grows.
  • Adding indexes to every column "just in case," without understanding write-performance trade-offs.

How to Avoid Them

  • Use transactions for any operation that updates more than one table together.
  • Use Prisma's include to eager-load related data instead of querying in a loop.

Learn to make your API faster and more efficient using caching and pagination.

  4. Pagination — limit/offset pagination (and cursor-based for very large datasets).
  5. Compression — using the compression middleware to reduce response size.

Daily Study Plan

  • Not paginating large datasets, causing slow responses and high memory usage.
