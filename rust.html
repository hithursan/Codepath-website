About This Curriculum

This is a 16-week, structured Rust curriculum designed to take a complete beginner to a professional-level systems programmer. The roadmap moves from toolchain setup, syntax, types, and ownership through control flow, functions, structs, enums, and pattern matching, into traits, generics, error handling, and crates, all the way to lifetimes, macros, custom iterators, concurrency, async/await, and three complete capstone projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics including Rust's signature ownership and borrowing system, Phase 2 covers control flow, functions, modules, and custom data structures, Phase 3 covers intermediate and advanced core topics like traits, generics, error handling, and crates, and Phase 4 covers master-level lifetimes, macros, concurrency, and production Rust applications.

By Week 16, you will have built three complete, portfolio-ready applications and be confident building production-grade Rust programs across systems, web, and concurrent domains.

Level: Beginner → Master  |  Duration: 16 weeks  |  Stack: Rust (stable), Cargo, Tokio  |  Prerequisites: Basic programming logic

Phase 1 — Absolute Basics

Weeks 1–4: Fundamentals, data types, ownership, and collections intro.

Week 1 — Setting Up & First Steps

Toolchain setup, project structure, and basic syntax.

Core Focus

Setting up the Rust toolchain, understanding project structure with Cargo, and writing your first Rust programs.

Topics & Subtopics

  1. Installing Rust & Cargo — Rust is installed via rustup, which also bundles Cargo, Rust's build tool and package manager. Cargo handles compiling, dependency management, and project scaffolding from day one — there is no separate setup required.
  2. Variables & Mutability — Variables in Rust are immutable by default, meaning their value cannot change after assignment. Adding the mut keyword explicitly allows a variable to be reassigned — this design pushes you toward safer code by default.
  3. Comments & println! Formatting — Comments use // for single lines or /* */ for blocks, and are ignored by the compiler. The println! macro supports placeholders ({}) for formatted, readable output.

Why This Week Matters: Cargo is one of the best build tools in any language — it handles compilation, dependency resolution, testing, and documentation from a single command. Understanding that variables are immutable by default from Week 1 introduces the Rust philosophy: make the safe choice the easy choice.

Code Example

// src/main.rs - the entry point of every Rust binary
fn main() {
    println!("Hello, Rust World!"); // macro that prints to console
}

fn main() {
    let name = "Alice";       // immutable by default
    let mut age = 25;         // mutable variable
    age += 1;
    println!("{} is {}", name, age);
}

fn main() {
    // Single-line comment
    /* Multi-line
       comment block */
    println!("{} + {} = {}", 2, 3, 2 + 3); // formatted output
}

Week 2 — Data Types & the Type System

Rust's strict, static type system and scalar types.

Core Focus

Understanding Rust's strict, static type system and working with all scalar types.

Topics & Subtopics

  1. Integers & Floats — Rust offers explicit-width integer types like i32 and u64, plus floating-point types f32 and f64. This precision avoids ambiguity and helps the compiler catch overflow and type errors at compile time rather than at runtime.
  2. Booleans & Characters — The bool type holds only true or false, used extensively in conditions and pattern matching. The char type represents a single Unicode scalar value — always exactly 4 bytes in memory, unlike C's single-byte char.
  3. Type Inference & Annotations — Rust's compiler can infer types automatically in most cases, reducing boilerplate while staying fully statically typed. Explicit type annotations are required when the compiler cannot determine the type unambiguously from context.

Why This Week Matters: Rust's integer overflow behavior — panic in debug mode, wrapping in release — is fundamentally different from C's silent undefined behavior. Understanding the explicit integer width types (i32, u64, etc.) and when to use each is the foundation for writing correct, portable numeric code in Rust.

Code Example

fn main() {
    let whole: i32 = 42;
    let decimal: f64 = 3.14;
    println!("{}", whole as f64 + decimal); // explicit cast with 'as'
}

fn main() {
    let is_active: bool = true;
    let grade: char = 'A';
    println!("Active: {}, Grade: {}", is_active, grade);
}

fn main() {
    let inferred = 10;            // inferred as i32
    let explicit: u8 = 255;       // explicitly typed
    println!("{} {}", inferred, explicit);
}

Week 3 — Operators & Strings

Expressions, operators, and Rust's two string types.

Core Focus

Working with operators and understanding Rust's two string types — one of the most important distinctions in the language.

Topics & Subtopics

  1. Arithmetic & Comparison Operators — Arithmetic operators (+, -, *, /, %) perform calculations, while comparison operators (==, <, >=) evaluate relationships and return booleans. Rust enforces matching types on both sides of these operations — no silent coercion.
  2. String vs &str — String is a growable, heap-allocated, owned text type, while &str is an immutable borrowed string slice. Understanding this distinction is essential to grasping Rust's ownership model and is referenced in virtually every Rust function signature.
  3. String Concatenation & Formatting — Strings can be joined using the + operator, push_str, or the format! macro for cleaner composition. format! builds a new String without printing — useful for constructing reusable text values.

Why This Week Matters: The String vs &str distinction is the single most confusing topic for Rust beginners — and for good reason. It is the first concrete manifestation of the ownership system: one is owned data on the heap, the other is a borrowed view into existing data. Getting comfortable with this distinction here makes everything in Phase 3 dramatically clearer.

Code Example

fn main() {
    let (a, b) = (17, 5);
    println!("{} {} {}", a / b, a % b, a > b); // 3 2 true
}

fn main() {
    let owned: String = String::from("Hello");
    let borrowed: &str = &owned;  // borrowing a slice of owned
    println!("{} / {}", owned, borrowed);
}

fn main() {
    let first = String::from("Rust");
    let combined = format!("{} is fun!", first); // builds a String
    println!("{}", combined);
}

Week 4 — Ownership & Collections Intro

Rust's signature ownership system and basic collections.

Core Focus

Mastering Rust's signature ownership system and working with the primary dynamic collection type — Vec<T>.

Topics & Subtopics

  1. Ownership & Move Semantics — Every value in Rust has a single owner, and when that owner goes out of scope, the value is dropped automatically — no garbage collector needed. Assigning a non-Copy value to another variable "moves" ownership, invalidating the original binding.
  2. Borrowing & References (&) — Borrowing lets you access a value without taking ownership, using & for immutable references and &mut for mutable ones. The borrow checker enforces that you cannot have a mutable and immutable reference to the same data at the same time.
  3. Vectors (Vec<T>) — A Vec<T> is a growable, heap-allocated list of elements of the same type — Rust's primary dynamic array structure. Vectors support indexing, iteration, and methods like push and pop.

Why This Week Matters: Ownership is the feature that makes Rust unique among all mainstream languages — it provides memory safety without a garbage collector and data-race freedom without a runtime. Every concept in the next 12 weeks — traits, generics, lifetimes, concurrency — builds directly on ownership. There is no shortcut through this week.

Code Example

fn main() {
    let s1 = String::from("hello");
    let s2 = s1;               // s1 is MOVED into s2
    println!("{}", s2);        // s1 can no longer be used here
}

fn print_length(s: &String) {  // borrows — does not own
    println!("Length: {}", s.len());
}
fn main() {
    let s = String::from("Rustacean");
    print_length(&s);          // pass a reference
    println!("{}", s);         // s is still valid here
}

fn main() {
    let mut numbers: Vec<i32> = vec![1, 2, 3];
    numbers.push(4);
    println!("{:?}", numbers); // [1, 2, 3, 4]
}

Phase 2 — Control Flow & Logic

Weeks 5–8: Functions, modules, and custom data structures.

Week 5 — Conditional Logic

Decision-making constructs in Rust.

Core Focus

Making decisions in Rust using if expressions, exhaustive match, and the concise if let shortcut.

Topics & Subtopics

  1. if / else if / else — Conditional expressions in Rust evaluate boolean conditions to choose which code block executes and do not require parentheses around the condition. Since if is an expression, it can directly return a value — enabling clean conditional assignments.
  2. match Expressions — match compares a value against a series of patterns and executes the matching branch, similar to a powerful switch. It is exhaustive — the compiler forces you to handle every possible case, eliminating an entire class of logic bugs.
  3. if let & Pattern Matching Shortcuts — if let provides a concise way to match a single pattern, commonly used with Option or Result types. It avoids the verbosity of a full match when only one case actually matters.

Why This Week Matters: Rust's match is exhaustive — the compiler will refuse to compile your program if you forget to handle a case. This single property eliminates the most common class of logic bugs: forgetting an edge case. Combined with Option<T> in Week 8, exhaustive match is the replacement for null pointer bugs in Rust.

Code Example

fn main() {
    let score = 75;
    let grade = if score >= 90 { "A" }
                else if score >= 70 { "B" }
                else { "C" };
    println!("Grade: {}", grade);
}

fn main() {
    let day = 3;
    match day {
        1 => println!("Monday"),
        2..=5 => println!("Midweek"),  // inclusive range pattern
        _ => println!("Weekend"),       // catch-all — required
    }
}

fn main() {
    let value: Option<i32> = Some(7);
    if let Some(n) = value {
        println!("Got value: {}", n);   // unwraps safely
    }
}

Week 6 — Loops & Iteration

Repetition constructs and iterator basics.

Core Focus

Repeating actions using Rust's three loop types and composing lazy transformations with iterators.

Topics & Subtopics

  1. loop, while, and for — loop repeats indefinitely until explicitly broken, while repeats while a condition holds, and for iterates over a range or collection. Rust's for loop is the idiomatic choice for iterating — avoiding manual index management and bounds errors.
  2. Breaking with Values (loop as Expression) — Unlike most languages, Rust's loop can return a value via break value;, making it usable as an expression. This is useful for retry logic where you need to return the eventual successful result from the loop itself.
  3. Iterators & iter() — Iterators provide a lazy, composable way to process sequences of values using methods like map and filter. Calling .iter() on a collection produces an iterator without consuming the original collection.

Why This Week Matters: Rust's iterator combinators — map, filter, fold, collect — are zero-cost abstractions. They compile to the same machine code as hand-written loops but express intent far more clearly. Idiomatic Rust code uses iterator chains constantly — getting comfortable with them now pays dividends every remaining week.

Code Example

fn main() {
    for i in 1..=5 {        // inclusive range
        println!("Count: {}", i);
    }
}

fn main() {
    let mut counter = 0;
    let result = loop {
        counter += 1;
        if counter == 5 { break counter * 2; } // loop returns 10
    };
    println!("{}", result);
}

fn main() {
    let nums = vec![1, 2, 3, 4];
    let doubled: Vec<i32> = nums.iter().map(|x| x * 2).collect();
    println!("{:?}", doubled); // [2, 4, 6, 8]
}

Week 7 — Functions & Modules

Code organization through functions and modules.

Core Focus

Organizing code into reusable functions, anonymous closures, and structured modules with controlled visibility.

Topics & Subtopics

  1. Function Definitions & Return Types — Functions are declared with fn, and the return type is specified after ->. The final expression in a function body is implicitly returned without needing the return keyword — a common source of confusion for newcomers.
  2. Closures — Closures are anonymous functions that can capture variables from their enclosing scope, similar to lambdas. They are commonly passed as arguments to iterator methods like map and filter, and their types are inferred by the compiler.
  3. Modules (mod) & Visibility — The mod keyword groups related code into namespaces, helping organize larger codebases. The pub keyword controls visibility, exposing items to code outside the module — everything is private by default.

Why This Week Matters: Rust's implicit return (the last expression without a semicolon) is a common trip-up point — adding a semicolon to the last line changes its type from the return value to a unit () and causes a compile error. Understanding this distinction makes reading and writing Rust functions natural rather than confusing.

Code Example

fn add(a: i32, b: i32) -> i32 {
    a + b  // implicit return — NO semicolon
}
fn main() {
    println!("{}", add(3, 4));
}

fn main() {
    let factor = 3;
    let multiply = |x: i32| x * factor; // closure captures 'factor'
    println!("{}", multiply(5));          // 15
}

mod math_utils {
    pub fn square(x: i32) -> i32 { x * x } // pub = visible outside
}
fn main() {
    println!("{}", math_utils::square(6));
}

Week 8 — Structs, Enums & Pattern Matching

Custom data structures and exhaustive pattern handling.

Core Focus

Modeling domain data with structs and enums, and handling all possible cases exhaustively with pattern matching.

Topics & Subtopics

  1. Structs — A struct groups related named fields into a single custom type, similar to a class without inherited behavior. Methods can be attached to structs using impl blocks — keeping data and behavior together in an organized way.
  2. Enums & Option<T> — Enums define a type as one of several distinct variants, optionally carrying data per variant. Option<T> is a built-in enum (Some(T) or None) that eliminates null-pointer bugs entirely from the language.
  3. Destructuring & Match Guards — Destructuring extracts fields from structs or enums directly into variables, making code more readable and concise. Match guards add extra if conditions onto match arms for finer-grained control over which arm executes.

Why This Week Matters: Option<T> is Rust's answer to null — and it is one of the greatest improvements to software reliability in modern language design. You cannot accidentally dereference None without explicitly handling it. Combined with exhaustive match, this makes "null pointer exception" a compile-time error rather than a production crash.

Code Example

struct Point { x: f64, y: f64 }
impl Point {
    fn distance_from_origin(&self) -> f64 {
        (self.x.powi(2) + self.y.powi(2)).sqrt()
    }
}

fn find_even(nums: &[i32]) -> Option<i32> {
    for &n in nums {
        if n % 2 == 0 { return Some(n); } // wrap result
    }
    None  // no even number found
}

fn main() {
    let pair = (5, -5);
    match pair {
        (x, y) if x + y == 0 => println!("Opposites!"), // match guard
        _ => println!("Not opposites"),
    }
}

Phase 3 — Intermediate & Advanced Core

Weeks 9–12: Traits, generics, error handling, and crates.

Week 9 — Structs & Traits I

Behavior sharing through traits — Rust's interfaces.

Core Focus

Sharing behavior across unrelated types using traits — Rust's powerful and flexible alternative to inheritance.

Topics & Subtopics

  1. Defining & Implementing Traits — A trait defines a set of method signatures that types can implement, similar to interfaces in other languages. This allows different types to share common behavior while remaining structurally distinct — no class hierarchy required.
  2. Default Trait Methods — Traits can provide default method implementations that types automatically inherit unless explicitly overridden. This reduces duplication when most implementers share the same base behavior while allowing customization where needed.
  3. Trait Objects (dyn Trait) — Trait objects allow you to store different types implementing the same trait behind a single pointer type like Box<dyn Trait>. This enables runtime polymorphism when concrete types aren't known until execution.

Why This Week Matters: Traits are the backbone of Rust's entire standard library — Iterator, Display, From, Into, Clone, Send, Sync are all traits. Understanding how to define, implement, and use traits with both static dispatch (generics) and dynamic dispatch (dyn Trait) is essential for writing idiomatic, reusable Rust code.

Code Example

trait Speak {
    fn speak(&self) -> String; // method signature only — must implement
}
struct Dog;
impl Speak for Dog {
    fn speak(&self) -> String { String::from("Woof!") }
}

trait Greet {
    fn greet(&self) -> String { String::from("Hello!") } // default impl
}
struct Robot;
impl Greet for Robot {} // uses the default — no override needed

trait Shape { fn area(&self) -> f64; }
fn print_area(shape: &dyn Shape) { // accepts ANY Shape implementer
    println!("Area: {}", shape.area());
}

Week 10 — Generics & Advanced Traits

Writing flexible, reusable, type-safe code.

Core Focus

Writing flexible, reusable, type-safe abstractions using generics, trait bounds, and operator overloading.

Topics & Subtopics

  1. Generic Functions & Structs — Generics allow functions and structs to operate over multiple types using a placeholder like <T>, while the compiler still enforces full type safety at compile time. This avoids duplicating logic for each concrete type — Rust's generics are monomorphized, so there is zero runtime cost.
  2. Trait Bounds — Trait bounds restrict generic types to those implementing specific traits, ensuring required operations (like comparison or formatting) are available inside the generic body. Written using <T: TraitName> or the cleaner where clause syntax.
  3. Operator Overloading — Rust allows custom types to support operators like + by implementing traits such as std::ops::Add. This lets domain types like vectors or money amounts behave naturally with familiar operator syntax.

Why This Week Matters: Rust's generics are zero-cost — the compiler monomorphizes them into concrete type-specific code at compile time, with no runtime overhead. This is the key difference from Java's type-erased generics and Python's duck typing. Writing generic code in Rust gives you both flexibility and maximum performance simultaneously.

Code Example

struct Wrapper<T> { value: T }  // generic struct

fn largest<T: PartialOrd>(a: T, b: T) -> T {
    if a > b { a } else { b }
}

fn print_all<T: std::fmt::Debug>(items: &[T]) {
    for item in items { println!("{:?}", item); }
}

use std::ops::Add;
#[derive(Debug, Clone, Copy)]
struct Point { x: i32, y: i32 }
impl Add for Point {
    type Output = Point;
    fn add(self, other: Point) -> Point {
        Point { x: self.x + other.x, y: self.y + other.y }
    }
}

Week 11 — Error Handling & File I/O

Robust failure handling without exceptions.

Core Focus

Handling errors explicitly and robustly using Result<T, E>, custom error types, and the filesystem.

Topics & Subtopics

  1. Result<T, E> & the ? Operator — Result represents either success (Ok(T)) or failure (Err(E)), forcing explicit handling of fallible operations. The ? operator propagates errors upward automatically, reducing boilerplate while keeping error paths explicit.
  2. Custom Error Types — Defining custom error enums lets you represent domain-specific failure modes clearly, often implementing the std::error::Error trait. This improves error messages and allows pattern matching on specific failure causes.
  3. File Reading & Writing (std::fs) — The std::fs module provides functions like read_to_string and write for interacting with the filesystem. Combined with Result, file operations force you to explicitly handle missing files or permission errors.

Why This Week Matters: Rust has no exceptions — all errors are values. The ? operator is the elegance that makes this approach practical: it propagates errors with no boilerplate while keeping the success path clean and readable. This approach means that in Rust, you literally cannot silently ignore an error — the compiler won't let you.

Code Example

fn divide(a: f64, b: f64) -> Result<f64, String> {
    if b == 0.0 { return Err(String::from("Division by zero")); }
    Ok(a / b)
}

#[derive(Debug)]
enum AppError { InsufficientFunds }
fn withdraw(balance: f64, amount: f64) -> Result<f64, AppError> {
    if amount > balance { return Err(AppError::InsufficientFunds); }
    Ok(balance - amount)
}

use std::fs;
fn main() -> std::io::Result<()> {
    fs::write("notes.txt", "Hello, file!")?; // ? propagates errors
    let contents = fs::read_to_string("notes.txt")?;
    println!("{}", contents);
    Ok(())
}

Week 12 — Crates, Modules & Smart Pointers

Dependency management and advanced memory tools.

Core Focus

Leveraging the crates.io ecosystem, organizing multi-file projects, and using smart pointers for scenarios basic ownership cannot handle.

Topics & Subtopics

  1. Cargo & External Crates — Crates are Rust's term for packages, declared as dependencies inside Cargo.toml and downloaded from crates.io. Running cargo build automatically resolves, downloads, and compiles all dependencies — the smoothest dependency management in any systems language.
  2. Organizing Multi-File Projects — Larger projects split code across multiple files using mod declarations that map to filenames, keeping each module's logic isolated. The use keyword brings items from other modules into scope concisely.
  3. Smart Pointers (Box, Rc, RefCell) — Box<T> allocates a value on the heap with single ownership, Rc<T> enables multiple shared owners via reference counting, and RefCell<T> allows mutable borrows checked at runtime. These tools handle scenarios the basic ownership model cannot express directly.

Why This Week Matters: The crates.io ecosystem is one of Rust's greatest strengths — serde, tokio, reqwest, axum are all external crates that transform what Rust can do. Smart pointers like Rc<T> and RefCell<T> handle real-world data structures like graph nodes and trees that the simple ownership model cannot express.

Code Example

// Cargo.toml:
// [dependencies]
// serde = { version = "1.0", features = ["derive"] }

// Multi-file: src/utils.rs
pub fn greet() -> String { String::from("Hi from utils!") }

// src/main.rs
// mod utils;
// fn main() { println!("{}", utils::greet()); }

use std::rc::Rc;
fn main() {
    let shared = Rc::new(String::from("shared data"));
    let clone_a = Rc::clone(&shared);  // increments reference count
    println!("{} (count: {})", clone_a, Rc::strong_count(&shared));
}

Phase 4 — Master Level Concepts

Week 13 — Lifetimes & Advanced Generics

Memory safety guarantees beyond basic ownership.

Core Focus

Using lifetime annotations to guarantee reference validity and writing complex generic functions with where clauses.

Topics & Subtopics

  1. Lifetime Annotations ('a) — Lifetimes describe how long references remain valid, allowing the compiler to prevent dangling references at compile time. Annotations like 'a are needed when a function returns a reference derived from its inputs — telling the compiler which input the output borrows from.
  2. Lifetimes in Structs — Structs holding references must declare lifetime parameters to ensure the struct cannot outlive the data it borrows. This guarantees compile-time safety for any struct that stores a reference alongside other fields.
  3. Advanced Trait Bounds (where clauses) — where clauses provide a cleaner syntax for specifying complex trait bounds on generic functions, especially with multiple constraints. They improve readability compared to cramming all bounds into angle brackets on one line.

Why This Week Matters: Lifetimes are the feature that allows Rust to guarantee memory safety without a garbage collector. They feel intimidating at first but represent a profound insight: the compiler can verify at compile time that no reference outlives the data it points to. Once you understand this, the concept of a dangling pointer becomes a compile error, not a runtime crash.

Code Example

fn longest<'a>(x: &'a str, y: &'a str) -> &'a str {
    if x.len() > y.len() { x } else { y } // lifetime tied to inputs
}

struct Excerpt<'a> { part: &'a str } // struct borrowing a string slice
fn main() {
    let novel = String::from("Call me Ishmael.");
    let excerpt = Excerpt { part: &novel[0..8] };
    println!("{}", excerpt.part);
}

fn summarize<T>(items: &[T]) -> usize
where
    T: std::fmt::Debug,  // where clause — cleaner for multiple bounds
{
    items.len()
}

Week 14 — Macros & Iterators Deep Dive

Metaprogramming and lazy, composable sequences.

Core Focus

Writing code-generating macros and building custom iterators that integrate seamlessly with Rust's iterator ecosystem.

Topics & Subtopics

  1. Declarative Macros (macro_rules!) — Declarative macros let you write code that generates other code at compile time, reducing repetition for patterns the type system cannot express. They are invoked with a !, like println! or custom macros you define yourself.
  2. Derive Macros (#[derive(...)]) — Derive macros automatically implement common traits like Debug, Clone, or PartialEq for structs and enums with a single annotation. This eliminates boilerplate while keeping generated implementations consistent and correct.
  3. Custom Iterators (impl Iterator) — Implementing the Iterator trait for your own type lets it work seamlessly with for loops and all iterator methods like map and collect. Only the next() method needs to be defined — all other methods come from the trait's default implementations.

Why This Week Matters: Custom iterators are Rust's answer to generators in other languages. Implementing Iterator on your own type gives it access to the entire iterator adaptor ecosystem — map, filter, take, zip, chain — for free. This is zero-cost lazy evaluation that rivals Python's generators in expressiveness while matching C in performance.

Code Example

macro_rules! square {
    ($x:expr) => { $x * $x }; // pattern-based code generation
}
fn main() { println!("{}", square!(5)); } // 25

#[derive(Debug, Clone, PartialEq)] // auto-implements three traits
struct User { name: String }
fn main() { println!("{:?}", User { name: "Sam".into() }); }

struct Counter { count: u32 }
impl Iterator for Counter {
    type Item = u32;
    fn next(&mut self) -> Option<u32> {
        self.count += 1;
        if self.count <= 5 { Some(self.count) } else { None }
    }
}

Week 15 — Concurrency: Threads, Channels & Async

Safe parallelism and asynchronous programming.

Core Focus

Achieving safe parallelism with threads and channels, and writing non-blocking async code with Tokio.

Topics & Subtopics

  1. Threads (std::thread) — Rust's standard library allows spawning OS threads with thread::spawn, and the ownership system statically prevents common data races at compile time — without a mutex. .join() waits for a spawned thread to finish before continuing.
  2. Channels (mpsc) for Message Passing — Channels allow safe communication between threads by sending owned data instead of sharing mutable memory. The mpsc module provides multi-producer, single-consumer channels — the Rust idiomatic alternative to shared mutable state.
  3. Async/Await with Tokio — async/await lets you write non-blocking code that reads sequentially, ideal for I/O-bound tasks like network requests. The tokio crate provides the runtime needed to actually execute async functions concurrently.

Why This Week Matters: Rust's ownership system makes it the only language where data races are a compile error — not a runtime bug. The compiler statically verifies that data sent between threads is Send and that shared data is Sync. Combined with Tokio's async runtime, Rust can handle hundreds of thousands of concurrent connections with safety guarantees no other language provides.

Code Example

use std::thread;
fn main() {
    let handle = thread::spawn(|| println!("Hello from a thread!"));
    handle.join().unwrap(); // wait for thread to finish
}

use std::sync::mpsc;
use std::thread;
fn main() {
    let (tx, rx) = mpsc::channel();
    thread::spawn(move || tx.send("data from thread").unwrap());
    println!("{}", rx.recv().unwrap()); // blocks until received
}

// Cargo.toml: tokio = { version = "1", features = ["full"] }
#[tokio::main]
async fn main() {
    let result = async { "data loaded" }.await;
    println!("{}", result);
}

How to Use This Curriculum

  • Always run cargo clippy on your code — Clippy catches non-idiomatic patterns and teaches you better Rust as you write it.
  • Push every weekly project to a public GitHub repository — your commit history shows consistent progress and your projects demonstrate real Rust competence.
  • When the borrow checker rejects your code, read the error message carefully — Rust's compiler errors are the best in any language and almost always explain exactly what went wrong and why.
  • After Week 16, explore async web frameworks like Axum or Actix, systems programming with unsafe Rust, or embedded development to apply these skills in real-world production domains.
