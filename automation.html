About This Roadmap

This roadmap takes a complete beginner to a confident, portfolio-ready Python Automation developer in 15 weeks of full-time study (~35–40 hours/week). Python is the dominant language for automation because it is readable, richly supported, cross-platform, and fast to prototype.

The path runs: Python Foundations → File & System Automation → Web Automation → Advanced Automation → Packaging, Deployment & Capstone.

By Week 15 you will have built approximately 13 mini-projects and one complete, production-ready automation suite — AutoPilot, a modular personal productivity automation system.

Week 1 — Python Basics: Variables, Control Flow & Functions

Write real Python programs using variables, data types, conditions, loops, and functions — the building blocks of every automation script.

Topics & Subtopics

  1. Python environment — running Python: interactive REPL, .py scripts, Jupyter Notebook; print(), input(), comments.
  2. Variables and data types — int, float, str, bool; type conversion; type() function.
  3. Strings (critical for automation) — .upper(), .lower(), .strip(), .replace(), .split(), .join(); f-strings; slicing; in operator.
  4. Collections — lists (.append(), .remove(), .sort()), dictionaries (.get(), .keys(), .items()), tuples, sets.
  5. Control flow — if/elif/else; for loop; while loop; break, continue, pass; list comprehensions.
  6. Functions — def, parameters, return values, default parameters, *args, **kwargs, scope, lambda functions.
  7. Useful built-in functions — len(), range(), enumerate(), zip(), sorted(), map(), filter().

Why This Week Matters: Every Python automation script uses variables to store data, loops to process lists of files or records, conditionals to handle different cases, and functions to structure reusable logic. This week is the foundation — every subsequent week builds directly on these concepts.

Practice Tasks

  • Write a function that takes a price as a string ("$19.99") and returns it as a float (19.99).
  • Write a script that accepts a sentence as input and outputs word count, character count, and whether a specific word is present.

🛠 Mini Project

A Text Report Generator — a Python script that accepts a block of text as input (simulating a log file or data extract) and outputs: total line count, word count, count of lines containing "ERROR", a list of all unique words found, and the 5 most common words — all using basic Python with no external libraries — pushed to GitHub.

Common Mistakes

  • Using = for comparison instead of ==.
  • Confusing .append(item) (adds one item) with .extend(list) (adds all items from another list).
  • Indentation errors — Python uses indentation for code blocks, unlike most other languages.
  • Modifying a list while iterating over it — causes skipped items.

How to Avoid Them

  • Read comparison expressions aloud: "if x equals-equals 5" — the double equals means compare.
  • Use append() for a single item; extend() or + to merge two lists.
  • Configure VS Code to show whitespace characters — indentation problems become visible.
  • Never modify a list in a for loop — iterate over a copy: for item in list.copy().

Week 2 — Python Intermediate: OOP, Modules & Error Handling

Write structured, maintainable Python automation code using classes, modules, error handling, and the standard library.

Topics & Subtopics

  1. Object-Oriented Programming (OOP) — classes and objects, __init__(), self, instance variables, methods, inheritance, __str__().
  2. Modules and imports — import module vs from module import function; creating your own modules; if __name__ == "__main__":.
  3. Standard library overview — os, sys, pathlib, datetime, time, math, random, json, csv, re.
  4. Error handling — try/except/else/finally; catching specific exceptions; raise; custom exception classes.
  5. datetime module — datetime.now(), timedelta, strftime()/strptime(), timestamped filenames.
  6. json module — json.loads(), json.dumps(), reading/writing JSON files.
  7. logging module — logging.basicConfig(), logging.info()/.warning()/.error(); why logging beats print().

Why This Week Matters: Real automation projects have multiple files, handle errors gracefully (network timeouts, missing files, malformed data), and run unattended. Without OOP you can't structure complex automation. Without error handling, any unexpected input crashes the script. Without logging, debugging a script that ran at 2 AM is impossible.

Practice Tasks

  • Write a function that reads a JSON config file and returns default values if the file is missing or malformed.
  • Write a script that deliberately triggers 5 different errors and catches each with a helpful message.

🛠 Mini Project

A Task Logger — a Python script using OOP: a Task base class with name, status, created_at; subclasses FileTask and WebTask; a TaskManager class with add_task(), complete_task(), export_to_json() methods; full error handling; logging to both console and a tasks.log file; timestamped JSON export — pushed to GitHub.

Common Mistakes

  • Forgetting self as the first parameter in every class method.
  • Catching bare except: without specifying an exception type — silently swallows unexpected errors.
  • Not using finally when opening files — leaves files open if an exception occurs.
  • Using print() for everything in scripts meant to run unattended.

How to Avoid Them

  • Rule: every method definition starts with def method_name(self, ...).
  • Always catch specific exceptions: except FileNotFoundError, except ValueError. Use except Exception as e as a catch-all at the outermost level only.
  • Use context managers (with open(...) as f:) — Python closes the file automatically even on exception.
  • Use logging from Week 2 onward for any script that might run unattended.

Week 3 — Regular Expressions, pathlib & the Standard Library

Master regular expressions for text extraction and transformation, and become fluent with pathlib and key standard library modules used in every automation project.

Topics & Subtopics

  1. Regular Expressions (re module) — basic patterns (., *, +, ?, ^, $), character classes (\d, \w, \s), groups; re.search(), re.findall(), re.sub(), re.compile(); practical patterns: email, phone, URL, dates.
  2. pathlib module — Path object; path operations with / operator; path.exists(), path.is_file(); path.stem, path.suffix, path.parent; path.glob(), path.mkdir(), path.rename(), path.read_text().
  3. os and shutil modules — os.listdir(), os.walk(), os.environ, os.system(); shutil.copy(), shutil.move(), shutil.rmtree().
  4. csv module — csv.reader(), csv.DictReader(), csv.writer(), csv.DictWriter(); encoding issues.
  5. sys module — sys.argv, sys.exit(), sys.path.
  6. subprocess module — subprocess.run(), capturing output, running OS commands.
  7. argparse module — building CLI interfaces; ArgumentParser, add_argument(), parse_args().

Why This Week Matters: Regex is the Swiss Army knife of text extraction — pulling emails from a log file, finding all prices on a page, validating file names. pathlib is how every modern Python automation script handles files. argparse turns your scripts into professional tools usable by anyone from the command line. These three topics appear in virtually every automation script after this week.

Daily Study Plan

Day Focus
Day 1 Regex basics — re.search, re.findall, basic patterns; match emails and phone numbers in sample text.
Day 2 Regex groups + re.sub + re.compile; extract dates and URLs from a log file.
Day 3 pathlib — Path creation, navigation, .glob(), file properties; build a directory scanner.
Day 4 pathlib file operations (read/write/rename/delete) + shutil copy/move; build a file sorter.
Day 5 csv module — read + write; process a sample CSV (filter rows, add a column, save result).
Day 6 subprocess.run + os.environ + sys.argv; run a shell command and capture output.
Day 7 argparse — add CLI flags to 3 previous scripts + mini project + GitHub commit.

Practice Tasks

  • Write a regex pattern that extracts all email addresses from a multi-line text string.
  • Write a script that walks a directory tree and reports all files larger than 1MB.
  • Write a script with argparse that accepts --input, --output, and --format flags.

🛠 Mini Project

A File Inventory Reporter — a CLI tool (argparse) that accepts a directory path, recursively scans with pathlib.glob() and os.walk(), groups files by extension, lists files above a configurable size threshold, extracts dates from filenames using regex, and writes a CSV report with columns: filename, extension, size_kb, date_extracted, full_path — pushed to GitHub with a README showing usage examples.

Common Mistakes

  • Forgetting r"" raw strings for regex patterns — backslashes are misinterpreted.
  • Using os.path.join() instead of pathlib — older style, less readable.
  • Using csv.reader and accessing columns by number (fragile) instead of DictReader with column names (robust).
  • Not handling CSV encoding — UnicodeDecodeError on Excel-generated files.

How to Avoid Them

  • Always write regex patterns as raw strings: r"\d{4}-\d{2}-\d{2}".
  • Default to pathlib.Path for all path operations from Week 3 onward.
  • Always use csv.DictReader when column names are available — row['Name'] beats row[0].
  • For CSV files from Excel: open(file, encoding='utf-8-sig') — handles the BOM character Excel adds.

Common Mistakes

  • Not using context managers (with open(...) as f) — leaves files open on error.

  • Before moving a file, check: if dest_path.exists(): handle_duplicate().

Week 6 — System & Process Automation: Scheduling, Monitoring & OS Interaction

Automate system-level tasks — running commands, monitoring system health, scheduling scripts, managing environment variables, and sending desktop notifications.

🛠 Mini Project

A System Automation Dashboard — runs continuously; monitors CPU, RAM, and disk usage every 30 seconds; logs to CSV; sends desktop notifications for threshold breaches; watches a "drop folder" with watchdog and runs the Week 4 file organizer on any new file; runs a scheduled daily system report at a set time — pushed to GitHub with a .env.example and README.

Common Mistakes

  • Running schedule without an infinite loop — the script exits before any scheduled jobs run.
  • Hardcoding absolute file paths in scheduled scripts — breaks if the working directory changes.
  • Not adding time.sleep(1) inside the scheduler loop — causes 100% CPU usage.
  • Running watchdog and schedule in the same thread — one blocks the other.

How to Avoid Them

  • schedule always needs: while True: schedule.run_pending(); time.sleep(1).
  • Use Path(__file__).parent as the base path — always relative to the script's location.
  • For combining watchdog + schedule: run each in its own thread using threading.Thread.

Week 7 — HTTP Requests & Web Scraping with BeautifulSoup

Fetch data from the web programmatically using the requests library and extract specific data from HTML pages using BeautifulSoup.

Topics & Subtopics

  1. HTTP and the web — request → server → HTML response; HTTP methods (GET, POST); status codes (200, 403, 404, 429, 500); request/response headers.
  2. requests library — requests.get(url); response.status_code, response.text, response.json(); params, headers, timeouts; requests.Session(); response.raise_for_status().
  3. HTML basics for scraping — tags, attributes (id, class, href); using browser DevTools (F12) to find elements.
  4. BeautifulSoup4 (bs4) — soup.find(), soup.find_all(); CSS selectors (soup.select(), soup.select_one()); .text, .get_text(strip=True), tag['href'].
  5. Scraping best practices and ethics — checking robots.txt; rate limiting with time.sleep(); User-Agent headers; static vs dynamic pages.
  6. Common scraping challenges — pagination, handling None from .find(), saving to CSV and JSON.

Why This Week Matters: Web scraping is one of the most practically valuable automation skills — price monitoring, research data collection, job listing aggregation, sports statistics, and real estate data are all commonly scraped. requests + BeautifulSoup is the standard Python stack for static page scraping and is sufficient for the majority of public data sources.

Important Note on Web Scraping Ethics

  • Always check https://targetsite.com/robots.txt before scraping any site.
  • Scraping legality varies by jurisdiction and website terms of service — always check a site's ToS before scraping.
  • This roadmap uses books.toscrape.com and quotes.toscrape.com — legal, purpose-built scraping practice sites.

Automate web browsers — logging into websites, filling forms, clicking buttons, and scraping JavaScript-rendered content.
