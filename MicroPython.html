MicroPython: Beginner to Master

Complete 12-Week Curriculum — From Firmware Flashing to Concurrent IoT Dashboards & MQTT Systems

About This Curriculum

This is a 12-week, project-driven MicroPython curriculum designed to take a complete beginner to a professional-level embedded IoT developer. The roadmap moves from firmware flashing, the REPL, primitive types, and GPIO basics through control flow, functions, PWM, OOP, and I2C/SPI sensor communication, into inheritance, error handling, file storage, Wi-Fi, and MQTT networking, all the way to async concurrency with uasyncio, generators, decorators, and three complete capstone IoT projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics and hardware fundamentals, Phase 2 covers control flow, functions, and sensor protocols, Phase 3 covers intermediate and advanced core topics including OOP and networking, and Phase 4 covers master-level concurrency, functional patterns, and full IoT system projects.

By Week 12, you will have built three complete, hardware-running IoT applications and be confident designing production-grade MicroPython embedded systems from scratch.

Level: Beginner → Master  |  Duration: 12 weeks  |  Hardware: ESP32 / Raspberry Pi Pico  |  Prerequisites: Basic Python syntax (no embedded experience needed)

Phase 1 — Absolute Basics

Weeks 1–3: Firmware setup, syntax, primitive types, GPIO basics, and collections.

Week 1 — MicroPython Environment, Syntax & Primitive Types

Set up your board and toolchain, and learn MicroPython's basic syntax and core data types.

Core Focus

Setting up your board and toolchain, and learning MicroPython's basic syntax and core data types.

Topics & Subtopics

  1. Flashing Firmware & the REPL — MicroPython runs as firmware flashed onto a microcontroller (like an ESP32 or Raspberry Pi Pico), turning it into a tiny Python interpreter. The REPL (Read-Eval-Print Loop) lets you type Python commands directly into the board and see results instantly — your first live hardware interaction.
  2. Tools: Thonny / mpremote / ampy — Thonny is a beginner-friendly IDE with built-in MicroPython support for uploading and running scripts. mpremote and ampy are command-line tools for copying files to a board and interacting with its REPL from a terminal.
  3. Variables & Type Inference — MicroPython is dynamically typed, meaning a variable's type is determined automatically from the value assigned to it. Unlike full Python, MicroPython is optimized to use far less memory — a critical consideration on constrained hardware with only kilobytes of RAM.
  4. Primitive Types (int, float, str, bool) — MicroPython supports the same core primitive types as standard Python: whole numbers, decimals, text, and true/false values. These types behave almost identically to CPython, easing the transition for Python developers moving to embedded development.

Why This Week Matters: The firmware + REPL setup is your entire development environment for embedded work — if this is broken, nothing else works. Understanding that MicroPython is memory-constrained from Day 1 shapes every design decision you will make for the rest of the curriculum.

Code Example

# Basic variables and types on a microcontroller
name       = "MicroPython"   # string
pin_count  = 30              # integer
voltage    = 3.3             # float
is_powered = True            # boolean

print(name, pin_count, voltage, is_powered)
# MicroPython 30 3.3 True

# Type checking
print(type(voltage))         # <class 'float'>
print(type(is_powered))      # <class 'bool'>

Week 2 — Operators, Constants & GPIO Basics

Master Python operators and take your first steps controlling real hardware via GPIO.

Core Focus

Mastering Python operators and taking your first steps controlling real hardware via GPIO.

Topics & Subtopics

  1. Arithmetic & Comparison Operators — MicroPython supports standard operators (+, -, *, /, %, ==, <, >) identical to desktop Python. These are essential for calculations like sensor value scaling, threshold comparisons, and duty cycle computation.
  2. The machine Module & Pin Class — The machine module is MicroPython's gateway to hardware control, providing classes like Pin to configure GPIO pins as inputs or outputs. This is the foundation for everything from blinking LEDs to reading button states and controlling motors.
  3. Digital Output — Blinking an LED — Setting a Pin to OUT mode lets you drive it HIGH or LOW, directly controlling components like LEDs. Combined with time.sleep(), this produces the classic "blink" pattern used to test any new board setup.
  4. Constants & Naming Conventions — While MicroPython has no true const keyword for general code, uppercase naming convention signals an unchanging value, and the micropython.const() function optimizes integer constants for memory. This is especially important for fixed pin numbers used throughout a script.

Why This Week Matters: The blinking LED is the "Hello, World!" of embedded development — it proves your board, wiring, and code path all work together. The machine.Pin class is used in virtually every MicroPython project ever written, and understanding HIGH/LOW, input/output modes, and pull resistors is foundational to all hardware control.

Code Example

from machine import Pin
import time
from micropython import const

LED_PIN = const(2)       # Memory-optimized constant — GPIO 2

led = Pin(LED_PIN, Pin.OUT)   # Configure GPIO 2 as a digital output

for _ in range(5):
    led.value(1)          # Drive pin HIGH → LED on
    time.sleep(0.5)
    led.value(0)          # Drive pin LOW  → LED off
    time.sleep(0.5)

# Arithmetic for blink speed calculation
base_delay = 1.0
fast_delay = base_delay / 4   # 0.25 seconds — arithmetic operator in action
print(f"Fast delay: {fast_delay}s")

Week 3 — Collections: Lists, Tuples & Dictionaries

Learn Python's core data structures, essential for organizing sensor readings and device configuration data.

Core Focus

Learning Python's core data structures, essential for organizing sensor readings and configuration data on constrained hardware.

Topics & Subtopics

  1. Lists (Ordered, Mutable Collections) — A list stores an ordered sequence of values that can be changed, added to, or removed from after creation. Lists are commonly used to store sequences of sensor readings over time, buffering data before processing or transmitting it.
  2. Tuples (Ordered, Immutable Collections) — A tuple is like a list but cannot be modified after creation, making it useful for fixed groupings like RGB color values or pin configuration pairs. Their immutability also makes them slightly more memory-efficient on constrained devices.
  3. Dictionaries (Key-Value Pairs) — A dict stores data as key-value pairs, ideal for representing configuration settings or labeled sensor data. Dictionaries allow fast lookups by a meaningful name instead of a numeric index, making code far more readable.
  4. Reading Analog Input (ADC) — The machine.ADC class reads analog voltage levels from a pin (0–65535 range on most boards), commonly used for sensors like potentiometers, light-dependent resistors, or thermistors. Readings are typically stored in a list or dictionary for later processing.

Why This Week Matters: Real sensor applications always involve collections — you never take one reading, you take many and process them. Understanding when to use a list (ordered stream of readings) vs a dictionary (labeled configuration) vs a tuple (fixed hardware constants like pin pairs) is a fundamental embedded design skill.

Code Example

from machine import ADC, Pin

sensor   = ADC(Pin(34))              # Analog input on GPIO 34
readings = []                         # List: mutable, ordered readings buffer

# Tuple: immutable pin pair — won't change after definition
I2C_PINS = (21, 22)                   # (SDA, SCL)

# Take 5 readings and store in list
for _ in range(5):
    readings.append(sensor.read_u16())  # 0–65535 range

# Dictionary: labeled summary data
summary = {
    "sensor": "LDR",
    "unit":   "raw",
    "min":    min(readings),
    "max":    max(readings),
    "avg":    sum(readings) // len(readings)
}
print(summary)

Phase 2 — Control Flow & Logic

Weeks 4–6: Conditionals, loops, functions, PWM, classes, and I2C/SPI sensor protocols.

Week 4 — Control Flow: Conditionals & Loops

Direct your program's logic and create reactive hardware behavior using conditionals and the embedded main loop.

Core Focus

Directing your program's logic and creating reactive hardware behavior.

Topics & Subtopics

  1. if / elif / else Statements — Conditional statements let your program make decisions based on sensor readings or button states. MicroPython uses identical conditional syntax to standard Python, including indentation-based blocks — no braces, no special embedded syntax.
  2. for Loops & range() — A for loop iterates over a sequence or a range() of numbers, perfect for repeating an action a fixed number of times like blinking an LED N times or taking N sensor readings. This is one of the most common patterns in embedded scripts.
  3. while Loops for Continuous Operation — A while True: loop runs indefinitely — this is the standard pattern for a microcontroller's "main loop" that continuously checks sensors and responds to inputs. This mirrors how embedded devices operate: they run forever until powered off.
  4. Reading Digital Input — Buttons — Configuring a Pin as IN with a pull-up or pull-down resistor lets you detect button presses by reading HIGH/LOW states. Combining this with if statements creates simple, responsive interactive behavior.

Why This Week Matters: The while True: main loop is the heartbeat of every embedded application — it never stops. Understanding the difference between polling (checking repeatedly in a loop) and interrupts (covered in Week 10) is the foundation of all embedded responsiveness design.

Code Example

from machine import Pin
import time

button = Pin(0, Pin.IN, Pin.PULL_UP)   # Input with internal pull-up resistor
led    = Pin(2, Pin.OUT)

# Standard embedded main loop — runs forever
while True:
    if button.value() == 0:            # Pressed = LOW (pull-up inverts logic)
        led.value(1)
        print("Button pressed — LED on!")
    else:
        led.value(0)
    time.sleep(0.1)                    # Debounce delay

# for loop: blink N times (used outside main loop for startup sequence)
for i in range(5):
    led.value(1); time.sleep(0.2)
    led.value(0); time.sleep(0.2)

Week 5 — Functions, Modules & PWM

Organize code into reusable functions, split logic across files, and control component intensity with PWM.

Core Focus

Organizing code into reusable functions, splitting logic across module files, and controlling component intensity with PWM.

Topics & Subtopics

  1. Function Declaration & Parameters — Functions group reusable logic under a name, accepting parameters and optionally returning a value using def. Breaking hardware logic into functions like blink_led() or read_sensor() makes scripts far easier to read, test, and maintain.
  2. Default Arguments & Multiple Return Values — A function parameter can have a default value, letting you omit it for common cases. Python functions can also return multiple values at once using comma-separated returns (packed as a tuple), useful for returning both a sensor value and a status flag.
  3. Creating & Importing Custom Modules (.py files) — Code can be split across multiple .py files uploaded to the board, then imported with import filename. This separates reusable logic (like sensor drivers or LED utilities) from your main application script, keeping code organized as projects grow.
  4. PWM (Pulse Width Modulation) — machine.PWM rapidly switches a pin on and off to simulate variable voltage, used to dim LEDs or control motor and servo speed. The duty_u16() value (0–65535) controls the effective power level delivered to the component.

Why This Week Matters: PWM is used in virtually every non-trivial embedded project — LED dimming, servo control, motor speed, buzzer tones. Custom modules are how professional embedded projects stay maintainable: a sensors.py, a network_utils.py, and a clean main.py that imports them both.

Code Example

# led_utils.py — custom module uploaded to board
from machine import Pin, PWM
import time

def fade(pin_num, steps=50, delay=0.02):
    """Smoothly fades an LED from off to full brightness."""
    pwm = PWM(Pin(pin_num))
    pwm.freq(1000)
    for i in range(steps):
        pwm.duty_u16(int(65535 * i / steps))
        time.sleep(delay)
    return pwm                # Return PWM object for further control

def set_brightness(pin_num, level=32767):
    """Set LED brightness. level: 0 (off) to 65535 (full)."""
    pwm = PWM(Pin(pin_num))
    pwm.freq(1000)
    pwm.duty_u16(level)
    return pwm

# main.py — imports the module
import led_utils
led_utils.fade(5)             # Smooth fade using default steps
led_utils.set_brightness(5, level=50000)

Week 6 — Custom Data Structures & I2C/SPI Sensors

Model structured sensor data with classes and communicate with external sensors over I2C and SPI protocols.

Core Focus

Modeling structured sensor data and communicating with external sensors over standard hardware protocols.

Topics & Subtopics

  1. Classes & Objects (Intro to OOP) — A class bundles related data (attributes) and behavior (methods) into a single reusable blueprint, used to model real-world things like a "TemperatureSensor." Creating an instance of a class produces an object you can interact with through its methods.
  2. The __init__ Constructor — The __init__ method runs automatically when a new object is created, typically used to set up initial attribute values like pin numbers and protocol objects. This ensures every object starts in a valid, ready-to-use hardware state.
  3. I2C Communication — I2C is a two-wire communication protocol (SDA/SCL) allowing a microcontroller to talk to multiple sensors on the same bus using unique 7-bit device addresses. MicroPython's machine.I2C class handles all the low-level signaling, letting you focus on reading data.
  4. SPI Communication — SPI is a faster, four-wire protocol commonly used for displays and high-speed sensors, using separate MOSI/MISO data lines and a chip-select pin to address individual devices. MicroPython's machine.SPI class provides a simple interface for reading and writing SPI devices.

Why This Week Matters: I2C is the most common sensor protocol in the embedded world — the BMP280 pressure sensor, SSD1306 OLED display, MPU6050 accelerometer, and dozens more all use I2C. Wrapping sensor communication in a class (rather than raw I2C calls scattered throughout main.py) is the first step toward maintainable embedded software architecture.

Code Example

from machine import I2C, Pin, SPI

class TempSensor:
    """Wraps I2C communication with a temperature sensor."""
    def __init__(self, sda=21, scl=22, bus=0):
        self.i2c = I2C(bus, sda=Pin(sda), scl=Pin(scl), freq=400000)
        self.addr = self._find_device()

    def _find_device(self):
        devices = self.i2c.scan()
        if not devices:
            raise OSError("No I2C device found!")
        return devices[0]

    def read_raw(self):
        """Read 2 bytes from the sensor register."""
        return self.i2c.readfrom(self.addr, 2)

    def temperature(self):
        raw = self.read_raw()
        return (raw[0] << 8 | raw[1]) * 0.0625   # Example conversion

sensor = TempSensor(sda=21, scl=22)
print("Devices found:", sensor.i2c.scan())
print("Temperature:", sensor.temperature(), "°C")

Phase 3 — Intermediate & Advanced Core

Weeks 7–9: OOP inheritance, error handling, file storage, Wi-Fi networking, and MQTT.

Week 7 — Object-Oriented Programming & Inheritance

Build sophisticated, reusable hardware abstractions using full OOP principles including inheritance and polymorphism.

Core Focus

Building more sophisticated, reusable hardware abstractions using full OOP principles.

Topics & Subtopics

  1. Inheritance & Subclasses — Inheritance lets a new class reuse and extend the behavior of an existing class, avoiding duplicated code. For example, an RGBLed class could inherit from a base Led class and add color-mixing methods, reusing all the GPIO setup logic without rewriting it.
  2. Method Overriding — A subclass can redefine (override) a method from its parent class to provide specialized behavior while keeping the same method name. This allows different sensor types to share a common interface (read()) but return values in their own format.
  3. Encapsulation & Properties — Encapsulation hides an object's internal details behind methods, exposing only what is necessary through a clean interface. The @property decorator lets you access a method like a simple attribute, transparently adding validation or unit conversion logic.
  4. Polymorphism with Hardware Abstractions — Polymorphism allows different classes (e.g., different sensor types) to be used interchangeably as long as they share a common method name like read(). This makes it easy to write generic code that works with any compatible sensor in a list, regardless of the underlying hardware.

Why This Week Matters: Polymorphism is what makes embedded sensor code truly scalable — you can loop over a list of different sensor objects and call sensor.read() on all of them without knowing or caring what hardware each one represents. This is the architecture behind every professional sensor framework and driver library.

Code Example

from machine import Pin, ADC

class Sensor:
    """Base class — defines the shared interface for all sensors."""
    def __init__(self, name):
        self.name = name

    def read(self):
        raise NotImplementedError("Subclasses must implement read()")

    def __repr__(self):
        return f"{self.name}: {self.read()}"

class LightSensor(Sensor):
    def __init__(self, pin):
        super().__init__("LightSensor")
        self.adc = ADC(Pin(pin))

    def read(self):                        # Overrides base read()
        return self.adc.read_u16()

class DigitalSensor(Sensor):
    def __init__(self, pin):
        super().__init__("DigitalSensor")
        self.pin = Pin(pin, Pin.IN, Pin.PULL_UP)

    def read(self):                        # Different override
        return self.pin.value()

# Polymorphism: same call on different objects
sensors = [LightSensor(34), DigitalSensor(0)]
for s in sensors:
    print(s.name, "→", s.read())          # Works for both!

Week 8 — Error Handling & File Storage

Write robust embedded code that handles hardware failures gracefully and persists data to flash storage.

Core Focus

Writing robust embedded code that handles failures gracefully and persists data to the board's onboard flash storage.

Topics & Subtopics

  1. try / except / finally — A try block lets you attempt code that might fail, while except catches and handles specific errors without crashing the device. finally ensures cleanup code (like turning off a motor or releasing a bus) always runs, regardless of whether an error occurred.
  2. Custom Exceptions — You can define your own exception classes by inheriting from Exception, allowing you to raise and catch domain-specific errors like SensorTimeoutError or StorageFullError. This makes debugging hardware issues far clearer than catching generic Exception messages.
  3. The os Module & Onboard Flash Storage — MicroPython boards have a small onboard filesystem accessible via the os module, letting you list, create, and delete files just like a computer filesystem. This is commonly used to store configuration files, firmware parameters, or logged sensor data that survives reboots.
  4. Reading & Writing Files — Standard Python file operations (open(), .write(), .read()) work identically in MicroPython, letting scripts persist data across reboots. This is essential for standalone data-logging applications where no USB connection is available.

Why This Week Matters: Embedded devices run unattended — if a sensor fails or storage fills up and your code crashes, nobody is there to restart it. Robust try/except error handling is what separates a weekend prototype from a device you can deploy in the field and trust to keep running for months.

Code Example

import os

class SensorTimeoutError(Exception):
    """Raised when a sensor fails to respond within the timeout."""
    pass

def log_reading(value, filename="data.csv"):
    """Append a sensor reading to CSV file with error handling."""
    try:
        with open(filename, "a") as f:      # 'a' = append mode (preserve history)
            f.write(f"{value}\n")
    except OSError as e:
        if e.args[0] == 28:                  # errno 28 = ENOSPC (storage full)
            print("Storage full! Deleting oldest log...")
            os.remove(filename)
        else:
            raise                            # Re-raise unexpected OS errors
    finally:
        print("Write attempt completed.")   # Always runs, even on error

# Check available filesystem space
stat = os.statvfs("/")
free_kb = (stat[0] * stat[3]) // 1024
print(f"Free storage: {free_kb} KB")

# Read back and print file contents
try:
    with open("data.csv", "r") as f:
        print(f.read())
except OSError:
    print("No data file found yet.")

Week 9 — Networking, Libraries & MQTT

Connect your device to Wi-Fi and the wider Internet of Things ecosystem using HTTP and MQTT.

Core Focus

Connecting your device to Wi-Fi and the wider Internet of Things ecosystem.

Topics & Subtopics

  1. Wi-Fi Connectivity (network Module) — The network module lets a board like the ESP32 connect to a Wi-Fi network as a station (STA_IF) or create its own access point (AP_IF). This is the gateway to all internet-connected MicroPython projects — without it, your device is an island.
  2. Making HTTP Requests (urequests) — The urequests library (a lightweight version of Python's requests) lets your device send HTTP GET/POST requests to web APIs. This enables devices to fetch live data like weather forecasts or send sensor readings to a cloud dashboard.
  3. MQTT for IoT Messaging — MQTT is a lightweight publish-subscribe messaging protocol ideal for low-power IoT devices, using a central "broker" to route messages between devices. The umqtt.simple library provides an easy MicroPython client for publishing sensor data and subscribing to remote control commands.
  4. JSON Parsing (ujson) — The ujson module encodes Python objects into JSON strings and decodes JSON strings back into Python objects, just like CPython's json module. This is essential for exchanging structured data with web APIs, MQTT brokers, and cloud platforms.

Why This Week Matters: Wi-Fi + MQTT transforms a standalone sensor into an IoT node that can send data to dashboards, receive remote commands, and integrate with other devices — the foundation of every smart home, industrial monitoring, and environmental sensing system built on MicroPython.

Code Example

import network, time, ujson
import urequests
from umqtt.simple import MQTTClient

# Connect to Wi-Fi
wlan = network.WLAN(network.STA_IF)
wlan.active(True)
wlan.connect("MyWiFi", "password123")
while not wlan.isconnected():
    time.sleep(0.5)
print("Connected:", wlan.ifconfig())    # ('192.168.1.x', ...)

# HTTP GET — fetch data from a public API
response = urequests.get("http://api.example.com/temperature")
data = ujson.loads(response.text)
print("Temperature:", data["temp"])
response.close()                         # Free memory — critical on MCU!

# MQTT publish sensor reading
client = MQTTClient("esp32_01", "broker.hivemq.com")
client.connect()
payload = ujson.dumps({"temp": 24.5, "humidity": 60})
client.publish(b"sensors/living_room", payload.encode())
client.disconnect()

Phase 4 — Master Level Concepts

Weeks 10–12: Async concurrency with uasyncio, hardware interrupts, generators, decorators, and three complete capstone IoT projects.

Week 10 — Concurrency: uasyncio & Interrupts

Handle multiple tasks simultaneously on a single-core constrained device using async/await and hardware interrupts.

Core Focus

Handling multiple tasks "simultaneously" on a single-core constrained device.

Topics & Subtopics

  1. uasyncio & async/await — uasyncio is MicroPython's asynchronous I/O library, letting you write non-blocking code using async def and await, similar to CPython's asyncio. This allows a device to "wait" for a sensor or network response without freezing every other task in the program.
  2. Creating & Scheduling Tasks — asyncio.create_task() schedules a coroutine to run concurrently alongside others, all managed by a single event loop. This lets one board blink an LED, read a sensor, and maintain an MQTT connection simultaneously without true multithreading.
  3. Hardware Interrupts (Pin.irq) — An interrupt instantly calls a handler function when a hardware event occurs (like a button press or sensor signal), without needing to constantly poll the pin in a loop. This is far more efficient and responsive than checking pin state repeatedly in a while True loop.
  4. Timers (machine.Timer) — machine.Timer triggers a function at precise, regular intervals completely independent of your main program loop, useful for tasks like periodic sensor sampling or watchdog resets. Hardware timers run with high precision, unaffected by any blocking code elsewhere in your program.

Why This Week Matters: A real IoT device must do many things at once — blink a status LED, sample sensors, maintain a network connection, and respond instantly to button presses. uasyncio is how professional MicroPython applications achieve this without needing a full RTOS. Hardware interrupts are the only way to guarantee a response within microseconds regardless of what the main code is doing.

Code Example

import uasyncio as asyncio
from machine import Pin, Timer

led1   = Pin(2, Pin.OUT)
button = Pin(0, Pin.IN, Pin.PULL_UP)

# Interrupt handler — called INSTANTLY on button press (not in event loop)
def button_isr(pin):
    print("Button event! (from interrupt)")  # Keep ISRs SHORT!

button.irq(trigger=Pin.IRQ_FALLING, handler=button_isr)

# Async task 1: blink LED every 0.5s without blocking
async def blink(pin, delay):
    while True:
        pin.value(not pin.value())
        await asyncio.sleep(delay)     # Yields control to other tasks

# Async task 2: print sensor reading every 2s
async def read_sensor():
    while True:
        print("Sensor:", 42)           # Replace with real read()
        await asyncio.sleep(2)

# Main: run both tasks concurrently
async def main():
    asyncio.create_task(blink(led1, 0.5))
    asyncio.create_task(read_sensor())
    await asyncio.sleep(30)            # Run for 30 seconds

asyncio.run(main())

Week 11 — Generators, Decorators & Functional Patterns

Apply advanced Python patterns to write more elegant, memory-efficient embedded code.

Core Focus

Applying advanced Python patterns to write more elegant, memory-efficient embedded code.

Topics & Subtopics

  1. Generators (yield) — A generator function uses yield instead of return to produce a sequence of values lazily, one at a time, without storing them all in memory simultaneously. This is extremely valuable on memory-constrained microcontrollers for streaming sensor data, reading files line by line, or producing infinite sequences.
  2. Decorators (Function Wrapping) — A decorator wraps a function to add extra behavior (like timing, logging, or retry logic) without modifying the original function's code. They are written using the @decorator_name syntax placed directly above a function definition and are a cornerstone of professional Python code organization.
  3. Context Managers (with statement) — A context manager (using __enter__/__exit__) ensures setup and cleanup code always runs correctly — like properly releasing an I2C bus lock or closing a file — even if an exception occurs inside the with block. This is the professional standard for managing hardware resources safely.
  4. Lambda Functions & Functional Patterns — A lambda defines a small, anonymous, single-expression function inline, useful for short callbacks like sort keys or interrupt handlers. Combined with map() and filter(), lambdas support concise functional-style data processing on sensor reading lists.

Why This Week Matters: Generators are the memory-efficient streaming solution on a device with 256KB of RAM — you never load a full dataset at once, you generate values on demand. Decorators are how you add logging, timing, and retry logic to sensor reads without cluttering your business logic — a technique used in every professional embedded Python codebase.

Code Example

import time

# Generator: yields sensor values lazily — no list allocation!
def sensor_stream(n, base=20.0):
    for i in range(n):
        yield base + (i * 0.1)     # Simulated reading; real: yield sensor.read()

# Decorator: adds timing to any function
def timed(func):
    def wrapper(*args, **kwargs):
        start = time.ticks_ms()
        result = func(*args, **kwargs)
        elapsed = time.ticks_diff(time.ticks_ms(), start)
        print(f"{func.__name__} took {elapsed}ms")
        return result
    return wrapper

@timed
def process_readings(stream):
    """Consumes a generator and returns min/max."""
    readings = list(stream)                # Materialize lazily when needed
    return min(readings), max(readings)

# Lambda with filter: keep only "hot" readings above threshold
readings = [20.1, 22.3, 19.8, 25.6, 30.1]
hot = list(filter(lambda x: x > 24.0, readings))
print("Hot readings:", hot)              # [25.6, 30.1]

# Use the generator and decorated function
stream = sensor_stream(10)
lo, hi = process_readings(stream)
print(f"Min: {lo}, Max: {hi}")

Code Example

import uasyncio as asyncio
import ujson
from umqtt.simple import MQTTClient
from machine import Pin

led = Pin(2, Pin.OUT)

def on_command(topic, msg):
    """MQTT subscription callback — handles remote commands."""
    cmd = ujson.loads(msg)
    if cmd.get("action") == "toggle_led":
        led.value(not led.value())
        print("LED toggled via MQTT!")

async def publish_sensor(client, topic, interval=5):
    """Async task: publish sensor reading every interval seconds."""
    while True:
        payload = ujson.dumps({"temp": 24.5, "humidity": 60})
        try:
            client.publish(topic, payload.encode())
        except Exception as e:
            print("MQTT publish failed:", e)
        await asyncio.sleep(interval)

async def mqtt_loop(client):
    """Async task: check for incoming MQTT messages."""
    while True:
        client.check_msg()             # Non-blocking MQTT poll
        await asyncio.sleep(0.1)

async def main():
    client = MQTTClient("esp32_01", "broker.hivemq.com")
    client.set_callback(on_command)
    client.connect()
    client.subscribe(b"device/esp32_01/commands")
    asyncio.create_task(publish_sensor(client, b"sensors/temp"))
    asyncio.create_task(mqtt_loop(client))
    await asyncio.sleep(3600)          # Run for 1 hour

asyncio.run(main())
