About This Curriculum

This is a 15-week, phase-based Lua curriculum designed to take a complete beginner to a professional-level Lua developer. The roadmap moves from syntax, types, variables, and operators through control flow, functions, tables as data structures, and modules, into OOP with metatables, error handling, file I/O, and LuaRocks libraries, all the way to coroutines, advanced metaprogramming, performance optimization, the C API, and three complete capstone projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics including Lua's unique dynamic typing and table system, Phase 2 covers control flow, advanced functions, and modules, Phase 3 covers intermediate and advanced core topics like OOP, error handling, and real-world libraries, and Phase 4 covers master-level coroutines, metaprogramming, performance, and production projects.

By Week 15, you will have built three complete, portfolio-ready applications and be confident building production-grade Lua programs across game engines, embedded systems, and scripting domains.

Level: Beginner → Master  |  Duration: 15 weeks  |  Stack: Lua 5.4, LuaRocks, LuaJIT  |  Prerequisites: Basic programming logic

Phase 1 — Absolute Basics

Weeks 1–3: Setup, core types, variables, and operators.

Week 1 — Introduction to Lua & Environment Setup

Understand what Lua is, where it's used, and write your first working script.

Core Focus

Understanding what Lua is, where it is used in the real world, and writing your very first working Lua script.

Topics & Subtopics

  1. What is Lua & Why Use It? — Lua is a lightweight, fast, embeddable scripting language designed to be integrated into applications written in C and other languages. It powers game engines (Roblox, World of Warcraft addons, Love2D), embedded systems, Neovim configuration, and Redis scripting due to its tiny footprint and speed.
  2. Installing Lua & Running Scripts — Lua can be installed via package managers (brew install lua, apt install lua5.4) or built from source at lua.org. Scripts run with the lua interpreter directly from the command line, or interactively through the Lua REPL for quick experimentation.
  3. Lua Syntax Basics — Statements, Comments & Blocks — Lua statements don't require semicolons (though they're allowed as optional separators). Single-line comments use --; multi-line comments use --[[ ... ]]. Blocks of code are delimited by keywords like do...end, not braces or indentation.
  4. Variables & Dynamic Typing — Lua variables require no type declaration — the type is determined dynamically by the value assigned. Variables are global by default unless declared with local, which is considered best practice to avoid polluting the global namespace.

Why This Week Matters: Lua's defining characteristic — being embeddable in C applications — is what puts it inside World of Warcraft, Roblox, Redis, Nginx, and Neovim. Understanding this from Week 1 gives every subsequent concept a clear purpose. The local vs global variable distinction is not stylistic — it is a performance and correctness concern that affects every line of real Lua code.

Code Example

-- Lua's classic "Hello World" — no semicolons required
print("Hello, Lua!")
print("Lua version: " .. _VERSION)   -- .. is string concatenation

-- Single-line comment
--[[
  This is a multi-line comment
  spanning several lines
]]
do
  local message = "Blocks group statements"
  print(message)
end   -- message is no longer visible here

local name = "Alice"      -- local variable, recommended
local age = 30
global_score = 100        -- global variable (avoid in real code)
print(name, age, global_score)

Week 2 — Data Types, Operators & Strings

Learn Lua's eight core types, perform calculations with operators, and manipulate strings.

Core Focus

Learning Lua's eight core types, performing calculations with operators, and manipulating strings with the standard library.

Topics & Subtopics

  1. Lua's Core Types (nil, boolean, number, string, table, function) — Lua has eight basic types, but the six you'll use daily are nil (absence of value), boolean, number (Lua 5.3+ distinguishes integers and floats internally), string, table (Lua's only data structure), and function. Use the built-in type() function to inspect any value's type at runtime.
  2. Arithmetic, Relational & Logical Operators — Lua supports standard arithmetic (+, -, *, /, %, ^ for exponent, // for floor division), relational (==, ~=, <, >), and logical (and, or, not) operators — note Lua uses ~= for "not equal" instead of !=.
  3. Truthiness — nil, false and Everything Else — In Lua, only nil and false are considered "falsy" — every other value, including 0 and empty string "", is "truthy." This differs from many languages and is a common source of bugs for newcomers from JavaScript or Python.
  4. String Manipulation Basics — Strings are immutable in Lua and support concatenation with .., length with #, and a rich standard library (string.upper, string.sub, string.format) for formatting and manipulation. Long strings use [[...]] to avoid escaping.

Why This Week Matters: Lua's truthiness rule — that 0 and "" are truthy — trips up virtually every developer coming from C, JavaScript, or Python. Getting this wrong causes silent logic bugs that are extremely difficult to track down. The ~= not-equal operator is another Lua-specific quirk you must internalize before writing any conditional logic.

Code Example

print(type(nil))        -- nil
print(type(true))       -- boolean
print(type(42))         -- number
print(type("hello"))    -- string
print(type({}))         -- table
print(type(print))      -- function

local a, b = 17, 5
print(a / b)     -- 3.4   — always produces a float
print(a // b)    -- 3     — floor division
print(a % b)     -- 2     — modulo
print(a ~= b)    -- true  — not equal (not !=)

if 0 then print("0 is truthy in Lua!") end        -- this WILL print
if "" then print("empty string is truthy too!") end -- this WILL print

local s = "Hello, Lua!"
print(#s)                          -- 12
print(string.upper(s))             -- HELLO, LUA!
print(s:sub(1, 5))                 -- Hello
print(string.format("Score: %d%%", 95))  -- Score: 95%

Week 3 — Tables Basics & The Nil Trap

Understand Lua's single, powerful data structure and learn to avoid common nil-related pitfalls.

Core Focus

Understanding Lua's single, powerful data structure — the table — and learning to avoid common nil-related pitfalls.

Topics & Subtopics

  1. Tables as Arrays (Sequences) — When tables use sequential integer keys starting at 1, they behave like arrays — called "sequences" in Lua terminology. Indexing starts at 1 (not 0), which is a critical difference from most other programming languages.
  2. Tables as Dictionaries (Key-Value Maps) — Tables can also use string (or any non-nil value) keys to behave like a dictionary or hash map. Dot notation (t.key) and bracket notation (t["key"]) are equivalent ways to access string keys.
  3. Mixed Tables & Table Constructors — A single table can mix array-style and dictionary-style entries simultaneously, since Lua tables are fundamentally just associative arrays under the hood. Table constructors ({ }) are the idiomatic way to build complex nested structures in one expression.
  4. Assigning nil & The Nil Trap — Setting a table key to nil effectively removes it from the table — there is no real distinction between "absent" and "set to nil." This means #table (the length operator) becomes unreliable on tables with holes — a classic Lua gotcha.

Why This Week Matters: The table is Lua's only composite data structure — it replaces arrays, dictionaries, sets, objects, modules, and namespaces all at once. Understanding the nil trap and why #t becomes unreliable on sparse tables is the most important Lua-specific knowledge for avoiding runtime bugs in real programs.

Code Example

local fruits = {"apple", "banana", "cherry"}
print(fruits[1])         -- "apple" — indexing starts at 1!
print(#fruits)            -- 3
fruits[4] = "date"
print(#fruits)             -- 4

local person = { name = "Alice", age = 30 }
print(person.name)            -- "Alice" — dot notation
print(person["age"])          -- 30 — bracket notation
print(person.missing)         -- nil — no error, just nil

local config = {
  "first item",          -- index [1]
  width = 800,           -- key "width"
  nested = { a = 1 }    -- nested table
}
print(config[1], config.width, config.nested.a)

local t = {1, 2, 3, 4, 5}
t[3] = nil                     -- creates a "hole"
print(#t)                       -- undefined behavior! could be 2 or 5
for k, v in pairs(t) do print(k, v) end  -- safe iteration

Phase 2 — Control Flow & Logic

Weeks 4–7: Functions, tables as data structures, and modules.

Week 4 — Control Flow & Functions

Direct program execution with Lua's control structures and define reusable functions.

Core Focus

Directing program execution with Lua's control structures and defining reusable functions with multiple return values.

Topics & Subtopics

  1. Conditionals — if, elseif, else — Lua's if statement requires then and is terminated with end — there are no braces. elseif (a single keyword, not else if) chains additional conditions cleanly without nested blocks.
  2. Loops — while, repeat...until, Numeric for — while loops run while a condition holds; repeat...until is Lua's do-while equivalent, always running at least once with the condition checked after the body. The numeric for loop (for i = start, stop, step do) is the idiomatic way to iterate a known range.
  3. Generic for with pairs and ipairs — The generic for loop with ipairs iterates array-style tables in order, stopping at the first nil. pairs iterates every key-value pair in any table (unordered), making it the safe choice for dictionary-style or sparse tables.
  4. Functions — Declaration, Parameters & Multiple Returns — Functions are first-class values in Lua, declared with function name(params) ... end. Uniquely, Lua functions can return multiple values in a single return statement — a feature used extensively throughout the standard library.

Why This Week Matters: Multiple return values from a single function is one of Lua's most distinctive and practical features — it eliminates the need for output parameters, result structs, or exceptions just to return a status alongside a value. The ipairs vs pairs distinction is critical: using ipairs on a sparse table silently stops early, causing hard-to-debug logic errors.

Code Example

local score = 82
if score >= 90 then
  print("Grade: A")
elseif score >= 80 then
  print("Grade: B")
else
  print("Grade: F")
end

local i = 1
while i <= 3 do print("while: " .. i); i = i + 1 end

local j = 1
repeat print("repeat: " .. j); j = j + 1 until j > 3

for k = 10, 1, -2 do print("for: " .. k) end   -- 10, 8, 6, 4, 2

local fruits = {"apple", "banana", "cherry"}
for index, value in ipairs(fruits) do print(index, value) end

local function divmod(a, b)
  return a // b, a % b          -- multiple return values
end
local quotient, remainder = divmod(17, 5)
print(quotient, remainder)       -- 3  2

Week 5 — Functions in Depth & Variadic Arguments

Master advanced function features — default-like patterns, varargs, closures, and recursion.

Core Focus

Mastering advanced function features including variadic arguments, default parameter patterns, closures, and tail-call optimization.

Topics & Subtopics

  1. Variadic Functions (...) — Lua functions can accept a variable number of arguments using ... (the vararg expression). Inside the function, {...} packs all extra arguments into a table, and select('#', ...) returns the count of varargs passed.
  2. Default Argument Patterns — Lua has no native default parameter syntax, but the idiomatic pattern uses the or operator: if an argument is nil (not passed), the or falls through to the default value. This works cleanly because nil is falsy.
  3. Closures — Functions That Capture State — A closure is a function that "remembers" variables from its enclosing scope even after that scope has finished executing. Lua's first-class functions and lexical scoping make closures simple and powerful — used for counters, iterators, and callbacks.
  4. Recursion & Tail Calls — Lua functions can call themselves recursively to solve problems naturally expressed in terms of smaller subproblems. Lua guarantees proper tail-call optimisation, meaning tail-recursive functions run in constant stack space — unlike most scripting languages.

Why This Week Matters: Closures are the foundation of Lua's OOP system (Week 8), its module pattern (Week 7), and its coroutine-based generators (Week 12). The param = param or default idiom appears in virtually every Lua codebase. Tail-call optimization is Lua's answer to deep recursion — understanding it unlocks functional programming patterns without stack overflow risk.

Code Example

local function sum(...)
  local total = 0
  for _, value in ipairs({...}) do
    total = total + value
  end
  return total
end
print(sum(1, 2, 3, 4, 5))    -- 15

local function greet(name, greeting)
  greeting = greeting or "Hello"     -- idiomatic default value
  return greeting .. ", " .. name .. "!"
end
print(greet("Alice"))              -- Hello, Alice!
print(greet("Bob", "Hi"))           -- Hi, Bob!

local function makeCounter()
  local count = 0                      -- captured by closure
  return function()
    count = count + 1
    return count
  end
end
local counter = makeCounter()
print(counter(), counter(), counter())  -- 1  2  3

local function factorial(n, acc)
  acc = acc or 1
  if n <= 1 then return acc end
  return factorial(n - 1, n * acc)   -- tail call
end
print(factorial(10))   -- 3628800

Week 6 — Tables as Data Structures

Use tables to implement classic data structures — stacks, queues, sets, and sorting.

Core Focus

Using tables to implement classic data structures — stacks, queues, sets — and sorting with custom comparators.

Topics & Subtopics

  1. Implementing a Stack with Tables — A stack (LIFO) is trivially implemented using a table's array part with table.insert and table.remove operating on the end of the sequence — both are O(1) operations when applied to the last index.
  2. Implementing a Queue with Tables — A queue (FIFO) can be implemented with table.insert at the end and table.remove(t, 1) at the front. For performance-critical queues, a head/tail index pattern avoids the O(n) shifting cost of removing from position 1.
  3. Sets Using Tables as Hash Maps — Lua has no built-in set type, but tables are commonly used as sets by mapping each element to true as the value — giving O(1) membership testing via simple key lookup, far faster than scanning an array.
  4. Sorting Tables with table.sort — table.sort sorts a sequence in place using the default < comparison, or a custom comparator function for complex sorting logic — such as sorting objects by a specific field or in descending order.

Why This Week Matters: The set pattern — set[value] = true — is one of the most used Lua idioms in game scripting. Checking membership in an array requires O(n) scanning; checking in a table-as-set is O(1). This performance difference is what separates Lua code that runs at 60fps from code that drops frames in a game loop.

Code Example

local stack = {}
table.insert(stack, "first")
table.insert(stack, "second")
table.insert(stack, "third")
print(table.remove(stack))    -- "third" — pop last
print(#stack)                 -- 2

local queue = {}
table.insert(queue, "first")
table.insert(queue, "second")
print(table.remove(queue, 1)) -- "first" — dequeue from front

local function makeSet(list)
  local set = {}
  for _, value in ipairs(list) do set[value] = true end
  return set
end
local fruitSet = makeSet({"apple", "banana", "cherry"})
print(fruitSet["banana"])    -- true — O(1) membership test
print(fruitSet["mango"])     -- nil — not in set

local numbers = {5, 3, 8, 1, 9, 2}
table.sort(numbers)
print(table.concat(numbers, ", "))           -- 1, 2, 3, 5, 8, 9
table.sort(numbers, function(a, b) return a > b end)
print(table.concat(numbers, ", "))           -- 9, 8, 5, 3, 2, 1

Week 7 — Modules, Scope & String Patterns

Organise code into reusable modules, master scoping rules, and use Lua's lightweight pattern-matching system.

Core Focus

Organizing code into reusable modules, mastering scope rules, and using Lua's lightweight pattern-matching system for text processing.

Topics & Subtopics

  1. Creating Modules with require — A Lua module is simply a .lua file that returns a table of public functions and values. require loads and caches the module (running it only once even if required multiple times), making modules the standard way to organise multi-file Lua projects.
  2. Scope Rules — Local vs Global, Blocks & Shadowing — Variables declared with local are scoped to the enclosing block and are not visible outside it. Lua allows shadowing — declaring a new local with the same name as an outer variable temporarily hides the outer one within the inner scope.
  3. String Patterns — Lua's Lightweight Regex — Lua doesn't use full regular expressions but provides "patterns" — a simpler, faster matching syntax via string.match, string.find, and string.gmatch. Common classes include %d (digit), %a (letter), %s (whitespace), and +/* for repetition.
  4. string.gsub for Find-and-Replace — string.gsub performs pattern-based substitution, optionally limited to a maximum number of replacements, and returns both the resulting string and the count of substitutions made — useful for sanitising and transforming text data.

Why This Week Matters: The module pattern — a file that returns a table — is how every real multi-file Lua project is organized. Lua's pattern system is intentionally simpler than full regex, making it faster and more embeddable. Knowing when %d+ is sufficient versus when you need a full regex library is the mark of a practical Lua developer.

Code Example

-- File: mathutils.lua
local M = {}
function M.square(x) return x * x end
function M.cube(x) return x * x * x end
return M

-- File: main.lua
-- local mathutils = require("mathutils")
-- print(mathutils.square(5))   -- 25

local x = 10
do
  local x = 20         -- shadows the outer x
  print(x)               -- 20
end
print(x)                 -- 10 — outer x unaffected

local text = "Order #4521 placed on 2026-06-30"
local orderId = string.match(text, "#(%d+)")
print(orderId)                          -- "4521"

for word in string.gmatch("one two three", "%a+") do
  print(word)                            -- one / two / three
end

local sentence = "The cat sat on the mat"
local result, count = string.gsub(sentence, "at", "og")
print(result)    -- "The cog sog on the mog"
print(count)      -- 3

Phase 3 — Intermediate & Advanced Core

Week 8 — Object-Oriented Programming with Metatables

Implement OOP patterns in Lua using tables and metatables — Lua has no built-in class system, so you build one.

Core Focus

Implementing OOP patterns from scratch using Lua's metatable system — Lua's most powerful and distinctive feature.

Topics & Subtopics

  1. Metatables & the __index Metamethod — A metatable is a table that controls another table's behavior for operations like indexing, arithmetic, and comparison. The __index metamethod, set via setmetatable, is the foundation of Lua's "fake" inheritance — when a key is missing, Lua looks it up in the metatable instead.
  2. Simulating Classes & Constructors — By convention, a Lua "class" is a table holding shared methods, with a constructor function that creates instances and links them via setmetatable. The colon syntax (function Class:method()) is sugar for an implicit self first parameter.
  3. Operator Overloading with Metamethods — Metamethods like __add, __eq, __lt, __tostring, and __call let custom tables respond to operators (+, ==, <) and built-in functions (tostring, calling the table like a function) — enabling expressive custom types.
  4. Encapsulation Patterns — Private State with Closures — Lua has no private keyword, but true encapsulation can be achieved using closures: state captured in a constructor's local scope is inaccessible from outside, only reachable through methods the constructor explicitly returns.

Why This Week Matters: Every major Lua-powered game engine — Roblox, Love2D, World of Warcraft's addon system — uses metatable-based OOP. Understanding how __index chains create inheritance, and how setmetatable links instances to their "class" table, is the prerequisite for reading and contributing to any real Lua game codebase.

Code Example

local Animal = { sound = "..." }
Animal.__index = Animal

local function newAnimal(name)
  local self = setmetatable({}, Animal)
  self.name = name
  return self
end
function Animal:speak() return self.name .. " says " .. self.sound end

local Dog = setmetatable({}, { __index = Animal })
Dog.__index = Dog
Dog.sound = "Woof"
function Dog.new(name)
  return setmetatable({ name = name }, Dog)
end
local rex = Dog.new("Rex")
print(rex:speak())   -- Rex says Woof

local Vector = {}
Vector.__index = Vector
Vector.__add = function(a, b)
  return setmetatable({x = a.x + b.x, y = a.y + b.y}, Vector)
end
Vector.__tostring = function(v)
  return string.format("(%d, %d)", v.x, v.y)
end
local v1 = setmetatable({x = 1, y = 2}, Vector)
local v2 = setmetatable({x = 3, y = 4}, Vector)
print(tostring(v1 + v2))   -- (4, 6)

Week 9 — Error Handling & Defensive Programming

Handle errors gracefully using Lua's protected call mechanism and write robust, defensive code.

Core Focus

Handling errors gracefully using Lua's protected call mechanism and writing robust, defensive code that fails fast with clear messages.

Topics & Subtopics

  1. error() and assert() — error(message) raises a Lua error, unwinding the call stack until caught. assert(condition, message) is shorthand that raises an error if the condition is falsy — the idiomatic way to validate function preconditions.
  2. pcall and xpcall — Protected Calls — pcall(function, ...) calls a function in "protected mode," catching any error and returning false, errorMessage instead of crashing the program. xpcall additionally accepts a custom error handler function, useful for attaching stack traces via debug.traceback.
  3. Custom Error Objects — Beyond simple string errors, error() can raise any Lua value — including a table — allowing structured error objects with error codes, context, and metadata, which calling code can inspect after a pcall catches them.
  4. Defensive Programming — Type Checking & Guard Clauses — Because Lua is dynamically typed with no compile-time checks, defensive functions validate their inputs explicitly at the start (guard clauses) to fail fast with clear errors rather than producing confusing downstream bugs.

Why This Week Matters: Lua's error model — raise any value, catch with pcall — is both simpler and more flexible than exception hierarchies in Java or Python. Game engines use pcall to sandbox untrusted mod scripts: if a mod's Lua code crashes, pcall catches it and keeps the game running. This is a real production pattern you will use immediately.

Code Example

local function withdraw(balance, amount)
  assert(type(amount) == "number", "amount must be a number")
  assert(amount <= balance, "insufficient funds")
  return balance - amount
end
print(withdraw(100, 50))  -- 50

local function risky(x)
  if x < 0 then error("negative input not allowed") end
  return math.sqrt(x)
end
local success, result = pcall(risky, -4)
if success then print("Result: " .. result)
else print("Caught error: " .. result) end

local function validateAge(age)
  if age < 0 or age > 120 then
    error({ code = "INVALID_AGE", message = "Age out of range", value = age })
  end
  return age
end
local ok, err = pcall(validateAge, -5)
if not ok then print(err.code .. ": " .. err.message) end

local function safeDivide(a, b)
  if type(a) ~= "number" or type(b) ~= "number" then
    return nil, "both arguments must be numbers"
  end
  if b == 0 then return nil, "division by zero" end
  return a / b
end
local result, err = safeDivide(10, 0)
print(result, err)   -- nil  division by zero

Week 10 — File I/O & Standard Library Deep Dive

Read and write files, and master Lua's essential standard library modules.

Core Focus

Reading and writing files with Lua's I/O library and mastering the essential standard library modules used in every real Lua project.

Topics & Subtopics

  1. File I/O — Reading Files — io.open(filename, mode) opens a file and returns a file handle (or nil plus an error message on failure). file:read("*a") reads the entire file, file:lines() iterates line by line — always close files with file:close().
  2. File I/O — Writing Files — Writing uses mode "w" (overwrite) or "a" (append). file:write(...) accepts multiple string/number arguments written sequentially. Wrapping file operations in pcall guards against permission or disk errors.
  3. The os Library — System Interaction — The os library provides system-level utilities: os.time() returns the current Unix timestamp, os.date() formats dates, os.clock() measures CPU time for benchmarking, and os.getenv() reads environment variables.
  4. The table and math Libraries — The table library provides table.insert, table.remove, table.concat, and table.sort for sequence manipulation. The math library provides math.floor, math.ceil, math.random, and trigonometric functions for numeric computation.

Why This Week Matters: File I/O and the os library are what make Lua useful for CLI tools, game save systems, and configuration management. The pattern of checking if not file then error(...) end immediately after io.open — combined with pcall from Week 9 — is the production-standard way to handle all file operations in real Lua codebases.

Code Example

local file, err = io.open("data.txt", "r")
if not file then error("Could not open file: " .. err) end
for line in file:lines() do print(line) end
file:close()

local file = io.open("output.txt", "w")
file:write("Line 1\n")
file:write("Line 2\n", "Line 3\n")
file:close()

local appendFile = io.open("output.txt", "a")
appendFile:write("Appended line\n")
appendFile:close()

print(os.date("%Y-%m-%d %H:%M:%S"))
local startTime = os.clock()
for i = 1, 1000000 do end
print(string.format("Elapsed: %.4f seconds", os.clock() - startTime))

math.randomseed(os.time())
print(math.random(1, 100))
print(math.floor(3.7), math.ceil(3.2))   -- 3  4
local nums = {5, 2, 8}
print(table.concat(nums, ", "))            -- "5, 2, 8"

Topics & Subtopics

Embedding Lua in C Applications (Conceptual Overview) — Lua's defining feature is its small, clean C API for embedding — a host application creates a lua_State, loads scripts, and exchanges data via a virtual stack. This explains why Lua dominates game scripting, configuration, and extensible application plugins.

Why This Week Matters: JSON is the universal data exchange format — without it, Lua cannot communicate with web APIs, read modern config files, or implement save-game systems that load correctly after a game update changes the format. Week 12 — Coroutines: Lua's Cooperative Concurrency

Master Lua's unique coroutine system for cooperative multitasking, generators, and asynchronous-style patterns.

Core Focus

Mastering Lua's unique coroutine system for cooperative multitasking, generators, and task scheduling in game engines.

Topics & Subtopics

  1. Creating & Resuming Coroutines — coroutine.create(function) wraps a function as a coroutine — a separate thread of execution that runs cooperatively, not preemptively. coroutine.resume starts or continues it; the coroutine runs until it finishes, errors, or yields.
  2. coroutine.yield — Pausing Execution — coroutine.yield(...) pauses the coroutine at exactly that point, returning control and any yielded values back to the caller. The next resume call continues execution right where yield left off — this is the foundation of generators and cooperative scheduling.
  3. Building Generators & Iterators with Coroutines — Coroutines naturally implement generator-style iterators using coroutine.wrap, which returns a plain function instead of a thread object — calling it resumes the coroutine and returns its yielded value directly, perfect for for loops.
  4. Cooperative Task Scheduling — Because Lua coroutines yield control explicitly (cooperative, not preemptive), they're ideal for building simple task schedulers — common in game engines for managing multiple "simultaneous" scripted behaviors without real OS threads.

Why This Week Matters: Coroutines are how Roblox scripts run "simultaneously" — each NPC, cutscene, and tween is a coroutine yielding control back to the engine's scheduler each frame. Understanding how yield and resume pass values bidirectionally makes coroutine-based generators and schedulers completely demystified instead of magical.

Code Example

local co = coroutine.create(function(a)
  print("Started with: " .. a)
  local b = coroutine.yield(a + 1)
  print("Resumed with: " .. b)
end)
print(coroutine.resume(co, 10))    -- Started with: 10  → true  11
print(coroutine.resume(co, 99))    -- Resumed with: 99  → true

local function counter()
  for i = 1, 3 do coroutine.yield(i) end
  return "done"
end
local co2 = coroutine.create(counter)
while coroutine.status(co2) ~= "dead" do
  print(coroutine.resume(co2))     -- true 1 / true 2 / true 3 / true done
end

local function range(n)
  return coroutine.wrap(function()
    for i = 1, n do coroutine.yield(i) end
  end)
end
for value in range(5) do print(value) end   -- 1 2 3 4 5

Week 13 — Advanced Metatables & Metaprogramming

Push Lua's metatable system to its limits — proxies, read-only tables, and dynamic dispatch.

Core Focus

Pushing Lua's metatable system to its limits with write interception, callable tables, weak references, and sandboxing proxies.

Topics & Subtopics

  1. __newindex — Intercepting Assignment — While __index intercepts reads of missing keys, __newindex intercepts writes to missing keys — enabling read-only tables, validation on assignment, and automatic logging of all mutations to an object.
  2. __call — Making Tables Callable — The __call metamethod allows a table to be invoked like a function using normal call syntax table(...). This is used to build "functor" objects — stateful callable objects, common in factory and command patterns.
  3. Weak Tables & Garbage Collection Hints — Weak tables (set via __mode = "k", "v", or "kv" in the metatable) allow the garbage collector to reclaim entries whose keys/values are otherwise unreferenced — essential for building caches that don't leak memory.
  4. Proxy Tables & Sandboxing Concepts — Combining __index, __newindex, and a separate hidden storage table builds proxy objects that fully control access to underlying data — the technique used to build secure sandboxes that restrict what untrusted scripts can read or modify.

Why This Week Matters: The sandboxing pattern — proxy tables with __index/__newindex controlling access — is literally how game engines isolate mod scripts from game internals. Weak tables are the solution to cache memory leaks in long-running game servers. These are not academic techniques — they are the tools that make production Lua systems robust and secure.

Code Example

local readOnly = {}
local proxy = setmetatable({}, {
  __index = readOnly,
  __newindex = function(t, k, v)
    error("Attempt to modify read-only table")
  end
})
readOnly.value = 42
print(proxy.value)    -- 42

local Multiplier = {}
Multiplier.__call = function(self, x) return x * self.factor end
local double = setmetatable({ factor = 2 }, Multiplier)
print(double(21))    -- 42 — calling a table like a function

local cache = setmetatable({}, { __mode = "k" })  -- weak keys
local function memoize(fn)
  return function(x)
    if cache[x] == nil then cache[x] = fn(x) end
    return cache[x]
  end
end
local fastSquare = memoize(function(x) return x * x end)
print(fastSquare(5))   -- 25 (cached after first call)

Week 14 — Performance, the C API & Embedding Lua

Optimise Lua code for performance and understand how Lua integrates with host applications via its C API.

Core Focus

Optimizing Lua code for performance and understanding how Lua integrates with host applications via its C API and LuaJIT.

Topics & Subtopics

  1. Performance Profiling & Common Bottlenecks — Lua performance issues typically stem from excessive table creation in loops, unnecessary global variable access (slower than local), and string concatenation in loops (creates many intermediate strings). os.clock() and manual instrumentation are the primary profiling tools.
  2. Locals vs Globals — Performance Implications — Accessing a local variable is a fast, direct register/stack access; accessing a global requires a table lookup in _G every single time. Caching frequently used globals (like math.floor) as locals inside hot loops yields measurable speedups.
  3. LuaJIT — Just-In-Time Compilation — LuaJIT is a high-performance alternative implementation of Lua that compiles hot code paths to native machine code at runtime, often running 10–50x faster than standard Lua for compute-heavy workloads. It's widely used in performance-critical game engines.
  4. The Lua C API — Embedding Conceptual Deep Dive — The C API revolves around a virtual stack shared between C and Lua: C code pushes/pops values, calls Lua functions, and registers C functions callable from Lua. Understanding this stack-based model explains how every embedding — games, Redis, Nginx/OpenResty — actually works under the hood.

Why This Week Matters: In a game loop running at 60fps, Lua code has approximately 16 milliseconds per frame. Understanding that global lookups are table accesses and string concatenation in loops is quadratic — and knowing how to fix both — is the difference between a smooth game and one that stutters. LuaJIT adoption is widespread enough that understanding it is a professional requirement.

Code Example

-- BAD: string concatenation in a loop — O(n²)
local s = ""
for i = 1, 1000 do s = s .. i end

-- GOOD: build table and concat once — O(n)
local parts = {}
for i = 1, 1000 do parts[i] = tostring(i) end
local result = table.concat(parts)

-- Cache global as local before hot loop
local floor = math.floor
local sum = 0
for i = 1, 1000000 do
  sum = sum + floor(i / 2)  -- fast local access
end

-- Run with: luajit script.lua
-- LuaJIT FFI: ffi.cdef[[ int printf(const char *fmt, ...); ]]

-- C API conceptual: this Lua function is callable from a C host
function on_player_damage(playerId, amount)
  print(string.format("Player %d took %d damage", playerId, amount))

-- Coroutine-driven tween system — real-world game engine pattern
local function tweenTo(entity, targetX, targetY, duration)
  return coroutine.wrap(function()
    local startX, startY, elapsed = entity.x, entity.y, 0
    while elapsed < duration do
      local t = elapsed / duration
      entity.x = startX + (targetX - startX) * t
      entity.y = startY + (targetY - startY) * t
      elapsed = elapsed + coroutine.yield()  -- yields dt each frame
    end
    entity.x, entity.y = targetX, targetY
  end)
end
