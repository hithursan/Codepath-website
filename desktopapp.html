Desktop App Development with Electron.js

Complete 15-Week Beginner-to-Advanced Roadmap — Build Cross-Platform Apps for Windows, macOS & Linux

Intro W1 W2 W3 W4 W5 W6 W7 W8 W9 W10 W11 W12 W13 W14 W15 Capstone

About This Roadmap

This roadmap presents a complete, structured, beginner-to-advanced learning path for Desktop Application Development using Electron.js, JavaScript, HTML, and CSS — built and documented as a college project.

Whether you have never written a single line of code before, or you're a student looking for a practical, hands-on technical project — this roadmap takes you step by step from the very basics to building, packaging, and distributing a real desktop application that runs on Windows, macOS, and Linux.

Duration: 15 weeks  |  Pace: Full-time (~35 hrs/week)  |  Stack: Electron.js · JavaScript (ES6+) · HTML5 · CSS3 · Node.js · SQLite

Hands-On: 14 weekly mini-projects + 1 complete, packaged capstone app (NoteVault)  |  Platform: Cross-platform (Windows, macOS, Linux)  |  Cost: 100% free tools and resources

5 Phases Overview

  • Phase 1 (Weeks 1–4): Foundations — Terminal, HTML, CSS, JavaScript
  • Phase 2 (Weeks 5–7): Electron Core — App structure, IPC, file system
  • Phase 3 (Weeks 8–10): Desktop Features — UI, database, native OS APIs
  • Phase 4 (Weeks 11–13): Advanced — Security, packaging, testing, CI
  • Phase 5 (Weeks 14–15): Capstone build, polish, and final presentation

By Week 15, you will have a real, installable desktop application — "NoteVault," a cross-platform note-taking app — that you can distribute as a downloadable installer.

Common Mistakes

  • Typing git add . without first checking git status.

  • Create .gitignore with node_modules on first line before writing any code.

Common Mistakes

  • Not closing tags (e.g., <p>text instead of <p>text</p>).
  • Using <br> for spacing instead of CSS margin/padding.

How to Avoid Them

  • Always close every tag before writing inside it: <p></p> first, content after.
  • Reserve <br> only for intentional line breaks (poem, address), not spacing.
  • Accept early that structure (HTML) and appearance (CSS) are deliberately separate.

Common Mistakes

  • Confusing = (assignment), == (loose), === (strict) — almost always use ===.
  • Forgetting map/filter return NEW arrays, don't modify original.
  • Trying to change HTML before DOM has loaded.
  • Forgetting await before async call (Promise object appears instead of value).

How to Avoid Them

  • Remember array.map(...) doesn't change array — store in variable.
  • Place <script> at bottom of <body>, or use DOMContentLoaded.
  • Pair every async function with await on every async call, wrap in try/catch.

Week 6 — IPC Communication, preload.js & Context Isolation

Learn how the main process and renderer process communicate — the most important architectural concept in Electron.

Topics & Subtopics

  1. Why separation — Renderer is like a browser; unrestricted Node.js access is a security risk.
  2. preload.js — Runs in special context with access to both Node.js and renderer's window; acts as gatekeeper.
  3. contextBridge.exposeInMainWorld — Only safe way to pass functions from preload/Node.js to renderer.
  4. ipcRenderer.invoke('channel') — Renderer sends message to main and awaits reply (Promise-based).
  5. ipcMain.handle('channel', handler) — Main process listens and returns result.
  6. win.webContents.send('channel', data) — Main pushes message to renderer (no request needed).
  7. ipcRenderer.on('channel', handler) — Renderer listens for push from main.

Why This Week Matters: IPC is the nervous system of your Electron application. Every time the UI needs to read a file, save data, or trigger an OS feature, it uses IPC. Misunderstanding IPC is the #1 reason Electron apps end up insecure or architecturally broken.

Daily Study Plan

Day Focus
Day 1 Read Electron Process Model docs. Understand why context isolation exists.
Day 2 Create preload.js, link in webPreferences, expose simple function using contextBridge.
Day 3 Call exposed function from renderer.js. Confirm in DevTools console.
Day 4 Implement ipcRenderer.invoke + ipcMain.handle for "ping/pong" message.
Day 5 Implement win.webContents.send + ipcRenderer.on for push from main to renderer.
Day 6 Combine both patterns: renderer requests, main responds and pushes. Add real button.
Day 7 Review + mini project.

Practice Tasks

  • Expose 3 different functions via contextBridge, call each from renderer.js.
  • Send a number from renderer to main, double it in main, display result back in UI.
  • Trigger push message from main using setTimeout, show in renderer.

🛠 Mini Project

Build a two-window Electron app. Window 1: form with text input + "Send to Main" button. On click, message sent to main via IPC, main reverses text and sends back, reversed text appears in UI. Window 2: log window where main pushes new log entry every time IPC is used. All via contextBridge + ipcMain.handle + ipcRenderer.invoke. Push to GitHub.

Common Mistakes

  • Enabling nodeIntegration: true to avoid learning IPC — serious security mistake.
  • Calling ipcRenderer directly in renderer without going through contextBridge.
  • Forgetting to pass preload path in webPreferences.
  • Confusing ipcMain.handle (request-response with invoke) with ipcMain.on (fire-and-forget with send).

How to Avoid Them

  • Never set nodeIntegration: true or contextIsolation: false — outdated and insecure.
  • All renderer ↔ main communication must go through preload.js and contextBridge.
  • Use invoke/handle for request-response; send/on only for one-way pushes.
  • Always use path.join(__dirname, 'preload.js').

Give your Electron app real-world usefulness by reading, writing, and managing files on the user's computer.

  3. dialog API — dialog.showOpenDialog(), dialog.showSaveDialog(), dialog.showMessageBox().

Why This Week Matters: This is what separates a desktop app from a website — the ability to interact with the user's real files. File system access enables saving notes, loading documents, reading configuration. It's also the first week heavily combining IPC (Week 6) with Node.js.

🛠 Mini Project

Build a "File Reader" feature as part of NoteVault. Renderer has "Open File" button triggering dialog.showOpenDialog (via IPC) to let user pick any .txt or .md file. File content read using fs.readFile in main process and sent to renderer where it displays in <pre> or <textarea>. "Save" button writes changes back to same file. Push to GitHub.

Common Mistakes

  • Using fs functions directly in renderer process (should only run in main, called via IPC).
  • Building file paths by string concatenation instead of path.join().
  • Calling dialog from renderer instead of main process.
  • Not handling case where dialog.showOpenDialog returns empty (user clicked Cancel).

How to Avoid Them

  • All fs and dialog calls belong in main process — call from renderer through IPC only.
  • Use path.join() for every file path construction, every time, no exceptions.
  • Always check result.canceled after dialog.showOpenDialog().
  • Wrap every fs operation in try/catch and return meaningful error through IPC.

Week 9 — Data Persistence with SQLite

Replace in-memory or JSON-file storage with a proper local SQLite database, giving NoteVault reliable, queryable, persistent data storage.

Topics & Subtopics

  1. SQLite — File-based, serverless SQL database; entire database is single .db file; no separate server needed.
  2. better-sqlite3 — Synchronous SQLite driver for Node.js (simpler to use correctly in Electron main process).
  3. Schema — Creating notes table with columns: id, title, content, createdAt, updatedAt.
  4. CRUD in SQL — INSERT INTO, SELECT * FROM, UPDATE ... SET ... WHERE, DELETE FROM ... WHERE.
  5. Initialization — CREATE TABLE IF NOT EXISTS on first run.
  6. DB location — Storing .db in app.getPath('userData') for correct OS-specific folder.
  7. Search — SELECT ... WHERE title LIKE '%search%' for basic search.

Why This Week Matters: JSON files work for small apps but break down quickly — no efficient searching, risk of corruption on bad save, no relational capability. SQLite is the standard choice for local desktop app data and is built into many operating systems.

Daily Study Plan

Day Focus
Day 1 SQLite concept + install better-sqlite3. Write test script creating database and table.
Day 2 SQL practice: INSERT, SELECT, UPDATE, DELETE in isolation.
Day 3 Create db.js with createNote, getAllNotes, getNoteById functions.
Day 4 Add updateNote and deleteNote. Test all 5 functions from standalone script.
Day 5 Create IPC channels for each operation (ipcMain.handle('db:createNote', ...)).
Day 6 Connect Week 8 UI to real SQLite data. Display notes from DB in sidebar.
Day 7 Implement search. Review + mini project.

Practice Tasks

  • Create SQLite database with notes table; manually insert/update/delete rows.
  • Write getAllNotes and createNote functions, test from Node.js script.
  • Connect "New Note" button to IPC call that inserts note into SQLite.
  • Load all notes from SQLite on startup and display in sidebar.

🛠 Mini Project

Replace all in-memory/JSON-based note storage with SQLite. NoteVault sidebar should load real notes from database on startup, editor should save to database, deleting a note should remove it from both UI and database. Notes must persist across app restarts. Push to GitHub.

Common Mistakes

  • Calling better-sqlite3 from renderer process (must only run in main).
  • Not closing database connection properly.
  • Forgetting CREATE TABLE IF NOT EXISTS — app crashes on second launch.
  • Building .db file path with string concatenation instead of path.join.

How to Avoid Them

  • All database code belongs in db.js inside main process — never import in renderer.
  • Always use CREATE TABLE IF NOT EXISTS in initialization query.
  • Always use path.join(app.getPath('userData'), 'notes.db') for DB path.
  • Separate database functions (pure SQL logic) from IPC handler setup.

NoteVault — Cross-Platform Note-Taking Desktop Application

NoteVault is a fully-featured, installable cross-platform note-taking desktop application — the kind of tool you'd actually use daily. Built incrementally across Phases 1–4 and finalized in Weeks 14–15.

Core Features

  1. Note Management — Create, view, edit, delete notes with title, content, created/updated timestamps. All stored in local SQLite database.
  2. Markdown Editor with Live Preview — Split-panel editor (textarea on left, rendered HTML on right) using Marked.js for live markdown rendering.
  3. Sidebar Note List — All notes displayed in scrollable sidebar; click to open in editor; sorted by last updated.
  4. Search — Live search across note titles (and optionally content) with SQL LIKE queries.
  5. File I/O — Import .md/.txt files via native open dialog; export notes via native save dialog.
  6. Native Application Menu — File menu (New, Open, Save, Quit), Edit menu (Undo, Redo, Cut, Copy, Paste), View menu (Toggle Theme, DevTools), Help menu — all with keyboard accelerators.
  7. System Tray Icon — Tray icon with quick actions: Show App, New Quick Note, Quit.
  8. Desktop Notifications — Notification on note save and other significant actions.
  9. Native Dialogs — Confirmation dialog before delete; error dialogs on failures.
  10. Right-Click Context Menus — Context menu on note list with Copy Title, Delete, etc.
  11. Dark/Light Theme — Toggle between themes; preference persisted across sessions in userData config.
  12. Keyboard Shortcuts — Ctrl/Cmd+N (new note), Ctrl/Cmd+S (save), Ctrl/Cmd+F (search), and more.
  13. Unsaved Changes Detection — Prompt before closing window or switching notes with unsaved changes.
  14. Cross-Platform Installers — NSIS for Windows, DMG for macOS, AppImage for Linux — all generated via electron-builder.

Technical Architecture

Process Separation: Main process (Node.js — file system, SQLite, dialogs, menus, tray) ⇄ Preload script (contextBridge) ⇄ Renderer process (HTML/CSS/JS — UI only).

IPC Channels: All renderer ↔ main communication via ipcRenderer.invoke / ipcMain.handle through contextBridge-exposed APIs.

Data Layer: SQLite database via better-sqlite3, stored in app.getPath('userData'). Single notes table with full CRUD.

Security: nodeIntegration: false, contextIsolation: true, all IPC inputs validated, Content Security Policy header set.

Tech Stack Summary

Electron.js · JavaScript (ES6+) · HTML5 · CSS3 · Node.js · SQLite (better-sqlite3) · Marked.js · electron-builder · Jest · Playwright · GitHub Actions CI

Deliverables

  • Working cross-platform desktop app (Windows, macOS, Linux)
  • Packaged installers (NSIS / DMG / AppImage) via electron-builder
  • GitHub Release with all 3 platform installers attached for download
  • Complete GitHub repository with professional README (screenshots, features, install instructions, architecture overview)
  • DISTRIBUTION.md — non-technical install guide for evaluators
  • System design document with architecture diagram
  • Passing test suite (Jest unit tests + Playwright e2e tests)
  • GitHub Actions CI pipeline (matrix build: Windows, macOS, Linux)
  • Rehearsed 5-minute demo walkthrough

Important: NoteVault is not built only in Weeks 14–15 — it is the consolidation of skills built throughout Weeks 5–13. Each week's mini project directly extends the same Library API. Week 14 is integration and polish; Week 15 is packaging, documentation, and presentation prep.
