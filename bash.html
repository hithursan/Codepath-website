About This Curriculum

This is a 15-week, phase-based Bash curriculum designed to take a complete beginner to a professional-level DevOps and systems engineer. The roadmap moves from shell setup, variables, quoting, and basic I/O through control flow, functions, arrays, and modular scripting, into file handling, error handling, text processing with grep/sed/awk, and process management, all the way to heredocs, process substitution, concurrency, professional CLI patterns, and three complete capstone projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics including the shell environment and variable system, Phase 2 covers control flow, functions, arrays, and reusable script modules, Phase 3 covers intermediate and advanced core topics like file operations, strict mode error handling, and the text-processing trinity, and Phase 4 covers master-level I/O, concurrency, professional tooling patterns, and production projects.

By Week 15, you will have built three complete, portfolio-ready automation tools and be confident writing production-grade Bash scripts for monitoring, log analysis, and deployment orchestration.

Level: Beginner → Master  |  Duration: 15 weeks  |  Stack: Bash 5+, GNU coreutils, bats-core  |  Prerequisites: Basic computer literacy

Phase 1 — Absolute Basics

Weeks 1–3: Shell setup, variables, and core data concepts.

Week 1 — Introduction to Bash & the Shell Environment

Understand what Bash is, how the shell works, and run your first commands and scripts.

Core Focus

Understanding what Bash is, how the shell works, and running your first commands and scripts.

Topics & Subtopics

  1. What is Bash & the Role of the Shell? — Bash (Bourne Again SHell) is a command-line interpreter and scripting language that acts as the interface between you and the operating system kernel. It's the default shell on most Linux distributions and macOS (pre-Catalina), making it essential for automation, system administration, and DevOps work.
  2. The Shebang, Script Permissions & Execution — The shebang (#!/bin/bash) on a script's first line tells the OS which interpreter to use when executed directly. Scripts must be marked executable with chmod +x before running with ./script.sh, or alternatively run explicitly via bash script.sh.
  3. Navigating the Filesystem (pwd, cd, ls) — pwd prints the current working directory; cd changes it (cd .. for parent, cd ~ for home); ls lists directory contents, with -la showing hidden files and detailed permissions. These three commands are the foundation of all shell navigation.
  4. Command History & Basic Keyboard Shortcuts — Bash remembers previously executed commands in history, navigable with Up/Down arrow keys and searchable with Ctrl+R. Essential shortcuts like Ctrl+C (interrupt), Ctrl+D (EOF/logout), and Ctrl+L (clear screen) dramatically speed up terminal work.

Why This Week Matters: Bash is the universal language of Unix/Linux automation — running on every server, container, and CI/CD pipeline in the world. The shebang, chmod +x, and the distinction between running a script directly vs. sourcing it are concepts referenced in every professional script you will ever read or write.

Code Example

#!/bin/bash
# This is a comment — Bash ignores everything after #
echo "Hello, Bash!"            # echo prints to standard output
echo "Shell: $SHELL"            # $SHELL is an environment variable

pwd                  # print current working directory
cd /tmp               # change to /tmp
ls -la                # list all files with details
cd -                  # return to the PREVIOUS directory
cd ~                  # go to home directory

history               # show command history
history | grep ls     # search history
!42                    # re-run command #42
!!                     # re-run the previous command

Week 2 — Variables & Basic Data Handling

Learn how Bash stores and uses data through variables and perform basic string and arithmetic operations.

Core Focus

Learning how Bash stores and uses data through variables and performing basic string and arithmetic operations.

Topics & Subtopics

  1. Declaring & Using Variables — Bash variables are untyped by default and store everything as strings internally. Assignment uses name=value with no spaces around the =; access uses $name or ${name} (braces recommended for clarity and safety inside strings).
  2. Environment Variables & export — Environment variables (like $HOME, $PATH, $USER) are set by the system and inherited by child processes. The export keyword promotes a local shell variable to an environment variable, making it visible to any scripts or programs the shell launches.
  3. String Operations & Quoting Rules — Double quotes ("$var") allow variable expansion; single quotes ('$var') treat everything literally with no expansion at all. Understanding this distinction is critical, as unquoted variables are a leading cause of Bash scripting bugs especially with filenames containing spaces.
  4. Arithmetic Operations — Bash arithmetic is performed inside $(( expression )) or with the let builtin. Integer-only by default; floating point requires external tools like bc or awk.

Why This Week Matters: The "no spaces around =" rule for variable assignment is Bash's most violated syntax rule — adding a space before or after = means Bash interprets it as a command, not an assignment. The double-quote vs single-quote distinction is the most important quoting rule in shell scripting and the root cause of the majority of real-world Bash bugs.

Code Example

name="Alice"              # no spaces around = !
age=30
echo "Name: $name"
echo "Age: ${age}"          # braces — recommended in strings
readonly PI=3.14159          # cannot be reassigned

echo "$HOME"                    # built-in environment variable
MY_VAR="custom value"
export MY_VAR                     # now visible to child processes

name="World"
echo "Hello, $name!"        # Hello, World! — double quotes expand
echo 'Hello, $name!'         # Hello, $name! — single quotes are literal
echo "Length: ${#name}"       # Length: 5 — string length
echo "${name^^}"               # WORLD — uppercase expansion

a=17; b=5
echo $((a + b))           # 22
echo $((a / b))            # 3  — integer division
echo $((a % b))             # 2  — modulo
result=$(echo "$a / $b" | bc -l)   # 3.40000... — float via bc
echo "$result"

Week 3 — Input/Output, Quoting & Special Variables

Read user input, redirect output, and master Bash's special parameter variables.

Core Focus

Reading user input, redirecting output streams, and mastering Bash's special parameter variables for argument handling.

Topics & Subtopics

  1. Reading User Input with read — The read builtin captures user input from standard input into a variable. Flags like -p (prompt), -s (silent, for passwords), and -a (read into an array) make it flexible for interactive scripts.
  2. Output Redirection (>, >>, 2>) — > redirects standard output to a file (overwriting); >> appends instead. 2> redirects standard error separately from standard output — essential for separating normal output from error logs in automation scripts.
  3. Pipes (|) & Command Chaining — The pipe operator (|) sends one command's standard output directly into the next command's standard input, enabling powerful command composition. && runs the next command only on success; || runs it only on failure.
  4. Special Variables ($0, $1, $#, $@, $?) — Bash provides special variables for script arguments: $0 is the script name, $1-$9 are positional arguments, $# is the argument count, $@ expands to all arguments, and $? holds the exit status of the last executed command.

Why This Week Matters: $? — the exit status of the last command — is the fundamental communication mechanism between commands and scripts in Unix. Every CI/CD pipeline decision, every && and || chain, and every if statement in Bash is ultimately checking exit codes. Understanding this model is the prerequisite for all professional shell scripting.

Code Example

read -p "Enter your name: " name
echo "Hello, $name!"

read -sp "Enter password: " password
echo
echo "Password length: ${#password}"

echo "Log entry 1" > log.txt        # overwrite
echo "Log entry 2" >> log.txt        # append
ls /nonexistent 2> errors.txt         # stderr only
ls /nonexistent &> all_output.txt     # stdout + stderr

ls -la | grep ".txt"
echo "Compiling..." && echo "Done!"    # && runs 2nd only on success
mkdir newdir || echo "mkdir failed!"    # || runs 2nd only on failure

#!/bin/bash
echo "Script name: $0"
echo "First arg: $1"
echo "Argument count: $#"
echo "All arguments: $@"
ls /tmp > /dev/null
echo "Exit status of ls: $?"     # 0 = success

Week 4 — Conditionals & Control Flow

Direct script execution using Bash's conditional structures and comparison tests.

Core Focus

Directing script execution using Bash's conditional structures, comparison operators, file tests, and case statements.

Topics & Subtopics

  1. if/elif/else & Test Conditions — Bash conditionals use if [[ condition ]]; then ... fi syntax. The double-bracket [[ ]] is the modern, recommended test syntax (over the older single-bracket [ ]), supporting safer string handling and pattern matching.
  2. Comparison Operators (String vs. Numeric) — Bash uses different operators for string comparison (==, !=, <, > inside [[ ]]) versus numeric comparison (-eq, -ne, -lt, -gt, -le, -ge) — confusing these is one of the most common Bash scripting mistakes.
  3. File Test Operators — Bash includes built-in operators for testing file properties: -e (exists), -f (regular file), -d (directory), -r/-w/-x (readable/writable/executable), and -s (non-empty) — essential for safe, defensive scripting before file operations.
  4. case Statements — The case statement provides clean, readable pattern matching against a single value — Bash's equivalent of switch, supporting wildcard patterns (*) and multiple matches per branch (pattern1|pattern2).

Why This Week Matters: The string vs. numeric operator distinction — using == for strings and -eq for numbers — is the most common source of subtle logic bugs in Bash. Using == to compare numbers performs a lexicographic string comparison, so [[ 10 == 9 ]] is false but [[ "10" == "9" ]] is also false for the wrong reason. This distinction must be internalized before writing any real conditional logic.

Code Example

score=82
if [[ $score -ge 90 ]]; then
    echo "Grade: A"
elif [[ $score -ge 80 ]]; then
    echo "Grade: B"
else
    echo "Grade: F"
fi

str1="apple"; str2="banana"
[[ "$str1" == "$str2" ]] && echo "Equal" || echo "Not equal"

num1=10; num2=20
[[ $num1 -lt $num2 ]] && echo "$num1 is less than $num2"
(( num1 < num2 )) && echo "Also valid: arithmetic context"

file="data.txt"
if [[ -e "$file" ]]; then echo "$file exists"; fi
if [[ -d "/tmp" ]]; then echo "/tmp is a directory"; fi
[[ -s "$file" ]] && echo "$file is non-empty"

read -p "Enter a fruit: " fruit
case "$fruit" in
    apple|pear)   echo "Pome fruit" ;;
    banana)       echo "Tropical fruit" ;;
    *)            echo "Unknown fruit" ;;
esac

Week 5 — Loops & Iteration

Automate repetitive tasks using Bash's loop constructs and loop control statements.

Core Focus

Automating repetitive tasks using Bash's full range of loop constructs and safe file-reading patterns.

Topics & Subtopics

  1. for Loops — Lists, Ranges & Files — for loops iterate over explicit word lists, numeric ranges ({1..10}), command output, or glob-matched filenames — making them extremely versatile for batch file processing, a common Bash use case.
  2. C-Style for Loops — Bash also supports a C-style for (( init; condition; increment )) loop, familiar to programmers from other languages, useful when explicit index control or non-1 step sizes are needed.
  3. while and until Loops — while loops run while a condition is true; until loops run until a condition becomes true (the inverse of while) — useful for polling patterns, like waiting for a file or service to become available.
  4. Loop Control — break, continue & Reading Files Line-by-Line — break exits a loop immediately; continue skips to the next iteration. The idiomatic, safe pattern for reading a file line-by-line is while IFS= read -r line; do ... done < file, which correctly handles spaces and backslashes.

Why This Week Matters: The while IFS= read -r line; do ... done < file pattern is one of the most important idioms in professional Bash — the naive alternative (for line in $(cat file)) silently word-splits on spaces, breaking any filename or log line containing spaces. Getting this right from the start prevents the most common batch-processing bug in shell scripts.

Code Example

for fruit in apple banana cherry; do
    echo "Fruit: $fruit"
done

for i in {1..5}; do echo "Number: $i"; done

for file in *.txt; do echo "Found: $file"; done    # glob-matched files

for (( i=0; i<10; i+=2 )); do
    echo "Even number: $i"
done

count=1
while [[ $count -le 3 ]]; do
    echo "While: $count"
    ((count++))
done

count=1
until [[ $count -gt 3 ]]; do echo "Until: $count"; ((count++)); done

while IFS= read -r line; do          # safe line-by-line file reading
    [[ "$line" == "STOP" ]] && break   # exit on sentinel
    [[ -z "$line" ]] && continue        # skip empty lines
    echo "Processing: $line"
done < "input.txt"

Week 6 — Functions & Script Arguments

Encapsulate reusable logic in functions and build scripts that accept flexible command-line arguments.

Core Focus

Encapsulating reusable logic in functions with proper local scoping, and building scripts that accept flexible command-line arguments with getopts.

Topics & Subtopics

  1. Defining & Calling Functions — Bash functions are declared with function name() { ... } or simply name() { ... } (both equivalent), and called just like any other command — by name, without parentheses. They must be defined before they're called in the script.
  2. Function Parameters & Return Values — Inside a function, $1, $2, etc. refer to the function's arguments (not the script's). Bash functions don't "return" values like other languages — return sets a numeric exit status (0–255); use echo and command substitution to get actual data back.
  3. local Variables & Scope — By default, variables in Bash are global, even inside functions — a major gotcha for newcomers. The local keyword scopes a variable to the function it's declared in, preventing accidental overwrites of global state.
  4. Parsing Script Arguments with getopts — getopts is Bash's built-in utility for parsing command-line flags (-v, -f filename) in a standardised, POSIX-compliant way — far more robust than manually checking $1, $2 for scripts accepting multiple optional flags.

Why This Week Matters: Global variables inside Bash functions are one of the most common sources of subtle bugs in longer scripts — a function modifying a variable called file or count accidentally overwrites the caller's variable of the same name. The local keyword is the mandatory fix, and making it a habit from this week forward is what separates professional shell scripts from fragile ones.

Code Example

greet() {
    echo "Hello, $1!"
}
greet "Alice"
greet "Bob"

add() {
    local result=$(( $1 + $2 ))    # local — scoped to this function
    echo "$result"                    # "return" data via echo
}
sum=$(add 5 3)
echo "Sum: $sum"                       # Sum: 8

counter=100
incrementLocal() {
    local counter=0                # shadows global — scoped here
    ((counter++))
    echo "Inside function: $counter"
}
incrementLocal                      # Inside function: 1
echo "Global unchanged: $counter"     # Global unchanged: 100

#!/bin/bash
while getopts "n:v" opt; do
    case $opt in
        n) name="$OPTARG" ;;
        v) verbose=true ;;
        *) echo "Usage: $0 -n name [-v]"; exit 1 ;;
    esac
done
echo "Name: $name, Verbose: $verbose"

Week 7 — Arrays, String Manipulation & Sourcing Scripts

Work with indexed and associative arrays, master advanced string operations, and structure multi-file Bash projects.

Core Focus

Working with indexed and associative arrays, mastering advanced string operations, and structuring multi-file Bash projects with sourcing.

Topics & Subtopics

  1. Indexed Arrays — Bash arrays are declared with name=(value1 value2 value3), indexed from 0. Access individual elements with ${array[index]}, all elements with ${array[@]}, and the count with ${#array[@]}.
  2. Associative Arrays (Bash 4+) — Associative arrays (declared with declare -A) work like dictionaries/hash maps, using string keys instead of numeric indices — essential for structured configuration data and lookup tables in scripts.
  3. Advanced Parameter Expansion & String Manipulation — Bash's parameter expansion syntax (${var#pattern}, ${var%pattern}, ${var//search/replace}) performs substring removal and replacement without spawning external processes like sed, making scripts faster and more portable.
  4. Sourcing Scripts (source / .) for Code Reuse — The source command (or its shorthand .) executes another script's content within the current shell rather than spawning a subshell — meaning variables and functions defined in the sourced file become available in the calling script, the basis for Bash "libraries."

Why This Week Matters: Parameter expansion for string manipulation — ${filename%.gz} to strip an extension, ${path##*/} to get the basename — runs entirely within the shell with zero process overhead. Replacing sed or awk calls with parameter expansion in hot loops can make scripts dramatically faster, and it's the idiomatic Bash approach that ShellCheck recommends.

Code Example

fruits=("apple" "banana" "cherry")
echo "${fruits[0]}"             # apple — 0-indexed
echo "${fruits[@]}"               # apple banana cherry — all elements
echo "${#fruits[@]}"                # 3 — array length
fruits+=("date")                     # append element

declare -A colors
colors["apple"]="red"
colors["banana"]="yellow"
echo "${colors[apple]}"                # red
for fruit in "${!colors[@]}"; do echo "$fruit: ${colors[$fruit]}"; done

filename="document.tar.gz"
echo "${filename%.gz}"             # document.tar — remove shortest suffix
echo "${filename%%.*}"              # document — remove longest suffix
echo "${filename/tar/zip}"           # document.zip.gz — replace first match
path="/usr/local/bin"
echo "${path##*/}"                    # bin — basename via longest prefix removal

# File: mathlib.sh
add() { echo $(( $1 + $2 )); }

# File: main.sh
source ./mathlib.sh
result=$(add 5 3)
echo "Result: $result"                 # Result: 8

Week 8 — File Handling & Filesystem Operations

Master file and directory manipulation, permissions, and filesystem-aware scripting patterns.

Core Focus

Mastering file and directory manipulation, Unix permissions, finding files with find, and inspecting disk usage.

Topics & Subtopics

  1. Creating, Copying, Moving & Deleting Files/Directories — mkdir -p creates nested directories without error if they exist; cp -r copies directories recursively; mv renames or moves; rm -rf deletes recursively and forcefully — the most destructive command in Bash, requiring extreme caution.
  2. File Permissions (chmod, chown) & the Permission Model — Unix permissions are represented as three triads (owner/group/other) each with read(4)/write(2)/execute(1) bits, summed into octal notation (e.g., 755). chmod changes permissions; chown changes ownership.
  3. Finding Files with find — find recursively searches directories matching criteria — name patterns, file type, size, modification time — and can execute commands on each match via -exec, making it one of the most powerful tools in the Bash toolkit.
  4. Disk Usage & Filesystem Inspection (du, df, stat) — du -sh summarises directory size in human-readable form; df -h shows filesystem-level disk space usage; stat reveals detailed metadata (permissions, timestamps, inode) about a specific file.

Why This Week Matters: rm -rf with a variable expansion bug — like rm -rf "$dir/" where $dir is empty — has deleted production servers. File test operators (-e, -d) and safe find usage with -exec are the professional patterns that prevent catastrophic data loss. This week's defensive habits protect both test environments and production systems.

Code Example

mkdir -p project/src/utils        # -p creates parents, no error if exists
cp -r project backup_project        # recursive copy
mv backup_project archived          # rename/move
rm -rf archived/temp                 # ⚠️ recursive force delete — use with CAUTION

chmod 755 script.sh           # rwxr-xr-x
chmod +x script.sh             # symbolic: add execute for all
chmod u+w,g-w file.txt          # add write for user, remove for group
ls -l script.sh                  # verify permissions

find . -name "*.log"                          # all .log files recursively
find . -type d -name "node_modules"            # directories named node_modules
find /tmp -mtime +7 -type f                      # files older than 7 days
find . -name "*.tmp" -exec rm {} \;                # delete each match

du -sh /var/log                  # summary, human-readable
df -h                              # disk free on all filesystems
stat file.txt                       # detailed metadata
du -ah . | sort -rh | head -5         # top 5 largest items

Week 9 — Error Handling & Defensive Scripting

Write robust, fail-safe Bash scripts using exit codes, strict mode, and proper error trapping.

Core Focus

Writing robust, fail-safe Bash scripts using exit codes, strict mode, signal trapping, and defensive guard clauses.

Topics & Subtopics

  1. Exit Codes & exit — Every command returns an exit code: 0 for success, any non-zero value (1–255) for failure, accessible via $?. Scripts can explicitly exit N to signal success/failure to whatever called them — critical for CI/CD pipelines and automation.
  2. Strict Mode — set -e, set -u, set -o pipefail — set -e exits immediately if any command fails; set -u errors on undefined variable usage (catching typos); set -o pipefail makes a pipeline fail if any command in it fails (not just the last). Together these form "Bash strict mode" — a professional best practice for every script.
  3. Trapping Signals & Cleanup with trap — trap registers a command or function to run when the script receives a signal (like EXIT, INT for Ctrl+C, or ERR) — the idiomatic way to guarantee cleanup (temp files, locks) happens even if the script is interrupted or crashes.
  4. Validating Input & Defensive Guard Clauses — Defensive Bash scripts validate arguments, check command availability (command -v), and verify preconditions before proceeding — failing fast with clear error messages rather than producing confusing downstream errors.

Why This Week Matters: set -euo pipefail — the three-option strict mode — is the single most important line in any production Bash script. Without set -e, a script happily continues after a command silently fails. Without set -u, a typo in a variable name runs silently with an empty string. Without pipefail, a failing command in a pipeline is invisible. All three are required for production-quality scripts.

Code Example

#!/bin/bash
ls /nonexistent_dir
echo "Exit code: $?"

if [[ ! -f "required_file.txt" ]]; then
    echo "Error: required file missing" >&2     # >&2 sends to stderr
    exit 1
fi

#!/bin/bash
set -euo pipefail            # strict mode — fail fast

#!/bin/bash
tempfile=$(mktemp)
cleanup() { rm -f "$tempfile"; echo "Cleaned up $tempfile"; }
trap cleanup EXIT              # runs on ANY script exit — even crashes

#!/bin/bash
set -euo pipefail
if [[ $# -lt 1 ]]; then
    echo "Usage: $0 " >&2; exit 1
fi
if ! command -v jq &> /dev/null; then
    echo "Error: 'jq' is required but not installed" >&2; exit 1
fi
echo "All preconditions met — proceeding"

Week 10 — Text Processing with grep, sed & awk

Master the three pillars of Unix text processing — searching, transforming, and extracting structured data.

Core Focus

Mastering the three pillars of Unix text processing — searching with grep, transforming with sed, and extracting structured data with awk.

Topics & Subtopics

  1. Pattern Searching with grep — grep searches text for lines matching a pattern (literal or regex). Common flags: -i (case-insensitive), -v (invert match), -r (recursive directory search), -n (show line numbers), -E (extended regex).
  2. Stream Editing with sed — sed (stream editor) transforms text line-by-line using commands, most commonly substitution (s/pattern/replacement/). The g flag replaces all matches per line (not just the first); -i edits files in place.
  3. Field Processing with awk — awk is a full text-processing language built for column/field-based data, automatically splitting each line into fields ($1, $2, ...) by whitespace (or a custom delimiter via -F). It's the go-to tool for extracting and computing on structured/tabular text.
  4. Combining Tools with Pipelines — Real-world Bash text processing chains grep, sed, awk, sort, uniq, and cut together via pipes — each tool doing one job well, composed into a powerful data-processing pipeline.

Why This Week Matters: grep + sed + awk is the trinity of Unix text processing — used in every DevOps pipeline, every log analysis script, and every data extraction task on Linux systems worldwide. Knowing all three and when to use each (grep for filtering, sed for substitution, awk for column extraction and aggregation) is the defining competency of a professional shell scripter.

Code Example

grep "error" app.log                     # find lines containing "error"
grep -i "ERROR" app.log                    # case-insensitive
grep -v "debug" app.log                     # invert: lines NOT containing debug
grep -rn "TODO" ./src                        # recursive, with line numbers
grep -E "^(WARN|ERROR)" app.log               # extended regex

echo "hello world" | sed 's/world/Bash/'       # hello Bash
echo "aaa bbb aaa" | sed 's/aaa/xxx/g'           # xxx bbb xxx — all matches
sed -i 's/foo/bar/g' config.txt                    # edit file in place
sed -n '2,4p' file.txt                               # print only lines 2-4

echo "Alice 30 Engineer" | awk '{print $1, $3}'    # Alice Engineer
awk -F',' '{print $2}' data.csv                      # comma-delimited
awk '{sum += $2} END {print "Total:", sum}' sales.txt  # aggregate
awk '$3 > 25 {print $1}' people.txt                    # conditional filter

# Real-world pipeline: top 5 IPs by request frequency
cat access.log | awk '{print $1}' | sort | uniq -c | sort -rn | head -5

Week 11 — Process Management, Environment & Networking Basics

Monitor and control running processes, manage shell environments, and perform basic network diagnostics from Bash.

Core Focus

Monitoring and controlling running processes, managing shell environments, and performing basic network diagnostics and HTTP requests from Bash.

Topics & Subtopics

  1. Process Inspection (ps, top, htop) & Job Control — ps aux lists all running processes with resource usage; top/htop provide a live, interactive view. Job control (& to background, jobs, fg, bg, Ctrl+Z) lets you manage multiple processes within a single terminal session.
  2. Killing Processes & Signals — kill sends signals to processes by PID; SIGTERM (15, default) requests graceful shutdown, SIGKILL (9) forces immediate termination, and SIGHUP (1) often triggers config reload. pkill/killall target processes by name instead of PID.
  3. Environment & Shell Configuration Files — .bashrc runs for every interactive non-login shell; .bash_profile/.profile runs for login shells. alias creates command shortcuts; PATH modifications here make custom scripts globally executable.
  4. Basic Networking from the Shell (curl, wget, ping) — curl and wget fetch data from URLs; curl is more flexible for APIs (supports custom headers, methods, JSON bodies), while wget excels at simple recursive downloads. ping and nc (netcat) test basic connectivity.

Why This Week Matters: Process management from the shell — checking if a service is running, sending SIGTERM for graceful shutdown, using SIGHUP to reload config — is how DevOps and SRE professionals manage production systems without rebooting. The curl -sf URL; echo $? pattern for health checking is the foundation of every monitoring script and deployment health check in the industry.

kill %1                                # terminate job 1

Week 12 — Advanced I/O: Process Substitution, Heredocs & File Descriptors

Master Bash's most advanced I/O redirection features for sophisticated data pipelines and templating.

Core Focus

Mastering Bash's most advanced I/O redirection features — heredocs, process substitution, custom file descriptors, and tee — for sophisticated data pipelines and templating.

Topics & Subtopics

  1. Here Documents (Heredocs) & Here Strings — A heredoc (<<EOF ... EOF) feeds multi-line text as input to a command, ideal for generating config files or multi-line prompts inline in a script. A here string (<<<) feeds a single string as input without needing a pipe.
  2. Process Substitution (<(...) and >(...)) — Process substitution lets you treat a command's output as if it were a file, passed via a temporary file descriptor — enabling powerful patterns like comparing the output of two commands directly with diff, without manual temp files.
  3. File Descriptors & Custom Redirection — Beyond standard stdin(0)/stdout(1)/stderr(2), Bash supports custom file descriptors (3-9) for advanced redirection scenarios — such as temporarily swapping stdout and stderr, or holding a file descriptor open across multiple commands.
  4. tee — Splitting Output to Multiple Destinations — tee reads from standard input and writes simultaneously to standard output and one or more files — essential for logging command output to a file while still seeing it live in the terminal.

Why This Week Matters: Process substitution — diff <(ls dir1) <(ls dir2) — eliminates the need for temporary files in hundreds of common scripting patterns. It's the feature that makes complex Bash pipelines possible without polluting the filesystem with intermediate files. Combined with heredocs for config templating, these are the tools that make professional Bash scripting genuinely elegant.

Code Example

cat < config.txt
server_name=myapp
port=8080
environment=production
EOF
echo "Config file generated"

grep "pattern" <<< "search this single string for pattern"

diff <(ls dir1) <(ls dir2)                     # compare directory listings
diff <(sort file1.txt) <(sort file2.txt)         # compare sorted versions

while read -r line; do echo "Got: $line"; done < <(curl -s https://example.com/data)

exec 3> custom_log.txt              # open FD 3 for writing
echo "Custom log entry" >&3            # write to FD 3
exec 3>&-                                # close FD 3

echo "Build started" | tee build.log                # prints AND writes
long_running_command | tee -a build.log                # -a appends
some_command 2>&1 | tee full_output.log                  # stdout + stderr
ls -la | tee listing.txt | grep ".sh"                       # tee mid-pipeline

Week 13 — Concurrency, Background Jobs & Parallel Execution

Run multiple tasks concurrently in Bash to dramatically speed up batch processing and automation workflows.

Core Focus

Running multiple tasks concurrently using background jobs, wait, GNU parallel, file locking, and named pipes for inter-process communication.

Topics & Subtopics

  1. Background Processes & wait — Appending & runs a command in the background, returning control immediately; $! captures the PID of the last backgrounded process. wait (with no args) blocks until all background jobs complete — the foundation of simple Bash parallelism.
  2. Parallel Execution with GNU parallel and xargs -P — xargs -P N runs up to N commands concurrently from a list of inputs; GNU parallel is a more feature-rich dedicated tool for parallelising shell commands across multiple cores, with built-in progress tracking and job control.
  3. Race Conditions & File Locking with flock — When multiple background processes write to the same shared file or resource concurrently, race conditions can corrupt data. flock provides advisory file locking, ensuring only one process accesses a critical section at a time.
  4. Named Pipes (FIFOs) for Inter-Process Communication — A named pipe (created with mkfifo) acts as a special file that connects two processes — one writes, another reads — enabling simple inter-process communication patterns without network sockets.

Why This Week Matters: Processing 100 files sequentially takes 100x longer than processing them in parallel — and Bash's background job pattern with wait makes parallelisation a 3-line change to any sequential loop. flock for preventing concurrent deployments from corrupting shared state is used in production deployment systems worldwide — it's the shell-level equivalent of a database transaction lock.

Code Example

process_file() { sleep 2; echo "Processed: $1"; }

process_file "file1.txt" &        # background
process_file "file2.txt" &        # background
process_file "file3.txt" &        # background
wait                                 # block until ALL background jobs finish
echo "All files processed"

# xargs -P: process files in parallel, 4 at a time
find . -name "*.jpg" | xargs -P 4 -I {} convert {} {}.png

# GNU parallel:
ls *.csv | parallel -j 4 'process_csv.sh {}'

(
    flock -x 200                          # acquire exclusive lock
    echo "Critical section: $(date)" >> shared.log
    sleep 1
) 200>/tmp/my.lock                          # lock file for FD 200

mkfifo /tmp/mypipe
echo "Hello via FIFO" > /tmp/mypipe &
cat < /tmp/mypipe                           # reads: "Hello via FIFO"
rm /tmp/mypipe                                # cleanup

Week 14 — Advanced Scripting Patterns & Tool Building

Apply professional patterns for building robust, distributable command-line tools in Bash.

Core Focus

Applying professional patterns for building robust, distributable command-line tools — proper CLI design, logging frameworks, layered configuration, and testable Bash with bats-core.

Topics & Subtopics

  1. Building a Proper CLI Tool — Argument Parsing & Help Text — Professional Bash CLI tools combine getopts with a structured usage() function, supporting --help, validating required vs. optional flags, and providing clear error messages — turning a script into a genuinely usable tool.
  2. Logging Frameworks & Verbosity Levels — A reusable logging function with severity levels (DEBUG, INFO, WARN, ERROR) — controllable via a verbosity flag — is a hallmark of professional Bash tooling, replacing scattered echo statements with consistent, filterable output.
  3. Configuration via Files & Environment Variable Precedence — Professional scripts support layered configuration: hardcoded defaults, overridable by a config file, overridable by environment variables, overridable by CLI flags — giving users maximum flexibility while keeping sensible defaults.
  4. Writing Testable Bash with bats-core — bats-core (Bash Automated Testing System) is the standard framework for unit-testing Bash scripts and functions, using a syntax similar to other testing frameworks (@test, assertions) — bringing real test coverage to shell scripting.

Why This Week Matters: The layered configuration pattern — defaults < config file < environment variables < CLI flags — is the standard design of every professional Unix tool (from git to docker to kubectl). Building your own tools with this pattern makes them usable by others without modification. bats-core tests are what separate "scripts that work on my machine" from "scripts that ship to production."

Code Example

#!/bin/bash
usage() {
    cat <&2
}
log INFO "Starting process"
log ERROR "Something failed"

PORT_DEFAULT=8080
[[ -f "./app.conf" ]] && source ./app.conf
PORT="${PORT:-$PORT_DEFAULT}"
while getopts "p:" opt; do [[ $opt == "p" ]] && PORT="$OPTARG"; done
echo "Using port: $PORT"

# File: mathlib.bats
#!/usr/bin/env bats
load 'mathlib.sh'
@test "add returns correct sum" {
    result=$(add 2 3)
    [ "$result" -eq 5 ]
}
# Run with: bats mathlib.bats
