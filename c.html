About This Curriculum

This is a 16-week, project-driven C curriculum designed to take a complete beginner to a professional-level systems programmer. The roadmap moves from compilation, syntax, variables, and basic I/O through control flow, functions, arrays, and strings, into pointers, structs, file handling, and dynamic memory, all the way to function pointers, state machines, POSIX threads, and three complete capstone projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics including the compilation pipeline and primitive types, Phase 2 covers control flow, functions, and data structures, Phase 3 covers intermediate and advanced core topics like pointers, structs, file I/O, and dynamic memory, and Phase 4 covers master-level function pointers, linked lists, concurrency, and production systems programming.

By Week 16, you will have built three complete, portfolio-ready applications and be confident building production-grade C programs — from command-line tools to multithreaded systems.

Level: Beginner → Master  |  Duration: 16 weeks  |  Stack: C (C11/C17), GCC, POSIX  |  Prerequisites: Basic programming logic

Phase 1 — Absolute Basics

Weeks 1–4: Fundamentals, data types, operators, and basic I/O.

Week 1 — Setting Up & Your First Program

Environment setup, compilation pipeline, and basic program structure.

Core Focus

Setting up your environment, understanding the compilation pipeline, and writing your very first C program.

Topics & Subtopics

  1. The Compilation Process — C code is translated into machine instructions through preprocessing, compiling, assembling, and linking. Understanding this pipeline explains errors like "undefined reference" and "implicit declaration" that you will encounter later.
  2. Program Structure (main, headers) — Every C program starts execution at main(), and headers like stdio.h give access to library functions. This is the skeleton that every C file shares — without it, nothing compiles.
  3. printf and Basic Output — printf formats and prints text to the console using format specifiers like %d, %f, and %s. It is the primary way to see what your program is doing during development and debugging.

Why This Week Matters: C is the only language where you need to understand the compilation pipeline to effectively debug your programs. Knowing that preprocessing handles #include, that the linker resolves function references, and that return 0 communicates success to the OS — these mental models prevent hours of confusion in every future week.

Code Example

#include <stdio.h>      // Provides printf()

int main(void) {
    printf("Hello, World!\n"); // \n moves to next line
    return 0;                  // Tells OS the program succeeded
}

Week 2 — Variables & Primitive Data Types

Storing and labeling data in memory.

Core Focus

Understanding how C stores data in memory and working with all primitive types and the sizeof operator.

Topics & Subtopics

  1. Variables & Declaration — A variable is a named location in memory that holds a value of a specific type. In C, you must declare a type before using a variable — there is no type inference like in modern languages.
  2. Primitive Types (int, float, double, char) — C provides built-in types for whole numbers, decimals, and single characters. Each type uses a different amount of memory and has a different value range — critical knowledge for embedded and systems work.
  3. sizeof Operator — sizeof tells you how many bytes a type or variable occupies in memory at compile time. This is essential for understanding memory usage, writing portable code, and working with pointers correctly later on.

Why This Week Matters: Unlike high-level languages, C gives you no automatic type coercion and no runtime type information. Understanding exactly how many bytes each type occupies — and what happens when you overflow them — is the difference between a C programmer who writes correct code and one who writes code that silently corrupts data.

Code Example

#include <stdio.h>

int main(void) {
    int age = 25;
    float price = 9.99f;
    char grade = 'A';
    printf("Size of int: %zu bytes\n", sizeof(age));
    printf("Size of float: %zu bytes\n", sizeof(price));
    printf("Age: %d, Price: %.2f, Grade: %c\n", age, price, grade);
    return 0;
}

Week 3 — Operators & Type Casting

Manipulating data with arithmetic and logical operations.

Core Focus

Manipulating data with arithmetic and logical operations, and explicitly controlling type conversions.

Topics & Subtopics

  1. Arithmetic & Assignment Operators — These perform math (+ - * / %) and update variables (+= -=) in a single step. The modulus % operator returns the remainder of integer division — essential for range-wrapping and even/odd checks.
  2. Relational & Logical Operators — Relational operators (== != < >) compare values and return true/false (1 or 0 in C), while logical operators (&& || !) combine multiple conditions. These form the basis of all decision-making in code.
  3. Type Casting — Casting explicitly converts a value from one type to another, such as int to float. Without it, integer division silently truncates decimal results — one of the most common C bugs for beginners.

Why This Week Matters: Integer division truncation is responsible for countless silent bugs in C programs. 7 / 2 returning 3 instead of 3.5 is correct C behavior — but only if you know it's happening. Mastering explicit casting and understanding when implicit promotion occurs is a foundational C skill.

Code Example

#include <stdio.h>

int main(void) {
    int a = 7, b = 2;
    float result = (float)a / b; // Cast prevents integer division
    printf("7 / 2 as float = %.2f\n", result);
    printf("7 %% 2 = %d\n", a % b);
    printf("a > b: %d\n", a > b);       // 1 (true)
    printf("a == 7 && b < 5: %d\n", a == 7 && b < 5); // 1
    return 0;
}

🛠 Weekly Task

Week 4 — Input Handling & Constants

Making programs interactive and protecting fixed values.

Core Focus

Making programs interactive with scanf and protecting fixed values using constants and preprocessor macros.

Topics & Subtopics

  1. scanf for Input — scanf reads formatted input from the keyboard into variables using format specifiers. Always match the specifier to the variable's type and always pass the variable's address using & to avoid undefined behavior.
  2. Constants (const, #define) — Constants are values that cannot change after being set, improving safety and readability. #define creates a preprocessor macro (no type, no scope), while const creates a typed, scoped constant — prefer const in modern C.
  3. Format Specifiers Deep Dive — Format specifiers like %d, %f, %c, and %s tell printf/scanf how to interpret data. Mismatching them with the wrong type causes undefined behavior — one of the most dangerous classes of C bugs.

Why This Week Matters: The & address-of operator in scanf is most students' first encounter with the concept of memory addresses in C — the same concept that underpins all of pointer theory in Week 9. Getting comfortable with "passing an address" here makes pointers dramatically less confusing later.

Code Example

#include <stdio.h>
#define PI 3.14159   // Preprocessor constant — no type, no semicolon

int main(void) {
    int radius;
    printf("Enter radius: ");
    scanf("%d", &radius);      // & gives the memory address of radius
    printf("Area = %.2f\n", PI * radius * radius);
    return 0;
}

Phase 2 — Control Flow & Logic

Weeks 5–8: Functions, modules, and core data structures.

Week 5 — Conditional Statements

Making decisions in code.

Core Focus

Making decisions in code using C's full set of conditional and branching constructs.

Topics & Subtopics

  1. if / else if / else — These statements execute different code blocks depending on whether a condition is true or false. They are the foundation of all branching logic in C and every other programming language.
  2. switch Statements — switch compares one variable against multiple constant values, offering a cleaner alternative to long if-else chains. The break statement at the end of each case is mandatory — forgetting it causes fall-through to the next case.
  3. Ternary Operator — The ternary operator (condition ? a : b) is a compact one-line way to write simple if-else logic. It is best used for short, readable value assignments — avoid nesting ternaries as it destroys readability.

Why This Week Matters: The switch statement's fall-through behavior — where execution continues into the next case unless you write break — is one of C's most famous gotchas. Understanding it here prevents real bugs. The ternary operator is used constantly in C for compact conditional assignments, especially inside printf calls.

Code Example

#include <stdio.h>

int main(void) {
    int score = 85;
    char grade = (score >= 90) ? 'A' : (score >= 80) ? 'B' : 'C';
    printf("Grade: %c\n", grade);

    int day = 3;
    switch (day) {
        case 1: printf("Monday\n");    break;
        case 2: printf("Tuesday\n");   break;
        case 3: printf("Wednesday\n"); break;
        default: printf("Other day\n");
    }
    return 0;
}

Week 6 — Loops & Iteration

Repeating actions efficiently.

Core Focus

Repeating actions efficiently using all three of C's loop constructs and controlling iteration precisely.

Topics & Subtopics

  1. for Loops — A for loop repeats code a known number of times using an initializer, condition, and increment. It is ideal for counting, array traversal, and any scenario where the number of iterations is determined before the loop starts.
  2. while and do-while Loops — while repeats as long as a condition holds, checking before each iteration; do-while checks after, guaranteeing at least one run. Choose based on whether the first execution should be conditional or guaranteed.
  3. break and continue — break exits a loop immediately, while continue skips the rest of the current iteration and jumps to the next check. They give you fine-grained control over loop flow without needing complex condition expressions.

Why This Week Matters: Loops over arrays and strings are the most fundamental operation in all of C programming. Every algorithm you will write in Weeks 8–16 — searching, sorting, counting words, processing files — is built on a loop. Getting completely fluent with for, while, break, and continue now is essential.

Code Example

#include <stdio.h>

int main(void) {
    // for loop with continue
    for (int i = 1; i <= 5; i++) {
        if (i == 3) continue; // Skip printing 3
        printf("%d ", i);
    }
    printf("\n");

    // do-while — runs at least once
    int n = 0;
    do {
        printf("n = %d\n", n);
        n++;
    } while (n < 3);

    return 0;
}

Week 7 — Functions

Organizing code into reusable blocks.

Core Focus

Organizing code into reusable, testable, and maintainable blocks using C functions.

Topics & Subtopics

  1. Function Declaration & Definition — A function is a named block of code that performs a task and can be called repeatedly. Declaring a prototype (signature without body) at the top of the file lets you call a function before it is defined below — standard practice in C.
  2. Parameters & Return Values — Functions can accept input via parameters and send back a result via return. This enables modular, reusable logic — the fundamental building block of every large C program.
  3. Pass by Value vs. Scope — C passes arguments by value by default, meaning functions work on copies of the original variables, not the originals themselves. Variables declared inside a function are local and disappear when the function returns — understanding scope prevents "variable undefined" errors.

Why This Week Matters: "Pass by value" is C's default and a constant source of confusion for new programmers who expect a function to modify their variable. Understanding that C copies the value — not the reference — is the exact conceptual gap that makes pointers (Week 9) make sense. Functions are also the unit of organization in all real C programs.

Code Example

#include <stdio.h>

// Function prototype — declared before main
int square(int n);

int main(void) {
    printf("Square of 5: %d\n", square(5));
    printf("Square of 9: %d\n", square(9));
    return 0;
}

int square(int n) {     // Takes a COPY of the argument
    return n * n;       // Original variable in main is untouched
}

Week 8 — Arrays & Strings

Storing and manipulating collections of data.

Core Focus

Storing collections of data using arrays and understanding how C represents strings as null-terminated character arrays.

Topics & Subtopics

  1. One-Dimensional Arrays — An array stores multiple values of the same type in contiguous memory, accessed via an index starting at 0. Arrays have a fixed size once declared — there is no automatic resizing like a Python list or JavaScript array.
  2. Multi-Dimensional Arrays — A 2D array is essentially an array of arrays, useful for representing grids, matrices, or tables. Access uses two indices, like grid[row][col], and they are stored in row-major order in memory.
  3. Strings as Char Arrays — In C, a string is just an array of characters ending with a null terminator (\0). Functions like strlen, strcpy, and strcmp from string.h operate on this convention — knowing the null terminator exists is critical for avoiding buffer overflows.

Why This Week Matters: Buffer overflows — writing past the end of a char array — are the most infamous class of C security vulnerabilities. Understanding that strings in C are just char arrays with a \0 at the end, and that C will not stop you from writing past the end, is the knowledge that prevents the most dangerous class of C bugs.

Code Example

#include <stdio.h>
#include <string.h>

int main(void) {
    // Integer array
    int scores[5] = {85, 92, 78, 95, 88};
    printf("First score: %d\n", scores[0]);

    // String as char array with null terminator
    char name[20] = "Claude";
    printf("Name: %s, Length: %zu\n", name, strlen(name));

    // 2D array
    int grid[2][3] = {{1, 2, 3}, {4, 5, 6}};
    printf("grid[1][2] = %d\n", grid[1][2]); // 6
    return 0;
}

Week 9 — Pointers: The Heart of C

Direct memory manipulation using pointers.

Core Focus

Understanding and applying C's most powerful and most feared feature — direct memory manipulation through pointers.

Topics & Subtopics

  1. Pointer Basics — A pointer is a variable that stores the memory address of another variable. The & operator gets an address, and * dereferences a pointer to access the value stored at that address.
  2. Pointers & Functions (Pass by Reference) — Passing a pointer to a function allows the function to modify the original variable, simulating "pass by reference". This is essential for functions that need to return multiple values or modify data in place.
  3. Pointer Arithmetic — Adding to a pointer moves it forward by the size of its pointed-to type, not just one byte. This is how C traverses arrays internally — an array name is itself a pointer to its first element.

Why This Week Matters: Pointers are the feature that separates C from every other high-level language — and the reason C is used for operating systems, embedded systems, and performance-critical code. Every concept in the remaining weeks — dynamic memory, linked lists, function pointers, threads — requires pointer fluency. There is no shortcut here: you must understand pointers completely.

Code Example

#include <stdio.h>

void increment(int *ptr) { // Receives the address of the variable
    (*ptr)++;               // Dereference to modify the original
}

int main(void) {
    int x = 10;
    int *p = &x;           // p stores the address of x
    printf("x = %d, address = %p\n", x, (void *)p);
    increment(&x);
    printf("x after increment = %d\n", x); // 11

    // Pointer arithmetic — moves by sizeof(int)
    int arr[3] = {10, 20, 30};
    int *ap = arr;
    printf("arr[1] = %d\n", *(ap + 1)); // 20
    return 0;
}

Week 10 — Structs, Unions & OOP Patterns in C

Modeling real-world entities — C's equivalent of classes.

Core Focus

Modeling real-world entities using structs and unions, and applying object-oriented-like patterns in C.

Topics & Subtopics

  1. struct — Custom Data Types — A struct groups related variables of different types under one name, similar to a class's fields in OOP. This is C's primary tool for object-like data modeling and is used in virtually every real C program.
  2. Struct Pointers & typedef — typedef creates an alias for a struct type, removing the need to repeat the struct keyword everywhere. Struct pointers let you pass large objects efficiently to functions and modify them in place using the -> arrow operator.
  3. Unions — A union stores different data types in the same memory location, with only one member valid at a time. It saves memory when you need to represent one of several possible value types — commonly used in network protocol headers and embedded systems.

Why This Week Matters: Structs are the foundation of every real C data model — from Linux kernel data structures to SQLite's internal types. The -> arrow operator for accessing struct members through a pointer is syntax you will use constantly in all remaining weeks and in every real C codebase you encounter.

Code Example

#include <stdio.h>

typedef struct {
    char name[30];
    int age;
} Person;  // typedef removes need to write "struct Person" everywhere

void birthday(Person *p) {
    p->age++;  // -> dereferences pointer AND accesses member
}

int main(void) {
    Person alice = {"Alice", 30};
    printf("Before: %s is %d\n", alice.name, alice.age);
    birthday(&alice);
    printf("After:  %s is %d\n", alice.name, alice.age);
    return 0;
}

Week 11 — File Handling & Error Handling

Persisting data and writing robust programs.

Core Focus

Persisting data to disk and writing robust, production-grade programs that handle failure gracefully.

Topics & Subtopics

  1. File I/O (fopen, fprintf, fscanf, fclose) — fopen opens a file in a mode (read/write/append), and you must always fclose it when done or data may not be written. fprintf/fscanf write and read formatted data exactly like printf/scanf.
  2. Error Detection (errno, return codes) — Most C library functions signal failure through return values like NULL or -1, and the global errno variable provides more detail on exactly what went wrong. Always check these return values before trusting a result.
  3. perror and Defensive Programming — perror prints a human-readable description of the last error alongside your custom message — essential for debugging file and system call failures. Checking pointers and inputs before use prevents the most common class of C crashes.

Why This Week Matters: C gives you no exceptions — there is no try-catch. Every function can fail, and in C, it is entirely your responsibility to check every return value. Programs that skip error checking appear to work in testing and crash in production. This week instills the defensive programming mindset that characterizes professional C developers.

Code Example

#include <stdio.h>

int main(void) {
    FILE *fp = fopen("data.txt", "w");
    if (fp == NULL) {           // ALWAYS check fopen's return value
        perror("File error");   // Prints: "File error: No such file..."
        return 1;
    }
    fprintf(fp, "Score: %d\n", 95);
    fclose(fp);                 // ALWAYS close — flushes buffers to disk

    // Read it back
    fp = fopen("data.txt", "r");
    char buf[100];
    fgets(buf, sizeof(buf), fp);
    printf("Read: %s", buf);
    fclose(fp);
    return 0;
}

Week 12 — Dynamic Memory & Standard Libraries

Manual memory management and the C standard library ecosystem.

Core Focus

Managing heap memory manually and leveraging the C standard library for common operations.

Topics & Subtopics

  1. malloc, calloc, realloc, free — These functions allocate memory on the heap at runtime, letting you create data structures whose size is not known at compile time. Every allocation must be paired with a free to avoid memory leaks — one of the most costly bugs in long-running C programs.
  2. Common Standard Libraries (stdlib.h, math.h, time.h) — stdlib.h provides memory and conversion utilities, math.h provides mathematical functions, and time.h handles dates and randomness seeding. Knowing the standard library prevents reinventing the wheel in every project.
  3. Memory Leaks & Dangling Pointers — A memory leak occurs when allocated memory is never freed; a dangling pointer points to memory that has already been freed and may be reallocated to something else. Tools like Valgrind detect both classes of bugs before they reach production.

Why This Week Matters: Dynamic memory allocation is what makes linked lists, dynamic arrays, hash maps, and trees possible in C — every data structure in Week 14 depends on it. Memory leaks in server software cause gradual memory exhaustion and crashes hours or days after deployment. Understanding malloc/free correctly is a defining skill of a competent C programmer.

Code Example

#include <stdlib.h>
#include <stdio.h>

int main(void) {
    int n;
    printf("How many numbers? ");
    scanf("%d", &n);

    int *arr = malloc(n * sizeof(int)); // Heap allocation — runtime size
    if (!arr) { perror("malloc"); return 1; }

    for (int i = 0; i < n; i++) {
        scanf("%d", &arr[i]);
    }
    printf("First: %d\n", arr[0]);

    free(arr);   // ALWAYS release heap memory
    arr = NULL;  // Prevent dangling pointer
    return 0;
}

Week 13 — Function Pointers & Decorator Patterns

Treating functions as data to build flexible, wrapped behavior.

Core Focus

Treating functions as first-class data using function pointers and building decorator-style behavior wrapping — C's equivalent of higher-order functions.

Topics & Subtopics

  1. Function Pointers — A function pointer stores the address of a function, allowing it to be passed around, stored in structs, and called indirectly. This enables callbacks, pluggable behavior, and runtime-selectable algorithms — the foundation of C plugin systems.
  2. Decorator-Style Wrapping — By wrapping a function pointer inside another function that adds logic (timing, logging, validation) before and after calling it, you replicate Python-style decorators in C. This pattern is used in Linux's kernel module system and many embedded frameworks.
  3. qsort and Callbacks in the Standard Library — qsort is the standard library's real-world example of function pointers — it accepts a custom comparator function to sort any data type. Understanding it cements how callbacks work and is directly applicable to everyday C programming.

Why This Week Matters: Function pointers are how C achieves runtime polymorphism — the same mechanism used to implement virtual function tables (vtables) in C++ and object systems in languages like Python and Ruby. Every callback-based API in C (signal handlers, qsort, GTK events) uses function pointers. This is master-level C.

Code Example

#include <stdio.h>
#include <stdlib.h>

void greet(void) { printf("Hello!\n"); }
void farewell(void) { printf("Goodbye!\n"); }

// "Decorator" — wraps any void function with logging
void logger(void (*fn)(void)) {
    printf("[LOG] Calling function...\n");
    fn();                          // Call through the function pointer
    printf("[LOG] Done.\n");
}

// Comparator for qsort — function pointer callback
int compare_ints(const void *a, const void *b) {
    return (*(int *)a - *(int *)b);
}

int main(void) {
    logger(greet);     // Pass function as argument
    logger(farewell);

    int nums[] = {5, 2, 8, 1, 9};
    qsort(nums, 5, sizeof(int), compare_ints);
    for (int i = 0; i < 5; i++) printf("%d ", nums[i]);
    return 0;
}

Week 14 — Generators via State Machines & Linked Lists

Lazy iteration and dynamic data structures in C.

Core Focus

Simulating generator-style lazy iteration using state structs and building dynamic data structures with linked lists.

Topics & Subtopics

  1. Generator-Style Iterators (State Structs) — Since C lacks yield, a generator is simulated with a struct holding state (current value, step, limit) and a next() function that advances it. This produces values lazily, one at a time — the same behavior as Python generators but implemented manually.
  2. Singly Linked Lists — A linked list is a chain of nodes where each node holds data and a pointer to the next node. It allows dynamic growth without resizing or copying, unlike arrays — and is the foundational data structure for stacks, queues, and hash table chaining.
  3. Recursion — A recursive function calls itself to solve smaller instances of a problem until reaching a base case. It is a natural fit for tree-like and divide-and-conquer problems and is commonly used for traversing linked lists and implementing merge sort.

Why This Week Matters: Linked lists are one of the most commonly asked-about data structures in C interviews — and understanding how to manage malloc/free correctly for each node, avoiding leaks and dangling pointers, demonstrates full mastery of dynamic memory from Week 12. State-machine generators are used in embedded systems and network protocol parsers everywhere.

Code Example

#include <stdio.h>
#include <stdlib.h>

// Generator-style counter using a state struct
typedef struct { int current; int limit; } Counter;
int has_next(Counter *c) { return c->current < c->limit; }
int next_value(Counter *c) { return c->current++; } // "yields" next number

// Linked list node
typedef struct Node { int data; struct Node *next; } Node;
Node *create_node(int data) {
    Node *n = malloc(sizeof(Node));
    n->data = data; n->next = NULL;
    return n;
}

int main(void) {
    Counter c = {0, 5};
    while (has_next(&c)) printf("%d ", next_value(&c));
    printf("\n");
    return 0;
}

Week 15 — Concurrency with POSIX Threads

Running code in parallel safely using pthreads.

Core Focus

Achieving true parallelism using POSIX threads and protecting shared state from race conditions with mutexes.

Topics & Subtopics

  1. Creating Threads (pthread_create) — pthread_create spawns a new thread that runs a function concurrently with the main program. pthread_join waits for that thread to finish before the main program continues — preventing the program from exiting before threads complete.
  2. Race Conditions & Mutexes — A race condition occurs when multiple threads access shared data simultaneously without coordination, causing unpredictable and incorrect results. A mutex (mutual exclusion lock) ensures only one thread touches shared data at a time — pthread_mutex_lock and pthread_mutex_unlock bracket the critical section.
  3. Thread-Safe Design Principles — Designing thread-safe code means minimizing shared mutable state and protecting everything that must be shared with proper locking. This prevents deadlocks (two threads each waiting for the other's lock) and data corruption in production systems.

Why This Week Matters: Multithreading is how servers handle thousands of simultaneous connections and how scientific computing utilizes all CPU cores. Race conditions are invisible in testing — a shared counter incremented by two threads without a mutex may give the correct answer 99% of the time and silently produce a wrong answer 1% of the time in production. Mutexes are the solution.

Code Example

#include <pthread.h>
#include <stdio.h>

pthread_mutex_t lock = PTHREAD_MUTEX_INITIALIZER;
int shared_counter = 0;

void *increment_task(void *arg) {
    for (int i = 0; i < 100000; i++) {
        pthread_mutex_lock(&lock);    // Only one thread at a time
        shared_counter++;
        pthread_mutex_unlock(&lock);  // Release for next thread
    }
    return NULL;
}

int main(void) {  // Compile with: gcc -o prog main.c -lpthread
    pthread_t t1, t2;
    pthread_create(&t1, NULL, increment_task, NULL);
    pthread_create(&t2, NULL, increment_task, NULL);
    pthread_join(t1, NULL);
    pthread_join(t2, NULL);
    printf("Counter: %d\n", shared_counter); // Should be 200000
    return 0;
}

How to Use This Curriculum

  • Always compile with -Wall -Wextra -g flags — warnings are errors in disguise, and -g enables debugging symbols for GDB.
  • Push every weekly project to a public GitHub repository — your commit history is visible proof of consistent, serious progress.
  • Run Valgrind or AddressSanitizer (-fsanitize=address) on every program from Week 9 onward — do not trust code you haven't checked for memory errors.
  • By Week 16, you should have three complete, compilable, leak-free projects and the confidence to read and contribute to real-world C codebases like the Linux kernel, Redis, or SQLite.
