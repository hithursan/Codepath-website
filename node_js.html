About This Curriculum

This is a 16-week, phase-based Node.js curriculum designed to take a complete beginner to a professional-level Node.js developer. The roadmap moves from the Node.js runtime, variables, npm, and data structures through functions, closures, async/await, and core built-in modules, into OOP, Express.js, MongoDB, authentication, and error handling, all the way to generators, Worker Threads, clustering, testing, Docker, and three complete capstone projects.

The curriculum is organized into four phases: Phase 1 covers the Node.js runtime and absolute JavaScript basics, Phase 2 covers control flow, functions, modules, and asynchronous programming, Phase 3 covers OOP, Express.js, MongoDB, and authentication, and Phase 4 covers master-level concurrency, testing, deployment, and full project work.

By Week 16, you will have built three complete, portfolio-ready applications and be confident building production-grade Node.js applications from scratch.

Level: Beginner → Master  |  Duration: 16 weeks  |  Stack: Node.js 20+ (LTS), npm, Express, MongoDB  |  Prerequisites: Basic JavaScript knowledge

Week 1 — Environment Setup & The Node.js Runtime

Understanding what Node.js is, installing it, and running your first scripts outside the browser.

Core Focus

Understanding what Node.js is, installing it, and running your first scripts outside the browser.

Topics & Subtopics

  1. What is Node.js & Why It Matters — Node.js is a JavaScript runtime built on Chrome's V8 engine that lets you run JavaScript outside the browser — on servers, in CLI tools, and in build scripts. Its non-blocking, event-driven architecture makes it exceptionally efficient at handling many concurrent connections.
  2. Installing Node.js & Using NVM — NVM (Node Version Manager) lets you install and switch between multiple Node.js versions on the same machine — essential because different projects often require different Node versions.
  3. The REPL & Running Scripts — The Node REPL (Read-Eval-Print Loop) is an interactive shell for testing JavaScript snippets instantly, accessed by typing node with no filename. For real programs, you save code in .js files and execute them with node filename.js.
  4. Global Objects: process, __dirname, __filename — Node.js provides special global variables not found in browsers. process gives you info about the current running program, while __dirname and __filename give you absolute paths to the current file and folder.

Why This Week Matters: Node.js is not just a language — it's a runtime with a completely different model from browser JavaScript. Understanding the event-driven, non-blocking architecture from Day 1 is the mental foundation that makes every subsequent concept — async/await, streams, HTTP servers — feel logical rather than arbitrary.

Code Example

// hello.js — Run with: node hello.js
// Unlike browser JS, Node.js has no `window` or `document`
console.log("Hello from Node.js!");
console.log("Node version:", process.version);
console.log("Current directory:", process.cwd());

// info.js — Node-specific globals
console.log(__dirname);                          // Absolute path to this file's folder
console.log(__filename);                         // Absolute path to this file itself
console.log(process.argv);                       // Array of command-line arguments
console.log(process.env.NODE_ENV || "development"); // Environment variable

Week 2 — Variables, Data Types & Operators

Mastering JavaScript's variable declarations, primitive types, and operators within the Node.js context.

Core Focus

Mastering JavaScript's variable declarations, primitive types, and operators within the Node.js context.

Topics & Subtopics

  1. let, const & var — Modern JavaScript uses const for values that never get reassigned and let for variables that will change. var is the legacy keyword with confusing function-scoping rules and should be avoided in new code.
  2. Primitive Data Types — JavaScript has seven primitive types: string, number, boolean, null, undefined, bigint, and symbol. Unlike many languages, JavaScript uses a single number type for both integers and decimals.
  3. Template Literals — Template literals (backtick strings) allow embedding expressions directly inside strings using ${'{}'} syntax, eliminating clunky string concatenation. They also natively support multi-line strings without escape characters.
  4. Operators (Comparison, Logical, Nullish Coalescing) — JavaScript distinguishes between loose equality (==, which type-converts) and strict equality (===, which doesn't) — always prefer ===. The nullish coalescing operator (??) provides a fallback only for null/undefined, unlike || which also triggers on falsy values like 0.

Why This Week Matters: The ?? vs || distinction is one of the most common sources of subtle bugs in Node.js APIs — returning 0 or an empty string that gets incorrectly treated as "missing." Understanding strict equality and the nullish coalescing operator prevents these bugs from day one.

Code Example

const appName = "MyApp";   // Cannot be reassigned
let userCount = 0;          // Can be reassigned
userCount = userCount + 1;  // Valid: 1

const name      = "Node.js";        // string
const version   = 20;               // number
const isStable  = true;             // boolean
const notSet    = undefined;        // undefined
const empty     = null;             // null — intentional absence
const bigNumber = 123456789012345n; // bigint

const user  = "Alice";
const score = 95;
const message = `Hello, ${user}! Your score is ${score}/100.`;
console.log(message);

const userInput = 0;
console.log(userInput || 10);  // 10 — || treats 0 as falsy (bug!)
console.log(userInput ?? 10);  // 0  — ?? only falls back on null/undefined

Week 3 — npm, Packages & Project Structure

Initializing Node.js projects, managing dependencies with npm, and understanding the module ecosystem.

Core Focus

Initializing Node.js projects, managing dependencies with npm, and understanding the module ecosystem.

Topics & Subtopics

  1. npm init & package.json — Every Node.js project starts with a package.json file, which tracks the project's metadata, dependencies, and scripts. npm init -y generates this file instantly with sensible defaults, ready for customization.
  2. Installing & Managing Packages — npm install downloads packages from the npm registry into node_modules and records them in package.json. The --save-dev flag marks a package as a development-only dependency (like testing tools).
  3. CommonJS vs ES Modules — Node.js supports two module systems: the original CommonJS (require/module.exports) and modern ES Modules (import/export, set via "type": "module" in package.json). ES Modules are now the recommended standard for new projects.
  4. package.json Scripts — The scripts section of package.json defines shortcuts for common commands like starting the server or running tests. You execute them with npm run <script-name> or just npm start/npm test for those two special names.

Why This Week Matters: npm is the world's largest software registry with over 2 million packages. Understanding the difference between CommonJS and ES Modules — and when to use each — prevents hours of confusing import errors. Mastering package.json scripts is the foundation of every Node.js developer's workflow.

Code Example

// Terminal setup
// npm init -y              → Creates package.json
// npm install express       → Production dependency
// npm install --save-dev nodemon → Dev-only dependency

// math.mjs — ES Module syntax
export function add(a, b) { return a + b; }
export default function multiply(a, b) { return a * b; }

// app.mjs — importing
import multiply, { add } from './math.mjs';
console.log(add(2, 3), multiply(2, 3)); // 5 6

Week 4 — Arrays, Objects & JSON

Working with JavaScript's core data structures and the JSON format that powers nearly all Node.js APIs.

Core Focus

Working with JavaScript's core data structures and the JSON format that powers nearly all Node.js APIs.

Topics & Subtopics

  1. Arrays & Common Array Methods — Arrays store ordered lists of values and come with powerful built-in methods. .map() transforms each element, .filter() selects matching elements, and .reduce() collapses an array into a single value — these three are the backbone of functional JS.
  2. Objects & Object Destructuring — Objects store key-value pairs representing structured data, similar to a dictionary or hash map. Destructuring lets you extract multiple properties into individual variables in a single concise line.
  3. The Spread Operator — The spread operator (...) expands an array or object into individual elements. It's the standard way to copy arrays/objects, merge them together, or pass array elements as individual function arguments.
  4. JSON: JSON.stringify & JSON.parse — JSON (JavaScript Object Notation) is the universal data format for APIs and config files. JSON.stringify() converts a JS object into a JSON string for storage/transmission, and JSON.parse() converts it back into a usable object.

Why This Week Matters: .map(), .filter(), and .reduce() are the three most important array methods in JavaScript — virtually every Express route handler and database query result is processed with them. JSON is the wire format of every REST API you will ever build or consume.

Code Example

const prices = [10, 25, 5, 40, 15];
const taxed   = prices.map(p => p * 1.08);
const cheap   = prices.filter(p => p < 20);
const total   = prices.reduce((sum, p) => sum + p, 0);
console.log(taxed, cheap, total);

const user = { name: "Bob", age: 32, role: "admin", email: "bob@app.com" };
const { name, role } = user;         // destructuring
const { name: n, ...rest } = user;   // rest pattern

const base    = { theme: "dark", lang: "en" };
const updated = { ...base, lang: "fr" };  // spread + override

const settings   = { theme: "dark", notifications: true };
const jsonString = JSON.stringify(settings, null, 2);
const parsedBack = JSON.parse(jsonString);
console.log(parsedBack.theme); // "dark"

Week 5 — Functions & Scope

Mastering function declarations, arrow functions, closures, and the rules of variable scope.

Core Focus

Mastering function declarations, arrow functions, closures, and the rules of variable scope.

Topics & Subtopics

  1. Function Declarations vs Arrow Functions — Function declarations are hoisted (usable before their definition in the file) and have their own this binding. Arrow functions are more concise and inherit this from their surrounding scope — making them ideal for callbacks.
  2. Default Parameters & Rest Parameters — Default parameters provide a fallback value when an argument is omitted, avoiding manual undefined checks. Rest parameters (...args) collect any number of extra arguments into a real array.
  3. Closures — A closure is a function that "remembers" the variables from the scope it was created in, even after that outer function has finished executing. Closures are the foundation of private state, factory functions, and the module pattern.
  4. Variable Scope (Block, Function, Global) — JavaScript has three scope levels: global (accessible everywhere), function (accessible within a function), and block (accessible within {}, introduced by let/const). Understanding scope prevents accidental variable leaks and naming collisions.

Why This Week Matters: Closures are the foundation of nearly every advanced Node.js pattern — middleware functions, event handlers, factory functions, and the module pattern all rely on closures. Understanding lexical scope is what separates developers who write predictable Node.js code from those who guess and debug.

Code Example

// Function declaration — hoisted, has its own `this`
function add(a, b) { return a + b; }

// Arrow function — concise, lexical `this`
const multiply = (a, b) => a * b;

// Default and rest parameters
function greet(name = "Guest", greeting = "Hello") {
    return `${greeting}, ${name}!`;
}
function sumAll(...numbers) {
    return numbers.reduce((sum, n) => sum + n, 0);
}
console.log(greet());              // "Hello, Guest!"
console.log(sumAll(1, 2, 3, 4));   // 10

// Closure — private state
function createCounter() {
    let count = 0;
    return {
        increment: () => ++count,
        getCount:  () => count,
    };
}
const counter = createCounter();
counter.increment();
counter.increment();
console.log(counter.getCount()); // 2

Week 6 — Modules, the Event Loop & Asynchronous Basics

Organizing code into modules and understanding Node's single-threaded, non-blocking execution model.

Core Focus

Organizing code into modules and understanding Node's single-threaded, non-blocking execution model.

Topics & Subtopics

  1. Creating & Importing Custom Modules — Splitting code into separate files (modules) improves organization and reusability. Each file is its own module scope — you explicitly choose what to export and what to import elsewhere.
  2. The Event Loop & Non-Blocking I/O — Node.js runs JavaScript on a single thread but handles I/O operations (file reads, network calls) asynchronously via the Event Loop, delegating slow work to the system and resuming your code when it completes. This is why Node.js can handle thousands of concurrent connections efficiently.
  3. Callbacks — A callback is a function passed as an argument to be executed later, often after an asynchronous operation completes. While foundational to Node.js, deeply nested callbacks ("callback hell") are why Promises and async/await were introduced.
  4. Promises — A Promise represents the eventual result of an asynchronous operation — it can be pending, fulfilled, or rejected. Promises let you chain .then() for success and .catch() for errors, avoiding deeply nested callbacks.

Why This Week Matters: The Event Loop is the single most important concept in Node.js — it's the reason Node can serve thousands of simultaneous requests on one thread. Not understanding it leads to accidentally blocking the loop with synchronous CPU work, making your entire server unresponsive.

Code Example

// Custom module
export function formatCurrency(amount) {
    return `$${amount.toFixed(2)}`;
}
export const TAX_RATE = 0.08;

import { formatCurrency } from './utils.js';
console.log(formatCurrency(19.999)); // "$20.00"

// Event Loop — async always deferred
console.log("1: Start");
setTimeout(() => console.log("3: Timeout callback"), 0);
console.log("2: End");
// Output: 1 → 2 → 3 (sync always runs first)

// Promise pattern
function fetchUserData(id) {
    return new Promise((resolve, reject) => {
        setTimeout(() => {
            if (id > 0) resolve({ id, name: "Alice" });
            else reject(new Error("Invalid user ID"));
        }, 500);
    });
}
fetchUserData(1)
    .then(user => console.log("Got user:", user))
    .catch(err => console.error("Failed:", err.message));

Week 7 — Async/Await & Error Handling

Writing clean asynchronous code with async/await syntax and handling errors robustly.

Core Focus

Writing clean asynchronous code with async/await syntax and handling errors robustly.

Topics & Subtopics

  1. async/await Syntax — The async/await keywords let you write asynchronous code that looks synchronous, dramatically improving readability over chained .then() calls. Any function marked async automatically returns a Promise, and await pauses execution until that Promise resolves.
  2. try/catch with Async Code — When using await, errors from rejected Promises are caught using standard try/catch blocks — exactly like synchronous error handling. This is far cleaner than chaining multiple .catch() calls.
  3. Promise.all & Promise.allSettled — Promise.all() runs multiple async operations concurrently and waits for all to succeed (rejecting immediately if any fails). Promise.allSettled() waits for all to finish regardless of success or failure, giving you a status report for each.
  4. Custom Error Classes — Extending the built-in Error class lets you create domain-specific error types carrying extra context (like an HTTP status code). This makes error handling in larger apps far more precise and informative.

Why This Week Matters: Promise.allSettled vs Promise.all is one of the most impactful API choices in concurrent Node.js code. Promise.all short-circuits on the first failure — great for dependent operations, catastrophic for independent ones. Custom error classes are what separate professional APIs from ones that return vague "Something went wrong" messages.

Code Example

// async/await — reads like synchronous code
async function runSequence() {
    console.log("Starting...");
    await new Promise(resolve => setTimeout(resolve, 1000));
    console.log("1 second passed!");
    return "Done";
}
runSequence().then(result => console.log(result));

// try/catch with async
async function run() {
    try {
        const result = await fetchData(true); // throws
        console.log(result);
    } catch (err) {
        console.error("Caught error:", err.message);
    }
}

// Promise.allSettled — never short-circuits
const fetchA = Promise.resolve("Result A");
const fetchC = Promise.reject(new Error("Failed C"));
Promise.allSettled([fetchA, fetchC]).then(console.log);

// Custom error class
class ValidationError extends Error {
    constructor(message, field) {
        super(message);
        this.name  = "ValidationError";
        this.field = field;
    }
}
throw new ValidationError("Email is required", "email");

Week 8 — Built-in Modules: File System, Path & OS

Interacting with the file system and operating system using Node's core built-in modules.

Core Focus

Interacting with the file system and operating system using Node's core built-in modules.

Topics & Subtopics

  1. The fs Module (Async & Promise-based) — The fs (File System) module reads and writes files. The modern fs/promises API provides Promise-based versions of every method, making file operations work seamlessly with async/await instead of callbacks.
  2. The path Module — The path module provides utilities for working with file and directory paths in a way that's safe across operating systems (Windows uses \, Unix uses /). Always use path.join() instead of manually concatenating path strings.
  3. The os Module — The os module provides information about the operating system the Node.js process is running on — CPU architecture, available memory, home directory, and more. It's commonly used for system diagnostics and platform-specific logic.
  4. Streams for Large Files — Streams let you process data piece-by-piece instead of loading an entire file into memory — essential for handling large files efficiently. A ReadStream reads chunks, and .pipe() sends them directly to a destination like a write stream or HTTP response.

Why This Week Matters: Streams are one of Node.js's most powerful and misunderstood features. Loading a 1GB file with readFile crashes your server with an out-of-memory error; streaming it with createReadStream handles it in constant memory. This distinction becomes critical in production file upload/download APIs.

Code Example

import { readFile, writeFile } from 'node:fs/promises';
import path from 'node:path';
import os from 'node:os';
import { createReadStream, createWriteStream } from 'node:fs';

// fs/promises — works with async/await
async function processFile() {
    await writeFile('notes.txt', 'Hello, Node.js file system!');
    const content = await readFile('notes.txt', 'utf-8');
    console.log("File content:", content);
}

// path — OS-safe path building
const filePath = path.join(__dirname, 'data', 'users.json');
console.log(path.extname(filePath));   // ".json"
console.log(path.basename(filePath));  // "users.json"

// os — system information
console.log("Platform:", os.platform());
console.log("CPU cores:", os.cpus().length);
console.log("Free memory (MB):", Math.round(os.freemem() / 1024 / 1024));

// Streams — memory-efficient for huge files
const readStream  = createReadStream('large-input.txt');
const writeStream = createWriteStream('large-output.txt');
readStream.pipe(writeStream);
readStream.on('end', () => console.log("File copy complete!"));

Week 9 — Object-Oriented JavaScript & Building an HTTP Server

Modeling data with ES6 classes and building a raw HTTP server with Node's built-in http module.

Core Focus

Modeling data with ES6 classes and building a raw HTTP server with Node's built-in http module.

Topics & Subtopics

  1. ES6 Classes & Constructors — JavaScript classes provide clean, familiar syntax for object-oriented programming, though they're syntactic sugar over JS's underlying prototype system. The constructor method initializes new instances with their starting properties.
  2. Inheritance with extends & super — The extends keyword lets a class inherit properties and methods from a parent class. super() calls the parent's constructor, and super.methodName() calls the parent's version of an overridden method.
  3. Building a Server with the http Module — Before frameworks like Express existed, Node.js's built-in http module was used to build servers directly. Understanding this raw API helps you appreciate what frameworks abstract away — handling requests, setting headers, and sending responses.
  4. Private Class Fields & Getters/Setters — Modern JavaScript classes support truly private fields using the # prefix — these are completely inaccessible from outside the class. Getters and setters let you define computed properties that look like regular property access.

Why This Week Matters: Building an HTTP server with the raw http module before learning Express reveals exactly what Express is abstracting — routing, body parsing, header management. Developers who understand the raw layer debug Express issues 10x faster than those who only know the framework.

Code Example

import http from 'node:http';

// ES6 class
class User {
    constructor(name, email) {
        this.name = name;
        this.email = email;
        this.createdAt = new Date();
    }
    getSummary() { return `${this.name} (${this.email})`; }
}

// Inheritance
class Animal {
    constructor(name) { this.name = name; }
    speak() { return `${this.name} makes a sound.`; }
}
class Dog extends Animal {
    speak() { return `${super.speak()} Specifically, ${this.name} barks!`; }
}

// Private fields
class BankAccount {
    #balance;
    constructor(initial) { this.#balance = initial; }
    get balance() { return `$${this.#balance.toFixed(2)}`; }
    deposit(amount) { this.#balance += amount; }
}

// Raw HTTP server
const server = http.createServer((req, res) => {
    res.writeHead(200, { 'Content-Type': 'text/plain' });
    res.end(`You requested: ${req.url} via ${req.method}`);
});
server.listen(3000, () => console.log("Server running on port 3000"));

Week 10 — Express.js Fundamentals

Building structured REST APIs rapidly using the Express.js framework — Node's most popular web framework.

Core Focus

Building structured REST APIs rapidly using the Express.js framework — Node's most popular web framework.

Topics & Subtopics

  1. Setting Up Express & Basic Routing — Express dramatically simplifies server creation compared to the raw http module, providing clean routing methods (.get(), .post(), etc.) mapped directly to HTTP verbs and URL paths.
  2. Route Parameters & Query Strings — Express makes extracting dynamic URL segments (:id) and query string parameters (?key=value) trivial via req.params and req.query. These are the two primary ways clients pass data through a GET request's URL.
  3. Middleware (express.json() & Custom Middleware) — Middleware functions run between the incoming request and the final route handler, used for parsing request bodies, logging, authentication, and more. express.json() is built-in middleware that parses JSON request bodies into req.body.
  4. Handling POST, PUT, DELETE Requests — A REST API exposes different HTTP verbs for different operations on the same resource: GET to read, POST to create, PUT/PATCH to update, and DELETE to remove. Express maps each verb to its own handler method on the same path.

Why This Week Matters: Express middleware is one of the most powerful concepts in the Node.js ecosystem — it's how logging, authentication, rate limiting, body parsing, and CORS all work. Understanding the request-response cycle and how next() passes control through the middleware chain is essential for every Express application.

Code Example

import express from 'express';
const app = express();
app.use(express.json());

// Custom logging middleware
app.use((req, res, next) => {
    console.log(`${req.method} ${req.url}`);
    next(); // Must call next() to continue
});

// Route with params and query string
app.get('/users/:id', (req, res) => {
    const { id } = req.params;
    const { include } = req.query;
    res.json({ userId: id, including: include });
});

// POST — create
let tasks = [{ id: 1, title: "Learn Express" }];
app.post('/tasks', (req, res) => {
    const task = { id: tasks.length + 1, ...req.body };
    tasks.push(task);
    res.status(201).json(task);
});

// DELETE — remove
app.delete('/tasks/:id', (req, res) => {
    tasks = tasks.filter(t => t.id !== Number(req.params.id));
    res.status(204).send();
});

app.listen(3000, () => console.log("Express server on port 3000"));

Week 11 — Databases: MongoDB with Mongoose

Persisting application data in MongoDB using Mongoose's schema-based modeling.

Core Focus

Persisting application data in MongoDB using Mongoose's schema-based modeling and ODM API.

Topics & Subtopics

  1. Connecting to MongoDB with Mongoose — Mongoose is an Object Data Modeling (ODM) library that provides schema validation and a clean API for interacting with MongoDB. The connection is established once at app startup, typically using a connection string from an environment variable.
  2. Defining Schemas & Models — A Mongoose Schema defines the shape, types, and validation rules for documents in a collection. A Model is then compiled from the schema and provides the actual interface for querying and saving documents to MongoDB.
  3. CRUD Operations with Mongoose — Mongoose Models expose intuitive methods for all CRUD operations: .create() to insert, .find()/.findById() to read, .findByIdAndUpdate() to update, and .findByIdAndDelete() to remove documents.
  4. Mongoose Relationships (Population) — Mongoose models relationships between collections using references (ObjectId). The .populate() method automatically replaces a referenced ID with the full document it points to, similar to a SQL JOIN.

Why This Week Matters: Moving from an in-memory array to MongoDB is the single biggest architectural step in Node.js development. Mongoose's .populate() for relationships and schema-level validation for data integrity are two features that prevent entire categories of bugs that plague raw MongoDB users.

Code Example

import mongoose from 'mongoose';

async function connectDB() {
    await mongoose.connect(process.env.MONGO_URI || 'mongodb://localhost:27017/myapp');
    console.log("MongoDB connected!");
}

// Schema with validation
const userSchema = new mongoose.Schema({
    name:  { type: String, required: true },
    email: { type: String, required: true, unique: true },
    age:   { type: Number, min: 0, default: 18 },
});
const User = mongoose.model('User', userSchema);

// CRUD operations
const newUser  = await User.create({ name: "Alice", email: "alice@app.com" });
const allUsers = await User.find();
const oneUser  = await User.findById(newUser._id);

Week 12 — Authentication, Validation & Robust Error Handling

Securing APIs with JWT authentication, validating input, and implementing centralized error handling.

Core Focus

Securing APIs with JWT authentication, validating input, and implementing centralized error handling.

Topics & Subtopics

  1. Password Hashing with bcrypt — Never store plain-text passwords. bcrypt hashes passwords with a computationally expensive algorithm, making them effectively impossible to reverse even if your database is compromised. bcrypt.compare() safely verifies a login attempt against the stored hash.
  2. JWT Authentication — JSON Web Tokens (JWT) are signed tokens issued after login, containing user info that the server can verify without a database lookup on every request. The client sends the token in the Authorization header on subsequent requests.
  3. Auth Middleware & Protected Routes — Authentication middleware intercepts requests, verifies the JWT from the Authorization header, and either attaches the decoded user to req.user or rejects the request with a 401 status. This pattern protects routes without repeating auth logic everywhere.
  4. Centralized Error Handling with Express — Express supports a special 4-argument error-handling middleware that catches errors from anywhere in the app, ensuring consistent error responses. Placing it last (after all routes) lets you call next(error) from any route handler to delegate error formatting.

Why This Week Matters: Authentication and authorization are the most security-critical parts of any API. Storing plain-text passwords, using weak JWT secrets, or forgetting to protect routes are the vulnerabilities that make the OWASP Top 10. Getting these patterns right from the start is non-negotiable for production applications.

Code Example

import bcrypt from 'bcrypt';
import jwt from 'jsonwebtoken';

// Password hashing — never store plain text
const hashedPassword = await bcrypt.hash(plainPassword, 10);
const isValid = await bcrypt.compare(plainPassword, storedHash);

// Issue JWT on login
const token = jwt.sign(
    { userId: "abc123", role: "admin" },
    process.env.JWT_SECRET,
    { expiresIn: "1h" }
);

// Auth middleware
function authMiddleware(req, res, next) {
    const token = req.headers.authorization?.split(' ')[1];
    if (!token) return res.status(401).json({ error: "No token provided" });
    try {
        req.user = jwt.verify(token, process.env.JWT_SECRET);
        next();
    } catch {
        res.status(401).json({ error: "Invalid or expired token" });
    }
}

// Centralized error handler — 4 params, registered LAST
function errorHandler(err, req, res, next) {
    console.error(err.stack);
    const status = err.statusCode || 500;
    res.status(status).json({ error: err.message || "Internal Server Error" });
}
// app.use(errorHandler);

Week 13 — Generators, Iterators & Advanced Async Patterns

Mastering generator functions, custom iterators, and advanced asynchronous control flow.

Core Focus

Mastering generator functions, custom iterators, and advanced asynchronous control flow patterns.

Topics & Subtopics

  1. Generator Functions — A generator function (marked with function*) can pause its execution at yield points and resume later, producing a sequence of values lazily — one at a time — instead of computing them all upfront. This is invaluable for working with large or infinite sequences.
  2. Custom Iterators with Symbol.iterator — Any object can become "iterable" (usable in for...of loops and spread syntax) by implementing the Symbol.iterator method. This is how built-in types like Array and Map support iteration, and you can add the same capability to your own classes.
  3. Async Generators & for await...of — Async generators combine generators with Promises, allowing you to yield values that arrive asynchronously over time — perfect for paginated API results or streaming data sources. The for await...of loop consumes them naturally.
  4. The EventEmitter Pattern — EventEmitter is Node's built-in implementation of the observer pattern — objects can emit named events, and other parts of your code can subscribe to react to them. It's the foundation underlying streams, HTTP servers, and countless Node.js APIs.

Why This Week Matters: Async generators are the cleanest solution to paginated API consumption and data streaming — yielding each page as it arrives rather than buffering everything in memory. EventEmitter is the backbone of Node.js itself — every stream, HTTP server, and TCP socket is an EventEmitter under the hood.

Code Example

// Generator function
function* numberSequence() {
    yield 1;
    yield 2;
    yield 3;
}
const gen = numberSequence();
console.log(gen.next()); // { value: 1, done: false }
console.log([...numberSequence()]); // [1, 2, 3]

// Custom iterator
class Range {
    constructor(start, end) { this.start = start; this.end = end; }
    [Symbol.iterator]() {
        let current = this.start, end = this.end;
        return { next: () => current <= end
            ? { value: current++, done: false }
            : { value: undefined, done: true } };
    }
}
console.log([...new Range(1, 5)]); // [1, 2, 3, 4, 5]

// Async generator + for await
async function* fetchPages(totalPages) {
    for (let page = 1; page <= totalPages; page++) {
        await new Promise(r => setTimeout(r, 200));
        yield { page, data: `Results for page ${page}` };
    }
}
for await (const result of fetchPages(3)) {
    console.log(result);
}

// EventEmitter
import { EventEmitter } from 'node:events';
class OrderSystem extends EventEmitter {}
const orders = new OrderSystem();
orders.on('orderPlaced', (order) => console.log(`New order: ${order.id}`));
orders.emit('orderPlaced', { id: 101, total: 49.99 });

Week 14 — Worker Threads, Clustering & Concurrency

Scaling CPU-bound and high-traffic Node.js applications using parallelism and multi-core architectures.

Core Focus

Scaling CPU-bound and high-traffic Node.js applications using parallelism and multi-core architectures.

Topics & Subtopics

  1. Understanding Concurrency vs Parallelism in Node.js — Node.js is single-threaded for JavaScript execution but achieves concurrency through its event loop and parallelism through Worker Threads or Clustering. Understanding this distinction is critical for knowing when to reach for these advanced tools.
  2. Worker Threads for CPU-Bound Tasks — Worker Threads let you run JavaScript on separate threads, ideal for CPU-intensive tasks (image processing, complex calculations) that would otherwise block the main event loop. Each worker runs independently and communicates via message passing.
  3. Clustering for Multi-Core HTTP Servers — The cluster module lets a Node.js app spawn multiple worker processes (one per CPU core) that all share the same server port, dramatically increasing the number of requests your app can handle concurrently in production.
  4. Rate Limiting & Backpressure — Backpressure occurs when data is produced faster than it can be consumed — critical to manage in streams and high-throughput APIs to prevent memory exhaustion. Rate limiting middleware protects your API from being overwhelmed by too many requests from a single client.

Why This Week Matters: A single CPU-intensive synchronous operation — like parsing a huge CSV or computing a complex report — can freeze your entire Node.js server for seconds, dropping every concurrent request. Worker Threads solve this at the code level; Clustering solves it at the process level. Both are essential tools for production Node.js services.

Code Example

// worker.js — runs on a separate thread
import { parentPort, workerData } from 'node:worker_threads';
function fib(n) { return n <= 1 ? n : fib(n - 1) + fib(n - 2); }
parentPort.postMessage(fib(workerData.n));

// main.js — spawns the worker
import { Worker } from 'node:worker_threads';
const worker = new Worker('./worker.js', { workerData: { n: 40 } });
worker.on('message', (result) => console.log("Fibonacci result:", result));

// Clustering — one worker process per CPU core
import cluster from 'node:cluster';
import os from 'node:os';
if (cluster.isPrimary) {
    const cpuCount = os.cpus().length;
    for (let i = 0; i < cpuCount; i++) cluster.fork();
    console.log(`Primary spawned ${cpuCount} workers`);
}

// Rate limiting
import rateLimit from 'express-rate-limit';
const limiter = rateLimit({
    windowMs: 15 * 60 * 1000,
    max: 100,
    message: { error: "Too many requests, please try again later." },
});
// app.use(limiter);

Week 15 — Testing, Logging, Deployment & DevOps

Writing automated tests, structured logging, containerizing with Docker, and deploying to production.

Core Focus

Writing automated tests, adding structured logging, containerizing with Docker, and deploying to production.

Topics & Subtopics

  1. Unit & Integration Testing with Vitest — Automated tests verify your code behaves correctly and prevent regressions as your app grows. Unit tests check individual functions in isolation, while integration tests verify that multiple parts (like API routes and a database) work together correctly.
  2. Testing Express APIs with Supertest — Supertest lets you send simulated HTTP requests directly to your Express app in tests, without needing to actually start a server on a port. This makes API integration testing fast and reliable.
  3. Structured Logging with Pino — Production applications need structured, leveled logging (info, warn, error) written to files or external services — console.log alone doesn't scale. Pino produces fast, structured JSON output that works with log aggregation tools.
  4. Containerizing with Docker — Docker packages your Node.js app and all its dependencies into a portable container that runs identically on any machine — solving the classic "it works on my machine" problem. A Dockerfile defines the steps to build your app's image.

Why This Week Matters: Without tests, every deployment is a gamble. Without structured logging, production debugging is guesswork. Without Docker, deployment is a manual, error-prone process. These three practices — testing, logging, containerization — are what separate a hobby project from a production service that other engineers can operate confidently.

Code Example

// Vitest unit test
import { describe, it, expect } from 'vitest';
import { sum } from './math.js';
describe('sum function', () => {
    it('adds two positive numbers', () => {
        expect(sum(2, 3)).toBe(5);
    });
});

// Supertest integration test
import request from 'supertest';
import app from './app.js';
describe('GET /health', () => {
    it('returns 200 and status ok', async () => {
        const res = await request(app).get('/health');
        expect(res.status).toBe(200);
        expect(res.body).toEqual({ status: 'ok' });
    });
});

// Pino structured logging
import pino from 'pino';
const logger = pino();
logger.info("Server started successfully");
logger.warn({ userId: 42 }, "Unauthorized action attempted");
logger.error({ err: new Error("DB lost") }, "Critical failure");
