3D Game Development with Unity & C#

Complete 15-Week Beginner-to-Advanced Roadmap — Build Real 3D Games from Scratch

About This Roadmap

This roadmap takes a complete beginner to a confident, portfolio-ready Unity 3D game developer in 15 weeks of full-time study (~35–40 hours/week). It is structured in five phases.

Unity is the world's most widely used real-time 3D development engine, powering games across PC, console, mobile, VR, and web platforms. Notable games built with Unity include Hollow Knight, Cuphead, Hearthstone, Monument Valley, and Cities: Skylines.

C# (pronounced "C-sharp") is Unity's primary scripting language. Skills you build in C# transfer directly to other domains including desktop app development (.NET), backend development (ASP.NET), and mobile development.

Engine: Unity (LTS version)  |  Language: C#  |  Duration: 15 Weeks (Full-Time)  |  Focus: 3D Games (First-Person, Third-Person, Open World)

5 Phases Overview

  • Phase 1 (Weeks 1–2): C# programming fundamentals and Unity editor orientation.
  • Phase 2 (Weeks 3–6): Core 3D mechanics — movement, physics, cameras, lighting, materials, and basic AI.
  • Phase 3 (Weeks 7–10): Intermediate systems — UI, audio, scene management, animation, raycasting, and inventory.
  • Phase 4 (Weeks 11–13): Advanced topics — procedural generation, save/load systems, shader basics, and performance optimization.
  • Phase 5 (Weeks 14–15): Capstone game build, polish, WebGL publishing, and portfolio/interview preparation.

By Week 15, you will have built approximately 13 mini-projects and one complete, playable, published 3D capstone game — "EchoWaste," a third-person survival exploration game set in a ruined open world.

Week 1 — C# Programming Fundamentals

Learn enough C# to read, understand, and write basic Unity scripts — becoming comfortable with variables, methods, conditions, loops, and classes.

Topics & Subtopics

  1. C# and .NET ecosystem — What C# is, how Unity uses it (Mono runtime), Console.WriteLine() vs Unity's Debug.Log().
  2. Variables and data types — Value types (int, float, bool, char), reference type string, var keyword, const/readonly, camelCase vs PascalCase.
  3. Operators — Arithmetic, comparison (==, !=), logical (&&, ||, !), compound assignment.
  4. Control flow — if/else if/else, switch/case, ternary ? :.
  5. Loops — for, while/do-while, foreach, break/continue.
  6. Methods (functions) — Defining, calling, parameters, return types, void, overloading.
  7. OOP basics — Classes, objects, fields, properties (get/set), constructors, public/private/protected, static vs instance.
  8. Collections — Arrays, List<T>, Dictionary<TKey, TValue>.
  9. Inheritance & polymorphism intro — Base/derived classes with :, override/virtual; why this matters (all scripts inherit from MonoBehaviour).

Why This Week Matters: Every Unity script is a C# class. Without understanding variables, methods, classes, and collections, you cannot read Unity documentation, understand tutorials, or fix your own bugs. Week 1 is the most important week in the roadmap — weaknesses here compound into every subsequent week.

Daily Study Plan

Day Focus
Day 1 C# setup + variables + data types + operators; write 10 small expressions and log results.
Day 2 Control flow — if/else/switch; build a "stat check" program.
Day 3 Loops — for, while, foreach; loop that generates 10 random enemy names.
Day 4 Methods — define 5 methods (damage calculator, level-up, health clamp); call them.
Day 5 Classes — build Player class with health/mana/speed fields and TakeDamage()/Heal()/LevelUp() methods.
Day 6 Collections — List<string> inventory, Dictionary<string, int> for quantities, array for hotbar.
Day 7 Inheritance — Enemy base class; Zombie/Skeleton overriding Attack(); review + mini project + GitHub commit.

Practice Tasks

  • Build a C# console "RPG Stat System" — Character class with name/health/attack/defense; methods Attack(Character target), Heal(int amount), IsAlive().
  • Build an Inventory class using Dictionary<string, int> — add/remove/display items.
  • Build a GameTimer class that counts down from 60 and logs "Time's up!".

🛠 Mini Project

A C# Console "Turn-Based Combat Simulator" — two Character objects (Player and Enemy) take turns attacking until one's health reaches zero; damage randomized within min/max range based on attack stat; console logs each turn; winner announced — pushed to GitHub.

Common Mistakes

  • Confusing = (assignment) with == (comparison) — causes silent logic bugs.
  • Forgetting new when creating objects and collections.
  • Making all fields public by default — start with private and expose only what's needed.
  • Skipping OOP because "it seems abstract" — this bites hard in Week 3.

How to Avoid Them

  • Read every conditional out loud: "if health equals-equals zero" vs "health equals zero."
  • Every List, Dictionary, custom class needs new.
  • Default rule: fields are private. If Inspector needs to see it, use [SerializeField] private.
  • Do the Player/Enemy/Inventory exercises — abstract concepts become concrete.

Week 2 — Unity Editor Deep Dive & First 3D Scripts

Navigate Unity's editor fluently, understand how GameObjects and components work, and write your first working 3D gameplay scripts.

Topics & Subtopics

  1. Unity Editor layout — Scene view (3D navigation: right-click+WASD, F focus, alt+drag orbit), Game view, Hierarchy, Inspector, Project, Console.
  2. GameObjects & components — Creating GameObjects, adding/removing components, Transform (position/rotation/scale), parent-child.
  3. First C# script — Creating a script, MonoBehaviour base class, Start() vs Awake(), Update(), Debug.Log/LogWarning/LogError.
  4. Component access — GetComponent<T>(), caching in Awake(), [SerializeField] for private fields in Inspector.
  5. Input System (Legacy intro) — Input.GetKey/GetKeyDown, Input.GetAxis, Input.GetMouseButton. (New Input System introduced in Week 4.)
  6. 3D Transform in code — transform.position, Translate(), rotation/Quaternion basics, Rotate(), LookAt(), Vector3 (forward/up/right).
  7. Time & frame-rate independence — Time.deltaTime, Time.time, Time.timeScale.
  8. Prefabs & Instantiation — Creating prefabs, Instantiate(prefab, position, rotation), Destroy(gameObject, delay).

Why This Week Matters: The Unity editor is your primary tool for 13 more weeks. The Update() loop, GetComponent<T>(), [SerializeField], Transform, Vector3, and Time.deltaTime are patterns you'll type thousands of times. Getting these automatic now means Week 3 can focus entirely on 3D mechanics.

Daily Study Plan

Day Focus
Day 1 Editor layout tour + navigation practice (fly mode, orbit, focus) + panel familiarity.
Day 2 Create 5 GameObjects; experiment with Transform in Inspector; build a simple scene.
Day 3 First script — MonoBehaviour, Start(), Update(), Debug.Log(); attach to cube.
Day 4 [SerializeField] + GetComponent<T>() + Input.GetAxis — move cube with arrow keys.
Day 5 transform.Translate + Time.deltaTime — smooth movement in all 3 axes.
Day 6 transform.Rotate rotating object; transform.LookAt() turret facing player.
Day 7 Prefabs + Instantiate/Destroy — spawn prefab on spacebar, destroy after 3s + mini project.

Practice Tasks

  • Make a sphere orbit around a cube using transform.RotateAround().
  • Build a "floating" script — GameObject bobs up/down using Mathf.Sin(Time.time).
  • Create a Prefab spawner instantiating random prefab from List<GameObject> every 2s using InvokeRepeating().

🛠 Mini Project

A "3D Object Showcase" scene — platform with 5 GameObjects each with unique script behavior: rotating cube, orbiting sphere, floating gem, bouncing ball (Mathf.PingPong), color-changing capsule (Mathf.Sin lerp between two colors) — well-organized hierarchy, [SerializeField] for parameters, pushed to GitHub.

Common Mistakes

  • Moving objects by setting transform.position directly without Time.deltaTime — frame-rate-dependent speed.
  • Calling GetComponent<T>() inside Update() — very expensive.
  • Script names with spaces or starting with numbers — must match class name exactly.
  • Confusing Scene view (editing) with Game view (runtime) — edits during Play mode are lost.

How to Avoid Them

  • Law of Time.deltaTime: If value changes per frame, multiply by Time.deltaTime.
  • Rule: GetComponent<T>() goes in Awake() or Start(), stored in private field.
  • Script file name = Class name. Both match exactly, including capitalization.
  • Never edit Inspector values while in Play mode — always exit Play first.

Week 3 — 3D Physics: Rigidbody, Colliders & Triggers

Build physically realistic 3D interactions using Unity's physics engine — gravity, collisions, forces, and trigger zones.

Topics & Subtopics

  1. Rigidbody — Mass/Drag/Angular Drag, useGravity, constraints, AddForce(), velocity, MovePosition(), isKinematic.
  2. Colliders — BoxCollider, SphereCollider, CapsuleCollider, MeshCollider (never on moving objects — expensive); Physics Materials (bounciness/friction).
  3. Collision callbacks — OnCollisionEnter/Stay/Exit; accessing collision.gameObject, collision.contacts, relativeVelocity.
  4. Trigger colliders — Is Trigger checkbox; OnTriggerEnter/Stay/Exit; uses: collectibles, checkpoints, trap zones.
  5. Tags & Layers in physics — other.CompareTag("Player") (more efficient than ==); Physics Layer Collision Matrix.
  6. FixedUpdate() vs Update() — ALL physics in FixedUpdate(); Time.fixedDeltaTime.
  7. Raycasting basics — Physics.Raycast, Debug.DrawRay() for visualization; uses: ground detection, shooting, mouse-over.

Why This Week Matters: Physics is the backbone of every 3D game. Character grounding, object pickup, bullet collision, death triggers, and area effects all rely on this week's systems. OnTriggerEnter and Raycast together account for a large proportion of all gameplay interactions in Unity 3D games.

Daily Study Plan

Day Focus
Day 1 Rigidbody — add to cube; experiment with mass/drag; AddForce on spacebar.
Day 2 Collider shapes — compare Box/Sphere/Capsule/Mesh; Physics Material (bouncy ball).
Day 3 OnCollisionEnter — change color when hitting floor; destroy on high-impact.
Day 4 Trigger zones — [SerializeField] area logging "Player entered"; healing zone.
Day 5 Tags + Layer Collision Matrix — only Player triggers zone, not enemies.
Day 6 FixedUpdate physics vs Update — observe jitter with framerate cap.
Day 7 Raycasting — ray from camera center detecting what it points at + mini project.

Practice Tasks

  • Build a "pinball-style" ramp scene using Physics Materials, forces, collision callbacks.
  • Create "lava floor" trigger that kills any GameObject tagged "Player".
  • Use Raycast from player downward to detect if grounded.

🛠 Mini Project

A "Physics Obstacle Course" — 3D course with: rolling ball controlled by AddForce, bounce pads (physics material + trigger), teleport zone (trigger that moves ball), destructible wall (objects destroyed on high-velocity collision), finish line trigger logging time — pushed to GitHub.

Common Mistakes

  • Putting physics code in Update() instead of FixedUpdate() — jittery physics.
  • Adding Rigidbody + MeshCollider to complex moving object — extremely expensive.
  • Using other.tag == "Player" instead of other.CompareTag("Player").
  • Forgetting Is Trigger on trigger collider — actual physical collision instead.

How to Avoid Them

  • Absolute rule: Rigidbody + physics in FixedUpdate(). Rendering + input in Update().
  • Moving objects use primitive colliders (Box/Sphere/Capsule), never MeshCollider.
  • Always use CompareTag() for tag comparison.
  • If OnTriggerEnter isn't firing: check Is Trigger enabled and at least one object has Rigidbody.

Week 4 — 3D Character Controller: Movement, Camera & New Input System

Build a complete, polished 3D player character with smooth movement and a functional third-person or first-person camera using Unity's Character Controller and New Input System.

Topics & Subtopics

  1. New Input System — Install via Package Manager; Input Actions asset; Action Maps/Actions/Bindings; PlayerInput component; InputAction.ReadValue<Vector2>(); why over Legacy.
  2. CharacterController component — controller.Move(velocity * Time.deltaTime), isGrounded, Slope Limit, Step Offset; trade-offs vs Rigidbody.
  3. Third-Person Movement — Reading input → world-space direction; movement relative to camera; character rotation with Quaternion.LookRotation; manual gravity (CharacterController doesn't apply).
  4. Jumping — Jump input detection; upward velocity + gravity over time; coyote time intro.
  5. Camera Systems (Cinemachine) — Install via Package Manager; CinemachineVirtualCamera; Third-Person Follow; CinemachineFreeLook; first-person camera child of player.
  6. Mouse look — Reading mouse delta; horizontal player rotation (Y axis); vertical camera rotation (X axis) with clamping; Cursor.lockState = CursorLockMode.Locked.

Why This Week Matters: Player movement and camera feel are the first things every player experiences. A character that feels responsive and a camera that doesn't fight the player are the difference between a playable game and an unplayable prototype. Cinemachine is Unity's professional, production-ready camera system.

Daily Study Plan

Day Focus
Day 1 New Input System setup — install, Input Actions asset, bind WASD + jump.
Day 2 CharacterController — attach to capsule; basic WASD movement without gravity.
Day 3 Add manual gravity — velocity.y -= gravity * Time.deltaTime; test isGrounded.
Day 4 Movement direction relative to camera; character rotation to face movement smoothly.
Day 5 Jump implementation — jump on button press only when grounded; tweak feel.
Day 6 Cinemachine — install; set up CinemachineFreeLook following player.
Day 7 Polish — smooth rotation, cursor lock, camera collision + mini project.

Practice Tasks

  • Add sprint (hold Shift to move faster, reduce FOV slightly with Cinemachine Brain).
  • Add crouch (CharacterController center Y offset + reduced speed).
  • Build simple obstacle course; time how long character takes to navigate.

🛠 Mini Project

A "3D Character Movement Demo" — polished capsule that walks, runs (sprint), jumps, and crouches on varied terrain with ramps/platforms/gaps; CinemachineFreeLook third-person camera; smooth rotation to face movement; all input via New Input System; all parameters via [SerializeField] — pushed to GitHub.

Common Mistakes

  • Using Legacy Input System in new projects — always use New Input System.
  • Not implementing manual gravity with CharacterController (character floats).
  • Putting camera inside player object without Cinemachine — gimbal lock and clipping.
  • Rotating on all axes instead of only Y for horizontal (character tips over).

How to Avoid Them

  • CharacterController requires manual gravity: velocity.y += Physics.gravity.y * Time.deltaTime.
  • Cinemachine handles camera collisions/smoothing that manual cameras get wrong — use it.
  • Horizontal rotation: only Y axis. Vertical look: only camera's X axis.

Week 5 — 3D Environments: Terrain, Lighting & Materials

Build believable 3D game worlds using Unity's Terrain system, global lighting, and physically-based materials.

Topics & Subtopics

  1. Unity Terrain — Create terrain; tools (Raise/Lower, Smooth, Paint Holes); height maps; texture painting (multiple layers); details (grass, rocks); trees (SpeedTree, mesh); settings.
  2. ProBuilder (intro) — Install via Package Manager; build simple geometry without Blender; UVs and textures; when to use vs Terrain vs imported meshes.
  3. Lighting in Unity 3D — Light types (Directional, Point, Spot, Area); intensity/color/range/shadows; Global Illumination concept; baked vs real-time vs mixed; Light Probes; Reflection Probes; Ambient light; Skybox.
  4. Lighting baking — Window → Rendering → Lighting; marking objects static; baking process; why baked is critical for performance.
  5. Materials & Shaders — Material = link between Mesh and Shader; Unity Standard Shader (Albedo, Metallic, Smoothness, Normal Map, Height Map, Emission); PBR; Standard Specular vs Metallic setup.
  6. Skybox — Assigning skybox material; 6-sided vs procedural; free HDRI from Poly Haven.
  7. Fog — Exponential vs linear; using for mood and performance.

Why This Week Matters: A beautiful, well-lit 3D environment turns a grey tech demo into a game world players want to explore. Lighting and materials account for the majority of a game's visual quality — a well-lit scene with basic meshes looks better than a poorly lit scene with high-detail models.

Daily Study Plan

Day Focus
Day 1 Terrain creation + height sculpting; build valley, hill, cliff formation.
Day 2 Terrain texture painting (3 layers: grass, dirt, rock); trees and grass details.
Day 3 ProBuilder — build ruined stone building or bunker interior on terrain.
Day 4 Lighting — Directional (sun), 3 Point Lights (interior), Skybox; adjust for time-of-day.
Day 5 Static objects + baked lightmaps; bake scene; compare before/after.
Day 6 Materials — create 5 materials (stone, metal, wood, glass, ground) using PBR; apply.
Day 7 Reflection Probes + Light Probes + Fog + mini project + GitHub commit.

Practice Tasks

  • Day-to-night lighting switch — script adjusting Directional Light rotation/color over time.
  • Create scene with 3 moods using only lighting: forest at dawn, cave interior, city at night.
  • Import free HDRI skybox from Poly Haven; set up with Reflection Probe.

🛠 Mini Project

A "3D Game Environment" — complete outdoor terrain scene (~500x500 units) with varied height, 3 texture layers, trees and grass, ProBuilder ruined structure, correctly baked lighting with Directional sun, fog effect, at least 5 custom materials — Week 4 character controller dropped in for exploration — pushed to GitHub with screenshots.

Common Mistakes

  • Using real-time shadows on every light — severe performance cost.
  • Not marking objects as Static before baking — bake has no effect.
  • Painting terrain at maximum resolution — crashes or long bake times.
  • Ignoring Light Probes — dynamic objects look unlit in baked scenes.

How to Avoid Them

  • Rule: static geometry → baked lighting. Moving objects → real-time + Light Probes.
  • Before baking, select all environment objects and enable Static.
  • Lower resolution lightmaps during dev (32/64); 512/1024 only for final builds.
  • Place Light Probe Group around player's movement area — essential for baked scenes.

Week 6 — Basic AI: NavMesh, Pathfinding & Enemy Behaviour

Build enemies and NPCs that navigate the 3D world intelligently using Unity's NavMesh system and implement state-based AI behaviour.

Topics & Subtopics

  1. NavMesh — Walkable surface representation; bake via Window → AI → Navigation → Bake; static marking; areas and costs; NavMeshObstacle for moving blockers.
  2. NavMeshAgent — SetDestination(Vector3); remainingDistance, stoppingDistance; speed, acceleration, angularSpeed; isStopped.
  3. State Machine — States: Idle, Patrol, Chase, Attack, Dead; enum EnemyState; switch for transitions; distance checks with Vector3.Distance.
  4. Patrol — Waypoint system (List<Transform>); cycling waypoints; waiting with WaitForSeconds coroutine.
  5. Detection & Chase — Physics.OverlapSphere detection radius; line-of-sight Raycast; chase by setting destination every frame.
  6. Attack — Range check with Vector3.Distance; cooldown using Time.time; player.GetComponent<Health>().TakeDamage(damage).
  7. Coroutines — IEnumerator + StartCoroutine(); yield return new WaitForSeconds(), yield return null; uses: attack anims, timed events, spawn delays.

Why This Week Matters: Enemies are in almost every 3D game. NavMesh-based pathfinding is the standard Unity approach used in professional games. The state machine pattern scales to complex AI in later projects — it's the foundation of every enemy AI system regardless of game genre.

Daily Study Plan

Day Focus
Day 1 NavMesh bake on flat terrain + 1 corridor; NavMeshAgent moving to static target.
Day 2 Waypoint patrol with List<Transform>; coroutine wait at each point.
Day 3 OverlapSphere detection; transition to Chase when player detected.
Day 4 Line-of-sight Raycast — enemy only chases if no wall between.
Day 5 Attack state — stop within attack range; deal damage on cooldown.
Day 6 Full state machine: Idle → Patrol → Chase → Attack → Dead.
Day 7 NavMeshObstacle on a door + enemy re-bake + mini project.

Practice Tasks

  • Build flanking enemy that runs to a different side of player before attacking.
  • Build "scared" enemy that runs away when health is low.
  • Build multiple enemy types with different speeds/ranges/damage using base Enemy class.

🛠 Mini Project

An "Enemy AI Arena" — walled arena with NavMesh baked, 3 enemy types (Patrol Guard, Aggressive Chaser, Ranged Shooter), Week 4 character as player, enemies with Health system (TakeDamage()), simple kill counter UI — enemies respawn after 5s with Instantiate/Destroy + coroutine — pushed to GitHub.

Common Mistakes

  • Baking NavMesh before marking objects as static — walkable surface generates incorrectly.
  • Calling agent.SetDestination() every frame when destination rarely changes.
  • Not stopping NavMeshAgent before destroying enemy — NavMesh error in console.
  • State machine growing into unreadable if/else chain — use switch.

How to Avoid Them

  • NavMesh workflow: mark static → bake → add NavMeshAgent → set destination → test.
  • Call SetDestination on a 0.2s timer (InvokeRepeating) rather than every frame.
  • Before destroying enemy: agent.isStopped = true; agent.enabled = false; then Destroy().
  • Enum state machine with switch — each case handles one state cleanly.

Build a complete game UI — HUD, main menu, pause menu, and game over screen — and connect them to real gameplay systems.

Topics & Subtopics

  1. Unity UI (Canvas) — Canvas render modes (Screen Space Overlay/Camera, World Space); RectTransform (anchoring, pivot); Canvas Scaler for multi-resolution.

  3. TextMeshPro — Why over legacy Text (sharper); Install TMP Essentials; TMP_Text type; runtime update.
  4. Health System — Health component with currentHealth, maxHealth, TakeDamage(), Heal(), Die(); UnityEvent for loose coupling to UI; screen flash on damage.
  5. HUD elements — Health bar, stamina bar, ammo count (TMP), crosshair, minimap (Render Texture from second camera).

A "Complete Game UI Demo" — integrating Week 4 character + Week 6 enemy arena with: full HUD (health, stamina, ammo, crosshair), Main Menu scene with Play/Quit, Pause Menu (Escape), Game Over triggered by death, Win triggered by killing all enemies — proper multi-scene project with SceneManager — pushed to GitHub.

Common Mistakes

  • Building UI without Canvas Scaler — breaks on different resolutions.
  • Using legacy Text instead of TextMeshPro — blurry text.

  • Update UI in response to events, not by polling every frame.
  • Time.timeScale = 0 pauses physics + Update. Never forget to restore to 1.

Week 8 — Audio: Sound Effects, Music & 3D Spatial Audio

Implement a complete audio system — background music, spatial 3D sound effects, audio management, and dynamic audio triggers.

Topics & Subtopics

  1. Unity Audio basics — AudioSource (volume, pitch, loop, spatial blend); AudioListener (Unity's "ear", on Main Camera); AudioClip (.wav/.mp3/.ogg); Play()/Stop()/PlayOneShot().
  2. 2D vs 3D audio (spatialBlend) — 0 = 2D (music, UI); 1 = 3D (in-world); minDistance/maxDistance; rolloff modes.
  3. Audio Manager (Singleton) — Singleton pattern; DontDestroyOnLoad(gameObject); AudioManager.Instance.PlaySFX(clip); music crossfade coroutine.
  4. Procedural & adaptive audio — Pitch variation for repeated sounds; volume variation; audio trigger zones.
  5. Footsteps system — Raycast downward for surface; different clips per surface (grass/stone/wood/metal); step timer based on movement speed.
  6. Reverb zones — AudioReverbZone for natural reverb (cave echo, room reverb).
  7. Mixer & categories — Audio Mixer asset; groups (Master/Music/SFX/Voice); routing AudioSources; volume control with mixer.SetFloat(); save with PlayerPrefs.

Why This Week Matters: Audio is the single most underrated element of game feel. A game with great audio and mediocre visuals consistently feels better than the reverse. 3D spatial audio is uniquely powerful in first-person and third-person games — the sound of footsteps approaching from behind, distant gunfire, and echoing caves all use this week's systems.

Daily Study Plan

Day Focus
Day 1 AudioSource on fireplace — loop ambient fire; spatial 3D blend test.
Day 2 PlayOneShot for non-looping SFX (button click, jump, landing).
Day 3 Audio Manager Singleton — DontDestroyOnLoad + PlaySFX()/PlayMusic().
Day 4 Music crossfade coroutine — fade out current, fade in new on scene load.
Day 5 Footstep system — Raycast surface detection + multiple clips + step timer.
Day 6 Audio Mixer — Master/Music/SFX groups + settings menu volume + PlayerPrefs save.
Day 7 Reverb zone in cave/tunnel + random pitch variation + mini project.

Practice Tasks

  • Build audio "zone system" — trigger areas crossfading between outdoor and indoor ambience.
  • Add pitch-varied footsteps to Week 4 character based on surface detection.
  • Add heartbeat SFX that gets louder and faster as player health decreases.

🛠 Mini Project

Integrate full audio into existing prototype (Weeks 4–7) — background music with crossfade between menu/gameplay, spatial 3D SFX for enemy footsteps/attacks, player footsteps on multiple surfaces, heartbeat SFX tied to health, Audio Mixer groups with settings menu volume slider saved via PlayerPrefs — pushed to GitHub.

Common Mistakes

  • Using audioSource.Play() for short SFX — cuts off previous sound.
  • Setting spatialBlend = 1 on background music — sounds quieter on one side.
  • Audio Manager getting duplicated on scene reload (missing singleton check).
  • Not setting min/max distance on 3D audio — sound audible across entire level.

How to Avoid Them

  • Short SFX: PlayOneShot(). Looping: audioSource.loop = true; Play().
  • Music: spatialBlend = 0 always. In-world: spatialBlend = 1 with min/max distance.
  • Singleton check: if (Instance != null && Instance != this) { Destroy(gameObject); return; }.
  • Every 3D AudioSource: set maxDistance to logical hearing range.

Week 9 — Animation: Animator, Blend Trees & Rigged Characters

Bring 3D characters to life with animation — walking, running, jumping, attacking — using Unity's Animator Controller and imported character rigs.

Topics & Subtopics

  1. Importing characters — FBX format; free rigged characters from Mixamo (Adobe, free account); import settings (Humanoid rig, Avatar, Apply Root Motion).
  2. Animator Controller — States = animation clips; transitions; conditions; parameters (Float, Int, Bool, Trigger); Has Exit Time; setting via scripts (SetFloat, SetBool, SetTrigger).
  3. Blend Trees — 1D (single float, e.g., speed: idle/walk/run); 2D (two axes, e.g., strafe + forward); Locomotion Blend Tree for player.
  4. Animation Layers — Base Layer (full body); Upper body layer for shooting while walking; weights and Avatar Masks.
  5. Root Motion vs Script-Driven — Root motion moves character via animation; script-driven moves via code with animation playing in place; when to use each.
  6. Animation Events — Trigger code at specific frame; uses: footstep sounds, attack hit detection, VFX spawning.
  7. IK (Inverse Kinematics intro) — OnAnimatorIK(); animator.SetIKPosition() for foot placement on uneven terrain.

Why This Week Matters: A character that glides across the ground or T-poses is immediately immersion-breaking. Animation is what makes characters feel alive. Blend Trees for locomotion and Animation Events for gameplay-triggered VFX/sound are used in virtually every 3D character in every commercial Unity game.

Day Focus

Day 2 Animator Controller — Idle and Walk states; transition with Bool parameter.
Day 3 1D Blend Tree — Idle/Walk/Run based on Speed float from velocity.magnitude.
Day 4 Jump Trigger — any state → Jump → Idle; sync with isGrounded.
Day 5 Attack animation — trigger from mouse click; Animation Event at hit frame calls DealDamage().
Day 6 Upper Body layer + Avatar Mask — shoot while walking (legs animate, upper body overlays).
Day 7 IK intro — foot IK on sloped terrain + mini project + GitHub commit.

Practice Tasks

  • Build 2D Blend Tree for strafing character (left/right while facing forward).

  • Create enemy with complete Animator Controller (Idle/Walk/Attack/Death).

Common Mistakes

  • Not setting Rig to Humanoid on import — Blend Trees and Avatars only work with Humanoid.
  • Leaving "Has Exit Time" on attack/jump transitions — character finishes idle before jumping.
  • Using Root Motion with CharacterController — character moves twice.
  • Setting animator parameters in Update() without caching Animator reference.

How to Avoid Them

  • Always set Rig to Humanoid on import settings of any character.
  • For jump/attack/death: turn OFF "Has Exit Time"; condition-based only.
  • With CharacterController: set "Apply Root Motion" to false; move via script.
  • Cache Animator in Awake(): animator = GetComponent<Animator>();.

Week 10 — Raycasting, Interaction & Inventory System

Build an interactive 3D world — players examine, pick up, use objects via raycasting — and implement a fully functional inventory system.

Topics & Subtopics

  1. Advanced Raycasting — Physics.Raycast with LayerMask; Physics.SphereCast; Physics.OverlapSphere; mouse-to-world with Camera.main.ScreenPointToRay; RaycastAll().
  2. Interaction system (interface-based) — IInteractable interface with Interact() method; any interactable implements it; clean decoupled design; "Press E to Pick Up" prompt UI.
  3. Pickup system — ItemPickup : MonoBehaviour, IInteractable; on Interact(): add to inventory, destroy/disable; different item types via inheritance or ScriptableObjects.
  4. ScriptableObjects for item data — ItemData : ScriptableObject defines name/icon/description/weight; lives as project asset; easy to edit; create instances in Editor.
  5. Inventory system — Inventory class with List<ItemData> items; AddItem(), RemoveItem(), HasItem(), UseItem(); stack-based with Dictionary<ItemData, int>.

Why This Week Matters: Interactive objects and inventory systems appear in the majority of 3D games — RPGs, survival, shooters (ammo pickup), puzzle games. The IInteractable interface pattern avoids spaghetti code. ScriptableObjects for item data is a Unity best practice that scales to large item databases.

Common Mistakes

  • Not using LayerMask on interaction Raycast — hits invisible colliders, triggers, UI.
  • Storing item data on pickup GameObject instead of ScriptableObjects — doesn't scale.

Week 11 — Procedural Generation & Open World Techniques

Generate game content algorithmically — terrain, object placement, dungeons — and learn streaming and LOD techniques for open-world performance.

Topics & Subtopics

  1. Procedural terrain with Perlin Noise — Mathf.PerlinNoise(x, y); generating height map; applying to procedural Mesh; Terrain.SetHeights() at runtime; octaves, lacunarity, persistence.
  2. Procedural object scatter — Placing trees/rocks/props based on terrain height and slope; rejection sampling; Poisson Disk Sampling intro.
  3. Procedural dungeon generation — Binary Space Partitioning (BSP); room-and-corridor generation using List<Rect>; instantiating room prefabs connected by corridors.
  4. Level of Detail (LOD) — LOD Group component; switching between high/medium/low-poly based on camera distance; creating LOD levels manually.
  5. Occlusion Culling — Not rendering objects hidden behind others; setup via Window → Rendering → Occlusion Culling → Bake; Occlusion Areas.
  6. Chunk-based world loading — Dividing world into chunks; loading/unloading based on player position; Additive scene loading with SceneManager.LoadSceneAsync().

Why This Week Matters: Procedural generation is what makes open-world, roguelike, and survival games feel different every time. LOD and Occlusion Culling are not optional extras — they're required for any open-world scene to run at playable framerates. This week gives you techniques used in games like Minecraft (procedural terrain), No Man's Sky (procedural everything), and Subnautica (chunk loading).

Daily Study Plan

Day Focus
Day 1 Perlin Noise 2D — generate texture showing noise as grey pixels; adjust scale/octaves.
Day 2 Apply Perlin to generate terrain heights: Terrain.SetHeights() with noise array.
Day 3 Procedural mesh generation — flat plane from vertices/triangles; apply Perlin for height.
Day 4 Procedural object scatter — place 200 trees/rocks based on height threshold and slope.
Day 5 BSP dungeon generator — create Room objects, connect with corridors, instantiate prefabs.
Day 6 LOD Group — create 3 LOD levels for tree mesh; observe draw call reduction.
Day 7 Occlusion Culling bake + additive chunk loading intro + mini project.

Practice Tasks

  • Build runtime terrain re-generator: button generates new terrain with random seed each press.
  • Add biome-based texture painting (height < X = sand, X–Y = grass, above Y = snow).
  • Add seed system to dungeon generator so same seed produces same dungeon.

🛠 Mini Project

A "Procedural Open World" demo — 512x512 unit world generated at runtime using Perlin with 3 octaves; biome-based texture painting; procedurally scattered trees/rocks/ruins; LOD applied to all large assets; Occlusion Culling baked; Week 4 character drops in for exploration; frame rate monitored with Profiler — pushed to GitHub with Profiler screenshots.

Common Mistakes

  • Generating procedural terrain without seeding — regeneration produces uncontrolled results.
  • Not using LOD on procedurally scattered objects — 200 high-poly trees tanks framerate.
  • Building procedural dungeons without boundary check — rooms overlap or go outside area.

How to Avoid Them

  • Always expose seed variable and use Random.InitState(seed) before generation.
  • Any object placed in large numbers (trees, rocks, grass) needs LOD Group.
  • Room boundary check: before placing, verify no intersection using Rect.Overlaps().

Week 12 — Save/Load System, PlayerPrefs & Game Data Management

Implement a robust save and load system that persists player progress, world state, and settings across game sessions.

Topics & Subtopics

  1. PlayerPrefs — GetInt/SetInt, GetFloat/SetFloat, GetString/SetString; Save(); good for simple settings (volume, quality, keybindings); NOT for complex state.
  2. JSON Serialization — JsonUtility.ToJson(object); JsonUtility.FromJson<T>(json); [System.Serializable] required; Newtonsoft.Json (Json.NET) for advanced cases.
  3. File I/O for save data — Application.persistentDataPath (correct cross-platform path); File.WriteAllText/ReadAllText; File.Exists.
  4. Save Data architecture — SaveData serializable container; SaveManager Singleton with Save(), Load(), DeleteSave(); multiple save slots.
  5. Saving complex data — Vector3 (native support); List<string> inventory; enemy states with Dictionary<string, bool> (needs Json.NET); [System.NonSerialized] to exclude.
  6. Loading & applying — Restore position/health/inventory on scene load; "Saveable" interface pattern with GetSaveData()/ApplySaveData().
  7. Auto-save & checkpoints — Checkpoint trigger zones; auto-save on timer using coroutines.

Why This Week Matters: Without a save system, no game can be played in sessions. This week's skills separate a demo from a real game product. JSON serialization and Application.persistentDataPath are the professional approach used in published Unity games — not PlayerPrefs for everything.

Daily Study Plan

Day Focus
Day 1 PlayerPrefs for volume, graphics quality, key bindings — these stay in PlayerPrefs.
Day 2 [System.Serializable] + JsonUtility.ToJson — serialize Player class to JSON and log.
Day 3 File I/O — write JSON to persistentDataPath; read back; verify file exists before loading.
Day 4 SaveData class capturing: position, health, inventory list, enemies defeated.
Day 5 SaveManager Singleton with Save() and Load(); call on button press / scene start.
Day 6 Multiple save slots (3 files) with slot selection UI.
Day 7 Checkpoint trigger auto-save + corrupt file error handling + mini project.

Practice Tasks

  • Save and restore which doors are open and which chests looted using Dictionary<string, bool>.
  • Build "Load Game" menu showing all 3 save slots with last-saved timestamp and player level.
  • Add "Are you sure?" confirmation before deleting a save.

🛠 Mini Project

A "Persistent World Demo" — Week 10 dungeon with complete save/load: position/health/inventory saved to JSON; chests looted and doors unlocked saved via unique IDs; 3 save slots accessible from Main Menu; checkpoint triggers that auto-save; corrupt/missing file handled gracefully with "No save found" — pushed to GitHub.

Common Mistakes

  • Using PlayerPrefs for inventory and world state — it's a simple key store, not structured data.
  • Not using Application.persistentDataPath — hardcoded paths break in builds.
  • Forgetting [System.Serializable] — JsonUtility silently serializes empty {}.
  • Not wrapping Load() in try/catch — one corrupt file crashes startup.

How to Avoid Them

  • PlayerPrefs = settings only. Game state = JSON files in persistentDataPath.
  • Always use Application.persistentDataPath — never hardcode paths.
  • Every class you serialize needs [System.Serializable].
  • Always try { Load(); } catch (Exception e) { Debug.LogError(e); CreateNewSave(); }.

Week 13 — Shaders, VFX & Performance Profiling

Add visual polish through shader effects and particle systems, and learn how to profile and optimize Unity 3D games to maintain target framerates.

Topics & Subtopics

  1. Shader Graph (URP intro) — Install via "3D URP" project template; nodes, properties, master stack; simple effects (color tint, UV scrolling, dissolve); vertex displacement.
  2. VFX (Particle System) — Built-in Particle System vs VFX Graph (CPU vs GPU); emission, shape, lifetime, velocity, size/color over lifetime; common VFX: explosion, fire, smoke, sparkle, dust, blood.
  3. Post-Processing (URP Volume) — Volume + Volume Profile; Bloom, Color Grading, Depth of Field, Vignette, Chromatic Aberration; global vs local volume.
  4. Unity Profiler — CPU Usage chart; GPU Usage; Memory section (allocations, GC pressure); Frame Debugger for every draw call.
  5. Performance Optimization — Draw call batching (Static, Dynamic, GPU Instancing); Object pooling; StringBuilder vs string concat; WaitForSeconds cached in coroutines; culling distance.
  6. Object Pooling — ObjectPool<T> (Unity 2021+ built-in); manual pool with Queue<GameObject> + GetFromPool/ReturnToPool; pooling bullets, enemies, particles.

Why This Week Matters: A game that runs at 15fps is unplayable regardless of how good it looks. This week teaches you to measure performance (Profiler) and fix it (batching, pooling, LOD, occlusion). Shader Graph and Post-Processing add the visual quality jump that makes screenshots look like a real commercial game.

Daily Study Plan

Practice Tasks

  • Build rain particle system attached to camera following player.
  • Create "damage flash" dissolve shader that plays on enemy hit.
  • Reduce a scene from 500+ draw calls to under 100 using batching and LOD.

🛠 Mini Project

Common Mistakes

  • Not using URP when learning Shader Graph (only works in URP or HDRP).
  • Creating 1000 bullets with Instantiate per second — causes massive GC spikes.
  • Using string concatenation in Update() — generates garbage every frame.
  • Adding every Post-Processing effect at maximum strength — overblown, unreadable image.

How to Avoid Them

  • Shader Graph needs URP or HDRP — use "3D (URP)" template for any Shader Graph project.
  • Any object created frequently at runtime: Object Pool, not Instantiate/Destroy.
  • Use $"Score: {score}" or StringBuilder, not + concatenation in Update().
  • Post-processing: use restraint — Bloom at 0.2 looks cinematic; 1.0 looks broken.
