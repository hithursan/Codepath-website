About This Curriculum

This is a 16-week, project-driven R curriculum designed to take a complete beginner to a professional-level R developer and data analyst. The roadmap moves from RStudio setup, vectors, data types, and data frames through control flow, functions, the apply family, and the tidyverse, into S3/S4/R6 OOP, file handling, error handling, ggplot2, and statistical modeling, all the way to functional programming with purrr, performance optimization, reproducible reporting, and three complete capstone projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics and R's core data structures, Phase 2 covers control flow, custom functions, and the tidyverse, Phase 3 covers intermediate and advanced core topics including OOP systems and statistical modeling, and Phase 4 covers master-level functional programming, performance, reporting, and package development.

By Week 16, you will have built three complete, portfolio-ready data analysis applications and be confident building production-grade R scripts, packages, and reproducible reports from scratch.

Level: Beginner → Master  |  Duration: 16 weeks  |  Stack: R 4.3+, RStudio, tidyverse, Quarto/R Markdown  |  Prerequisites: Basic programming logic (no R experience needed)

Week 1 — Environment Setup & R Fundamentals

Install R and RStudio, understand the R console, and write your first lines of R code.

Core Focus

Installing R and RStudio, understanding the R console, and writing your first lines of R code.

Topics & Subtopics

  1. What is R & Why Use It? — R is a programming language and environment built specifically for statistical computing, data analysis, and visualization, originally developed by statisticians for statisticians. Unlike general-purpose languages, R treats data analysis as a first-class citizen, with vectors and data frames built directly into its core syntax.
  2. Installing R & RStudio — R is the underlying language engine, while RStudio is the most popular IDE for writing and running R code, offering a console, script editor, and plotting pane in one window. Install R first from CRAN, then RStudio Desktop on top of it.
  3. The RStudio Interface: Console, Script, Environment, Plots — RStudio is divided into four key panes: the Console (where code executes immediately), the Script Editor (where you write reusable .R files), the Environment pane (showing all currently defined variables), and the Plots/Viewer pane (displaying graphs and output).
  4. Installing & Loading Packages — R's functionality is massively extended through packages — pre-built collections of functions hosted on CRAN. You install a package once with install.packages(), then load it into each session with library().

Why This Week Matters: The R and RStudio mental model — console vs script, environment pane, packages — is the foundation every R workflow builds on. Getting the tidyverse installed and loading correctly from Day 1 means you are ready for real data work from Week 2 onward.

Code Example

# R uses <- as the traditional assignment operator (= also works)
x <- 5                  # Assign the value 5 to x
print(x)                # Explicitly print x to the console
x                       # In RStudio/console, typing x alone also prints it

# Verify your setup:
R.version.string        # e.g. "R version 4.3.2"
getwd()                 # Shows your current working directory
sessionInfo()           # Detailed report: R version, OS, loaded packages

# Packages: install ONCE, load in every session
install.packages("tidyverse")   # Run once to download and install
library(tidyverse)              # Run in every script that uses it
search()                        # Lists all currently attached packages

Week 2 — Vectors & Basic Data Types

Understand R's foundational data structure — the vector — and the core atomic data types it can hold.

Core Focus

Understanding R's foundational data structure — the vector — and the core atomic data types it can hold.

Topics & Subtopics

  1. Atomic Data Types: numeric, character, logical, integer — R has four primary atomic types: numeric (decimals), integer (whole numbers, suffixed with L), character (text strings), and logical (TRUE/FALSE). Every value in R belongs to one of these types, checkable with class() or typeof().
  2. Creating Vectors with c() — A vector is R's fundamental data structure — an ordered collection of values that must all be the same type. The c() (combine) function is the primary way to create vectors, and R will automatically coerce mixed types to the most flexible common type.
  3. Vectorized Operations — R applies arithmetic operations to entire vectors element-by-element automatically — a concept called "vectorization" — eliminating the need for explicit loops in most cases. This is one of R's most powerful and distinctive features compared to general-purpose languages.
  4. Indexing & Subsetting Vectors — R uses 1-based indexing (not 0-based), and square brackets [] select one or more elements by position, by a logical condition, or by name. Negative indices exclude elements rather than selecting from the end.

Why This Week Matters: Vectors are the atom of R — every other data structure (matrix, data frame, list) is built from them. Vectorized operations are what make R code concise and fast — learning to think in vectors rather than loops is the single most important mental shift for becoming an effective R programmer.

Code Example

# Atomic types
age      <- 28L              # integer (L suffix)
price    <- 19.99            # numeric (double by default)
name     <- "Alice"          # character
is_admin <- TRUE             # logical
class(age); class(price); class(name); class(is_admin)

# Vectors with c()
numbers <- c(10, 20, 30, 40, 50)
mixed   <- c(1, "two", TRUE)   # Coercion: ALL become character!
length(numbers)                # 5

# Vectorized operations — no loop needed
prices     <- c(10, 20, 30, 40)
discounted <- prices * 0.9     # Multiplies EVERY element at once
total      <- sum(prices)      # Aggregate function

# Indexing (1-based!)
fruits <- c("Apple", "Banana", "Cherry", "Date")
fruits[1]                      # "Apple"
fruits[c(1, 3)]                # "Apple" "Cherry"
fruits[-2]                     # All EXCEPT 2nd element
fruits[fruits == "Cherry"]     # Logical subsetting

Week 3 — Data Structures: Matrices, Lists & Factors

Expand beyond simple vectors into R's richer multi-dimensional and heterogeneous data structures.

Core Focus

Expanding beyond simple vectors into R's richer multi-dimensional and heterogeneous data structures.

Topics & Subtopics

  1. Matrices — A matrix is a two-dimensional, rectangular data structure where every element must be the same type (like a vector, but with rows and columns). Matrices are created with matrix() and are essential for linear algebra and numeric computations.
  2. Lists — A list is R's most flexible data structure — unlike vectors, a list can hold elements of completely different types and lengths, including other lists. Lists use double square brackets [[ ]] to extract a single element's actual value, versus single brackets [ ] which return a sub-list.
  3. Factors (Categorical Data) — A factor represents categorical data with a fixed, known set of possible values ("levels") — like survey responses (Low/Medium/High) or blood types. Factors are R's way of efficiently storing and statistically handling categorical variables, especially in modeling.
  4. NULL, NA & Missing Data — NA represents a missing or unknown value within a vector, while NULL represents the complete absence of a value or an empty object. Most R functions require explicit handling of NA to avoid silently propagating missing-ness through calculations.

Why This Week Matters: Lists are the backbone of R's OOP systems and the return value of most modeling functions (like lm()). Factors prevent a classic data analysis bug — treating categories as numbers. Proper NA handling with na.rm = TRUE is in virtually every real-world analysis script.

Code Example

# Matrix
m <- matrix(1:6, nrow = 2, ncol = 3)
m[1, 2]      # Row 1, column 2 → 3
m[, 2]       # All rows, column 2 → 3 5
dim(m)       # 2 3

# List — different types, different lengths
person <- list(name = "Bob", age = 32, scores = c(85, 90, 78))
person$name           # "Bob"
person[["age"]]       # 32
person[["scores"]][2] # 90
str(person)           # Structure overview

# Ordered factor
sizes <- factor(c("Small", "Large", "Medium", "Small"),
                levels = c("Small", "Medium", "Large"),
                ordered = TRUE)
sizes[1] < sizes[2]   # TRUE — Small < Large
table(sizes)          # Frequency count

# NA handling
scores <- c(85, 90, NA, 78, NA)
sum(scores)                    # NA — poisoned by default!
sum(scores, na.rm = TRUE)      # 253 — NAs ignored
sum(is.na(scores))             # 2 — count of missing values

Week 4 — Data Frames & Importing Data

Work with R's most important data structure for real-world analysis — the data frame — and load external data.

Core Focus

Working with R's most important data structure for real-world data analysis — the data frame — and loading external data.

Topics & Subtopics

  1. Creating & Inspecting Data Frames — A data frame is R's equivalent of a spreadsheet or database table — a list of equal-length vectors (columns) that can each hold different types, displayed in a rectangular row/column format. It is the central data structure for nearly all data analysis work in R.
  2. Selecting Columns & Rows — Data frames support multiple selection syntaxes: $ for a single column by name, [ ] with row/column indices, and logical conditions for filtering rows. Understanding these is essential before moving to tidyverse's cleaner syntax.
  3. Reading CSV Files — read.csv() (base R) or readr::read_csv() (tidyverse, faster and more consistent) load external CSV files directly into a data frame, making file-based data immediately ready for analysis. The tidyverse version automatically guesses column types and prints a helpful summary.
  4. Writing Data Out (write.csv / write_csv) — Once you have cleaned or transformed data, write_csv() exports a data frame to a CSV file, preserving column names as the header row by default. Always confirm the file was written with file.exists().

Why This Week Matters: The data frame is the workhorse of all R data analysis. Every real-world dataset you will ever load — from CSV, Excel, SQL, or an API — lands in a data frame. The $, [ ], and logical-filter syntax are used in every single R analysis script ever written.

Code Example

library(readr)

# Create and inspect a data frame
df <- data.frame(
  name  = c("Alice", "Bob", "Carol"),
  age   = c(28, 32, 25),
  score = c(85.5, 90.2, 78.9)
)
head(df)          # Preview first rows
str(df)           # Structure: column types and sample values
nrow(df); ncol(df)  # 3 3

# Selection and filtering
df$name               # 'name' column as a vector
df[, "age"]           # Same result, matrix-style
df[1, ]               # First row, all columns
df[df$age > 26, ]     # Rows where age > 26

# Read and write CSV
sales_data <- read_csv("sales.csv")
summary(sales_data)   # Quick statistical summary

cleaned_data <- data.frame(product = c("Widget", "Gadget"),
                            revenue = c(1500, 2300))
write_csv(cleaned_data, "cleaned_output.csv")
file.exists("cleaned_output.csv")   # TRUE

Week 5 — Conditionals & Control Flow

Direct program execution using if/else logic and R's vectorized conditional functions.

Core Focus

Directing program execution using if/else logic and R's vectorized conditional functions.

Topics & Subtopics

  1. if, else if, else — R's conditional syntax evaluates a single logical condition and branches accordingly. Note that if/else in R works on a single value, not a whole vector — for vectorized conditionals, ifelse() is needed.
  2. Vectorized Conditionals with ifelse() — ifelse() applies a condition across an entire vector at once, returning a new vector where each element is chosen based on the corresponding logical test. This avoids writing a manual loop for simple element-wise conditional logic.
  3. switch() Statements — switch() selects one of several code branches based on matching a string or integer value, serving as R's alternative to long if/else if chains when comparing a single variable against multiple discrete options.
  4. Logical Operators (&, |, &&, ||, !) — R distinguishes between vectorized logical operators (&, |, which compare element-by-element) and scalar short-circuit operators (&&, ||, which compare exactly one value). Use &&/|| inside if() statements, and &/| when filtering vectors or data frames.

Why This Week Matters: The distinction between scalar if() and vectorized ifelse() is one of the most common sources of bugs for R beginners — using if() on a vector produces a warning and only checks the first element. Getting this right prevents silent, hard-to-detect errors in data processing pipelines.

Code Example

# Scalar if/else — one value only
score <- 85
if (score >= 90) {
  grade <- "A"
} else if (score >= 80) {
  grade <- "B"   # This branch executes
} else {
  grade <- "C"
}

# Vectorized ifelse() — works on the entire vector at once
scores  <- c(45, 92, 67, 88, 55)
results <- ifelse(scores >= 60, "Pass", "Fail")
# "Fail" "Pass" "Pass" "Pass" "Fail"

# switch() for discrete string matching
day_type <- function(day) {
  switch(day,
    "Mon" = , "Tue" = , "Wed" = , "Thu" = , "Fri" = "Weekday",
    "Sat" = , "Sun" = "Weekend",
    "Unknown day"
  )
}
day_type("Sat")  # "Weekend"

# Vectorized vs scalar logical operators
age      <- c(15, 22, 17, 30)
has_id   <- c(FALSE, TRUE, TRUE, TRUE)
can_enter <- age >= 18 & has_id   # Element-by-element &
if (age[2] >= 18 && has_id[2]) print("Entry granted")

Week 6 — Loops & The Apply Family

Iterate over data using traditional loops and R's idiomatic, more efficient apply family of functions.

Core Focus

Iterating over data using traditional loops and R's idiomatic, more efficient apply family of functions.

Topics & Subtopics

  1. for Loops — R's for loop iterates over each element of a vector or list, executing the loop body once per element. They remain useful for tasks with side effects like printing or file writing, though often slower than vectorized alternatives for large datasets.
  2. while Loops — while loops repeat a block of code as long as a condition remains true, checked before each iteration — useful when the number of iterations is not known in advance, unlike a for loop over a fixed-length vector.
  3. The sapply() & lapply() Functions — lapply() applies a function to every element of a list/vector and always returns a list; sapply() does the same but tries to simplify the result into a vector or matrix when possible. These are the idiomatic R replacement for many for loops, and are often faster.
  4. mapply() & vapply() — mapply() applies a function across multiple vectors simultaneously (element-wise, like a multi-argument sapply), while vapply() is a safer version of sapply() that requires you to specify the expected output type, preventing silent bugs from unexpected return shapes.

Why This Week Matters: In R, explicit for loops on large data are slow — the apply family calls optimized C code under the hood. vapply() over sapply() is the professional choice because it fails loudly if the return type is wrong, catching bugs at development time instead of silently corrupting results.

Code Example

# for loop — iterates directly over values
fruits <- c("Apple", "Banana", "Cherry")
for (fruit in fruits) {
  cat("I like", fruit, "\n")
}

# while loop
count <- 1; total <- 0
while (count <= 5) {
  total <- total + count
  count <- count + 1    # MUST update or infinite loop!
}
print(total)  # 15

# lapply vs sapply
numbers <- c(4, 9, 16, 25)
result_list <- lapply(numbers, sqrt)   # Returns a LIST
result_vec  <- sapply(numbers, sqrt)   # Returns a VECTOR (simplified)
class(result_list); class(result_vec)  # "list" "numeric"

# mapply: multi-vector element-wise
heights <- c(160, 170, 180)
weights <- c(60,  70,  85)
bmi <- mapply(function(h, w) w / (h/100)^2, heights, weights)

# vapply: enforces return type
safe_result <- vapply(heights, function(h) h / 100, FUN.VALUE = numeric(1))

Week 7 — Writing Functions

Create reusable, well-documented custom functions — the cornerstone of clean, maintainable R code.

Core Focus

Creating reusable, well-documented custom functions — the cornerstone of clean, maintainable R code.

Topics & Subtopics

  1. Defining Functions with function() — R functions are defined using the function() keyword, assigned to a name like any other variable, and automatically return the value of their last evaluated expression — no explicit return() required, though it is often clearer to include it.
  2. Default Arguments & Named Arguments — Function parameters can have default values, making them optional when calling the function. Arguments can be passed positionally (in order) or by name (in any order), with named arguments improving readability for functions with many parameters.
  3. Variable Scope & the <<- Operator — Variables created inside a function exist only in that function's local scope and disappear once it returns. The <<- "superassignment" operator can modify a variable in an enclosing (parent) scope, though it should be used sparingly as it can make code harder to reason about.
  4. Anonymous Functions & the Lambda Shorthand (\(x)) — Anonymous functions (functions without an assigned name) are commonly used inline within sapply()/lapply() calls. R 4.1+ introduced a compact lambda syntax \(x) ... as shorthand for function(x) ....

Why This Week Matters: Functions are the primary unit of reuse in R. The \(x) lambda shorthand appears everywhere in modern tidyverse code — in purrr::map(), sapply(), and inline mutate() calls. Understanding lexical scope prevents a whole class of bugs where functions unexpectedly read the wrong variable from the global environment.

Code Example

# Basic function — auto-returns last expression
calculate_bmi <- function(weight_kg, height_m) {
  weight_kg / (height_m^2)
}
calculate_bmi(70, 1.75)   # 22.86

# Default and named arguments
greet <- function(name, greeting = "Hello", punctuation = "!") {
  paste0(greeting, ", ", name, punctuation)
}
greet("Alice")                           # "Hello, Alice!"
greet("Bob", greeting = "Hey")           # "Hey, Bob!"
greet(punctuation = "?", name = "Carol") # Order doesn't matter when named

# Variable scope and <<- superassignment
counter <- 0
increment <- function() { counter <<- counter + 1 }
increment(); increment(); increment()
print(counter)  # 3 — global variable was modified

# Lambda shorthand (R 4.1+)
numbers     <- c(1, 2, 3, 4, 5)
squared_old <- sapply(numbers, function(x) x^2)
squared_new <- sapply(numbers, \(x) x^2)   # Identical result, less typing
print(squared_new)  # 1 4 9 16 25

Week 8 — Strings, Dates & The Tidyverse (dplyr/tidyr)

Manipulate text and dates, and learn the tidyverse's elegant grammar for transforming data frames.

Core Focus

Manipulating text and dates, and learning the tidyverse's elegant grammar for transforming data frames.

Topics & Subtopics

  1. String Manipulation with stringr — The stringr package (part of tidyverse) provides a consistent, predictable set of string functions, all prefixed with str_. Common operations include str_detect() (pattern matching), str_replace(), str_split(), and str_trim().
  2. Dates with lubridate — The lubridate package makes working with dates dramatically easier than base R, providing intuitive parsing functions (ymd(), mdy()) and extraction functions (year(), month(), wday()) for common date components.
  3. The Pipe Operator (%>% and |>) — The pipe operator passes the result of the expression on its left as the first argument to the function on its right, allowing you to chain multiple operations into a readable left-to-right sequence. R now has a native pipe (|>) alongside the original magrittr pipe (%>%) from tidyverse.
  4. dplyr Verbs: filter(), select(), mutate(), arrange() — dplyr provides a consistent "grammar of data manipulation" built around a handful of verbs. Chained together with the pipe, they form highly readable data transformation pipelines that read like English sentences.

Why This Week Matters: The pipe operator fundamentally changes how R code reads — transforming deeply nested function calls into clear, step-by-step left-to-right pipelines. The five core dplyr verbs (filter, select, mutate, arrange, summarize) cover 90% of all data manipulation tasks in any real analysis project.

Code Example

library(stringr); library(lubridate); library(dplyr)

# stringr — consistent str_ prefix
text <- "  Hello, R World!  "
str_trim(text)                          # "Hello, R World!"
str_to_upper(text)                      # "  HELLO, R WORLD!  "
str_detect(text, "World")               # TRUE
str_replace(text, "World", "Universe")  # "  Hello, R Universe!  "

# lubridate — intuitive date parsing
date1 <- ymd("2024-03-15")
date2 <- mdy("03/20/2024")
date2 - date1                           # Time difference of 5 days
year(date1); month(date1)               # 2024 3

# Pipe operator
numbers <- c(4, 16, 25, 9, 1)
result  <- numbers |> mean() |> sqrt() |> round(2)

# dplyr pipeline
df <- data.frame(name = c("A","B","C"), score = c(60,95,80), age = c(22,19,25))
result <- df |>
  filter(score >= 70) |>
  mutate(passed = TRUE) |>
  arrange(desc(score)) |>
  select(name, score)
print(result)

Week 9 — Object-Oriented R: S3 Classes

Understand R's most common and lightweight object-oriented system, S3, used throughout base R and many packages.

Core Focus

Understanding R's most common and lightweight object-oriented system, S3, used throughout base R and many packages.

Topics & Subtopics

  1. S3 Classes & structure() — S3 is R's simplest and most widely-used OOP system — it works by attaching a "class" attribute to an ordinary object (often a list), with no formal class definition required. Functions then behave differently based on that class attribute through a mechanism called "dispatch."
  2. Generic Functions & Method Dispatch — S3's power comes from "generic" functions like print() and summary(), which automatically dispatch to a class-specific method (e.g., print.employee) if one exists. This lets built-in functions behave appropriately for your custom object types.
  3. Writing Custom Generic Functions — You can create entirely new generic functions using UseMethod(), which tells R to look up the appropriate class-specific implementation at call time. This is how you build extensible, class-aware behavior into your own functions.
  4. Inheritance in S3 — S3 supports a simple form of inheritance by assigning an object multiple classes (a character vector), with R checking each class in order when dispatching a method. If no method exists for the first (most specific) class, R falls back to the next class in the vector.

Why This Week Matters: S3 is everywhere in R — lm() returns an S3 object, summary() and print() use S3 dispatch, and almost every major package uses S3 for its core types. Understanding S3 dispatch is what lets you write functions that work intelligently on different types of objects, just like base R does.

Code Example

# S3 object: a list with a class attribute
create_employee <- function(name, salary) {
  structure(list(name = name, salary = salary), class = "employee")
}
emp <- create_employee("Alice", 65000)
class(emp)   # "employee"

# Generic dispatch: print.employee called automatically for "employee" objects
print.employee <- function(x, ...) {
  cat("Employee:", x$name, "| Salary: $", x$salary, "\n")
}
print(emp)   # Calls print.employee(), not the default print!

# Custom generic function
describe <- function(x) UseMethod("describe")
describe.employee <- function(x) cat(x$name, "earns $", x$salary, "\n")
describe.default  <- function(x) cat("No specific description available.\n")
describe(emp)    # Dispatches to describe.employee()
describe(42)     # Dispatches to describe.default()

# S3 inheritance: multiple classes
create_manager <- function(name, salary, team_size) {
  structure(list(name = name, salary = salary, team_size = team_size),
            class = c("manager", "employee"))  # "manager" checked first
}
mgr <- create_manager("Carol", 95000, 6)
print(mgr)   # No print.manager → falls back to print.employee()

Week 10 — S4 & R6 Classes (Advanced OOP)

Move beyond S3 into R's more formal, structured object-oriented systems used in larger applications and packages.

Core Focus

Moving beyond S3 into R's more formal, structured object-oriented systems used in larger applications and packages.

Topics & Subtopics

  1. S4 Classes with setClass() — S4 is a more rigorous, formal OOP system than S3, requiring you to explicitly declare a class's structure (its "slots") and their expected types upfront using setClass(). This enforced structure provides better validation and is preferred for complex package development (e.g., Bioconductor packages).
  2. S4 Generic Methods with setGeneric() / setMethod() — Like S3, S4 supports generic functions that dispatch based on class, but with explicit declaration via setGeneric() and setMethod(). This formality provides stricter type-checking and clearer contracts than S3's more implicit convention-based approach.
  3. R6 Classes: Reference Semantics — R6 (a popular CRAN package) provides a more traditional, mutable, encapsulated OOP system closely resembling Python or Java classes, where objects are modified in-place ("reference semantics") rather than copied on every modification like S3/S4. This makes R6 ideal for stateful objects like database connections or simulations.
  4. R6 Private Fields & Methods — R6 supports true encapsulation via a private list, whose fields and methods are completely inaccessible from outside the object — only accessible from within the class's own public methods. This enables proper data hiding, unlike S3 and S4 which have no built-in privacy mechanism.

Why This Week Matters: R6 has become the standard for stateful, encapsulated objects in modern R packages — it is used by shiny, plumber, and testthat internally. Understanding the difference between S3's copy-on-modify semantics and R6's reference semantics prevents subtle bugs when passing objects between functions.

Code Example

library(R6)

# S4 class with typed slots
setClass("BankAccount", representation(
  owner   = "character",
  balance = "numeric"
))
account <- new("BankAccount", owner = "Alice", balance = 1000)
account@balance   # 1000 — S4 uses @ not $

# S4 generic and method
setGeneric("deposit", function(account, amount) standardGeneric("deposit"))
setMethod("deposit", "BankAccount", function(account, amount) {
  account@balance <- account@balance + amount
  account   # Return the updated object
})
account <- deposit(account, 500)
account@balance   # 1500

# R6 class: mutable, reference semantics
Counter <- R6Class("Counter",
  public = list(
    count = 0,
    increment = function() {
      self$count <- self$count + 1   # self = current instance
      invisible(self)                # Enables method chaining
    }
  )
)
c1 <- Counter$new()
c1$increment(); c1$increment()
c1$count   # 2 — modified IN PLACE

# R6 with private fields — true encapsulation
BankR6 <- R6Class("BankR6",
  public = list(
    initialize = function(b) { private$balance <- b },
    deposit    = function(a) { private$balance <- private$balance + a },
    getBalance = function()  { private$balance }
  ),
  private = list(balance = 0)   # acc$balance from outside → NULL (hidden!)
)
acc <- BankR6$new(100); acc$deposit(50); acc$getBalance()  # 150

Week 11 — File Handling & Error Handling

Read and write various file formats robustly, and gracefully handle errors and exceptional conditions.

Core Focus

Reading and writing various file formats robustly, and gracefully handling errors and exceptional conditions.

Topics & Subtopics

  1. Working with RDS Files (R's Native Format) — RDS is R's native binary file format for saving a single R object (of any type — data frame, list, model, etc.) to disk with perfect fidelity, preserving classes and structure exactly. Unlike CSV, RDS round-trips complex objects without any data-type loss.
  2. Reading JSON with jsonlite — The jsonlite package provides fromJSON() and toJSON() for converting between R objects and JSON, the standard format for web APIs and config files. It intelligently converts JSON arrays/objects into appropriate R vectors, lists, or data frames.
  3. tryCatch() for Error Handling — tryCatch() is R's primary mechanism for catching and handling errors gracefully, similar to try/catch in other languages. You specify handler functions for different condition types: error (fatal), warning (non-fatal), and finally (always runs).
  4. Custom Conditions with warning() and stop() — stop() immediately halts execution and raises an error, while warning() signals a potential problem without stopping the program. Both are "conditions" that tryCatch() can intercept. Writing clear, descriptive messages makes debugging far easier.

Why This Week Matters: Production R code fails gracefully — it does not crash and lose work. tryCatch() is what separates a beginner script from a production data pipeline. RDS is the professional standard for saving R model objects between sessions — it is how you save a trained machine learning model and load it months later with identical behavior.

Code Example

library(jsonlite)

# RDS: perfect round-trip for any R object
my_data <- list(name = "Alice", scores = c(85, 90, 78), active = TRUE)
saveRDS(my_data, "my_data.rds")
loaded  <- readRDS("my_data.rds")
identical(my_data, loaded)   # TRUE — identical structure

# JSON with jsonlite
json_text <- '{"name": "Bob", "age": 32, "skills": ["R", "Python"]}'
parsed    <- fromJSON(json_text)
parsed$skills                              # "R" "Python"
toJSON(parsed, pretty = TRUE)              # Convert back to formatted JSON

# tryCatch: graceful error handling
safe_divide <- function(a, b) {
  tryCatch({
    if (b == 0) stop("Cannot divide by zero!")
    a / b
  },
  error   = function(e) { cat("Error:", conditionMessage(e), "\n"); NA },
  finally = { cat("safe_divide called.\n") }
  )
}
safe_divide(10, 0)   # Prints error message, returns NA, runs finally

# Custom conditions
validate_age <- function(age) {
  if (!is.numeric(age)) stop("Age must be numeric, got: ", class(age))
  if (age < 0)          stop("Age cannot be negative: ", age)
  if (age > 120)        warning("Age seems unusually high: ", age)
  age
}
tryCatch(validate_age(150), warning = function(w) cat("Warning:", conditionMessage(w)))

Week 12 — Data Visualization & Statistical Modeling

Create publication-quality visualizations with ggplot2 and fit fundamental statistical models.

Core Focus

Creating publication-quality visualizations with ggplot2 and fitting fundamental statistical models.

Topics & Subtopics

  1. The Grammar of Graphics with ggplot2 — ggplot2 builds visualizations by layering components: data, aesthetic mappings (aes() — which columns map to x/y/color), and geometric objects (geom_*() — points, lines, bars). This layered "grammar of graphics" approach makes complex, customized plots remarkably composable.
  2. Common Plot Types: Bar, Line, Histogram, Boxplot — ggplot2 supports nearly every standard chart type through different geom_*() functions: geom_col() for bar charts, geom_line() for time series, geom_histogram() for distributions, and geom_boxplot() for comparing groups.
  3. Linear Regression with lm() — lm() fits a linear regression model using a formula syntax (y ~ x) that reads naturally as "y predicted by x." The resulting model object supports summary() for coefficients/significance and predict() for forecasting new values.
  4. Summary Statistics & summary() — The summary() function is a polymorphic generic that provides an appropriate statistical overview depending on the object passed — a five-number summary for numeric vectors, frequency counts for factors, or full model diagnostics for an lm object.

Why This Week Matters: ggplot2 is the most used data visualization library in data science — its grammar-of-graphics approach means you can build virtually any chart by composing layers. lm() is R's flagship statistical function, and understanding its output (coefficients, R², p-values) is fundamental for any data science or research role.

Code Example

library(ggplot2)

# Layered ggplot2: scatter + trend line

Week 13 — Functional Programming with purrr & Closures

Master R's functional programming paradigm using the tidyverse's purrr package and the closures that underpin it.

Core Focus

Mastering R's functional programming paradigm using the tidyverse's purrr package and the closures that underpin it.

Topics & Subtopics

  1. purrr::map() Family — purrr provides a more consistent, type-safe alternative to base R's apply family — map() always returns a list, while map_dbl(), map_chr(), and map_lgl() return simplified vectors of a guaranteed specific type, failing loudly if that type cannot be produced.
  2. Closures: Functions That Return Functions — A closure is a function that captures and "remembers" variables from its enclosing environment, even after that outer function has finished executing — enabling powerful patterns like function factories that generate customized functions on demand.
  3. Decorators: Wrapping Functions for Extra Behavior — Although R has no formal @decorator syntax, closures let you achieve the same effect — writing a function that wraps another function, adding behavior (logging, timing, caching) before and/or after the original runs, then returning the enhanced version.
  4. Memoization (Caching Function Results) — Memoization is a performance optimization, typically implemented via a closure, that caches a function's previous results so repeated calls with the same arguments return instantly from the cache instead of recomputing. This is especially valuable for expensive recursive functions.

Why This Week Matters: purrr's type-safe map_dbl()/map_chr() family prevents a whole class of silent type-coercion bugs that sapply() can introduce. Closures and decorators are the patterns behind every timing, logging, retry, and caching abstraction in professional R packages — understanding them makes you a far more powerful R programmer.

Code Example

library(purrr)

# purrr map family — type-guaranteed returns
numbers <- list(4, 9, 16, 25)
map(numbers, sqrt)                               # Always returns a LIST
map_dbl(numbers, sqrt)                           # DOUBLE vector guaranteed: 2 3 4 5
map_chr(numbers, ~ paste0("Value: ", .x))        # ~.x is purrr lambda shorthand

# Closure: function factory
make_multiplier <- function(factor) {
  function(x) x * factor    # Closes over `factor` permanently
}
double <- make_multiplier(2)
triple <- make_multiplier(3)
double(5); triple(5)  # 10  15 — each closure has its own private `factor`

# Decorator-style closure: adds timing to any function
with_timing <- function(f) {
  function(...) {
    start  <- Sys.time()
    result <- f(...)
    cat("Took:", Sys.time() - start, "\n")
    result
  }
}
slow_sum  <- function(n) sum(1:n)
timed_sum <- with_timing(slow_sum)
timed_sum(1000000)

# Memoization via closure
make_memoized_fib <- function() {
  cache <- c(1, 1)
  function(n) {
    if (n <= length(cache)) return(cache[n])   # Cache hit!
    result   <- Recall(n-1) + Recall(n-2)
    cache[n] <<- result                         # Store result
    result
  }
}
fib <- make_memoized_fib()
fib(20)   # Computed once; subsequent identical calls return instantly

Week 14 — Performance, Vectorization & Parallel Computing

Write efficient R code by avoiding common performance pitfalls and leveraging multi-core parallel processing.

Core Focus

Writing efficient R code by avoiding common performance pitfalls and leveraging multi-core parallel processing.

Topics & Subtopics

  1. Vectorization vs Explicit Loops (Performance) — R's vectorized operations (built on optimized C code under the hood) are dramatically faster than equivalent explicit for loops for large datasets, because they avoid R's per-iteration interpreter overhead. Profiling with system.time() reveals this difference concretely.
  2. Pre-allocating Memory — Growing a vector or list inside a loop (e.g., result <- c(result, new_value)) forces R to repeatedly copy the entire growing structure into new memory — a major performance trap. Pre-allocating the full-size container upfront with vector() or numeric() avoids this entirely.
  3. Profiling Code with Rprof() / microbenchmark — The microbenchmark package precisely times and statistically compares multiple implementations of the same task, running each thousands of times to produce reliable performance distributions. This is the standard tool for evidence-based optimization decisions.
  4. Parallel Computing with the parallel Package — The built-in parallel package's mclapply() (Mac/Linux) or parLapply() (cross-platform, with clusters) distributes independent computations across multiple CPU cores simultaneously, dramatically speeding up embarrassingly-parallel workloads like simulations or per-row data processing.

Why This Week Matters: The growing-vector-in-a-loop anti-pattern turns O(n) work into O(n²) — a script that takes 1 second on 1,000 rows takes 1,000 seconds on 1,000,000 rows. microbenchmark prevents premature optimization by showing you where the actual bottlenecks are, not where you guess them to be.

Code Example

library(microbenchmark); library(parallel)

n <- 1000000; x <- 1:n

# Vectorized vs loop — concrete performance comparison
system.time({
  result_loop <- numeric(n)
  for (i in 1:n) result_loop[i] <- x[i]^2   # SLOW: interpreter overhead per step
})
system.time({ result_vec <- x^2 })            # FAST: optimized C internally

# Pre-allocation — avoid growing vectors in loops
n <- 10000
# BAD: result <- c(); for (i in 1:n) result <- c(result, i^2)  ← AVOID
result <- numeric(n)                           # GOOD: allocate ALL space upfront
for (i in 1:n) result[i] <- i^2               # Fill by index — no copying

# microbenchmark: evidence-based comparison
x_bench <- 1:10000
microbenchmark(
  loop_version   = { s <- 0; for(i in x_bench) s <- s + i },
  sapply_version = sum(sapply(x_bench, identity)),
  vectorized     = sum(x_bench),
  times = 100
)

# Parallel computing
detectCores()
slow_square <- function(x) { Sys.sleep(0.01); x^2 }
results <- mclapply(1:20, slow_square, mc.cores = detectCores() - 1)

Week 15 — Reproducible Reporting & Package Development

Create shareable, reproducible analysis reports with R Markdown/Quarto and package your own R code for reuse.

Core Focus

Creating shareable, reproducible analysis reports with R Markdown/Quarto and packaging your own R code for reuse.

Topics & Subtopics

  1. R Markdown / Quarto Documents — R Markdown (and its successor, Quarto) lets you combine narrative text, executable R code, and rendered output (tables, plots) in a single document that compiles to HTML, PDF, or Word — the standard tool for reproducible data analysis reporting in R.
  2. Code Chunks & Chunk Options — Within an R Markdown/Quarto document, R code lives in "chunks" with optional settings controlling whether code, output, or warnings are displayed in the final rendered document. This fine-grained control is essential for producing clean, presentation-ready reports.
  3. Creating an R Package Structure — Packaging your reusable functions into a formal R package (using usethis::create_package()) provides a standardized structure for documentation, testing, and sharing — the same format used by every package on CRAN.
  4. Writing Function Documentation with roxygen2 — roxygen2 comments (prefixed with #') sit directly above a function definition and generate proper .Rd help documentation automatically, viewable via ?function_name once the package is built. This keeps documentation co-located with the code it describes.

Why This Week Matters: Reproducible reports are the gold standard in data science — a Quarto document can be re-run six months later on new data and produce an identical analysis automatically. Publishing a package on CRAN or GitHub is the highest form of R code sharing, and the skills here (roxygen2, testthat, usethis) are what separate a script writer from a professional R developer.

library(usethis)
