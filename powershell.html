PowerShell: Beginner to Master

Complete 12-Week Curriculum — From Cmdlets & the Pipeline to Parallel Automation, REST APIs & Production Scripts

About This Curriculum

This is a 12-week, project-driven PowerShell curriculum designed to take a complete beginner to a professional-level automation engineer. The roadmap moves from cmdlets, the pipeline, variables, and primitive types through control flow, functions, modules, and PowerShell classes, into inheritance, error handling, file system operations, REST APIs, and remoting, all the way to parallel concurrency with ForEach-Object -Parallel, functional patterns, and three complete capstone automation projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics and the PowerShell object model, Phase 2 covers control flow, functions, modules, and OOP, Phase 3 covers intermediate and advanced core topics including inheritance, error handling, and API integration, and Phase 4 covers master-level concurrency, functional patterns, and full automation system projects.

By Week 12, you will have built three complete, production-ready automation scripts and be confident designing enterprise-grade PowerShell solutions from scratch.

Level: Beginner → Master  |  Duration: 12 weeks  |  Stack: PowerShell 7+, VS Code, Windows/Linux/macOS  |  Prerequisites: Basic command-line familiarity (no PowerShell experience needed)

Phase 1 — Absolute Basics

Weeks 1–3: Syntax, types, the pipeline, and collections.

Week 1 — PowerShell Environment, Syntax & Primitive Types

Set up your shell environment and learn PowerShell's basic syntax and core data types.

Core Focus

Setting up your shell environment and learning PowerShell's basic syntax and core data types.

Topics & Subtopics

  1. PowerShell vs PowerShell 7+ & the Console/ISE/VS Code — Windows PowerShell (5.1) ships with Windows, while PowerShell 7+ is the modern, cross-platform, open-source edition you should use going forward. VS Code with the PowerShell extension is the recommended editor for writing, running, and debugging scripts on any operating system.
  2. Cmdlets & the Verb-Noun Naming Convention — A cmdlet is a lightweight PowerShell command following a strict Verb-Noun pattern, like Get-Process or Stop-Service. This consistent naming makes commands predictable and easy to discover using Get-Command — you can often guess a command before looking it up.
  3. Variables ($variable) — Variables in PowerShell are always prefixed with a $ sign and are dynamically typed, automatically inferring their type from the assigned value. PowerShell variables can hold anything from simple numbers to entire complex objects returned by cmdlets.
  4. Primitive Types (Int, Double, String, Bool) — PowerShell supports standard .NET primitive types under the hood, including integers, decimals, strings, and booleans. Because PowerShell is built on .NET, every value is actually a full .NET object with its own properties and methods accessible via dot notation.

Why This Week Matters: The Verb-Noun naming convention is what makes PowerShell self-documenting — once you know the pattern, you can discover commands without memorizing them. Understanding that every PowerShell value is a .NET object (not plain text like bash) is the single most important mental shift for becoming an effective PowerShell user.

Code Example

# Declaring variables — $ prefix always required
$name       = "PowerShell Learner"   # String (System.String)
$age        = 28                     # Int32
$pi         = 3.14159                # Double
$isLearning = $true                  # Boolean ($true/$false — lowercase in PS)

# String interpolation with double quotes
Write-Host "$name is $age years old. Learning: $isLearning"

# Every variable is a .NET object — properties and methods available
Write-Host $name.Length              # 18
Write-Host $name.ToUpper()          # POWERSHELL LEARNER
Write-Host $age.GetType().Name      # Int32

# Discover cmdlets
Get-Command Get-*                    # All cmdlets starting with Get-
Get-Help Get-Process -Examples      # Usage examples for Get-Process

Week 2 — Operators, the Pipeline & Type Conversion

Master PowerShell's operators and its single most powerful feature: the object pipeline.

Core Focus

Mastering PowerShell's operators and its single most powerful feature: the pipeline.

Topics & Subtopics

  1. Arithmetic & Comparison Operators — PowerShell supports +, -, *, /, % for arithmetic, but uses letter-based comparison operators like -eq, -ne, -lt, -gt instead of symbols. This avoids ambiguity with PowerShell's redirection operators and is consistent across all comparison scenarios.
  2. The Pipeline (|) — The pipeline passes the full output object from one cmdlet directly as input to the next, chaining commands together fluently. Unlike traditional shells that pipe plain text, PowerShell pipes rich, structured .NET objects — preserving all their properties and methods throughout the chain.
  3. Type Conversion & Casting — PowerShell can convert between types explicitly using cast syntax like [int]$value, or implicitly in many common operations. Understanding casting prevents unexpected errors when mixing strings and numbers, and is essential for processing user input safely.
  4. Get-Help & Discoverability — Get-Help <cmdlet> -Examples instantly shows usage examples for any cmdlet, making PowerShell remarkably self-documenting. Running Update-Help periodically ensures your local help files stay current with the latest documentation.

Why This Week Matters: The PowerShell pipeline is what separates it from every other shell — passing structured objects (not text) means you never write regex parsers to extract fields from command output. The letter-based comparison operators (-eq, -gt) are used in every conditional and filter in the language, so getting them instinctive from Week 2 is essential.

Code Example

# Letter-based comparison operators — not < > symbols
$x = 10; $y = 20
$x -lt $y          # True  (-lt = less than)
$x -eq 10          # True  (-eq = equal)
$x -ne $y          # True  (-ne = not equal)

# The pipeline: objects flow through, not text
Get-Process |
    Where-Object { $_.CPU -gt 10 } |      # $_ = current pipeline object
    Sort-Object CPU -Descending |
    Select-Object Name, CPU -First 5       # Top 5 by CPU

# Type casting — explicit and implicit
$value  = "42"
$number = [int]$value + 8    # Cast string → int, then add
Write-Host $number            # 50

# Size literals — PS understands KB, MB, GB
$ram = 50MB
Get-Process | Where-Object { $_.WorkingSet -gt $ram } | Select-Object Name, WorkingSet

# Self-documentation
Get-Help Where-Object -Examples
Update-Help                   # Refresh local help database

Week 3 — Collections: Arrays, Hashtables & Objects

Learn PowerShell's core data structures used to organize and manipulate data.

Core Focus

Learning PowerShell's core data structures used to organize and manipulate data.

Topics & Subtopics

  1. Arrays (@()) — An array stores an ordered collection of items, declared using @() syntax or comma-separated values. Arrays support indexing, slicing, and built-in methods for adding or filtering elements — and like everything in PowerShell, they are full .NET objects.
  2. Hashtables (@{}) — A hashtable stores unordered key-value pairs, ideal for representing configuration settings, lookup tables, or labeled data. Keys are accessed using either dot notation or bracket syntax, similar to dictionaries in Python or objects in JavaScript.
  3. Custom Objects ([PSCustomObject]) — [PSCustomObject] lets you create structured objects with named properties on the fly, similar to a lightweight struct or record. This is the idiomatic PowerShell way to build clean, readable, table-friendly output from your scripts.
  4. Object Properties & Methods (Get-Member) — Every value in PowerShell is a .NET object with its own properties (data) and methods (actions), discoverable using Get-Member. Understanding this object model is fundamental to knowing what you can do with any value returned by a cmdlet.

Why This Week Matters: [PSCustomObject] is the professional standard for structured output in PowerShell — it produces clean table output with Format-Table, serializes correctly to JSON and CSV, and integrates seamlessly with the pipeline. Every real-world script that produces reports or structured data uses it.

Code Example

# Array: ordered, indexed collection
$fruits = @("Apple", "Banana", "Mango")
$fruits[0]                          # "Apple" — 0-based indexing
$fruits[-1]                         # "Mango" — last element
$fruits.Count                       # 3
$fruits += "Orange"                 # Add element (creates new array)

# Hashtable: key-value pairs
$prices = @{
    Apple  = 1.50
    Banana = 0.50
    Mango  = 2.00
}
$prices["Apple"]                    # 1.50 — bracket access
$prices.Banana                      # 0.50 — dot access

# PSCustomObject: structured, table-friendly output
$item = [PSCustomObject]@{
    Name     = "Apple"
    Price    = 1.50
    InStock  = $true
}
$item.Name                          # "Apple"
$item | Format-Table                # Displays as a clean table row

# Get-Member: discover properties and methods
$item | Get-Member                  # Lists all properties and methods
"Hello" | Get-Member                # String's available methods

Phase 2 — Control Flow & Logic

Weeks 4–6: Conditionals, loops, functions, modules, and PowerShell classes.

Week 4 — Control Flow: Conditionals & Loops

Direct script execution using PowerShell's conditional and looping constructs.

Core Focus

Directing script execution using PowerShell's conditional and looping constructs.

Topics & Subtopics

  1. if / elseif / else Statements — Conditional statements let scripts branch logic based on Boolean expressions, using the letter-based comparison operators covered in Week 2. PowerShell's if blocks use curly braces similar to C-family languages, and conditions can include compound logic with -and, -or, and -not.
  2. switch Statements — PowerShell's switch statement evaluates a value against multiple conditions and even supports wildcard (-Wildcard) and regex (-Regex) matching directly in each case. It is a clean, powerful alternative to long if-elseif chains for discrete-value branching.
  3. foreach & ForEach-Object — The foreach keyword loops over a collection in script blocks efficiently, while ForEach-Object is the pipeline-friendly cmdlet version used after a |. Both iterate through items but are chosen based on context: script or pipeline.
  4. for & while Loops — A for loop repeats a fixed number of times using a counter variable, while a while loop repeats as long as a condition remains true. Both are essential for retry logic, polling, countdown timers, and counted iterations in automation scripts.

Why This Week Matters: PowerShell's switch statement with -Wildcard and -Regex support is uniquely powerful — it can match file extensions, log patterns, and error codes in a single clean block. The distinction between foreach (in-memory collection) and ForEach-Object (pipeline-streaming) affects both memory usage and performance at scale.

Code Example

# if/elseif/else with compound conditions
$score = 85
if ($score -ge 90) {
    Write-Host "Grade: A"
} elseif ($score -ge 80 -and $score -lt 90) {
    Write-Host "Grade: B"     # This branch executes
} else {
    Write-Host "Grade: C or below"
}

# switch with script block conditions
$numbers = 1..10
foreach ($n in $numbers) {
    switch ($n) {
        { $_ % 15 -eq 0 } { Write-Host "FizzBuzz"; break }
        { $_ % 3  -eq 0 } { Write-Host "Fizz";     break }
        { $_ % 5  -eq 0 } { Write-Host "Buzz";     break }
        default            { Write-Host $_ }
    }
}

# ForEach-Object in pipeline (streaming — memory efficient)
1..5 | ForEach-Object { "Item: $_" }

# while loop: retry pattern
$attempts = 0
while ($attempts -lt 3) {
    $attempts++
    Write-Host "Attempt $attempts"
}

Week 5 — Functions, Parameters & Modules

Structure scripts into reusable, parameterized functions and organize them into importable modules.

Core Focus

Structuring scripts into reusable, parameterized functions and organizing them into importable modules.

Topics & Subtopics

  1. Function Declaration (function) — Functions are declared with the function keyword followed by a Verb-Noun name and a script block. Following the cmdlet naming convention makes your custom functions feel like native PowerShell commands and work predictably with Get-Command.
  2. Parameters & param() Blocks — The param() block defines a function's expected inputs, including type constraints, default values, and validation attributes. PowerShell supports advanced parameter attributes like [Parameter(Mandatory=$true)] for automatic input validation with user prompts.
  3. Pipeline Input (ValueFromPipeline) — A function can accept input directly from the pipeline by marking a parameter with [Parameter(ValueFromPipeline=$true)] and using a process block to handle each piped object. This lets your custom functions integrate seamlessly into pipeline chains exactly like built-in cmdlets.
  4. Modules (.psm1) & Import-Module — A module is a .psm1 file (or folder with a manifest) bundling related functions together for reuse across scripts and sessions. Import-Module loads a module's exported functions into your current session, similar to importing a library in any other language.

Why This Week Matters: Pipeline-aware functions with ValueFromPipeline are what make custom functions feel native — they compose with other cmdlets exactly like Where-Object or Sort-Object. Modules with Export-ModuleMember are the professional standard for distributing PowerShell tooling — every enterprise script library uses this pattern.

Code Example

# Function with param block, type constraints, and default values
function Get-Square {
    param(
        [Parameter(Mandatory=$true)]
        [int]$Number,
        [bool]$Verbose = $false
    )
    if ($Verbose) { Write-Host "Computing square of $Number..." }
    return $Number * $Number
}
Get-Square -Number 6           # 36
Get-Square -Number 8 -Verbose $true

# Pipeline-aware function using process block
function ConvertTo-Celsius {
    param(
        [Parameter(ValueFromPipeline=$true)]
        [double]$Fahrenheit
    )
    process {
        [PSCustomObject]@{
            Fahrenheit = $Fahrenheit
            Celsius    = [math]::Round(($Fahrenheit - 32) * 5/9, 2)
        }
    }
}
# Works as pipeline: 32, 212, 98.6 passed as pipeline objects
32, 212, 98.6 | ConvertTo-Celsius | Format-Table

Week 6 — Custom Objects, Classes & Advanced Data Structures

Model structured data and introduce true object-oriented programming with PowerShell classes.

Core Focus

Modeling structured data and introducing true object-oriented programming with PowerShell classes.

Topics & Subtopics

  1. PowerShell Classes (class keyword) — PowerShell 5+ supports defining true class types with properties and methods, similar to C# or Java. Classes provide stronger structure and type safety compared to [PSCustomObject] for more complex, reusable applications and larger script projects.
  2. Constructors & Properties — A class constructor (a method with the same name as the class) initializes a new object's properties when it is created, ensuring every instance starts in a valid state. Properties define the typed data each instance of the class will hold throughout its lifetime.
  3. Methods on Classes — Methods define behavior that belongs to a class, letting objects perform actions using their own internal data via $this. This encapsulates logic directly with the data it operates on — a core OOP principle that makes code more organized and testable.
  4. Sorting, Filtering & Grouping Data — Sort-Object, Where-Object, and Group-Object are essential cmdlets for reshaping collections of data based on property values. These cmdlets work seamlessly with both built-in .NET objects and your own custom class instances.

Why This Week Matters: PowerShell classes integrate directly with the .NET type system — instances work as first-class pipeline objects, display correctly in Format-Table, and serialize properly to JSON and CSV without any special handling. Understanding $this and typed properties prevents the class of bugs that [PSCustomObject]'s dynamic nature can silently introduce.

Code Example

# PowerShell class with constructor and methods
class Rectangle {
    [double]$Width
    [double]$Height

    # Constructor
    Rectangle([double]$w, [double]$h) {
        $this.Width  = $w
        $this.Height = $h
    }

    # Methods using $this to access instance data
    [double] Area()      { return $this.Width * $this.Height }
    [double] Perimeter() { return 2 * ($this.Width + $this.Height) }

    [string] ToString() {
        return "Rectangle(${$this.Width} x ${$this.Height})"
    }
}

# Instantiate using ::new()
$r1 = [Rectangle]::new(4, 5)
$r2 = [Rectangle]::new(10, 3)
$r3 = [Rectangle]::new(6, 6)

Write-Host "Area: $($r1.Area())"   # 20

# Works seamlessly with pipeline cmdlets
$r1, $r2, $r3 |
    Where-Object { $_.Area() -gt 20 } |
    Sort-Object { $_.Area() } |
    Format-Table Width, Height, @{N="Area"; E={$_.Area()}}

Week 7 — Inheritance, Interfaces & Advanced OOP

Extend PowerShell's class system with inheritance and shared behavior contracts.

Core Focus

Extending PowerShell's class system with inheritance and shared behavior contracts.

Topics & Subtopics

  1. Class Inheritance — A PowerShell class can inherit from a parent class using a colon (class Child : Parent), automatically gaining all its properties and methods. This avoids duplicating shared logic across related classes and models "is-a" relationships like a Circle is-a Shape.
  2. Method Overriding — A subclass can redefine a parent's method to provide specialized behavior while keeping the same method signature. This allows different object types to share a common interface but return different results when the same method name is called, enabling polymorphic code.
  3. Hidden & Static Members — The hidden keyword prevents a property or method from appearing in default Format-Table output or Get-Member results, useful for internal implementation details. The static keyword defines members that belong to the class type itself rather than any individual instance.
  4. Polymorphism in Practice — Polymorphism allows objects of different subclasses to be treated uniformly through their shared parent type, simplifying code that works with collections of mixed but related object types. This is especially powerful when processing arrays containing different shape, sensor, or report types.

Why This Week Matters: Polymorphism is what enables truly reusable automation — a function that accepts a base Shape type works automatically with any current or future subclass. Static members like counters or factory methods are used throughout enterprise PowerShell tooling to track instances and enforce creation patterns.

Code Example

# Base class with virtual method
class Shape {
    hidden [string]$_type = "Unknown"   # hidden from default output
    static [int]$Count = 0              # shared across ALL instances

    Shape() { [Shape]::Count++ }        # increment on every instantiation

    [double] Area() { return 0 }        # base — intended to be overridden
    [string] Describe() { return "Shape with area $($this.Area())" }
}

# Subclass: inherits Shape, overrides Area()
class Circle : Shape {
    [double]$Radius
    Circle([double]$r) { $this.Radius = $r }
    [double] Area() { return [math]::Round([math]::Pi * $this.Radius * $this.Radius, 2) }
}

class Square : Shape {
    [double]$Side
    Square([double]$s) { $this.Side = $s }
    [double] Area() { return $this.Side * $this.Side }
}

# Polymorphism: same call, different behavior
$shapes = @([Circle]::new(3), [Square]::new(5), [Circle]::new(7))
$shapes | ForEach-Object { $_.Describe() }
Write-Host "Total shapes created: $([Shape]::Count)"

Week 8 — Error Handling & File System Operations

Write robust scripts that handle failures gracefully and manipulate the file system confidently.

Core Focus

Writing robust scripts that handle failures gracefully and manipulate files confidently.

Topics & Subtopics

  1. try / catch / finally — A try block lets you attempt risky code, catch handles any resulting terminating error (with $_ holding the error record), and finally runs cleanup code regardless of whether an error occurred. This is PowerShell's primary structured error-handling mechanism.
  2. $ErrorActionPreference & -ErrorAction — By default, many cmdlet errors in PowerShell are non-terminating and will not trigger a catch block unless explicitly configured. Setting -ErrorAction Stop on a cmdlet (or $ErrorActionPreference = 'Stop' globally) forces all errors to be catchable terminating errors.
  3. Custom Exceptions & throw — The throw keyword raises a terminating error, optionally with a custom message or exception object, which can then be caught by a try/catch block. This lets you signal specific failure conditions in your own functions with descriptive, debuggable error messages.
  4. File & Folder Operations (Get-ChildItem, Get-Content, Set-Content) — Get-ChildItem lists files and folders (with -Recurse and -Filter support), while Get-Content and Set-Content read and write file contents. Combined with Test-Path and New-Item, these cmdlets form the foundation of all PowerShell file system automation.

Why This Week Matters: The -ErrorAction Stop requirement is one of PowerShell's most common gotchas — newcomers write try/catch blocks that silently do nothing because the error was non-terminating. Production scripts always combine -ErrorAction Stop with try/catch/finally to ensure every failure is handled, logged, and cleaned up reliably.

Code Example

# Error handling with -ErrorAction Stop to make errors catchable
function Read-SafeFile {
    param(
        [Parameter(Mandatory=$true)]
        [string]$Path
    )
    if (-not (Test-Path $Path)) {
        throw [System.IO.FileNotFoundException]"File not found: $Path"
    }
    try {
        $content = Get-Content -Path $Path -ErrorAction Stop
        return $content
    } catch [System.IO.IOException] {
        Write-Warning "IO Error: $($_.Exception.Message)"
        return $null
    } catch {
        Write-Error "Unexpected error: $($_.Exception.Message)"
        throw   # Re-throw unhandled errors
    } finally {
        Write-Verbose "Read attempt for '$Path' complete."
    }
}

# File system operations
Get-ChildItem -Path "C:\Logs" -Filter "*.log" -Recurse |
    Where-Object { $_.LastWriteTime -lt (Get-Date).AddDays(-30) } |
    Select-Object Name, LastWriteTime

# Write summary to file
"Line count: $(Get-Content 'input.txt' | Measure-Object -Line | Select-Object -Expand Lines)" |
    Set-Content -Path "summary.txt"

Week 9 — Modules, Remoting & Working with APIs

Leverage PowerShell's ecosystem and connect to remote systems and web services.

Core Focus

Leveraging PowerShell's ecosystem and connecting to remote systems and web services.

Topics & Subtopics

  1. The PowerShell Gallery & Install-Module — The PowerShell Gallery is a public repository of community and Microsoft-published modules, installable via Install-Module. This vastly extends PowerShell's built-in capabilities for managing cloud services, Active Directory, databases, and virtually any enterprise system.
  2. PowerShell Remoting (Invoke-Command, Enter-PSSession) — PowerShell Remoting lets you run commands on remote machines over WinRM or SSH, either interactively (Enter-PSSession) or as a scripted one-off command (Invoke-Command). This is central to managing fleets of servers at scale without physical or RDP access.
  3. Working with REST APIs (Invoke-RestMethod) — Invoke-RestMethod sends HTTP requests and automatically parses JSON or XML responses into PowerShell objects with no manual parsing required. This makes consuming web APIs remarkably simple — the response is immediately ready for pipeline processing.
  4. JSON & CSV Handling (ConvertTo-Json, Import-Csv) — ConvertTo-Json/ConvertFrom-Json convert between PowerShell objects and JSON text, while Import-Csv/Export-Csv handle tabular CSV data effortlessly. These cmdlets make PowerShell excellent for data interchange, reporting tasks, and building lightweight ETL pipelines.

Why This Week Matters: Invoke-RestMethod automatically deserializes JSON into PowerShell objects — a response from the GitHub API becomes a proper PowerShell object you can pipe to Where-Object immediately. Combined with Export-Csv and ConvertTo-Json, this makes PowerShell one of the fastest languages for building lightweight API integration and data reporting tools.

Code Example

# PowerShell Gallery: install a community module
Install-Module -Name PSReadLine -Force -Scope CurrentUser

# REST API: automatic JSON → object deserialization
$response = Invoke-RestMethod -Uri "https://api.github.com/users/octocat" -Method GET
Write-Host "$($response.name) has $($response.public_repos) public repos"
Write-Host "Account created: $($response.created_at)"

# Save response as JSON file
$response | ConvertTo-Json -Depth 5 | Set-Content -Path "user.json"

# Read it back and process
$user = Get-Content "user.json" | ConvertFrom-Json
Write-Host "Loaded: $($user.login)"

# CSV: import tabular data and process with pipeline
Import-Csv -Path "employees.csv" |
    Where-Object { [int]$_.Salary -gt 50000 } |
    Sort-Object Salary -Descending |
    Export-Csv -Path "high_earners.csv" -NoTypeInformation

Week 10 — Concurrency: Jobs, Runspaces & ForEach-Object -Parallel

Run multiple operations simultaneously to dramatically speed up automation tasks.

Core Focus

Running multiple operations simultaneously to dramatically speed up automation tasks.

Topics & Subtopics

  1. Background Jobs (Start-Job) — Start-Job runs a script block in a separate background process, freeing up your current session to continue working while the job runs. Results are retrieved later using Receive-Job once the job completes, and jobs can be waited on with Wait-Job.
  2. ForEach-Object -Parallel (PowerShell 7+) — PowerShell 7 introduced -Parallel on ForEach-Object, allowing pipeline iterations to run concurrently across multiple threads with minimal syntax overhead. This is the simplest, most modern way to parallelize repetitive independent tasks like health checks or file processing.
  3. Runspaces & Runspace Pools — A runspace is a lightweight execution environment for PowerShell code, and a runspace pool manages many of them for true, high-performance parallel execution. This is the most advanced and efficient concurrency technique, used for demanding automation workloads that process thousands of items.
  4. Thread Safety & Synchronized Collections — When multiple threads access shared data simultaneously, race conditions can corrupt results unless synchronization is used. PowerShell supports thread-safe collections like [System.Collections.Concurrent.ConcurrentBag[object]] for safely aggregating parallel results without data corruption.

Why This Week Matters: Checking 100 servers sequentially takes 100× the time of checking 1 — ForEach-Object -Parallel with a -ThrottleLimit can reduce that to roughly 100/ThrottleLimit × time. This single technique can transform an overnight batch job into a 5-minute script, which is why parallel execution is the skill that most dramatically increases automation productivity at scale.

Code Example

# Background job: fire and forget
$job = Start-Job { Start-Sleep 3; "Job done!" }
Write-Host "Main thread continues..."
$result = $job | Wait-Job | Receive-Job
Write-Host $result

# ForEach-Object -Parallel: concurrent processing (PS 7+)
$servers = "Server01", "Server02", "Server03", "Server04", "Server05"
$results = $servers | ForEach-Object -Parallel {
    $ping = Test-Connection $_ -Count 1 -Quiet
    [PSCustomObject]@{
        Server = $_
        Online = $ping
    }
} -ThrottleLimit 3      # Max 3 threads running simultaneously

Week 11 — Generators, Decorators & Functional Patterns

Apply advanced functional patterns to write cleaner, more powerful PowerShell automation.

Core Focus

Applying advanced functional patterns to write cleaner, more powerful PowerShell automation.

Topics & Subtopics

  1. "Generators" via IEnumerable & Pipeline Streaming — PowerShell does not have a native yield keyword, but functions that write objects one at a time to the output stream via the process block achieve the same lazy, memory-efficient behavior as generators. This lets large datasets be processed item-by-item instead of loading everything into memory at once.
  2. "Decorators" via Function Wrapping & Proxy Functions — Since PowerShell lacks built-in decorator syntax, the same effect is achieved by wrapping a function (or creating a proxy command) that adds behavior like logging, timing, or retry logic before and after calling the original. This pattern is commonly used to add cross-cutting functionality without modifying existing cmdlets.
  3. Script Blocks as First-Class Values — A [scriptblock] is a chunk of PowerShell code stored in a variable, passable as a parameter and invoked later with & or .Invoke(). This enables powerful functional patterns like custom filters, callbacks, strategy patterns, and dynamically composed logic.
  4. Measure-Command & Performance Profiling — Measure-Command wraps any script block and reports precisely how long it took to execute, making it the essential tool for identifying performance bottlenecks. This is commonly combined with decorator-style wrapper functions for automatic performance logging across an entire script library.

Why This Week Matters: Script blocks as first-class values unlock PowerShell's full power as a functional language — you can pass behavior as data, compose functions dynamically, and build strategy patterns without inheritance. The decorator/wrapper pattern is how enterprise PowerShell tools add logging, auditing, and retry logic to every function call without touching the original function code.

Code Example

# Script block as first-class value — passed as parameter
function Invoke-WithLogging {
    param(
        [Parameter(Mandatory=$true)]
        [string]$Name,
        [Parameter(Mandatory=$true)]
        [scriptblock]$Action
    )
    Write-Host "[LOG] Starting: $Name"
    $result = & $Action             # Invoke the script block with &
    Write-Host "[LOG] Completed: $Name"
    return $result
}

Invoke-WithLogging -Name "GetDate" -Action { Get-Date }

# Decorator: wraps ANY function with timing
function Invoke-Timed {
    param([scriptblock]$Action)
    $elapsed = Measure-Command { $script:result = & $Action }
    Write-Host "Elapsed: $($elapsed.TotalMilliseconds)ms"
    return $script:result
}
Invoke-Timed { 1..1000 | ForEach-Object { $_ * 2 } | Measure-Object -Sum }

# Pipeline streaming: lazy, memory-efficient "generator" pattern
function Get-NumberStream {
    param([int]$From, [int]$To)
    process {                        # process block streams one at a time
        $From..$To | ForEach-Object { $_ * $_ }
    }
}
1 | Get-NumberStream -From 1 -To 1000000 | Select-Object -First 5

# Project 3: Parallel API/Server Health Monitor
$endpoints = @(
    "https://api.github.com",
    "https://www.microsoft.com",
    "https://jsonplaceholder.typicode.com/posts/1",
    "https://httpbin.org/get"
)

$bag = [System.Collections.Concurrent.ConcurrentBag[object]]::new()

$totalTime = Measure-Command {
    $endpoints | ForEach-Object -Parallel {
        $url    = $_
        $localBag = $using:bag
        try {
            $elapsed = Measure-Command {
                $r = Invoke-WebRequest -Uri $url -ErrorAction Stop
            }
            $localBag.Add([PSCustomObject]@{
                Url          = $url
                Status       = $r.StatusCode
                ResponseMs   = [math]::Round($elapsed.TotalMilliseconds, 1)
                Healthy      = $true
            })
        } catch {
            $localBag.Add([PSCustomObject]@{
                Url        = $url
                Status     = "Error"
                ResponseMs = -1
                Healthy    = $false
            })
        }
    } -ThrottleLimit 5
}

$bag | Sort-Object ResponseMs | Format-Table -AutoSize
Write-Host "Total wall-clock time: $($totalTime.TotalSeconds)s"
$bag | Export-Csv "health_report.csv" -NoTypeInformation
