About This Curriculum

This is a 15-week, phase-based Julia curriculum designed to take a complete beginner to a professional-level scientific computing and data engineering developer. The roadmap moves from syntax, numeric types, arrays, and broadcasting through control flow, multiple dispatch, collections, modules, and custom structs, into abstract types, file I/O, error handling, and the package ecosystem, all the way to metaprogramming, parallelism, performance engineering, and three complete capstone projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics including Julia's unique numeric type hierarchy and 1-based arrays, Phase 2 covers control flow, Julia's defining multiple dispatch paradigm, and modules, Phase 3 covers intermediate and advanced core topics like the type system, file I/O, and the data science package ecosystem, and Phase 4 covers master-level metaprogramming, parallelism, performance engineering, and production projects.

By Week 15, you will have built three complete, portfolio-ready applications and be confident building production-grade Julia programs across scientific computing, data engineering, and high-performance simulation domains.

Level: Beginner → Master  |  Duration: 15 weeks  |  Stack: Julia 1.10+, Pkg.jl, DataFrames.jl  |  Prerequisites: Basic programming logic

Phase 1 — Absolute Basics

Weeks 1–3: Setup, core types, variables, and operators.

Week 1 — Introduction to Julia & Environment Setup

Understand what Julia is, why it was created, and write your first working code.

Core Focus

Understanding what Julia is, why it was created, and writing your first working Julia scripts.

Topics & Subtopics

  1. What is Julia & Why Use It? — Julia is a high-level, high-performance language designed for numerical and scientific computing, combining the ease of Python with speed approaching C. It achieves this through Just-In-Time (JIT) compilation via LLVM, making it the language of choice for data science, machine learning research, and computational simulations.
  2. Installing Julia & juliaup — juliaup is Julia's official version manager, installed from julialang.org, that handles downloading, updating, and switching between Julia versions effortlessly. Once installed, the julia command launches the interactive REPL directly from your terminal.
  3. The Julia REPL & Package Manager Mode — The REPL has several special modes accessed by typing a character: ] enters Pkg mode for installing packages, ; enters shell mode for OS commands, and ? enters help mode for documentation lookup — each exited with backspace.
  4. Variables, Naming Conventions & Unicode Support — Julia variables require no type declaration and use dynamic typing by default. Uniquely, Julia supports Unicode characters in identifiers (typed via LaTeX-style tab completion, e.g., \alpha<TAB> becomes α) — embracing mathematical notation directly in code.

Why This Week Matters: Julia was created to solve the "two-language problem" — writing prototypes in Python and rewriting for performance in C/Fortran. Understanding this design philosophy from Week 1 explains every subsequent design decision: why types matter, why multiple dispatch exists, and why Julia's performance model is fundamentally different from scripting languages.

Code Example

# Julia's classic "Hello World" — no semicolons required
println("Hello, Julia!")
println("Julia version: ", VERSION)

# In the REPL (interactive modes):
# ]  → Pkg mode:    pkg> add DataFrames
# ;  → Shell mode:  shell> ls
# ?  → Help mode:   help?> println

name = "Alice"            # dynamically typed — no declaration needed
age = 30
α = 0.05                  # Unicode identifier — type \alpha then Tab
π_approx = 3.14159
println("$name is $age years old (α = $α)")

Week 2 — Data Types, Operators & String Interpolation

Learn Julia's numeric type hierarchy, perform calculations with operators, and format output with strings.

Core Focus

Learning Julia's rich numeric type hierarchy, performing calculations with operators including chained comparisons, and formatting output with string interpolation.

Topics & Subtopics

  1. Numeric Types (Int64, Float64, Rational, Complex) — Julia has a rich numeric type hierarchy: Int64 and Float64 are the defaults on most systems, Rational represents exact fractions (3//4), and Complex numbers are first-class citizens written with the im suffix. All numeric types support seamless arithmetic interop.
  2. Arithmetic, Comparison & Operators — Julia supports standard arithmetic (+, -, *, /, ÷ for integer division, % for modulo, ^ for power) plus chained comparisons (1 < x < 10) that read naturally like mathematical notation — a feature most languages lack.
  3. String Interpolation & Concatenation — Strings use double quotes; interpolation embeds expressions with $variable or $(expression). Concatenation uses the * operator (not +), reflecting Julia's view of strings as a mathematical "product" of sequences.
  4. Type Conversion, Promotion & typeof — convert(Type, value) performs explicit conversion; Julia's type promotion system automatically widens mixed-type arithmetic (e.g., Int + Float64 → Float64) to avoid precision loss. typeof() inspects any value's concrete runtime type.

Why This Week Matters: Julia's Rational type — representing exact fractions like 1//3 without any rounding error — is not available in Python or JavaScript. Chained comparisons (1 < x < 10) eliminate an entire class of logic bugs from numerical code. These features exist because Julia was designed by mathematicians, for mathematicians.

Code Example

x = 42                    # Int64 by default
y = 3.14                  # Float64 by default
r = 3//4                  # Rational{Int64} — exact fraction
c = 2 + 3im               # Complex{Int64}
println(typeof(x), " ", typeof(r))

a, b = 17, 5
println(a / b)       # 3.4   — always returns Float64
println(a ÷ b)        # 3     — integer division
println(a % b)        # 2     — remainder
println(1 < a < 100)  # true  — chained comparison

name = "Bob"; score = 88
println("User $name scored $(score)%")
greeting = "Hello, " * name * "!"    # * for concatenation
println(greeting)

x = 42
y = convert(Float64, x)    # explicit conversion → 42.0
println(typeof(y))           # Float64
z = 1 + 2.5                  # automatic promotion
println(typeof(z))            # Float64

Week 3 — Tuples, Arrays Basics & Booleans

Learn Julia's foundational collection types and understand 1-based indexing and column-major arrays.

Core Focus

Learning Julia's foundational collection types and understanding the critical design decisions around 1-based indexing and column-major array storage.

Topics & Subtopics

  1. Tuples & Named Tuples — Tuples ((1, 2, 3)) are fixed-size, immutable, ordered collections — extremely fast since their size is known at compile time. Named tuples ((x=1, y=2)) add field-name access, commonly used for lightweight structured data without defining a full type.
  2. Arrays — Creation & 1-Based Indexing — Julia arrays are mutable, dynamically resizable, and use 1-based indexing — a deliberate choice aligning with mathematical and scientific notation conventions. Functions ending in ! signal in-place mutation, like push!.
  3. Multi-Dimensional Arrays & Column-Major Order — Julia arrays support N dimensions natively (zeros(3, 4) creates a 3×4 matrix). Internally, arrays are stored in column-major order (unlike C/Python's row-major), meaning iterating column-by-column is significantly faster — a key performance consideration.
  4. Booleans & Comparison Operators — Julia's Bool type has exactly two values, true and false, and unlike many dynamic languages, no other value is implicitly "truthy" — conditions must explicitly evaluate to a Bool, catching a common class of bugs at the type level.

Why This Week Matters: Julia's column-major array storage is not an accident — it matches Fortran and MATLAB conventions, the languages that dominated scientific computing before Julia. Iterating over columns instead of rows can be 10x faster on large matrices due to CPU cache behavior. Understanding this layout now prevents the most common Julia performance mistake.

Code Example

point = (3, 4)                      # plain tuple
println(point[1])                    # 3 — 1-based indexing!

named = (x=3, y=4, label="origin")  # named tuple
println(named.x, " ", named.label)   # 3 origin

arr = [10, 20, 30, 40]
println(arr[1])              # 10 — first element
println(arr[end])             # 40 — last element
push!(arr, 50)                # mutate in place — ! signals mutation
println(arr)                  # [10, 20, 30, 40, 50]

matrix = [1 2 3; 4 5 6]             # 2×3 matrix
println(matrix[2, 3])                # 6 — row 2, column 3
println(size(matrix))                # (2, 3)
println(matrix[:, 1])                # [1, 4] — entire first column (fast, contiguous)

flag = true
println(typeof(flag))           # Bool
result = 5 > 3                   # comparison returns Bool
println(result && flag)          # true

Phase 2 — Control Flow & Logic

Weeks 4–7: Functions, collections, multiple dispatch, and modules.

Week 4 — Control Flow & Functions

Direct program logic with Julia's control structures and define functions — Julia's most central language feature.

Core Focus

Directing program logic with Julia's expressive control structures and defining functions with multiple return values, optional arguments, and keyword parameters.

Topics & Subtopics

  1. Conditionals — if, elseif, else — Julia's if blocks are terminated with end (no braces). Like many expression-oriented languages, if/else can also be used as an expression that returns a value, assignable directly to a variable.
  2. Loops — for and while — for loops iterate over ranges (1:10), arrays, or any iterable; while loops run while a condition holds. Julia's range syntax (start:stop or start:step:stop) is concise and used pervasively throughout the language.
  3. Function Definitions — Standard & One-Liner Syntax — Functions are declared with function name(args) ... end, or as compact one-liners using name(args) = expression for simple cases. Julia functions can return multiple values as a tuple, automatically destructured at the call site.
  4. Optional & Keyword Arguments — Julia distinguishes positional arguments from keyword arguments (separated by ; in the signature). Both support default values, making function signatures flexible without needing overloads.

Why This Week Matters: Julia's if as an expression — returning a value directly to a variable — eliminates the need for a ternary operator and produces cleaner code. Multiple return values via automatic tuple packing and destructuring is a pattern used constantly in Julia's standard library, making it essential to master before progressing to multiple dispatch.

Code Example

score = 82
grade = if score >= 90
    "A"
elseif score >= 80
    "B"
else
    "F"
end
println(grade)   # B — if used as an expression

for i in 1:2:10          # start:step:stop — odd numbers 1 to 9
    print(i, " ")
end

function divmod_julia(a, b)
    return a ÷ b, a % b              # multiple return values
end
quotient, remainder = divmod_julia(17, 5)
println(quotient, " ", remainder)     # 3 2

square(x) = x^2                       # one-liner function syntax
println(square(7))                    # 49

function greet(name, greeting="Hello"; punctuation="!")
    return "$greeting, $name$punctuation"
end
println(greet("Alice"))                          # Hello, Alice!
println(greet("Bob", "Hi"; punctuation="?"))      # Hi, Bob?

Week 5 — Collections: Arrays, Dicts, Sets & Comprehensions

Master Julia's core collection types and write elegant, vectorised code using comprehensions and broadcasting.

Core Focus

Mastering Julia's core collection types and writing elegant, vectorised code using comprehensions and the dot broadcasting syntax.

Topics & Subtopics

  1. Array Manipulation — push!, pop!, filter, map — Functions ending in ! (a Julia naming convention) indicate the function mutates its argument in place. map and filter are higher-order functions central to idiomatic Julia, often combined with anonymous functions using the -> syntax.
  2. Dictionaries (Dict) — Dict{KeyType, ValueType} stores key-value pairs with O(1) average lookup, created with Dict("key" => value, ...) syntax. Iterate with for (k, v) in dict, and check membership with haskey.
  3. Sets & Set Operations — Set stores unique, unordered elements with fast membership testing, and supports mathematical set operations directly: union, intersect, setdiff — reflecting Julia's mathematical design philosophy.
  4. Array Comprehensions & Broadcasting (the . Syntax) — Comprehensions ([expr for x in collection]) build arrays concisely. Broadcasting with the dot syntax (.+, sin.(), f.()) applies any function element-wise across arrays — one of Julia's most powerful and distinctive features.

Why This Week Matters: Broadcasting — the dot syntax — is Julia's answer to NumPy's vectorization. sqrt.(arr) applies sqrt to every element with no overhead, no loops in your code, and no Python-style object dispatch per element. Combined with comprehensions, this is how Julia achieves C-like performance with Python-like conciseness in numerical code.

Code Example

nums = [5, 3, 8, 1, 9]
push!(nums, 12)
filtered = filter(x -> x > 4, nums)
doubled = map(x -> x * 2, filtered)
println(filtered, " → ", doubled)

scores = Dict("Alice" => 95, "Bob" => 87)
scores["Carol"] = 100
println(scores["Alice"])                     # 95
println(haskey(scores, "Dave"))              # false
for (name, score) in scores
    println("$name: $score")
end

a = Set([1, 2, 3, 4])
b = Set([3, 4, 5, 6])
println(union(a, b))        # Set([1,2,3,4,5,6])
println(intersect(a, b))     # Set([3,4])

squares = [x^2 for x in 1:10 if x % 2 == 0]   # comprehension with filter
println(squares)                                 # [4, 16, 36, 64, 100]

arr1 = [1, 2, 3]; arr2 = [10, 20, 30]
println(arr1 .+ arr2)                            # [11, 22, 33]
println(sqrt.(arr1))                              # element-wise sqrt

Week 6 — Multiple Dispatch & Functions in Depth

Master Julia's defining paradigm — multiple dispatch — and write flexible, generic, reusable functions.

Core Focus

Mastering Julia's defining paradigm — multiple dispatch — and writing flexible, generic, and reusable functions with variadic arguments and do-block syntax.

Topics & Subtopics

  1. Multiple Dispatch — Julia's Core Paradigm — Unlike single-dispatch OOP languages (where a method belongs to one class), Julia selects which method to run based on the types of all arguments. This allows you to define multiple specialised versions of the same function name for different combinations of input types.
  2. Method Specificity & Ambiguity Resolution — When multiple methods could match a call, Julia selects the most specific applicable method based on the argument types. Understanding specificity rules prevents "method ambiguity" errors, which Julia raises explicitly rather than silently guessing.
  3. Variadic Functions & Splatting — Functions can accept a variable number of arguments using args... (the "slurp" pattern). The same ... syntax "splats" a collection into individual arguments when calling a function — the inverse operation.
  4. Anonymous Functions & Do-Block Syntax — Anonymous functions use x -> expression syntax for short expressions. For multi-line function arguments (common with map, open, reduce), Julia's do block syntax provides a readable alternative that avoids deeply nested closures.

Why This Week Matters: Multiple dispatch is the single feature that makes Julia unique among mainstream languages — and it is the reason Julia can be both generic and fast simultaneously. When you define area(s::Circle) and area(s::Rectangle), the compiler generates separate, optimised machine code for each. No virtual dispatch overhead, no runtime type checking — full specialisation at compile time.

Code Example

describe(x::Int) = println("$x is an integer")
describe(x::Float64) = println("$x is a float")
describe(x::String) = println("\"$x\" is a string")

describe(42)        # 42 is an integer
describe(3.14)       # 3.14 is a float
describe("hello")    # "hello" is a string

process(x::Number) = println("Generic number: $x")
process(x::Int) = println("Specific integer: $x")   # more specific — wins for Ints

process(5)     # Specific integer: 5
process(5.0)    # Generic number: 5.0

function mysum(args...)
    total = 0
    for x in args; total += x; end
    return total
end
println(mysum(1, 2, 3, 4))    # 10
nums = [1, 2, 3, 4]
println(mysum(nums...))         # 10 — splat

result = map(1:5) do x         # do-block syntax
    y = x^2
    y + 1
end
println(result)                 # [2, 5, 10, 17, 26]

Week 7 — Modules, Scope & Custom Structs

Organise code into modules, understand Julia's scoping rules, and define your own composite types.

Core Focus

Organizing code into modules, understanding Julia's scoping rules (including the global scope gotcha), and defining custom composite types with structs.

Topics & Subtopics

  1. Modules — module, export, and using — A module groups related functions, types, and constants under a namespace, preventing naming collisions across large codebases. export marks names as part of the module's public API, made available via using ModuleName.
  2. Scope Rules — Global vs Local — Julia distinguishes global scope (module-level) from local scope (functions, loops, blocks). Inside functions, variables are local by default; at the top level of a script, loops require the global keyword to modify outer variables — a common beginner gotcha.
  3. Structs — Custom Composite Types — struct defines a new immutable composite type with named, typed fields — Julia's primary building block for custom data structures. Fields are accessed with dot notation; instances are created by calling the struct name like a function.
  4. Mutable Structs & Constructors — mutable struct allows field reassignment after creation, at a small performance cost compared to immutable structs. Custom inner constructors (defined inside the struct body) enable validation logic to run whenever an instance is created.

Why This Week Matters: Julia's global scope gotcha — where a for loop at the top level of a script cannot modify outer variables without the global keyword — surprises virtually every Julia beginner. Understanding it now prevents hours of debugging. Immutable structs are the default because they enable compiler optimisations that can make code orders of magnitude faster than mutable alternatives.

Code Example

module MathUtils
export square, cube
square(x) = x^2
cube(x) = x^3
end
# using .MathUtils; println(square(5))   # 25

total = 0
function addToTotal(x)
    total = total + x    # creates a NEW local `total`!
    return total
end
println(addToTotal(5))   # 5 — local result
println(total)            # 0 — outer total is UNCHANGED

struct Point
    x::Float64
    y::Float64
end
p = Point(3.0, 4.0)
println(p.x, ", ", p.y)    # 3.0, 4.0
distance(p::Point) = sqrt(p.x^2 + p.y^2)
println(distance(p))         # 5.0

mutable struct Account
    balance::Float64
    function Account(balance)
        balance < 0 && error("Initial balance cannot be negative")
        new(balance)
    end
end
acc = Account(100.0)
acc.balance += 50.0
println(acc.balance)   # 150.0

Week 8 — Julia's Type System & Abstract Types

Master Julia's type hierarchy to write generic, extensible, and well-organised code.

Core Focus

Mastering Julia's type hierarchy to write generic, extensible, and well-organised code — Julia's alternative to classical object-oriented programming.

Topics & Subtopics

  1. Abstract Types & the Type Hierarchy — abstract type defines a node in Julia's type tree with no fields or instances of its own — purely an organisational concept for dispatch. Every concrete type has exactly one supertype, forming a strict tree traversable with supertype() and subtypes().
  2. Building Polymorphic Behavior with Dispatch — Instead of virtual methods and inheritance, Julia achieves polymorphism by defining functions that dispatch on the abstract type for shared logic, and on concrete types for specialised behavior — composition over inheritance, enforced at the language level.
  3. Parametric Types — Types can be parameterised over other types (struct Box{T}), enabling generic, type-stable containers similar to generics in other languages. The compiler specialises code for each concrete parameterisation, achieving both flexibility and performance.
  4. Interfaces via Duck Typing & Method Conventions — Julia has no formal interface keyword; instead, "interfaces" are informal conventions — a type "implements" an interface by defining the expected methods (e.g., Base.length, Base.iterate for iterables).

Why This Week Matters: Julia's type system is the reason it achieves C-level performance without C's manual memory management. When you write struct Box{T}, the compiler generates a completely separate, fully optimised version of Box for every concrete type T it's used with — zero overhead, zero boxing. This is the technical foundation of Julia's "walk like Python, run like C" promise.

Code Example

abstract type Shape end

struct Circle <: Shape     # <: means "is a subtype of"
    radius::Float64
end
struct Square <: Shape
    side::Float64
end
println(Circle <: Shape)           # true
println(supertype(Circle))          # Shape

area(s::Circle) = π * s.radius^2
area(s::Square) = s.side^2
describe(s::Shape) = println("Area: $(round(area(s), digits=2))")

describe(Circle(5.0))      # Area: 78.54
describe(Square(4.0))      # Area: 16.0

struct Box{T}
    value::T
end
intBox = Box{Int}(42)
strBox = Box{String}("hello")
println(typeof(intBox))            # Box{Int64}

struct Countdown; from::Int; end
Base.iterate(c::Countdown, state=c.from) = state < 0 ? nothing : (state, state - 1)
for n in Countdown(3); println(n); end   # 3, 2, 1, 0

Week 9 — File I/O & String Processing

Read and write files, and master Julia's string manipulation and regular expression capabilities.

Core Focus

Reading and writing files with automatic resource management, and mastering Julia's string manipulation and regular expression capabilities.

Topics & Subtopics

  1. File I/O — Reading & Writing — open(filename, mode) opens a file, returning a handle best used with do-block syntax to guarantee automatic closing. readlines(), read(file, String), and write() cover the most common file operations.
  2. Delimited Files & CSV with DelimitedFiles / CSV.jl — Julia's standard library DelimitedFiles module reads simple delimited data into arrays with readdlm. For real-world tabular data, the CSV.jl package paired with DataFrames.jl provides robust, fast, type-inferring CSV parsing.
  3. Regular Expressions — Julia uses PCRE-compatible regular expressions written with the r"pattern" string macro. match, occursin, and eachmatch provide standard regex operations, with capture groups accessible via indexing on the match result.
  4. String Manipulation Functions — Julia's string library provides split, join, strip, uppercase/lowercase, replace, and startswith/endswith — combined with the fact that strings are iterable and indexable.

Why This Week Matters: Julia's do-block syntax for file handling — open("file.txt", "r") do f ... end — guarantees the file is closed even if an exception is thrown inside the block. This is the same resource-management guarantee that Python's with statement provides, and it's essential for writing production-quality I/O code that doesn't leak file handles.

Code Example

open("data.txt", "w") do file
    write(file, "Line 1\n")
    write(file, "Line 2\n")
end

lines = readlines("data.txt")
for line in lines
    println(line)
end

using DelimitedFiles
data = readdlm("scores.csv", ',', header=true)
matrix, header = data
println(header)
println(matrix[1, :])

text = "Order #4521 placed on 2026-06-30"
m = match(r"#(\d+)", text)
println(m.captures[1])                      # "4521"
println(occursin(r"\d{4}-\d{2}-\d{2}", text)) # true

csv_row = "  alice, 30, engineer  "
fields = split(strip(csv_row), ",")
cleaned = [strip(f) for f in fields]
println(join(cleaned, " | "))               # "alice | 30 | engineer"

Week 10 — Error Handling & The Package Ecosystem

Handle errors idiomatically and learn to navigate Julia's package manager and core data science libraries.

Core Focus

Handling errors idiomatically with custom exception types and navigating Julia's package manager and core data science libraries.

Topics & Subtopics

  1. Exception Handling — try/catch/finally — Julia's try/catch/finally works similarly to other languages: code in try runs, catch e captures any thrown exception as e, and finally always executes for cleanup regardless of whether an error occurred.
  2. Custom Exceptions — Custom exception types are defined as structs inheriting from Exception, with a Base.showerror method to control how they're displayed. Throw them with throw(), providing structured, domain-specific error context.
  3. Pkg.jl — Julia's Package Manager — Pkg handles installing, updating, and managing project-specific dependencies via Project.toml and Manifest.toml files, ensuring fully reproducible environments — a major strength over many other ecosystems.
  4. Core Data Science Packages — DataFrames.jl & Statistics — DataFrames.jl is Julia's primary tabular data library (analogous to pandas), providing powerful, fast data manipulation. The standard library Statistics module provides mean, std, median, and cor for fundamental statistical analysis.

Why This Week Matters: Julia's Project.toml + Manifest.toml reproducibility model — where Pkg.instantiate() recreates an exact environment from a manifest — is more robust than Python's requirements.txt because it locks transitive dependency versions too. This is why Julia is used in production scientific computing where reproducibility is non-negotiable.

Code Example

function safeDivide(a, b)
    try
        return a / b
    catch e
        println("Error caught: $e")
        return NaN
    finally
        println("Division attempt complete")
    end
end
println(safeDivide(10, 0))    # Inf (Julia float division by zero — not an error!)

struct InsufficientFundsError <: Exception
    requested::Float64
    available::Float64
end
Base.showerror(io::IO, e::InsufficientFundsError) =
    print(io, "Requested \$$(e.requested) but only \$$(e.available) available")

# In Pkg mode (press ] in REPL):
# pkg> activate .
# pkg> add DataFrames CSV
# pkg> instantiate

using DataFrames, Statistics
df = DataFrame(name=["Alice","Bob","Carol"], score=[95,87,100])
println(mean(df.score))                       # 94.0

Week 11 — Working with Real-World Julia (Plotting, JSON, HTTP)

Apply Julia in real-world scenarios — data visualisation, structured data interchange, and web requests.

Core Focus

Applying Julia in real-world scenarios — data visualisation with Plots.jl, JSON data interchange, HTTP requests, and Python interop.

Topics & Subtopics

  1. Data Visualisation with Plots.jl — Plots.jl is Julia's flexible plotting front-end, supporting multiple backends (GR, Plotly, PythonPlot) through one unified API. Common plot types — line, scatter, bar, histogram — are created with concise, composable function calls.
  2. JSON Parsing with JSON3.jl — JSON3.jl provides fast, type-stable JSON encoding and decoding, commonly used for config files, API responses, and data interchange between systems. JSON3.read parses into Julia-native types automatically.
  3. HTTP Requests with HTTP.jl — HTTP.jl provides a full-featured client and server library for making web requests and building APIs. HTTP.get fetches a URL, returning a response object with .status and .body fields ready for further processing.
  4. Interfacing with Python via PyCall.jl — PyCall.jl allows calling Python libraries directly from Julia, useful for leveraging mature Python ecosystems (e.g., specific ML libraries) while writing primarily in Julia. Python objects and Julia objects convert automatically for common types.

Why This Week Matters: PyCall.jl — the ability to call Python libraries directly from Julia — is the practical bridge that makes Julia adoption feasible in real organizations. You can migrate critical performance bottlenecks from Python to Julia incrementally, calling existing Python libraries for everything else. This interop story is unique among compiled languages.

Week 12 — Metaprogramming & Macros

Use Julia's powerful metaprogramming system to write code that generates and transforms other code.

Core Focus

Using Julia's powerful metaprogramming system to write code that generates and transforms other code at parse time.

Topics & Subtopics

  1. Symbols & Expressions (Expr) — In Julia, code itself is data: a Symbol represents an identifier (:x), and Expr represents a parsed expression tree. The :() quoting syntax or quote...end blocks capture code without evaluating it — the foundation of all metaprogramming.
  2. eval & Dynamic Code Generation — eval(expr) evaluates an Expr in a given module's scope, executing it as if it were written directly in the source. This is powerful but should be used sparingly — most legitimate metaprogramming needs are better served by macros.
  3. Macros — macro, @, and Code Transformation — A macro takes Julia code (as unevaluated Expr syntax) as input and returns transformed code, which is then compiled and run — macros operate at parse time, before execution, unlike regular functions. Invoke macros with the @ prefix.
  4. Built-in Macros (@time, @show, @assert) — Julia's standard library uses macros extensively for developer ergonomics: @time measures execution time and allocations, @show prints both an expression and its value, and @assert validates conditions with a clear error on failure.

Why This Week Matters: Julia's metaprogramming is the reason packages like Plots.jl, DataFrames.jl, and Flux.jl can provide such expressive, domain-specific APIs without any language modifications — they extend Julia's syntax through macros. The @distributed, @threads, and @simd macros that unlock parallelism (Week 13) are all macros under the hood.

Code Example

sym = :x
expr = :(x + y * 2)
println(typeof(expr))   # Expr
println(expr.args)        # [:+, :x, :(y * 2)]

expr2 = :(2 + 3 * 4)
result = eval(expr2)
println(result)            # 14

macro sayhi(name)
    return :(println("Hello, ", $name, "!"))
end
@sayhi "Julia"           # prints: Hello, Julia!

@show 2 + 2                  # prints: 2 + 2 = 4
@assert 1 + 1 == 2 "Math is broken!"

@time begin
    sum(1:1_000_000)
end

Week 13 — Parallelism, Multithreading & Distributed Computing

Leverage Julia's first-class support for parallel and concurrent execution — a core strength of the language.

Core Focus

Leveraging Julia's first-class support for parallel and concurrent execution — from async tasks through multithreading to distributed computing across processes.

Topics & Subtopics

  1. Tasks & Asynchronous Programming (@async, @sync) — A Task (also called a "green thread" or coroutine) represents a unit of work that can be suspended and resumed cooperatively. @async schedules a task to run concurrently; @sync blocks until all enclosed @async tasks complete.
  2. Multithreading with Threads.@threads — Julia supports true OS-level multithreading (started with julia -t N or JULIA_NUM_THREADS=N). Threads.@threads parallelises a for loop across available threads — ideal for CPU-bound, data-parallel workloads like numerical array processing.
  3. Distributed Computing with Distributed.jl — For workloads too large for a single machine's threads, Distributed.jl spawns separate Julia processes (workers) locally or across a cluster, communicating via message passing. @distributed parallelises loops across worker processes.
  4. Thread-Safety & Race Conditions — When multiple threads mutate shared state simultaneously, race conditions can corrupt data. Julia provides Threads.Atomic{T} for lock-free atomic operations and ReentrantLock for explicit mutual exclusion, both essential for safe concurrent code.

Why This Week Matters: Julia's parallelism story is unique among scientific computing languages: @threads for shared-memory CPU parallelism, @distributed for multi-process cluster computing, and @async/@sync for I/O concurrency — all with a consistent, composable design. This is why Julia is chosen for large-scale climate models, financial simulations, and drug discovery workflows.

Code Example

function slowTask(id)
    sleep(1)
    println("Task $id done")
end
@sync begin
    for i in 1:3
        @async slowTask(i)
    end
end
println("All tasks complete")

using Base.Threads
results = zeros(10)
Threads.@threads for i in 1:10
    results[i] = i^2
end
println(results)
println("Running on $(nthreads()) threads")

using Distributed
addprocs(4)
result = @distributed (+) for i in 1:1000
    i^2
end
println(result)

counter = Atomic{Int}(0)
Threads.@threads for i in 1:1000
    atomic_add!(counter, 1)
end
println(counter[])   # reliably 1000

Week 14 — Performance Engineering & Type Stability

Write genuinely fast Julia code by understanding type stability, the compiler, and benchmarking tools.

Core Focus

Writing genuinely fast Julia code by understanding type stability, the compiler's inference system, and professional benchmarking tools.

Topics & Subtopics

  1. Type Stability — Julia's Performance Cornerstone — A function is "type-stable" if its return type can be inferred purely from its input argument types, without depending on runtime values. Type-unstable code forces the compiler to use slow, generic dynamic dispatch instead of fast, specialised machine code.
  2. Benchmarking with BenchmarkTools.jl — @time is useful for quick checks but includes compilation overhead and is noisy; BenchmarkTools.jl's @btime and @benchmark run a function many times, discard outliers, and report statistically reliable minimum/median execution time and memory allocations.
  3. Avoiding Allocations & Pre-allocating Arrays — Each array allocation triggers garbage collector pressure; pre-allocating output arrays and using in-place functions (ending in !, like sum!, map!) avoids unnecessary allocation in hot loops, often yielding 10-100x speedups.
  4. @code_warntype & Understanding Compiler Output — @code_warntype reveals Julia's type inference results for a function call, highlighting type instabilities in red/yellow — the primary diagnostic tool for hunting down performance bugs before they impact production code.

Why This Week Matters: Type stability is the single most important concept for writing fast Julia code. A type-unstable function can be 10-100x slower than its type-stable equivalent — not because of algorithmic differences, but because the compiler cannot specialise it. @code_warntype makes invisible type instabilities visible in seconds, making performance engineering in Julia dramatically more tractable than in other dynamic languages.

Code Example

# BAD: type-unstable — return type depends on runtime value
unstable(x) = x > 0 ? 1 : 1.0          # returns Int OR Float64!

# GOOD: type-stable — always returns the same type
stable(x) = x > 0 ? 1.0 : 0.0           # always Float64
@code_warntype stable(5.0)               # inspect — no warnings

using BenchmarkTools
function sumLoop(n)
    total = 0
    for i in 1:n; total += i; end
    return total
end
@btime sumLoop(1_000_000)    # reliable timing, e.g.: 0.521 ms (0 allocations)

# Allocates new array every call — slower
f_alloc(x) = x .* 2

# Reuses pre-allocated output — much faster for repeated calls
function f_inplace!(out, x)
    out .= x .* 2    # in-place broadcast, zero allocation
    return out
end

function processValue(x)
    if x > 0
        return x * 2      # type instability! mixes numeric and String
    else
        return "negative"
    end
end
@code_warntype processValue(5)    # reveals Union{Int, String} return type

abstract type SimulationModel end

struct StockPriceModel <: SimulationModel
    drift::Float64
    volatility::Float64
end

step!(m::StockPriceModel, price, dt) =
    price * exp((m.drift - 0.5 * m.volatility^2) * dt +
                m.volatility * sqrt(dt) * randn())

function runTrials(model::SimulationModel, nTrials::Int, nSteps::Int, dt::Float64)
    results = zeros(nTrials)
    Threads.@threads for trial in 1:nTrials
        price = 100.0
        for _ in 1:nSteps
            price = step!(model, price, dt)
        end
        results[trial] = price
    end
    return results
end
