About This Curriculum

This is a 12-week, phase-based Go curriculum designed to take a complete beginner to a professional-level Go developer. The roadmap moves from environment setup, primitive types, variables, and operators through control flow, functions, structs, and custom data structures, into interfaces, error handling, file I/O, and Go's standard library, all the way to goroutines, channels, generics, functional patterns, and three complete capstone projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics and Go fundamentals, Phase 2 covers control flow, functions, and structs, Phase 3 covers interfaces, error handling, file I/O, and packages, and Phase 4 covers master-level concurrency, generics, and full project work.

By Week 12, you will have built three complete, portfolio-ready applications and be confident building production-grade Go applications from scratch.

Level: Beginner → Master  |  Duration: 12 weeks  |  Stack: Go 1.21+  |  Prerequisites: Basic programming logic

Phase 1 — Absolute Basics

Weeks 1–3: Go environment, syntax, primitive types, arrays, slices, and maps.

Week 1 — Go Environment, Syntax & Primitive Types

Get Go installed, understand program structure, and learn primitive data types.

Core Focus

Getting Go installed, understanding program structure, and learning primitive data types.

Topics & Subtopics

  1. Go Installation & Workspace (go mod, go run) — Go projects are organized using modules, which track dependencies and versions. The go command-line tool compiles, runs, and manages your code without needing external build systems.
  2. Package Declaration & main Function — Every Go file belongs to a package, and executable programs must have a main package with a main function. This function is the entry point Go calls when your program starts.
  3. Variables & Type Inference (var, :=) — Go is statically typed, but it can infer types automatically using the := short declaration syntax. Variables declared with var are useful when you need an explicit type or a zero-value default.
  4. Primitive Types (int, float64, string, bool) — Go provides built-in primitive types representing numbers, text, and true/false values. Choosing the right type ensures memory efficiency and predictable arithmetic behavior.

Why This Week Matters: Go's module system and tooling are first-class features of the language. Understanding go mod, go run, and the package/main structure from day one builds the mental model needed for every Go concept that follows — from simple CLI tools to production-grade APIs.

Code Example

package main

import "fmt" // standard library for formatted I/O

func main() {
    var age int = 30      // explicit type declaration
    name := "Gopher"      // inferred type: string
    isLearning := true    // inferred type: bool
    fmt.Println(name, age, isLearning)
}

Week 2 — Operators, Constants & Type Conversion

Master arithmetic and logical operators and how Go handles strict typing.

Core Focus

Mastering arithmetic and logical operators and how Go handles strict typing.

Topics & Subtopics

  1. Arithmetic & Comparison Operators — Go supports standard operators like +, -, *, /, % for math and ==, !=, <, > for comparisons. Integer division truncates, so dividing two ints never produces a decimal.
  2. Constants (const) & iota — Constants hold values that cannot change during execution, improving safety and readability. The iota keyword auto-increments values, commonly used to build enumerated constants.
  3. Type Conversion (Explicit Casting) — Go never converts types automatically, even between numeric types — you must convert explicitly. This strictness prevents subtle bugs common in looser-typed languages.
  4. Strings & Runes (UTF-8 Basics) — Go strings are sequences of bytes encoded in UTF-8, and a rune represents a single Unicode code point. Understanding this distinction matters when working with non-ASCII text.

Why This Week Matters: Go's strict type system is intentional — it eliminates entire categories of runtime bugs caused by implicit conversions. Understanding iota for enumerations and explicit casting for numeric work is fundamental to writing correct, idiomatic Go from the start.

Code Example

package main

import "fmt"

func main() {
    const Pi = 3.14159              // constant value
    var x int = 10
    var y float64 = float64(x) / 3.0  // explicit conversion required
    fmt.Println(Pi, y, x%3)        // modulo operator

    // iota for enumerated constants
    const (
        Sunday = iota  // 0
        Monday         // 1
        Tuesday        // 2
    )
    fmt.Println(Sunday, Monday, Tuesday)
}

Week 3 — Composite Types: Arrays, Slices & Maps

Learn Go's core data containers used in almost every real program.

Core Focus

Learning Go's core data containers used in almost every real program.

Topics & Subtopics

  1. Arrays (Fixed-Size Collections) — An array has a fixed length defined at declaration and cannot grow or shrink. Arrays are rarely used directly in Go but form the foundation for slices.
  2. Slices (Dynamic Arrays) — A slice is a flexible, resizable view over an underlying array, making it Go's go-to list-like structure. You can grow a slice using the built-in append function.
  3. Maps (Key-Value Pairs) — A map stores values indexed by unique keys, similar to dictionaries in other languages. Maps are ideal for fast lookups by name, ID, or other identifiers.
  4. Zero Values & nil — Every Go type has a default "zero value" when not explicitly initialized — 0 for numbers, "" for strings, nil for pointers/slices/maps. Understanding zero values prevents unexpected nil-pointer style errors.

Why This Week Matters: Slices and maps are the workhorses of almost every Go program. Unlike arrays, slices grow dynamically and are passed by reference, while maps provide O(1) key lookups. Confusing arrays with slices or mishandling nil maps are among the most common Go beginner mistakes.

Code Example

package main

import "fmt"

func main() {
    nums := []int{1, 2, 3}        // slice literal
    nums = append(nums, 4)        // grow the slice dynamically
    fmt.Println(nums)             // [1 2 3 4]

    ages := map[string]int{"Sam": 25}  // map literal
    ages["Eve"] = 31                   // add new key-value pair

    for name, age := range ages {      // range over map
        fmt.Printf("%s is %d\n", name, age)
    }
}

Phase 2 — Control Flow & Logic

Weeks 4–6: Conditionals, loops, functions, structs, and custom data structures.

Week 4 — Control Flow: Conditionals & Loops

Direct program execution using Go's minimalist control structures.

Core Focus

Directing program execution using Go's minimalist control structures.

Topics & Subtopics

  1. if / else if / else Statements — Go's conditional statements don't require parentheses around the condition, keeping syntax clean. You can also declare a variable scoped only to the if block for cleaner logic.
  2. The Unified for Loop — Go has only one looping keyword, for, which can act as a traditional loop, a while-loop, or an infinite loop depending on syntax. This simplifies the language while remaining fully expressive.
  3. switch Statements — Go's switch is more powerful than in many languages — it doesn't require break and supports matching multiple values or conditions per case. It's often a cleaner alternative to long if-else chains.
  4. range for Iteration — The range keyword iterates over slices, arrays, maps, and strings, providing both index/key and value. It's the idiomatic way to loop through collections in Go.

Why This Week Matters: Go's decision to have only one loop keyword (for) and a switch without fall-through by default reflects Go's design philosophy: simple, unambiguous syntax. These constructs form the backbone of every Go program's logic and are used constantly in real codebases.

Code Example

package main

import "fmt"

func main() {
    // if with initialization statement
    if x := 42; x > 10 {
        fmt.Println("x is greater than 10")
    }

    // for as a while loop
    i := 0
    for i < 3 {
        fmt.Println("i =", i)
        i++
    }

    // switch — no break needed
    for i := 0; i < 3; i++ {
        switch {
        case i == 0:
            fmt.Println("first")
        default:
            fmt.Println("index:", i)
        }
    }
}

Week 5 — Functions, Multiple Returns & Closures

Structure code into reusable, composable units.

Core Focus

Structuring code into reusable, composable units using Go's function features.

Topics & Subtopics

  1. Function Declaration & Parameters — Functions in Go are declared with the func keyword and require explicit parameter and return types. This explicitness makes function signatures self-documenting.
  2. Multiple Return Values — Unlike many languages, Go functions can return more than one value natively, commonly used for returning a result alongside an error. This pattern is central to idiomatic Go error handling.
  3. Variadic Functions — A variadic function accepts a variable number of arguments using the ... syntax. This is useful for functions like fmt.Println that handle an unknown number of inputs.
  4. Closures & Anonymous Functions — A closure is a function that captures variables from its surrounding scope, even after that scope has exited. This enables powerful patterns like counters, generators, and callback logic.

Why This Week Matters: Multiple return values are Go's idiomatic alternative to exceptions — returning (result, error) pairs is the pattern used in virtually every Go standard library function. Understanding this convention is essential before moving to interfaces and error handling.

Code Example

package main

import "fmt"

// multiple return values — result + error pattern
func divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, fmt.Errorf("cannot divide by zero")
    }
    return a / b, nil
}

// variadic function
func sum(nums ...int) int {
    total := 0
    for _, n := range nums {
        total += n
    }
    return total
}

// closure — captures outer variable
func makeCounter() func() int {
    count := 0
    return func() int {
        count++
        return count
    }
}

func main() {
    result, err := divide(10, 2)
    fmt.Println(result, err)         // 5 <nil>

    fmt.Println(sum(1, 2, 3, 4, 5)) // 15

    counter := makeCounter()
    fmt.Println(counter(), counter()) // 1 2
}

Week 6 — Structs, Methods & Custom Data Structures

Model real-world entities using Go's struct system.

Core Focus

Modelling real-world entities using Go's struct system and method receivers.

Topics & Subtopics

  1. Defining Structs — A struct groups related fields into a single custom type, similar to a class without inheritance. Structs are the primary way to model structured data in Go.
  2. Methods on Structs — Methods are functions attached to a specific type via a "receiver," allowing behavior to be tied directly to data. This is how Go achieves object-like behavior without traditional classes.
  3. Pointer vs Value Receivers — A value receiver works on a copy of the struct, while a pointer receiver can modify the original instance. Choosing the right receiver type is critical for performance and correctness.
  4. Nested & Embedded Structs — Structs can contain other structs as fields, enabling composition instead of inheritance. Embedding promotes the inner struct's fields and methods directly onto the outer struct.

Why This Week Matters: Structs and methods are Go's answer to classes. The distinction between pointer and value receivers is one of the most important decisions in Go — getting it wrong leads to bugs where mutations silently don't persist. This pattern appears in every non-trivial Go program.

Code Example

package main

import "fmt"

type Car struct {
    Brand string
    Speed int
}

// pointer receiver — mutates the original struct
func (c *Car) Accelerate(amount int) {
    c.Speed += amount
}

// value receiver — works on a copy, read-only
func (c Car) Describe() string {
    return fmt.Sprintf("%s going %d km/h", c.Brand, c.Speed)
}

// embedded struct — composition over inheritance
type ElectricCar struct {
    Car               // embed Car — inherits fields and methods
    BatteryLevel int
}

func main() {
    myCar := Car{Brand: "Toyota", Speed: 0}
    myCar.Accelerate(50)
    fmt.Println(myCar.Describe())  // Toyota going 50 km/h

    tesla := ElectricCar{Car: Car{Brand: "Tesla", Speed: 0}, BatteryLevel: 80}
    tesla.Accelerate(100)          // promoted method from Car
    fmt.Println(tesla.Describe())  // Tesla going 100 km/h
}

Week 7 — Interfaces & Go's Approach to "OOP"

Understand polymorphism in Go through interfaces instead of class inheritance.

Core Focus

Understanding polymorphism in Go through interfaces instead of class inheritance.

Topics & Subtopics

  1. Defining & Implementing Interfaces — An interface defines a set of method signatures without implementation, and any type implementing those methods automatically satisfies it. Go uses implicit (structural) typing — there's no implements keyword.
  2. Polymorphism via Interfaces — Functions can accept an interface type to work with any struct that satisfies it, enabling flexible, decoupled code. This is how Go achieves polymorphism without traditional class hierarchies.
  3. The Empty Interface (interface{} / any) — The empty interface can hold a value of any type, useful for generic-style functions before Go added type parameters. Go 1.18+ aliases this as any for readability.
  4. Type Assertions & Type Switches — A type assertion extracts the concrete type from an interface value, while a type switch handles multiple possible types cleanly. Both are essential when working with values of unknown underlying type.

Why This Week Matters: Go's implicit interface satisfaction is one of its most elegant design decisions — you can satisfy an interface defined in a library you've never seen, just by implementing the right methods. This is the foundation of Go's entire standard library design and is key to writing testable, decoupled code.

Code Example

package main

import "fmt"

type Shape interface {
    Area() float64
}

type Circle struct{ Radius float64 }
type Rectangle struct{ Width, Height float64 }

func (c Circle) Area() float64    { return 3.14 * c.Radius * c.Radius }
func (r Rectangle) Area() float64 { return r.Width * r.Height }

// accepts any Shape — polymorphic
func printArea(s Shape) {
    fmt.Printf("Area: %.2f\n", s.Area())
}

// type switch — handle unknown concrete types
func describe(i any) {
    switch v := i.(type) {
    case int:    fmt.Println("int:", v)
    case string: fmt.Println("string:", v)
    default:     fmt.Println("unknown type")
    }
}

func main() {
    var s Shape = Circle{Radius: 4}
    printArea(s)                       // Area: 50.24
    printArea(Rectangle{Width: 3, Height: 5}) // Area: 15.00
    describe(42)                       // int: 42
}

Week 8 — Error Handling, Panic/Recover & File I/O

Write robust programs that handle failures and interact with the filesystem.

Core Focus

Writing robust programs that handle failures gracefully and interact with the filesystem.

Topics & Subtopics

  1. The error Interface & Custom Errors — Go treats errors as ordinary values implementing the error interface, rather than using exceptions. You can create custom error types for richer, more descriptive failure information.
  2. panic and recover — panic immediately stops normal execution for unrecoverable situations, while recover lets a deferred function regain control and prevent a crash. These are reserved for truly exceptional cases, not routine error handling.
  3. defer for Cleanup — The defer keyword schedules a function call to run right before the enclosing function returns, commonly used to close files or release resources. Deferred calls run in last-in-first-out order.
  4. Reading & Writing Files (os, bufio) — The os package provides functions to open, create, and manipulate files, while bufio adds efficient buffered reading and writing. Together they form Go's standard approach to file I/O.

Why This Week Matters: Go's error-as-value approach is intentionally different from exception-based languages — it forces you to handle every possible failure explicitly. defer prevents resource leaks that are a common source of bugs and production incidents. Together, these patterns define idiomatic, production-quality Go.

Code Example

package main

import (
    "bufio"
    "fmt"
    "os"
)

// custom error type with extra context
type FileError struct {
    Path string
    Err  error
}

func (e *FileError) Error() string {
    return fmt.Sprintf("file error at %s: %v", e.Path, e.Err)
}

func main() {
    // create file — defer guarantees Close even on error
    file, err := os.Create("notes.txt")
    if err != nil {
        fmt.Println("Error:", err)
        return
    }
    defer file.Close()  // runs when main() returns

    file.WriteString("Hello, Go file I/O!\n")
    file.WriteString("Second line.\n")

    // read file back with bufio.Scanner
    readFile, _ := os.Open("notes.txt")
    defer readFile.Close()

    scanner := bufio.NewScanner(readFile)
    for scanner.Scan() {
        fmt.Println(scanner.Text())
    }
}

Week 9 — Packages, Modules & Popular Standard Libraries

Organise larger codebases and leverage Go's rich standard library.

Core Focus

Organising larger codebases into packages and leveraging Go's rich standard library for JSON and HTTP.

Topics & Subtopics

  1. Creating & Importing Custom Packages — Splitting code into packages improves organization and reusability across a project. Exported identifiers (capitalized names) are accessible from other packages; lowercase ones stay private to their package.
  2. Go Modules & Dependency Management — go.mod tracks your module's dependencies and versions, while go get fetches external packages. This system replaced the older GOPATH-based workflow for dependency management.
  3. JSON Encoding/Decoding (encoding/json) — Go's encoding/json package converts structs to JSON (marshaling) and JSON back into structs (unmarshaling). Struct tags control how field names map to JSON keys.
  4. HTTP Requests & a Basic Server (net/http) — The net/http package lets you build web servers and make HTTP client requests with minimal setup. It's one of Go's most celebrated standard-library features for backend development.

Why This Week Matters: Go's standard library is exceptionally powerful — net/http alone is sufficient to build production HTTP servers without any third-party framework. Understanding struct tags for JSON mapping and the package visibility rules are essential for any Go backend developer.

Code Example

package main

import (
    "encoding/json"
    "fmt"
    "net/http"
)

type User struct {
    Name string `json:"name"` // struct tag maps Go field to JSON key
    Age  int    `json:"age"`
}

// JSON marshaling
func main() {
    data, _ := json.Marshal(User{Name: "Ana", Age: 28})
    fmt.Println(string(data))  // {"name":"Ana","age":28}

    // unmarshal JSON back to struct
    var u User
    json.Unmarshal(data, &u)
    fmt.Println(u.Name, u.Age) // Ana 28

    // basic HTTP server
    http.HandleFunc("/health", func(w http.ResponseWriter, r *http.Request) {
        w.Header().Set("Content-Type", "application/json")
        json.NewEncoder(w).Encode(map[string]string{"status": "ok"})
    })
    fmt.Println("Server running on :8080")
    http.ListenAndServe(":8080", nil)
}

Week 10 — Concurrency: Goroutines & Channels

Harness Go's signature feature: lightweight concurrent execution.

Core Focus

Harnessing Go's signature feature: lightweight concurrent execution with goroutines and channels.

Topics & Subtopics

  1. Goroutines — A goroutine is a lightweight thread managed by the Go runtime, started simply by prefixing a function call with go. Thousands of goroutines can run concurrently with minimal overhead — far cheaper than OS threads.
  2. Channels for Communication — Channels let goroutines safely send and receive values between each other, avoiding shared-memory race conditions. They embody Go's philosophy: "don't communicate by sharing memory; share memory by communicating."
  3. Buffered vs Unbuffered Channels — Unbuffered channels block until both sender and receiver are ready, enforcing synchronization. Buffered channels allow a fixed number of values to be queued without immediate blocking.
  4. select Statements — The select statement lets a goroutine wait on multiple channel operations simultaneously, proceeding with whichever is ready first. It's essential for building responsive concurrent systems and implementing timeouts.

Why This Week Matters: Concurrency is Go's most celebrated feature — it's why companies like Google, Cloudflare, and Uber adopted Go for high-throughput services. Understanding goroutines and channels, especially the difference between buffered and unbuffered channels, is what separates Go developers from Go masters.

Code Example

package main

import (
    "fmt"
    "time"
)

func worker(id int, ch chan string) {
    ch <- fmt.Sprintf("worker %d done", id)  // send result to channel
}

func main() {
    ch := make(chan string)
    go worker(1, ch)           // launch goroutine
    fmt.Println(<-ch)          // receive result: "worker 1 done"

    // buffered channel — queue up to 3 values without blocking
    buffered := make(chan int, 3)
    buffered <- 1
    buffered <- 2
    buffered <- 3

    // select with timeout — responsive concurrency
    done := make(chan bool)
    go func() {
        time.Sleep(2 * time.Second)
        done <- true
    }()
    select {
    case <-done:
        fmt.Println("completed!")
    case <-time.After(3 * time.Second):
        fmt.Println("timed out!")
    }
}

Week 11 — Sync Primitives, Generics & Functional Patterns

Manage shared state safely and write reusable, type-flexible code.

Core Focus

Managing shared state safely with sync primitives and writing reusable, type-flexible code with generics.

Topics & Subtopics

  1. sync.WaitGroup & sync.Mutex — WaitGroup lets the main goroutine wait until a set of other goroutines finish, while Mutex prevents simultaneous access to shared data. Together they solve the most common concurrency safety problems.
  2. Generics (Type Parameters) — Generics, introduced in Go 1.18, let you write functions and types that work across multiple data types while preserving compile-time type safety. They reduce code duplication compared to using interface{}.
  3. Functional Patterns: Higher-Order Functions — Go supports passing functions as arguments and returning them as values, enabling decorator-like and generator-like patterns. These patterns simulate features other languages implement natively.
  4. "Decorator" Pattern via Function Wrapping — Since Go lacks built-in decorators, the same effect is achieved by wrapping a function inside another function that adds behavior before/after the call. This is commonly used for logging, timing, and HTTP middleware.

Why This Week Matters: Race conditions are among the hardest bugs to debug. sync.Mutex and WaitGroup are the standard tools to prevent them. Generics (Go 1.18+) fundamentally changed what's possible in reusable Go code — understanding them is now essential for any Go developer working on modern codebases.

Code Example

package main

import (
    "fmt"
    "sync"
)

// sync.Mutex — protect shared counter
type SafeCounter struct {
    mu    sync.Mutex
    count int
}

func (c *SafeCounter) Increment() {
    c.mu.Lock()
    defer c.mu.Unlock()
    c.count++
}

// generic function — works with any comparable type
func Map[T, U any](items []T, f func(T) U) []U {
    result := make([]U, len(items))
    for i, v := range items {
        result[i] = f(v)
    }
    return result
}

// decorator pattern via function wrapping
func WithLogging(f func()) func() {
    return func() {
        fmt.Println("start")
        f()
        fmt.Println("end")
    }
}

func main() {
    nums := []int{1, 2, 3}
    squared := Map(nums, func(n int) int { return n * n })
    fmt.Println(squared)  // [1 4 9]

    logged := WithLogging(func() { fmt.Println("doing work") })
    logged()              // start / doing work / end
}

Code Example — Project 3 Pattern

package main

import (
    "fmt"
    "sync"
)

func fetch(url string, wg *sync.WaitGroup, results chan<- string) {
    defer wg.Done()                      // signal completion when done
    results <- "fetched: " + url         // send result to channel
}

func main() {
    urls    := []string{"a.com", "b.com", "c.com", "d.com", "e.com"}
    results := make(chan string, len(urls))  // buffered — no blocking
    var wg sync.WaitGroup

    for _, u := range urls {
        wg.Add(1)
        go fetch(u, &wg, results)  // launch one goroutine per URL
    }

    wg.Wait()       // wait for all goroutines to finish
    close(results)  // safe to close after all sends are done

    for r := range results {
        fmt.Println(r)
    }
}

Phase 2 4–6 Control Flow & Logic Direct program logic with for/switch, structure code with functions and closures, and model data with structs and methods
Phase 3 7–9 Intermediate & Advanced Core Design decoupled systems with interfaces, handle errors idiomatically, interact with files, and build HTTP servers and JSON APIs
Phase 4 10–12 Master Level Concepts Write concurrent programs with goroutines and channels, apply generics and functional patterns, and complete three production projects

How to Use This Curriculum

  • Complete each week's task before moving on — building projects reinforces concepts far better than reading alone.
  • Push every weekly project to a public GitHub repository — your commit history is visible proof of consistent progress.
  • Run go vet and golangci-lint on your code from Week 3 onward — get comfortable with idiomatic Go early.
  • Use go test -race on all concurrent code in Weeks 10–12 to detect race conditions before they become production bugs.
  • By Week 12, you should have three complete projects and the confidence to build any Go CLI tool, REST API, or concurrent service from scratch.
