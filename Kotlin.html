About This Curriculum

This is a 15-week, phase-based Kotlin curriculum designed to take a complete beginner to a professional-level Kotlin developer. The roadmap moves from syntax, variables, null safety, and collections through control flow, functions, data classes, and modules, into OOP, generics, file I/O, and error handling, all the way to coroutines, DSLs, reflection, multiplatform, and three complete capstone projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics including setup and null safety, Phase 2 covers control flow, functions, and core data structures, Phase 3 covers intermediate and advanced core topics like OOP, generics, and error handling, and Phase 4 covers master-level coroutines, DSLs, reflection, and production projects.

By Week 15, you will have built three complete, portfolio-ready applications and be confident building production-grade Kotlin applications across Android, backend, and multiplatform targets.

Level: Beginner → Master  |  Duration: 15 weeks  |  Stack: Kotlin 2.0+, IntelliJ IDEA, Gradle  |  Prerequisites: Basic programming logic

Phase 1 — Absolute Basics

Weeks 1–3: Setup, core types, variables, and null safety.

Week 1 — Introduction to Kotlin & Environment Setup

Understand what Kotlin is, how it relates to the JVM, and write your first working program.

Core Focus

Understanding what Kotlin is, how it relates to the JVM, and writing your first working program.

Topics & Subtopics

  1. What is Kotlin & Why Use It? — Kotlin is a modern, statically typed language developed by JetBrains that runs on the JVM, compiles to JavaScript, and compiles natively via Kotlin/Native. It is Google's preferred language for Android development and is increasingly popular for backend services due to its concise syntax and strong null safety.
  2. Installing Kotlin & Project Setup (IntelliJ IDEA / Gradle) — IntelliJ IDEA has first-class Kotlin support out of the box; Android Studio also bundles the Kotlin compiler. Projects are typically managed with Gradle, using a build.gradle.kts file written in Kotlin itself.
  3. Running Kotlin (Compiler, REPL, Kotlin Scripts) — kotlinc compiles .kt files to .jar or .class files runnable on the JVM. The Kotlin REPL lets you experiment interactively. Kotlin scripts (.kts) run directly without a formal project, ideal for quick prototypes.
  4. Variables — val vs var & Type Inference — val declares an immutable, read-only reference (preferred by default); var declares a mutable reference. Kotlin infers types from the assigned value, so explicit type annotations are optional but recommended at function boundaries.

Why This Week Matters: Everything in Kotlin is built on these fundamentals. Understanding the JVM relationship, the val/var distinction, and how type inference works forms the mental model you will rely on throughout every subsequent phase of this curriculum.

Code Example

// Kotlin entry point — no class wrapper required
fun main() {
    println("Hello, Kotlin!")
    println("Running on JVM: ${System.getProperty("java.version")}")
}

val name = "Alice"          // immutable — inferred as String
var age = 30                // mutable — inferred as Int
age += 1                    // allowed — var can change
// name = "Bob"             // compile error — val cannot be reassigned
val score: Double = 95.5    // explicit type annotation

Week 2 — Data Types, Operators & String Templates

Learn Kotlin's type system, perform calculations with operators, and format output with string templates.

Core Focus

Learning Kotlin's type system, performing calculations with operators, and formatting output with string templates.

Topics & Subtopics

  1. Basic Types (Int, Double, Boolean, Char, String) — Kotlin's basic types are all objects with no primitives at the language level, though the compiler optimises them to JVM primitives where possible. Every type is part of a unified type hierarchy rooted at Any.
  2. Arithmetic, Comparison & Logical Operators — Kotlin supports standard arithmetic (+, -, *, /, %), comparison (==, !=, <, >), and logical (&&, ||, !) operators — with == performing structural equality (calling .equals()) rather than reference equality, unlike Java.
  3. String Templates & Raw Strings — String templates embed expressions directly with $variable or ${expression} — no concatenation needed. Raw strings (triple-quoted """...""") preserve formatting and avoid escape characters, ideal for multi-line text, regex, or JSON.
  4. Type Conversion & Smart Casts — Kotlin requires explicit conversion functions (toInt(), toDouble(), toString()) with no implicit widening like Java. Smart casts automatically treat a variable as a more specific type after an is check, without an explicit cast.

Why This Week Matters: Kotlin's structural equality == vs reference equality === distinction eliminates a whole category of bugs that plague Java developers. String templates replace string concatenation entirely — you will use them in virtually every line of output code you write.

Code Example

val age: Int        = 30
val price: Double   = 19.99
val isActive: Boolean = true
val name: String    = "Alice"
println("$name is $age years old, price: $price")

val a = 17; val b = 5
println(a / b)            // 3 — integer division
println(a.toDouble() / b) // 3.4 — explicit conversion
println(a == 17)          // true — structural equality

val text = "42"
val number = text.toIntOrNull() ?: 0  // safe conversion with fallback
val obj: Any = "Hello"
if (obj is String) {
    println(obj.length)               // smart cast — obj is String here
}

Week 3 — Null Safety, Ranges & Nullable Types

Master Kotlin's signature null-safety system and use ranges for concise iteration and comparison logic.

Core Focus

Mastering Kotlin's signature null-safety system and using ranges for concise iteration and comparison logic.

Topics & Subtopics

  1. Nullable Types (?) & the Billion-Dollar Bug Fix — Appending ? to a type (String?) allows a variable to hold null; without it, the compiler guarantees the variable can never be null. This eliminates NullPointerException at compile time — Kotlin's most celebrated feature.
  2. Safe Calls (?.) & the Elvis Operator (?:) — The safe call operator (?.) calls a method or property only if the receiver isn't null, otherwise returning null. The Elvis operator (?:) provides a default value when the left-hand expression is null — together they eliminate verbose null-checking boilerplate.
  3. The Not-Null Assertion (!!) & Safe Casts — The !! operator forcibly asserts a nullable value is not null, throwing NullPointerException if wrong — use it sparingly. The safe cast operator (as?) returns null instead of throwing when a cast fails.
  4. Ranges, Progressions & in / !in — Ranges (1..10) represent a sequence of values and are used heavily in loops and when expressions. downTo, step, and until (exclusive end) control direction and stride. The in operator checks range membership concisely.

Why This Week Matters: Null safety is Kotlin's defining feature — the entire reason it was created. Understanding the difference between ?. (safe, returns null) and !! (unsafe, throws) and knowing when to use each is the most important skill for writing production Kotlin code.

Code Example

var name: String = "Alice"       // cannot be null — compiler enforced
var nickname: String? = null     // can be null — must be checked before use

val length = nickname?.length         // null — safe call short-circuits
val displayName = nickname ?: "Anonymous"  // Elvis fallback
println(displayName)                  // Anonymous

val maybeNull: String? = "Hello"
val definitelyString: String = maybeNull!!   // throws if null

for (i in 1..5)            print("$i ")       // 1 2 3 4 5
for (i in 10 downTo 1 step 3) print("$i ")   // 10 7 4 1
println(5 in 1..10)         // true

Phase 2 — Control Flow & Logic

Weeks 4–7: Functions, collections, data classes, and modules.

Week 4 — Functions & Control Flow

Direct program logic with Kotlin's expressive control structures and write clean, reusable functions.

Core Focus

Directing program logic with Kotlin's expressive control structures and writing clean, reusable functions.

Topics & Subtopics

  1. Function Syntax, Default & Named Arguments — Kotlin functions are declared with fun, support default parameter values, and allow named arguments at call sites for clarity — eliminating the need for method overloading in most cases.
  2. if/else and when as Expressions — In Kotlin, if/else and when can be used as expressions that return a value, not just statements — eliminating the need for a separate ternary operator. when is Kotlin's powerful pattern-matching replacement for switch.
  3. Loops — for, while, do-while — for loops iterate over ranges, collections, or anything implementing Iterable. break and continue support labels (break@outer) for controlling nested loops precisely.
  4. Single-Expression Functions & Lambda Expressions — Functions with a single expression body can omit braces and use = instead. Lambda expressions ({ params -> body }) are first-class values, often used as arguments to higher-order functions like map, filter, and forEach.

Why This Week Matters: Kotlin's when as an expression is one of its most powerful features — replacing entire chains of if-else with exhaustive, readable pattern matching. The implicit it parameter in single-argument lambdas is a pattern you will see in virtually every Kotlin codebase.

Code Example

fun greet(name: String, greeting: String = "Hello", punctuation: String = "!"): String =
    "$greeting, $name$punctuation"

println(greet("Alice"))                          // Hello, Alice!
println(greet("Bob", punctuation = "?"))          // Hello, Bob?

val score = 82
val grade = when {
    score >= 90 -> "A"
    score >= 80 -> "B"
    score >= 70 -> "C"
    else        -> "F"
}
println(grade)   // B

fun square(x: Int): Int = x * x                  // single-expression function
val numbers = listOf(1, 2, 3, 4)
println(numbers.map { it * it })                  // [1, 4, 9, 16]

Week 5 — Collections: List, Set, Map & Sequences

Store and manipulate groups of data using Kotlin's rich mutable and immutable collection types.

Core Focus

Storing and manipulating groups of data using Kotlin's rich, distinct mutable and immutable collection types.

Topics & Subtopics

  1. Immutable vs. Mutable Collections — Kotlin distinguishes between read-only interfaces (List, Set, Map) and their mutable counterparts (MutableList, MutableSet, MutableMap). listOf() creates immutable lists by default; mutableListOf() creates ones you can modify.
  2. List Operations & Functional Transformations — Kotlin's List provides a huge standard library of functional operations: map, filter, sortedBy, groupBy, reduce, and fold. These are chainable, expressive, and replace most manual loop-based logic.
  3. Set & Map — Uniqueness & Key-Value Storage — Set stores unique elements with no duplicates, ideal for membership testing and deduplication. Map<K, V> stores key-value pairs with convenient to infix syntax for creating pairs.
  4. Sequences — Lazy Evaluation for Large Datasets — Sequence<T> is Kotlin's lazy collection type — operations are not executed until a terminal operation (toList(), first(), sum()) is called, avoiding creating intermediate collections at each step.

Why This Week Matters: The distinction between immutable and mutable collections is one of Kotlin's most important safety features. Sequences are critical for performance on large datasets — understanding when to switch from eager List operations to lazy Sequence operations is a mark of a senior Kotlin developer.

Code Example

val readOnly: List<Int> = listOf(1, 2, 3)
val mutable: MutableList<Int> = mutableListOf(1, 2, 3)
mutable.add(4)                  // allowed

val numbers = listOf(5, 3, 8, 1, 9, 2)
val result = numbers
    .filter { it > 2 }
    .sortedDescending()
    .map { "[$it]" }
println(result)                  // [[9], [8], [5], [3]]

val scores = mutableMapOf("Alice" to 95, "Bob" to 87)
scores["Carol"] = 100

val lazyResult = (1..1_000_000).asSequence()
    .filter { it % 3 == 0 }
    .map { it * 2 }
    .take(5)
    .toList()                   // evaluation happens HERE
println(lazyResult)             // [6, 12, 18, 24, 30]

Week 6 — Data Classes, Sealed Classes & Object Declarations

Model domain data idiomatically using Kotlin's signature class types.

Core Focus

Modeling domain data idiomatically using Kotlin's signature class types — data classes, sealed classes, and objects.

Topics & Subtopics

  1. Data Classes — data class automatically generates equals(), hashCode(), toString(), and a copy() method from the declared constructor properties. They're the idiomatic way to model simple value-holding types — DTOs, domain models, and API payloads.
  2. Sealed Classes & Sealed Interfaces — A sealed class restricts the type hierarchy to a known, fixed set of subclasses declared in the same file. Combined with when, the compiler enforces exhaustiveness — catching missing cases at compile time, ideal for state machines and results.
  3. Enum Classes — Enum classes define a fixed set of named constants, optionally with properties and methods attached to each constant. They're more powerful than plain constants because they can implement interfaces and hold typed data.
  4. Object Declarations & Companion Objects — object creates a singleton — exactly one instance, lazily initialised on first access. companion object attaches "static-like" members to a class, accessible without an instance — Kotlin's replacement for Java's static keyword.

Why This Week Matters: Sealed classes are one of the most powerful features in Kotlin. The compiler-enforced exhaustiveness of when over a sealed hierarchy eliminates entire categories of runtime bugs. data class replaces hundreds of lines of Java boilerplate with a single line.

Code Example

data class User(val name: String, val age: Int)
val alice = User("Alice", 30)
val olderAlice = alice.copy(age = 31)
println(alice == User("Alice", 30))   // true — structural equality

sealed class Result<out T> {
    data class Success<T>(val data: T) : Result<T>()
    data class Error(val message: String) : Result<Nothing>()
}
fun <T> handle(result: Result<T>) = when (result) {
    is Result.Success -> println("Got: ${result.data}")
    is Result.Error   -> println("Failed: ${result.message}")
    // no else needed — compiler knows all cases
}

enum class Direction(val degrees: Int) {
    NORTH(0), EAST(90), SOUTH(180), WEST(270);
    fun opposite(): Direction = entries[(ordinal + 2) % entries.size]
}

object AppConfig { val version = "1.0.0" }  // singleton

Week 7 — Extension Functions, Scope Functions & Modules

Extend existing types without inheritance, simplify object configuration, and structure multi-file projects.

Core Focus

Extending existing types without inheritance, simplifying object configuration with scope functions, and structuring multi-file Kotlin projects.

Topics & Subtopics

  1. Extension Functions & Properties — Extension functions let you add new functions to existing classes — even ones you don't own like String or List — without modifying their source or using inheritance. They're resolved statically, making them safe and predictable.
  2. Scope Functions — let, run, with, apply, also — Scope functions execute a block of code on an object and return either the object itself or the block's result. apply for configuration, let for null-checks and transformations, also for side effects.
  3. Infix, Operator & Inline Functions — Infix functions enable readable, operator-like syntax (1 to 2 instead of 1.to(2)). Operator overloading lets custom types use +, -, ==, etc. inline functions eliminate lambda allocation overhead for performance-critical higher-order functions.
  4. Packages, Imports & Multi-File Project Structure — Kotlin organises code into packages matching the directory structure by convention. import brings external declarations into scope; aliased imports (import Foo as Bar) resolve naming conflicts cleanly.

Why This Week Matters: Extension functions and scope functions are the tools that make Kotlin code feel like a DSL. Every major Kotlin library — Ktor, Exposed, kotlinx — is built on these concepts. Knowing all five scope functions and when to use each is a mark of fluent Kotlin proficiency.

Code Example

fun String.isPalindrome(): Boolean {
    val cleaned = this.lowercase().filter { it.isLetter() }
    return cleaned == cleaned.reversed()
}
println("Racecar".isPalindrome())     // true

data class Person(var name: String = "", var age: Int = 0)
val person = Person().apply {         // apply: configure, returns object
    name = "Alice"
    age = 30
}
person.let { println("${it.name} is ${it.age}") }

data class Vector2(val x: Int, val y: Int) {
    operator fun plus(other: Vector2) = Vector2(x + other.x, y + other.y)
}
val v = Vector2(1, 2) + Vector2(3, 4)
println(v)                            // Vector2(x=4, y=6)

Week 8 — Object-Oriented Programming in Kotlin

Master Kotlin's class system — inheritance, interfaces, abstract classes, and visibility modifiers.

Core Focus

Mastering Kotlin's class system — inheritance, interfaces, abstract classes, and visibility modifiers.

Topics & Subtopics

  1. Classes, Constructors & Visibility Modifiers — Kotlin classes have a concise primary constructor declared in the class header, with an optional init block for additional setup logic. Visibility modifiers (public, private, protected, internal) default to public if omitted.
  2. Inheritance & open/override — Classes and members are final (non-inheritable) by default in Kotlin — you must explicitly mark a class open to allow inheritance and a member open to allow overriding. This "closed by default" design prevents fragile base class problems.
  3. Interfaces & Abstract Classes — Interfaces can contain both abstract methods and methods with default implementations. Abstract classes can hold constructor state (unlike interfaces) and are used when subclasses share both data and behaviour, not just a contract.
  4. Class Delegation (by) — Kotlin's class delegation lets a class delegate an interface's implementation to another object — composition made effortless, avoiding boilerplate forwarding methods. The by keyword generates all the delegation code automatically.

Why This Week Matters: Kotlin's "closed by default" design prevents the fragile base class problem that plagues Java. The by delegation keyword makes composition (the preferred design principle) trivially easy, replacing dozens of boilerplate forwarding methods with a single keyword.

Code Example

class BankAccount(val owner: String, private var balance: Double) {
    init { require(balance >= 0) { "Initial balance cannot be negative" } }
    fun deposit(amount: Double) { balance += amount }
    fun getBalance(): Double = balance
}

open class Animal(val name: String) {
    open fun sound(): String = "..."
}
class Dog(name: String) : Animal(name) {
    override fun sound(): String = "Woof"
}

interface Shape {
    fun area(): Double
    fun describe(): String = "Area is ${"%.2f".format(area())}"
}

interface Repository { fun findAll(): List<String> }
class InMemoryRepository : Repository { override fun findAll() = listOf("a", "b") }
class CachedRepository(inner: Repository) : Repository by inner  // delegation

Week 9 — Generics, Higher-Order Functions & Functional Patterns

Write reusable, type-safe abstractions using generics and Kotlin's first-class function support.

Core Focus

Writing reusable, type-safe abstractions using generics and Kotlin's first-class function support.

Topics & Subtopics

  1. Generic Classes & Functions — Generic type parameters (<T>) let classes and functions operate on any type while preserving compile-time type safety. Generic constraints (<T : Comparable<T>>) restrict which types are valid and unlock type-specific operations inside the body.
  2. Variance — in, out, and * (Star Projection) — out T marks a type parameter as covariant (producer-only), in T marks it contravariant (consumer-only). The star projection (*) is used when the specific type parameter is unknown or irrelevant.
  3. Higher-Order Functions & Function Types — Higher-order functions accept functions as parameters or return functions as results. Function types are written as (ParamType) -> ReturnType. This enables powerful composition patterns: strategy pattern, callbacks, and pipeline building.
  4. Reified Type Parameters & Inline Functions — Normally, generic type information is erased at runtime (type erasure), making T::class impossible inside a generic function. Marking an inline function's type parameter reified preserves the actual type at runtime, enabling type checks and reflection.

Why This Week Matters: reified type parameters are a uniquely Kotlin feature that solves a fundamental JVM limitation — type erasure. Understanding variance (in/out) is essential for working with Kotlin's standard library and for building generic frameworks that don't surprise their users with type errors.

Code Example

class Box<T>(private var content: T) {
    fun get(): T = content
    fun set(value: T) { content = value }
}
fun <T : Comparable<T>> maxOf(a: T, b: T): T = if (a > b) a else b
println(maxOf(5, 10))          // 10
println(maxOf("apple", "banana"))  // banana

interface Producer<out T> { fun produce(): T }
interface Consumer<in T> { fun consume(item: T) }
fun printAll(items: List<*>) = items.forEach { println(it) }

fun <T, R> List<T>.transform(operation: (T) -> R): List<R> = this.map(operation)

inline fun <reified T> isInstance(value: Any): Boolean = value is T
println(isInstance<String>("hello"))   // true

Week 10 — File I/O, Serialisation & Concurrency Basics

Read and write files, serialise data with kotlinx.serialization, and understand Kotlin's threading fundamentals.

Core Focus

Reading and writing files, serialising data with kotlinx.serialization, and understanding Kotlin's threading fundamentals before coroutines.

Topics & Subtopics

  1. File & Directory I/O (java.io / kotlin.io) — Kotlin provides idiomatic extension functions over Java's I/O classes: File("path").readText(), writeText(), and forEachLine simplify common file operations. Always use use { } to guarantee streams close properly, even on exceptions.
  2. JSON Serialisation with kotlinx.serialization — kotlinx.serialization is Kotlin's official multiplatform serialisation library. Annotate a data class @Serializable, then use Json.encodeToString() and Json.decodeFromString() for type-safe, reflection-free JSON conversion.
  3. Threads & the synchronized Keyword — Thread { }.start() spawns OS-level threads, which are expensive and limited in number. The synchronized keyword prevents race conditions by ensuring only one thread executes a critical section at a time.
  4. Why Coroutines Exist — The Problem with Threads — Threads are heavyweight (megabytes of stack memory each) and blocking I/O wastes them sitting idle. Coroutines are lightweight, suspendable computations that can pause without blocking the underlying thread — solving the scalability problem threads create for I/O-bound work.

Why This Week Matters: Understanding the raw thread model first makes coroutines make sense — you understand exactly what problem they solve. kotlinx.serialization is now the preferred JSON library for Kotlin Multiplatform, making it essential for any developer targeting more than one platform.

Code Example

import java.io.File
File("log.txt").writeText("Line 1\nLine 2\nLine 3")
File("log.txt").forEachLine { line -> println(line) }
println(File("log.txt").exists())   // true

import kotlinx.serialization.*
import kotlinx.serialization.json.*

@Serializable
data class Product(val name: String, val price: Double, val inStock: Boolean)

val json = Json.encodeToString(Product("Widget", 9.99, true))
val product = Json.decodeFromString<Product>(json)
println(product.name)   // Widget

var counter = 0
val lock = Object()
val threads = List(100) {
    Thread { synchronized(lock) { counter++ } }
}
threads.forEach { it.start() }
threads.forEach { it.join() }
println(counter)   // 100

Week 11 — Error Handling, Logging & Dependency Management

Handle exceptions idiomatically, model failures with Kotlin's Result type, and structure dependencies cleanly.

Core Focus

Handling exceptions idiomatically, modeling failures with Kotlin's Result type, and structuring dependencies cleanly.

Topics & Subtopics

  1. Exception Handling — try/catch as an Expression — Like if and when, try/catch in Kotlin can be used as an expression that returns a value. finally always executes for cleanup. Kotlin doesn't have checked exceptions — you're never forced to declare or catch a specific exception type.
  2. Custom Exceptions & require/check/error — Custom exceptions extend Exception or RuntimeException with domain-specific context. require() validates function arguments, check() validates state, and error() always throws — useful in exhaustive when branches.
  3. The Result<T> Type for Explicit Error Handling — Kotlin's built-in Result<T> wraps either a success value or a failure exception, avoiding exception-based control flow for expected failures. runCatching { } builds a Result automatically; .onSuccess, .onFailure provide functional handling.
  4. Logging with SLF4J / kotlin-logging — SLF4J is the standard JVM logging facade; kotlin-logging provides idiomatic Kotlin wrappers with lazy message evaluation — avoiding string construction cost when a log level is disabled. Configure log output and levels via Logback or Log4j2.

Why This Week Matters: Kotlin's Result<T> type and runCatching represent a fundamental shift in error handling philosophy — from exception-based control flow to explicit, typed error values. This pattern is used throughout modern Kotlin libraries and is essential for writing maintainable, predictable code.

Code Example

fun parseAge(input: String): Int = try {
    input.toInt()
} catch (e: NumberFormatException) {
    println("Invalid age: ${e.message}")
    -1                              // try/catch returns a value
} finally {
    println("Parsing attempt finished")
}

class InsufficientFundsException(val requested: Double, val available: Double) :
    Exception("Requested $requested but only $available available")

fun withdraw(amount: Double, balance: Double): Double {
    require(amount > 0) { "Amount must be positive" }
    if (amount > balance) throw InsufficientFundsException(amount, balance)
    return balance - amount
}

fun fetchUser(id: Int): Result<String> = runCatching {
    if (id <= 0) throw IllegalArgumentException("Invalid id")
    "User-$id"
}
fetchUser(5)
    .onSuccess { println("Found: $it") }
    .onFailure { println("Error: ${it.message}") }

Week 12 — Coroutines: Structured Concurrency & Flow

Master Kotlin's coroutine system for writing readable, non-blocking, structured concurrent code.

Core Focus

Mastering Kotlin's coroutine system for writing readable, non-blocking, and structured concurrent code.

Topics & Subtopics

  1. suspend Functions & Coroutine Builders (launch, async, runBlocking) — suspend marks a function as able to pause and resume without blocking a thread. launch starts a fire-and-forget coroutine returning a Job; async starts a coroutine returning a Deferred<T> you can await().
  2. Coroutine Scope, Context & Structured Concurrency — A CoroutineScope defines the lifetime boundary for child coroutines — when the scope is cancelled, all children are cancelled too. This "structured concurrency" guarantees no coroutine is ever leaked or forgotten, unlike raw thread management.
  3. Dispatchers — Default, IO, Main — Dispatchers determine which thread pool a coroutine runs on. Dispatchers.Default is for CPU-intensive work, Dispatchers.IO is optimised for blocking I/O, and Dispatchers.Main runs on the UI thread (Android/UI frameworks).
  4. Flow — Asynchronous Streams — Flow<T> represents a cold, asynchronous stream of values emitted over time — Kotlin's coroutine-based equivalent of reactive streams. Operators like map, filter, and collect compose pipelines; flows are lazy and only run when collected.

Why This Week Matters: Coroutines are the single most important feature that distinguishes Kotlin from Java for modern application development. Structured concurrency — the guarantee that no child coroutine outlives its parent scope — solves the hardest problems of concurrent programming with an elegant model.

    File("data.txt").readText()
}

    for (i in 1..5) { delay(100); emit(i) }
}

    numberFlow().map { it * it }.collect { println(it) }  // 1 4 9 16 25

Week 13 — DSLs, Reflection & Annotations

Build expressive internal Domain-Specific Languages, inspect types at runtime, and create custom annotations.

Core Focus

Building expressive internal Domain-Specific Languages, inspecting types at runtime, and creating custom annotations.

Topics & Subtopics

  1. Lambdas with Receivers & Type-Safe Builders — A lambda with a receiver (T.() -> Unit) executes as if it were a member of T, enabling the "builder pattern" used by Kotlin's own standard library (e.g., buildString { }). This is the foundation of Kotlin's signature DSL capability.
  2. Building a Domain-Specific Language (DSL) — Combining infix functions, extension functions, and lambdas with receivers, Kotlin can express highly readable internal DSLs — used in practice by Gradle Kotlin DSL, Ktor routing, and kotlinx.html. The goal is code that reads like natural domain language.
  3. Annotations & Annotation Processing Basics — Custom annotations (annotation class) attach declarative metadata to code, readable via reflection or compile-time processors (like KSP). @Target and @Retention control where an annotation can be applied and whether it survives to runtime.
  4. Reflection — Inspecting Classes at Runtime — Kotlin reflection (kotlin-reflect) provides a richer API over Java reflection, exposing Kotlin-specific concepts like nullability, data class properties, and KFunction. Use ::class to get a KClass<T> reference and inspect its members.

Why This Week Matters: Kotlin's DSL capabilities are what powers Gradle's build.gradle.kts, Ktor's routing, and Jetpack Compose's UI layout. Understanding lambdas with receivers reveals how all of these frameworks actually work under the hood — making you a far more effective user and builder of libraries.

Week 14 — Multiplatform, Testing & Performance

Understand Kotlin Multiplatform fundamentals, write comprehensive tests, and profile Kotlin code.

Core Focus

Understanding Kotlin Multiplatform fundamentals, writing comprehensive tests, and profiling and optimising Kotlin code.

Topics & Subtopics

  1. Kotlin Multiplatform (KMP) Fundamentals — KMP lets you share business logic across Android, iOS, web, and desktop from a single Kotlin codebase, using expect/actual declarations to bridge platform-specific implementations — networking, data models, and validation shared once.
  2. Unit Testing with Kotest / JUnit5 — Kotlin testing typically uses JUnit5 with idiomatic Kotlin assertion libraries like Kotest or AssertJ, providing readable, fluent assertions. Test functions can use backtick-quoted names for full human-readable descriptions.
  3. Mocking with MockK — MockK is Kotlin's idiomatic mocking library, with first-class support for coroutines, extension functions, and final classes (which other JVM mocking libraries struggle with). It enables isolated unit testing of classes with external dependencies.
  4. Performance — Inline Classes & Benchmarking with JMH — value class wraps a single value with zero runtime overhead — type-safe wrappers around primitives like UserId(Int) with no boxing cost. JMH provides statistically rigorous benchmarking for the JVM.

Why This Week Matters: Kotlin Multiplatform is the fastest-growing part of the Kotlin ecosystem — companies are adopting it to share business logic across Android and iOS, eliminating duplicated logic in two separate codebases. MockK's coroutine support makes it the only viable mocking library for modern Kotlin code.
