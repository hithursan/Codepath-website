About This Roadmap

Databases are the backbone of every software system ever built. Every app you use — banking systems, college portals, e-commerce sites, hospital records, social media platforms — stores its data in a relational database and queries it using SQL.

This roadmap takes you from understanding what a database even is, all the way to designing normalized schemas, writing complex multi-table queries, building stored procedures, and managing transactions — the skills that make you genuinely useful in any software or data role.

The path runs: Database Foundations → Basic SQL → Aggregates & JOINs → Subqueries & DML → ERD & Normalization → Schema Implementation → Views & Procedures → Transactions → Window Functions & CTEs → EduBase Capstone.

Phase 1 — Database Foundations & Basic SQL

Weeks 1 through 3 take you from understanding what a database is to mastering the complete SELECT statement with filtering, sorting, string functions, and date functions. By the end of Phase 1 you will confidently create tables, insert data, and retrieve exactly what you need from a single table.

Week 1 — What Is a Database? Setting Up MySQL & First Queries

Understand what databases are and why they exist, install MySQL Workbench, create your first database and table, and run your first SQL queries.

Topics & Subtopics

  1. What is a database and why it exists — databases vs spreadsheets; relational vs non-relational (conceptual); RDBMS examples (MySQL, PostgreSQL, SQLite, Oracle, SQL Server).
  2. MySQL installation and Workbench navigation — connecting to localhost, root user, interface overview.
  3. Creating a database — CREATE DATABASE, USE, SHOW DATABASES, DROP DATABASE.
  4. Creating a table — CREATE TABLE with column names, data types (INT, VARCHAR(n), DATE), DROP TABLE.
  5. Inserting data — INSERT INTO table (col1, col2) VALUES (val1, val2); single row and multi-row inserts.
  6. Retrieving data — SELECT * FROM table; result grid in Workbench.
  7. MySQL comment syntax — -- single line, /* multi-line */.

Why This Week Matters: This week establishes what you are working with and why. Every SQL command you write for the rest of the roadmap requires a database and tables to operate on. Without this foundation, no other week makes sense. Databases exist because spreadsheets have size limits, no concurrency, no referential integrity, and no access control — a database handles millions of rows, simultaneous users, relationships between data, and permissions natively.

Common Mistakes

  • Forgetting the semicolon ; at the end of SQL statements — MySQL usually gives an error or waits for more input.
  • Forgetting to run USE dbname; before creating tables — tables get created in the wrong database.
  • Using spaces in table or column names — use underscores: student_id, not student id.
  • Running DROP DATABASE without being certain — irreversible in most configurations.

How to Avoid Them

  • End every SQL statement with a semicolon — make it a reflex.
  • Always start every SQL script with USE dbname; as the first line.
  • Use snake_case for all identifiers (table names, column names).
  • Before running any DROP command, verify which database is active with SELECT DATABASE();

Week 2 — Data Types, Constraints & Table Design Basics

Understand MySQL's data types in depth, learn all standard table constraints, and design tables that prevent bad data from being inserted.

Topics & Subtopics

  1. MySQL data types — numeric (TINYINT, INT, BIGINT, DECIMAL(p,s), FLOAT); string (CHAR(n), VARCHAR(n), TEXT, ENUM); date/time (DATE, DATETIME, TIMESTAMP, YEAR).
  2. NULL and NOT NULL — NULL means "no value known"; never use = NULL; use IS NULL and IS NOT NULL.
  3. PRIMARY KEY — uniquely identifies each row; cannot be NULL; typically INT AUTO_INCREMENT.
  4. AUTO_INCREMENT — MySQL automatically assigns the next integer value on INSERT.
  5. UNIQUE constraint — prevents duplicate values in a column (e.g., email must be unique per user).
  6. DEFAULT values — value used if a column is not provided in an INSERT.
  7. CHECK constraint — MySQL 8.0.16+ enforces a condition on column values.
  8. ENUM and SET types — ENUM stores one value from a predefined list.
  9. DESCRIBE / DESC — shows the structure of a table.
  10. ALTER TABLE — ADD COLUMN, DROP COLUMN, MODIFY COLUMN, RENAME COLUMN, ADD CONSTRAINT.

Why This Week Matters: Poor data type and constraint choices are one of the most common sources of data quality bugs in real systems. Choosing VARCHAR(5) for a field that needs 200 characters causes truncation. Not using NOT NULL allows unexpected NULL values that break query logic. This week teaches you to design tables defensively.

Daily Study Plan

Day Focus
Day 1 Numeric data types — try INT, DECIMAL, FLOAT. Insert money values and observe rounding with FLOAT vs. DECIMAL.
Day 2 String types — CHAR vs VARCHAR performance concept; ENUM with gender/status examples.
Day 3 Date/time types — DATE, DATETIME, TIMESTAMP. Insert with NOW(), CURDATE(), date literals.
Day 4 NULL behavior — try WHERE column = NULL (wrong), then WHERE column IS NULL (correct). NOT NULL constraint.
Day 5 PRIMARY KEY + AUTO_INCREMENT. Insert without specifying the PK column; verify auto-assignment.
Day 6 UNIQUE, DEFAULT, CHECK constraints. ALTER TABLE — add, modify, drop columns.
Day 7 Review all types and constraints. Design a products table using all learned concepts. Mini project.

  • Try inserting a row with price = -5 — observe the CHECK constraint reject it.
  • Use ALTER TABLE to add a discount_percent DECIMAL(5,2) DEFAULT 0.00 column to the products table.

Common Mistakes

  • Using VARCHAR(255) for everything regardless of actual content length.
  • Using FLOAT for monetary values — causes precision errors; always use DECIMAL(10,2) for money.
  • Forgetting AUTO_INCREMENT on the primary key column and manually managing IDs.
  • Using = to check for NULL values instead of IS NULL.

How to Avoid Them

  • Match data type to actual data: names → VARCHAR(100), descriptions → TEXT, money → DECIMAL(10,2), status → ENUM.
  • Never use FLOAT or DOUBLE for financial calculations.
  • Always pair INT AUTO_INCREMENT PRIMARY KEY for surrogate primary keys.
  • Use IS NULL and IS NOT NULL exclusively for NULL comparisons.

Week 3 — SELECT in Depth: Filtering, Sorting & String Functions

Master the complete SELECT statement — every clause, operator, and built-in function needed to retrieve exactly the data you want from a single table.

Topics & Subtopics

  1. SELECT with specific columns — never use SELECT * in production; always name columns.
  2. WHERE clause and operators — =, !=, >, <, >=, <=; AND, OR, NOT.
  3. Special operators — BETWEEN a AND b (inclusive); IN (val1, val2); LIKE with % and _; IS NULL, IS NOT NULL.
  4. ORDER BY — ASC/DESC; multiple column sorting.
  5. LIMIT and OFFSET — return n rows starting from position m; pagination concept.
  6. DISTINCT — return only unique combinations of selected columns.
  7. Column aliases (AS) — rename a column or expression in the result set.
  8. String functions — UPPER, LOWER, LENGTH, SUBSTRING, CONCAT, TRIM, REPLACE.
  9. Numeric functions — ROUND, CEIL, FLOOR, ABS, MOD.
  10. Date functions — NOW(), CURDATE(), YEAR(), MONTH(), DAY(), DATEDIFF(), DATE_ADD().

Why This Week Matters: The WHERE clause is used in SELECT, UPDATE, and DELETE. Mastering it this week means it will be automatic for the rest of the roadmap. Most SQL problems — including all interview questions — require precise filtering, sorting, and column computation. This is the most used week's content in real SQL work.

Common Mistakes

  • Writing WHERE name = NULL instead of WHERE name IS NULL.
  • Using ORDER BY with a column number (ORDER BY 3) instead of the column name — unclear and error-prone.
  • Confusing % (zero or more characters) and _ (exactly one character) in LIKE.

How to Avoid Them

  • Always use IS NULL / IS NOT NULL for NULL checks — never = or != with NULL.
  • Always use column names in ORDER BY, not positional numbers.
  • Test LIKE patterns on a small dataset before using them on large tables.

Phase 2 — Core SQL: Aggregates, JOINs, Subqueries & DML

Weeks 4 through 7 cover the core SQL skills used in every real-world database application: summarizing data with GROUP BY, combining tables with JOINs, writing nested queries, and safely modifying data. By the end of Phase 2 you will be able to answer real interview questions and write production-quality SQL.

Week 4 — Aggregate Functions, GROUP BY & HAVING

Learn to summarize and analyze data using aggregate functions, group results by categories, and filter grouped results.

Topics & Subtopics

  1. Aggregate functions — COUNT, SUM, AVG, MIN, MAX; operate on a set of rows and return a single value.
  2. COUNT(*) vs COUNT(column) — COUNT(*) counts all rows including NULLs; COUNT(col) counts only non-NULL values.
  3. GROUP BY clause — divides rows into groups based on unique values; every column in SELECT must be in GROUP BY or wrapped in an aggregate function.
  4. HAVING clause — filters groups after aggregation; WHERE filters rows before grouping.
  5. Multiple column GROUP BY — one group per unique combination of columns.
  6. WITH ROLLUP — adds a grand total row at the end of GROUP BY results.
  7. ROUND with aggregates — ROUND(AVG(price), 2).

Why This Week Matters: Aggregate queries are the core of data analysis. Every report, dashboard, or analytical query in the real world uses GROUP BY + aggregates. "How many students are in each department?", "What is the average salary by role?", "Which product category has the highest revenue?" — all answered with what you learn this week.

Day 4 GROUP BY rule practice — deliberately violate it; observe the error/behavior in MySQL.

Day 7 Mini project.

Common Mistakes

  • Writing WHERE COUNT(*) > 5 instead of HAVING COUNT(*) > 5 — WHERE cannot reference aggregate functions.
  • Selecting a non-aggregated column not in the GROUP BY clause — MySQL may return a random value for it.
  • Confusing COUNT(col) (ignores NULLs) with COUNT(*) (counts all rows).

How to Avoid Them

  • Rule: if the filter condition uses an aggregate function, use HAVING; otherwise, use WHERE.
  • Set SET SESSION sql_mode = 'ONLY_FULL_GROUP_BY'; in MySQL Workbench to enforce the GROUP BY rule and catch mistakes.
  • Use COUNT(col) when you want to count non-NULL values specifically; otherwise default to COUNT(*).

Week 5 — JOINs: Combining Data from Multiple Tables

Master all types of JOINs — the most important SQL skill after basic SELECT — to combine data from related tables.

Topics & Subtopics

  1. Why JOINs exist — normalized databases split data across multiple tables; JOIN reunites them for queries.
  2. INNER JOIN — returns only rows where the join condition is satisfied in BOTH tables; unmatched rows are excluded.
  3. LEFT JOIN — returns ALL rows from the left table, matched rows from the right; NULLs fill unmatched right columns.
  4. RIGHT JOIN — mirror of LEFT JOIN; rarely needed (rewrite as LEFT JOIN with tables swapped).
  5. FULL OUTER JOIN — MySQL: simulate with LEFT JOIN UNION RIGHT JOIN.
  6. CROSS JOIN — Cartesian product; every combination of rows from both tables.
  7. SELF JOIN — joining a table with itself; requires table aliases; use case: employee-manager relationships.
  8. Multi-table JOIN — chain multiple JOINs: FROM orders o JOIN customers c ON ... JOIN products p ON ...
  9. Anti-join pattern — LEFT JOIN ... WHERE right_table.pk IS NULL finds all rows in left table with no match in right table.

Why This Week Matters: JOINs are the most tested SQL concept in every technical interview for every role. Almost every real query involves at least one JOIN. The LEFT JOIN + WHERE IS NULL anti-join pattern appears in dozens of interview problems. This is the most important week in the roadmap.

Daily Study Plan

  • Write a query to find all customers who have NEVER made a payment — LEFT JOIN + WHERE IS NULL on the payment table.
  • Write a query to list every film with its category name and the number of times it has been rented — requires 3+ JOINs + GROUP BY.

Common Mistakes

  • Forgetting the ON condition in a JOIN — accidentally creates a Cartesian product.
  • Joining on the wrong columns — mismatched semantics produce wrong results.
  • Using RIGHT JOIN everywhere — almost always rewritable as a clearer LEFT JOIN with tables swapped.
  • Not checking result row counts — unexpectedly many rows usually indicates a missing or wrong join condition.

How to Avoid Them

  • Always write the ON clause immediately after the JOIN keyword.
  • Before writing a JOIN: "What column in table A corresponds to what column in table B?" — join on those.
  • Prefer LEFT JOIN (put the "all rows" table on the left) for readability.

Week 6 — Subqueries: Queries Within Queries

Write nested SQL queries to solve problems that cannot be solved with a single-level query, and understand when subqueries are appropriate vs JOINs.

Topics & Subtopics

  1. What is a subquery — a complete SELECT statement nested inside another SQL statement; the inner query runs first.
  2. Subquery in WHERE clause — scalar result: WHERE salary > (SELECT AVG(salary) FROM employees).
  3. IN with subquery — outer column must match any value returned by the inner query.
  4. NOT IN with subquery — critical caveat: if the subquery returns ANY NULL value, NOT IN returns no rows at all; prefer NOT EXISTS.
  5. EXISTS and NOT EXISTS — returns TRUE if the subquery returns at least one row; safer than NOT IN when NULLs are possible.
  6. Scalar subqueries — a subquery returning exactly one row and one column; can appear anywhere a single value is expected.
  7. Correlated subqueries — inner query references a column from the outer query; executes once per outer row — use JOINs instead when performance matters.
  8. Derived table (subquery in FROM) — FROM (SELECT ...) AS alias — treats the inner query's result as a virtual table; useful for two-level aggregation.

Why This Week Matters: Many SQL problems — particularly interview problems — are most naturally expressed with subqueries. "Find students who scored above the class average" requires either a correlated subquery or a derived table. Understanding subqueries also prepares you for CTEs in Week 13, which are a cleaner syntax for the same concept.

Daily Study Plan

Day Focus

Day 2 Subquery with IN: find customers who rented films in a specific category (without JOIN).
Day 3 NOT IN and the NULL problem — experiment with this deliberately.

Common Mistakes

  • Using NOT IN on a column that can contain NULLs — this silently returns zero rows and appears to "work" while giving wrong results.
  • Writing deeply nested subqueries (4+ levels) when a JOIN or CTE would be cleaner.
  • Assuming the outer query logic based on the wrong understanding of what the inner query returns.

How to Avoid Them

  • Use NOT EXISTS instead of NOT IN when the subquery column could contain NULL values.
  • Limit nesting to 2 levels; use CTEs (Week 13) for anything deeper.
  • When debugging a subquery: run the inner query alone first; verify its output; then wrap it in the outer query.

Week 7 — DML: INSERT, UPDATE, DELETE & Safe Data Modification

Master all data modification statements — including safe patterns for UPDATE and DELETE — and understand how to work with data that references other tables.

Topics & Subtopics

  1. INSERT INTO ... SELECT ... — copies data from one query result into another table; used for backups and staging tables.
  2. UPDATE with WHERE — always include a WHERE clause; without it, EVERY row is updated.
  3. UPDATE with JOIN — update rows in one table based on matching rows in another.
  4. DELETE with WHERE — always include WHERE; without it, EVERY row is deleted (but the table structure remains).
  5. TRUNCATE TABLE — faster than DELETE with no WHERE; resets AUTO_INCREMENT; cannot be rolled back in MySQL.
  6. Safe DELETE/UPDATE practices — before running a DELETE, run the equivalent SELECT first to verify what will be deleted.
  7. Foreign key constraint behavior — RESTRICT (default), CASCADE, SET NULL, NO ACTION.
  8. INSERT ... ON DUPLICATE KEY UPDATE — upsert: inserts if the row doesn't exist, updates if a UNIQUE/PK constraint would be violated.

Why This Week Matters: UPDATE and DELETE without WHERE are among the most dangerous SQL mistakes possible — they affect every row in the table. This week teaches you the professional discipline to prevent data disasters. Foreign key behavior is also critical for understanding what happens when you delete parent records in a normalized database.

Daily Study Plan

Day Focus
Day 1 INSERT INTO ... SELECT — copy filtered rows from one table to another.
Day 2 UPDATE with WHERE — 10 targeted updates. Use SELECT first to verify what will be changed.
Day 3 UPDATE with JOIN. UPDATE without WHERE — inside a TRANSACTION so you can ROLLBACK.
Day 4 DELETE with WHERE — safe patterns. "SELECT first, DELETE second" discipline.
Day 5 TRUNCATE vs. DELETE — observe AUTO_INCREMENT difference. DELETE without WHERE inside a TRANSACTION.
Day 6 Foreign key cascade behaviors — set up parent/child tables with each FK action; test deleting parent rows.
Day 7 INSERT ON DUPLICATE KEY UPDATE — upsert pattern. Mini project.

🛠 Mini Project

Common Mistakes

  • Running UPDATE table SET col = val; without a WHERE clause — updates every single row.
  • Running DELETE FROM table; without a WHERE clause — deletes all rows.
  • Not testing DELETE/UPDATE with a SELECT equivalent first.
  • Confusing TRUNCATE (cannot rollback, resets AUTO_INCREMENT) with DELETE (can rollback, keeps AUTO_INCREMENT).

How to Avoid Them

  • Enable "Safe Updates" mode in MySQL Workbench: Edit → Preferences → SQL Editor → check "Safe Updates".
  • Before any UPDATE or DELETE, run the equivalent SELECT and verify the row count and content.
  • Wrap any risky UPDATE/DELETE in START TRANSACTION; ... ROLLBACK; during development; only change to COMMIT when verified.

Phase 3 — Database Design: ERD, Normalization, Keys & Constraints

Weeks 8 through 10 shift from writing queries to designing databases professionally. You will draw Entity-Relationship Diagrams before writing any SQL, systematically normalize schemas to eliminate redundancy, and implement a complete production-quality MySQL schema with all constraint types and performance indexes.

Week 8 — Entity-Relationship Diagrams (ERD) & Conceptual Design

Design the conceptual structure of a database using Entity-Relationship Diagrams before writing any SQL — the professional database design methodology.

Topics & Subtopics

  1. Entities and attributes — entity: a distinct real-world object (becomes a table); attribute: a property (becomes a column); primary key attribute: uniquely identifies each entity instance.
  2. Relationships — an association between two entities; named (e.g., Student "enrolls in" Course).
  3. Cardinality — ONE-TO-ONE (one person has one passport), ONE-TO-MANY (one department has many students), MANY-TO-MANY (students enroll in many courses; courses have many students).
  4. Crow's Foot notation — the most common ERD style; symbols at the ends of relationship lines indicate mandatory/optional and one/many.
  5. Junction table (many-to-many implementation) — e.g., enrollments(student_id FK, course_id FK); attributes describing the relationship belong in the junction table.
  6. Weak entities — cannot be uniquely identified without their relationship to another entity; PK includes the FK of the owner entity.
  7. Mapping ERD to relational schema — one-to-one → add FK to either table; one-to-many → FK on the "many" side; many-to-many → junction table.

Why This Week Matters: A poorly designed schema — discovered after the database is in production with real data — is extremely costly to fix. ERD design is what professional database architects and data engineers do before writing a single line of SQL. This week is where you learn to think like a database designer, not just a SQL writer.

  • Treating attributes of a junction table (like enrollment date, grade) as belonging to the parent entities.

How to Avoid Them

  • Any time two entities have a many-to-many relationship, immediately plan a junction table with two FK columns.
  • Attributes that describe the relationship (not either entity alone) belong in the junction table.
  • When uncertain about cardinality: ask both directions — "Can one A have many B?" AND "Can one B have many A?"

Week 9 — Normalization: 1NF, 2NF, 3NF & BCNF

Learn normalization theory and practice — systematically eliminate data redundancy and anomalies from database designs.

Topics & Subtopics

  1. Data anomalies — Update anomaly (one fact requires updating many rows), Insert anomaly (cannot insert valid data because unrelated required data is missing), Delete anomaly (deleting data accidentally destroys other valid data).
  2. Functional dependencies — "A → B" means A uniquely determines B; e.g., student_id → student_name.
  3. First Normal Form (1NF) — each column must contain only atomic (indivisible) values; no repeating groups or arrays in a cell; each row uniquely identifiable.
  4. Second Normal Form (2NF) — must be in 1NF AND every non-key attribute must be fully functionally dependent on the ENTIRE primary key; partial dependencies removed; only relevant when the PK is composite.
  5. Third Normal Form (3NF) — must be in 2NF AND no non-key attribute should transitively depend on the PK through another non-key attribute.
  6. Boyce-Codd Normal Form (BCNF) — stricter version of 3NF; every determinant must be a superkey; handles rare anomalies 3NF misses.
  7. Denormalization — deliberately introducing redundancy for performance in read-heavy systems; common in data warehouses.

Why This Week Matters: Normalization is the theoretical core of database design. Interview questions on normalization are extremely common in DBMS courses and technical interviews. More practically: a normalized schema prevents entire categories of bugs from ever existing. This week gives you both the theory (to pass exams) and the practical skill (to apply it in EduBase).

Daily Study Plan

Day 2 Functional dependencies — practice writing FD sets for 3 given relations.

Common Mistakes

  • Confusing partial dependency (2NF issue) with transitive dependency (3NF issue).
  • Forgetting that 2NF is only relevant when the primary key is composite.
  • Over-normalizing — creating too many small tables when the joined query is rarely needed.

How to Avoid Them

  • For each normal form, always start by asking: "What problem does this form solve?"
  • Draw the FD diagram; partial dependency means an FD arrow starts from only PART of the composite PK; transitive dependency means a non-key column points to another non-key column.
  • Normalization to 3NF is the standard for OLTP systems; going further should be justified by specific anomaly scenarios.

Week 10 — Implementing Schemas: Keys, Constraints, Indexes & Referential Integrity

Translate a normalized ERD into a complete, production-quality MySQL schema with proper keys, all constraint types, foreign key relationships, and performance indexes.

Topics & Subtopics

  1. Translating ERD to DDL — start with tables that have no FKs, then add tables with FKs in dependency order.
  2. Composite primary keys — PRIMARY KEY (col1, col2); common in junction tables.
  3. Foreign key creation syntax — FOREIGN KEY (col) REFERENCES other_table(other_col); always name FK constraints explicitly: CONSTRAINT fk_name FOREIGN KEY ...
  4. Referential integrity actions — RESTRICT (default), CASCADE, SET NULL, NO ACTION.
  5. Composite UNIQUE constraints — UNIQUE KEY uq_enrollment (student_id, course_id, semester) — the COMBINATION must be unique.
  6. Indexes: CREATE INDEX, SHOW INDEX — B-Tree index (default); UNIQUE index; index all FK columns manually (MySQL does NOT auto-index FK columns).
  7. When to add an index — columns frequently in WHERE, JOIN ON, and ORDER BY; FK columns; email columns. Index trade-off: speeds up reads, slows writes.
  8. EXPLAIN SELECT ... — shows query execution plan; key fields: type (ALL = full table scan = slow), key (which index was used), rows (estimated rows examined).

Why This Week Matters: This week bridges design and implementation. A perfectly normalized ERD that is poorly implemented in SQL — missing indexes on FK columns, no referential integrity constraints, wrong cascade behaviors — will cause performance problems and data integrity issues in production.

Week 11 — Views & Stored Procedures

Create reusable database objects — views for simplified querying and stored procedures for encapsulated, reusable database logic.

Topics & Subtopics

  1. Views — a saved SELECT query stored as a named database object; behaves like a virtual table; simplifies complex joins, hides sensitive columns, provides a stable interface while the underlying schema can change.
  2. CREATE VIEW, ALTER VIEW, DROP VIEW — query a view with SELECT * FROM view_name.
  3. Updatable vs non-updatable views — updatable views require a single base table with no aggregates, DISTINCT, GROUP BY, or joins.
  4. Stored procedures — encapsulated, reusable database logic; DELIMITER $$ required before CREATE PROCEDURE.
  5. Parameters: IN, OUT, INOUT — IN: value passed into procedure; OUT: return value to caller; INOUT: read and update.
  6. Variables in procedures — DECLARE v_count INT DEFAULT 0; inside the procedure body.
  7. Conditional logic — IF condition THEN ... ELSEIF ... ELSE ... END IF;
  8. Loops — WHILE, LOOP.
  9. User-defined functions (basic) — similar to a procedure but must return a value with RETURN; can be used inline in SQL expressions.
  10. Error handling — DECLARE EXIT HANDLER FOR SQLEXCEPTION ROLLBACK;

Why This Week Matters: Views and stored procedures are the building blocks of a database application layer. In production systems, applications often interact with the database exclusively through views and stored procedures — not direct table access. This enforces security, business rules, and a stable API even as the underlying schema evolves.

Common Mistakes

  • Forgetting DELIMITER $$ before CREATE PROCEDURE — causes syntax errors from the semicolons inside the body.

Week 12 — Transactions, ACID Properties & Concurrency

Understand and implement database transactions — ensuring that multi-step operations either fully succeed or fully fail, preventing partial data corruption.

Topics & Subtopics

  1. What is a transaction — a sequence of SQL statements treated as a single unit; either ALL succeed (COMMIT) or ALL are undone (ROLLBACK); example: bank transfer = debit account A AND credit account B — both must succeed or neither should.
  2. ACID properties — Atomicity (all or nothing), Consistency (valid state to valid state), Isolation (concurrent transactions behave as if serial), Durability (committed data survives crashes).
  3. START TRANSACTION, COMMIT, ROLLBACK — manual transaction control.
  4. SAVEPOINT and partial rollback — SAVEPOINT name; ROLLBACK TO SAVEPOINT name undoes back to that point without rolling back the entire transaction.
  5. Auto-commit mode — MySQL default is autocommit=1; every statement is automatically committed; use START TRANSACTION to manage manually.
  6. Concurrency problems — dirty read, non-repeatable read, phantom read.
  7. Isolation levels — READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ (MySQL InnoDB default), SERIALIZABLE.
  8. Deadlocks — two transactions each hold a lock the other needs; MySQL InnoDB detects and automatically rolls back the "lighter" transaction.
  9. Transactions in stored procedures — DECLARE EXIT HANDLER FOR SQLEXCEPTION ROLLBACK;

Why This Week Matters: Transactions are what make databases safe for financial and business-critical operations. Without proper transaction handling, a network failure during a bank transfer could debit one account without crediting the other — losing money permanently. Understanding ACID and isolation levels is also a standard interview topic for any backend or data engineering role.

Daily Study Plan

Day Focus

How to Avoid Them

  • Use DECLARE EXIT HANDLER FOR SQLEXCEPTION ROLLBACK; in stored procedures to auto-rollback on any error.

Week 13 — Window Functions, CTEs & Advanced Analytics

Learn SQL's most powerful analytical features — window functions and Common Table Expressions — to write elegant, high-performance analytical queries.

Topics & Subtopics

  1. Common Table Expressions (CTEs) with WITH — a named temporary result set visible only for the duration of the query; makes complex queries readable by breaking them into named steps.
  2. Recursive CTEs — a CTE that references itself; used for hierarchical data (org charts, category trees); requires an anchor member and a recursive member joined with UNION ALL.
  3. Window functions: OVER() clause — unlike GROUP BY which collapses rows, a window function computes an aggregate OVER a set of related rows while retaining all individual rows.
  4. PARTITION BY — divides rows into groups for the window function; similar to GROUP BY but doesn't collapse rows.
  5. Ranking functions — ROW_NUMBER() (unique sequential numbers), RANK() (ties get same rank, next rank skipped: 1,2,2,4), DENSE_RANK() (ties don't skip: 1,2,2,3), NTILE(n) (divide into n buckets).
  6. Aggregate window functions — SUM() OVER(), AVG() OVER(), COUNT() OVER() — running totals, moving averages.
  7. Offset functions — LAG(col, n) returns value from n rows before; LEAD(col, n) returns value from n rows after; used for "compared to previous period" calculations.
  8. Frame specifications — ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW — cumulative sum up to each row.

Why This Week Matters: Window functions and CTEs are SQL's most powerful analytical tools. "Rank students by GPA within each department", "show each course's enrollment alongside the department total", "running total of fees collected per month" — these require window functions. CTEs make deeply nested subqueries readable and maintainable. These are the features that separate intermediate SQL from advanced SQL.

Common Mistakes

  • Confusing RANK() and DENSE_RANK() — forgetting that RANK skips numbers after ties.
  • Using GROUP BY when window functions are needed — GROUP BY collapses rows; window functions preserve them.
  • Recursive CTEs without a termination condition — causes infinite recursion; always add a WHERE depth < n or similar limit.

How to Avoid Them

  • Rule: if you need per-row calculations that reference other rows in the same group → use window functions; if you need one result per group → use GROUP BY.
  • Always test recursive CTEs with a LIMIT clause first to observe the first few levels before removing the limit.
