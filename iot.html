About This Roadmap

This roadmap takes a complete beginner — with no prior electronics or programming experience — to a hands-on IoT developer capable of building, programming, and cloud-connecting a fully functional environmental monitoring system in 15 weeks of full-time study.

The path runs: Electronics Foundations → Arduino Basics → Environmental Sensors → Display & Alerts → Data Logging → Wi-Fi Connectivity → Cloud Dashboard → Capstone Build.

Every week builds directly on the previous one. By Week 15 you will have EnviroNode — a real, working IoT device measuring 5 environmental parameters, logging data to an SD card, and streaming live readings to a public cloud dashboard.

Phase 1 — Electronics Foundations & Arduino Basics

Weeks 1 through 3 take you from understanding what electricity is to reading real physical signals with Arduino's analog-to-digital converter. By the end of Phase 1 you will have built multiple circuits, written structured C++ sketches, and learned the core debugging tools used throughout this roadmap.

Week 1 — Electronics Fundamentals & First Arduino Program

Understand basic electronics concepts, set up the Arduino IDE, and build your first working circuit and program.

Topics & Subtopics

  1. What is electricity — voltage, current, resistance and their practical meaning.
  2. Ohm's Law — V = I × R and its use for LED resistor calculation.
  3. LEDs, resistors, and breadboards — how each works and how they connect.
  4. Arduino Uno anatomy — digital pins 0–13, analog pins A0–A5, 5V/3.3V/GND, USB, reset.
  5. Arduino IDE — writing, verifying, and uploading a sketch.
  6. Sketch structure — setup() runs once; loop() runs continuously.
  7. Digital output — pinMode(), digitalWrite(), delay().
  8. Serial Monitor — Serial.begin(), Serial.println() for debugging.

Why This Week Matters: Every project in this roadmap builds on these fundamentals. If you do not understand what a resistor does or why setup() runs differently from loop(), you will be copy-pasting code without understanding it. This week is the absolute foundation.

Daily Study Plan

Day Focus
Day 1 Read about voltage, current, resistance. Ohm's Law with a real LED calculation. Draw a simple LED circuit on paper.
Day 2 Install Arduino IDE. Connect Arduino to PC. Run the Blink example on the built-in LED (Pin 13).
Day 3 Build first circuit: LED + resistor on breadboard connected to Pin 13. Blink using Blink sketch.
Day 4 Modify the Blink sketch: 3 different blink patterns. Add a second LED on Pin 12.
Day 5 Open Serial Monitor. Print "LED ON" and "LED OFF" at the correct moments.
Day 6 Add a third LED. Create a traffic light pattern: green → yellow → red → repeat.
Day 7 Review Ohm's Law with 3 calculation examples. Complete mini project. Push sketch to GitHub.

Practice Tasks

  • Calculate the correct resistor for a red LED (Vf=2V), yellow LED (Vf=2.1V), blue LED (Vf=3.2V) at 5V, 10mA.
  • Build a circuit with 3 LEDs and write a sketch that blinks them in a specific repeating pattern.
  • Write a sketch that sends a count to Serial Monitor: "Loop: 1", "Loop: 2" ... every second.

🛠 Mini Project

Build a breadboard traffic light (red, yellow, green LEDs + resistors). Write a sketch that cycles: green (5 s) → yellow (2 s) → red (5 s) → repeat. Each transition must print a message to Serial Monitor: TRAFFIC LIGHT: GREEN, TRAFFIC LIGHT: YELLOW, etc. Draw the circuit diagram in Tinkercad. Push the sketch to GitHub with a comment explaining every line.

Common Mistakes

  • Connecting an LED directly to 5V and GND without a resistor — the LED will likely burn out immediately.
  • Plugging components across the breadboard's center gap, accidentally disconnecting them.
  • Selecting the wrong COM port in Arduino IDE.
  • Not pressing Save before uploading — uploading an older version of the sketch.

How to Avoid Them

  • Always use a resistor in series with an LED. When in doubt, use 330Ω for red/yellow or 220Ω for green/blue.
  • Check that component legs are in the same breadboard row — same column, same side of the center gap.
  • Go to Tools → Port after plugging in Arduino and select the newly appeared port.
  • Press Ctrl+S before every upload — make it a habit.

Week 2 — Digital Input, Buttons & PWM

Make your Arduino respond to physical inputs (buttons) and control output intensity using PWM.

Topics & Subtopics

  1. Digital input — pinMode(INPUT_PULLUP), digitalRead().
  2. Buttons and debouncing — why bouncing happens and how to fix it in software.
  3. Conditional logic in C++ — if/else, else if, boolean operators &&, ||, !.
  4. Variables and data types — int, bool, float, char, declaration and initialization.
  5. PWM output — analogWrite(pin, 0–255) on PWM-capable pins.
  6. for loops — repeat a block a set number of times; used for LED fade effects.

Why This Week Matters: Real IoT systems respond to buttons, switches, thresholds, and conditions. Understanding digital input and conditional logic is what allows your system to say "if temperature is above 35°C, turn on the alert." This week also introduces PWM, used for the EnviroNode status indicator LED.

Day 2 Build button circuit. Read button state, print to Serial Monitor. Toggle LED with button.
Day 3 Software debouncing — implement and observe the difference with and without it.
Day 4 C++ variables and data types review. Write a sketch using int, bool, float.

Practice Tasks

  • Build a circuit with 2 buttons and 2 LEDs. Button 1 toggles LED 1; Button 2 fades LED 2 up and down each press.
  • Add a Serial Monitor debug line every time debounce confirms a valid press: Button pressed (debounced).
  • Write a sketch that fades an LED from 0 to 255 over exactly 2 seconds, then off immediately.

🛠 Mini Project

LED brightness controller with 2 buttons. Button A increases brightness (by 50 per press: 0 → 50 → 100 → 150 → 200 → 255 → stays at 255). Button B decreases brightness (same steps in reverse). Current brightness printed to Serial Monitor on every press. Implement debouncing. Draw circuit in Tinkercad. Push to GitHub.

Common Mistakes

  • Using INPUT instead of INPUT_PULLUP — causes "floating pin" behavior where button press is detected randomly.
  • Writing if (buttonState = LOW) (assignment) instead of if (buttonState == LOW) (comparison) — a classic C++ bug.
  • Forgetting that PWM only works on PWM-capable pins (marked with ~ on the board: 3, 5, 6, 9, 10, 11 on Uno).
  • Using delay() for button debounce, which blocks the entire program.

How to Avoid Them

  • Always use pinMode(buttonPin, INPUT_PULLUP) — simpler and more reliable than an external pull-up resistor.
  • Always use == for comparison, never =.
  • Check the Uno pinout diagram: only pins marked with ~ support analogWrite().
  • Implement debounce with millis() instead of delay().

Week 3 — Analog Input: Reading the Physical World

Read continuously varying physical signals using Arduino's analog-to-digital converter — the skill used for every environmental sensor in Phase 2.

Topics & Subtopics

  1. Analog vs digital signals — continuous vs discrete.
  2. Analog-to-Digital Conversion (ADC) — 10-bit, 0–1023 range, 0V–5V mapping.
  3. analogRead() — reading from A0–A5 pins.
  4. Voltage divider circuits — Vmid = 5V × R2 / (R1 + R2).
  5. Light Dependent Resistors (LDRs) — resistance decreases as light increases.
  6. Potentiometers — variable resistor for practicing analog input.
  7. map() function — rescaling from one range to another.
  8. Serial Plotter — real-time graphing of analog data.

Why This Week Matters: Analog input is how your Arduino reads MQ-135 (air quality) and LDR (light level) values. The ADC and map() function are used in every analog sensor read in the capstone. Understanding the voltage divider is the foundation for understanding how resistive sensors work.

Daily Study Plan

Day Focus
Day 1 ADC concept — how analogRead() works. Voltage divider math on paper.
Day 2 Connect potentiometer. Read values, print to Serial Monitor. See full range 0–1023.
Day 3 Open Serial Plotter. Plot potentiometer as you turn it. Control LED brightness with map().
Day 4 Build LDR voltage divider circuit. Read light values. Plot in Serial Plotter.
Day 5 Threshold logic: turn LED on if light reading falls below 400 (ambient darkness detected).
Day 6 Combine potentiometer + LDR: one controls LED brightness, one triggers an alert.
Day 7 Draw circuit diagrams for both circuits. Mini project + GitHub commit.

Practice Tasks

  • Read the potentiometer and print a "bar graph" of asterisks to Serial Monitor proportional to the reading (100% = **********, 50% = *****).
  • Build a "night light" circuit: LED off when bright, LED on when dark, brightness inversely proportional to light level.
  • Calculate the LDR voltage divider midpoint voltage for 3 different light conditions using approximate LDR resistance values.

🛠 Mini Project

Ambient light level meter. LDR reads light level (analogRead 0–1023). Map the value to 5 LEDs on a bar graph: all 5 LEDs on = bright, 0 LEDs = dark. Print the raw ADC value and the number of active LEDs to Serial Monitor every second. Capture a Serial Plotter screenshot showing light levels changing as you move a flashlight near and away. Include the screenshot in your GitHub README.

Common Mistakes

  • Connecting the LDR directly between 5V and A0 without a series resistor — the pin floats and gives meaningless values.
  • Using Serial.print() with multiple values on the same line in Serial Plotter — Plotter requires each value with Serial.println().
  • Confusing map() output — it does not clamp values outside the input range; use constrain() too if needed.

How to Avoid Them

  • Always use a voltage divider for resistive sensors: 5V → LDR → junction → 10kΩ resistor → GND; read the junction with analogRead().
  • For Serial Plotter: Serial.print("Light:"); then Serial.println(lightValue); on separate lines.
  • Use constrain(value, min, max) before map() if raw reading might occasionally exceed expected range.

  • Sparkfun — Voltage Dividers (learn.sparkfun.com)
  • Last Minute Engineers — LDR Tutorial (lastminuteengineers.com)

Phase 2 — Environmental Sensors & Display

Weeks 4 through 7 add all five of EnviroNode's sensors one by one, introduce the I2C communication protocol, connect a 16×2 LCD display, and integrate everything into a single multi-sensor sketch with structured serial output and a rotating LCD display.

Week 4 — DHT22: Temperature & Humidity Sensor

Read accurate temperature and humidity data from the DHT22, handle sensor errors, and display clean formatted data on Serial Monitor and Serial Plotter.

Topics & Subtopics

  1. DHT22 sensor operation — single-wire digital protocol, thermistor + capacitive humidity element.
  2. DHT22 vs DHT11 — wider range (−40°C to +80°C), higher accuracy (±0.5°C, ±2% RH).
  3. Adafruit DHT library — dht.readTemperature(), dht.readHumidity().
  4. isnan() error checking — library returns NaN on failed read; always check before using.
  5. float data type — for decimal temperature and humidity values.
  6. millis() non-blocking timer — read sensor every 2 seconds without blocking other code.
  7. Structured Serial output — labeled columns, header row, formatted readings.

Why This Week Matters: The DHT22 is the primary temperature and humidity sensor in EnviroNode. This week introduces the complete real environmental sensor workflow: install library → wire sensor → read data → handle errors → format output. This exact pattern repeats for every sensor in the project.

Daily Study Plan

Day Focus
Day 1 DHT22 concept and datasheet. Identify the 4 pins (or 3 if using a module). Wire the sensor.
Day 2 Install DHT library + Adafruit Unified Sensor. Upload example. Verify readings appear.
Day 3 Add Fahrenheit conversion: float tempF = temperature * 9/5 + 32;
Day 4 Add isnan() checks. Simulate a read error by briefly disconnecting the data wire.
Day 5 Replace delay(2000) with a millis()-based timer. Verify non-blocking behavior.
Day 6 Format Serial output: print a header line once in setup, then labeled columns every 2 seconds.
Day 7 Mini project + GitHub commit with Serial Plotter screenshot in README.

Practice Tasks

  • Deliberately disconnect the DHT22 data wire mid-run and confirm isnan() correctly triggers the error message.
  • Use the Serial Plotter to plot temperature and humidity simultaneously as two separate series while breathing on the sensor.
  • Calculate heat index (feels-like temperature) from temperature and humidity using the formula in the DHT library's HeatIndex example.

🛠 Mini Project

Temperature and humidity monitor that: reads DHT22 every 2 seconds (non-blocking with millis()), prints readings in labeled columns (Temp: 27.4°C | Humidity: 65.2% | HeatIndex: 29.1°C), handles read errors gracefully, and plots all three values in Serial Plotter. Push to GitHub with Serial Plotter screenshot in README.

Common Mistakes

  • Connecting DHT22 data pin without a 10kΩ pull-up resistor (many module versions have it built-in; bare DHT22 requires it).
  • Using delay(2000) throughout — blocks all other code from running.
  • Not calling dht.begin() in setup() — results in zero or NaN readings.
  • Forgetting to pass the sensor type in the constructor: DHT dht(DHTPIN, DHT22);

How to Avoid Them

  • If using a bare DHT22 (not a module), add a 10kΩ resistor between the data pin and 5V.
  • Use millis() timing pattern from this week onward — it is the standard approach for all multi-sensor Arduino code.
  • Always call dht.begin() in setup() before the first read.

Week 5 — I2C Protocol, BMP280 & LCD Display

Learn the I2C communication protocol, read barometric pressure from the BMP280, and display all readings on a 16×2 LCD screen.

Topics & Subtopics

  1. I2C protocol — SDA (Serial Data), SCL (Serial Clock), device addresses, master/slave; on Uno: A4 (SDA), A5 (SCL).
  2. I2C addresses — BMP280 typically 0x76 or 0x77; I2C LCD typically 0x27 or 0x3F.
  3. I2C Scanner sketch — confirm addresses before writing library code.
  4. BMP280 sensor — barometric pressure (300–1100 hPa) and temperature; Adafruit BMP280 library.
  5. 16×2 LCD + I2C adapter — PCF8574 chip reduces 16-pin LCD to 2 wires; shares I2C bus with BMP280.
  6. LiquidCrystal I2C library — lcd.begin(), lcd.print(), lcd.setCursor(), lcd.clear(), lcd.backlight().
  7. LCD display rotation — millis() timer cycles between screens every 3–5 seconds.

Why This Week Matters: I2C is how professional embedded systems connect multiple peripherals on shared buses. EnviroNode uses I2C for both the BMP280 and the LCD simultaneously — if you do not understand I2C addressing, your two devices will conflict and neither will work.

Daily Study Plan

Day Focus
Day 1 I2C concept — two wires, addresses, master/slave. Read the Wire library reference.
Day 2 Wire BMP280. Run I2C Scanner. Note the address. Install Adafruit BMP280 library.
Day 3 Read and print pressure and temperature from BMP280. Compare temperature to DHT22 reading.
Day 4 Wire I2C LCD (sharing SDA/SCL with BMP280). Run I2C Scanner — confirm both appear.
Day 5 Install LiquidCrystal I2C library. Display "Hello IoT!" on LCD. Adjust contrast if needed.
Day 6 Display BMP280 readings on LCD. Rotate screens using millis() timer.
Day 7 Integrate DHT22 — 3-screen LCD rotation. Mini project + GitHub commit.

Practice Tasks

  • Use the I2C Scanner to find all I2C devices connected. What happens if you use the wrong address in code?
  • Display temperature from BMP280 and from DHT22 on separate LCD screens — do they agree?
  • Display barometric pressure formatted to 1 decimal place: 1013.2 hPa.

🛠 Mini Project

Environmental display station. LCD rotates every 4 seconds through 3 screens: Screen 1: Temp: 27.4C / Humidity: 65% (DHT22); Screen 2: Pressure: 1013hPa / Temp: 27.1C (BMP280); Screen 3: Status: OK / Uptime: 42s. All readings printed simultaneously to Serial Monitor. Push to GitHub with a photo of the LCD showing live readings.

Common Mistakes

  • Powering BMP280 from 5V instead of 3.3V — BMP280 is a 3.3V device (some modules have an onboard regulator; check yours).
  • Not finding the LCD in I2C Scanner because address is 0x3F but library is initialized with 0x27.
  • Calling lcd.clear() every loop — causes visible flickering; use lcd.setCursor(0,0) instead.

How to Avoid Them

  • Check your BMP280 module's voltage specification before wiring.
  • Always run I2C Scanner before writing any I2C library code — use the address the Scanner reports.
  • Use lcd.setCursor() to rewrite specific positions rather than clearing the entire screen each update.

Week 7 — MQ-135 Air Quality Sensor

Add MQ-135 air quality sensing, understand its operating principles and limitations, and implement a 3-level Air Quality Index indicator.

Topics & Subtopics

  1. MQ-135 operation — heating element + metal oxide surface; resistance changes in presence of target gases.
  2. Warm-up requirement — 24+ hours for first use; some minutes each power cycle. Readings immediately after power-on are unreliable.
  3. AOUT vs DOUT — use AOUT (analog) for precise readings; DOUT (digital) is only above/below threshold.
  4. Voltage conversion — float voltage = analogRead(A0) * (5.0 / 1023.0);
  5. AQI thresholds (relative) — Low (~0–300 raw) = Good, Mid (~300–600) = Moderate, High (>600) = Poor — calibrate to your environment.
  6. Gases detected — primarily CO₂, NH₃, benzene, alcohol, smoke — a general indicator, not a specific gas analyzer.
  7. Safety note — the heating element gets hot; do not touch the metal sensing mesh.

Why This Week Matters: Air quality monitoring is one of the most significant real-world applications of low-cost environmental IoT. Adding the MQ-135 completes EnviroNode's sensing suite. Understanding its limitations — being able to explain what your sensor can and cannot measure — shows technical depth during a viva.

Daily Study Plan

Day Focus
Day 1 MQ-135 concept and datasheet. Safety notes. Wire the sensor. Power on for warm-up.
Day 2 Read analog value. Print to Serial Monitor. Observe how values change near different sources.
Day 3 Convert raw ADC to voltage. Print voltage alongside raw value.
Day 4 Define AQI thresholds as const int variables. Implement 3-level classification.
Day 5 Add LED indicators: Green = Good, Yellow = Moderate, Red = Poor.
Day 6 Integrate into the multi-sensor sketch from Week 6.
Day 7 Real-world test near cooking smells, then fresh air. Mini project + GitHub commit.

🛠 Mini Project

Air quality indicator: MQ-135 reads continuously, classifies as Good/Moderate/Poor using definable thresholds, controls 3 LEDs (green/yellow/red), prints AQI: 412 (MODERATE) to Serial Monitor. LCD displays Air: MOD | 412. Include a clear warning comment at the top of the sketch: // Note: MQ-135 readings are relative, not calibrated AQI values.

Common Mistakes

  • Treating MQ-135 readings as standardized AQI values — they are raw voltages proportional to mixed gas concentrations.
  • Testing immediately after first power-on — readings are completely unstable until the sensor warms up.
  • Using DOUT for precise measurement — DOUT only gives a simple above/below threshold binary output.

How to Avoid Them

  • Always document in your code and README that MQ-135 values are relative indicators only.
  • Power the sensor and let it warm up for at least 1 hour before recording baseline values.
  • Use AOUT for all analog air quality measurements.

Week 8 — Threshold Alerts & Multi-Sensor Logic

Why This Week Matters: An environmental monitoring station without alerts is just a display. Threshold-based alerting is the core safety function of any real monitoring system — from greenhouse controllers to server room temperature alarms.

Week 9 — SD Card Data Logging

Log all sensor readings to a CSV file on a micro SD card — creating a persistent, downloadable record of environmental data over time.

Topics & Subtopics

  1. SPI protocol — 4 wires: MOSI, MISO, SCK, CS; on Uno: MOSI=11, MISO=12, SCK=13, CS=configurable.
  2. SD library — SD.begin(csPin), SD.open("log.csv", FILE_WRITE), file.println(data), file.close().
  3. CSV file format — Comma Separated Values; header row; importable into Excel or Google Sheets.
  4. Timestamp with millis() — relative time in seconds since startup (no RTC needed for this roadmap).
  5. SD card formatting — must be FAT32 for Arduino SD library; cards over 32GB may need special formatting.
  6. File handling safety — open → write → close on each log entry prevents data loss on power interruption.
  7. Error handling — detect SD card initialization failure, display on LCD, continue local operation.

Why This Week Matters: The SD card transforms EnviroNode from a live display into a proper data recorder. Logged CSV files are the direct evidence of your device working over time — they are what you bring to a viva to show hours of real environmental data collection.

Daily Study Plan

Day Focus
Day 1 SPI concept. Wire SD module (MOSI=11, MISO=12, SCK=13, CS=10). Format SD card as FAT32.
Day 2 SD library basics: open a file, write "Hello SD Card", close, verify on PC.
Day 3 Write the logging function: timestamp + all sensor readings in CSV format.
Day 4 Log every 30 seconds (millis() timer). Let it run for 5 minutes. Import CSV into a spreadsheet.
Day 5 Add SD card initialization error detection — print "SD FAIL" on LCD if SD not found.
Day 6 Integrate into the full multi-sensor sketch. Resolve any SPI pin conflicts.
Day 7 Let the full system run for 2 hours. Export CSV. Plot charts in Google Sheets. Mini project + GitHub commit.

🛠 Mini Project

Data logger: all 5 parameters logged to log.csv on SD card every 30 seconds. Header row: time_s,temp_C,humidity_pct,pressure_hPa,aqi_raw,light_pct. CSV file included in GitHub repo (real data from your sensor). Screenshot of data plotted as line charts in Google Sheets included in README.

Common Mistakes

  • SD card formatted as NTFS or exFAT — only FAT32 (or FAT16 for small cards) works with the Arduino SD library.
  • Not closing the file after each write — data may not be written to the physical card (buffered in RAM).
  • SPI pin conflicts — pins 10–13 are reserved for SPI; do not connect other components to them.

How to Avoid Them

  • Format the SD card on PC as FAT32 before first use every time.
  • Always call file.close() after writing each set of readings.
  • Reserve pins 10–13 exclusively for SPI/SD card; move all other components to other digital pins.

Week 10 — Serial Communication & Data Visualization

Stream all sensor data over Serial for real-time PC-side visualization and understand serial communication protocols in depth.

Topics & Subtopics

  1. UART in depth — Universal Asynchronous Receiver-Transmitter; full-duplex; TX and RX pins; Uno hardware UART on pins 0/1 shared with USB.

Week 11 — ESP-01 / ESP8266 Wi-Fi Module

Connect the ESP-01 module to Arduino and to a Wi-Fi network using AT commands, and perform your first HTTP request from the microcontroller.

Topics & Subtopics

  1. ESP8266 / ESP-01 overview — Wi-Fi enabled microcontroller from Espressif; used in AT-command mode controlled by Arduino.
  2. 3.3V power requirement (critical) — ESP-01 peaks at ~250mA; never use Arduino's built-in 3.3V pin; use an AMS1117 3.3V regulator from the 5V pin.
  3. ESP-01 pinout — VCC, GND, TX, RX, CH_PD, GPIO0, RST. CH_PD must be tied HIGH.
  4. AT commands — AT (test), AT+RST (reset), AT+CWMODE=1 (station), AT+CWJAP (connect to Wi-Fi), AT+CIFSR (get IP), AT+CIPSTART (open TCP), AT+CIPSEND (send data).
  5. SoftwareSerial for ESP-01 — use pins 3/4 so hardware Serial remains free for debugging.
  6. Wi-Fi credentials security — never commit real credentials to a public GitHub repo; use placeholder comments.

Why This Week Matters: Wi-Fi connectivity is what makes EnviroNode an IoT system rather than just a data logger. This week introduces the most challenging hardware of the roadmap — the ESP-01 is sensitive to power and wiring errors. Take extra care with every connection before powering on.

Daily Study Plan

Day 2 Build the 3.3V power supply using AMS1117 on breadboard. Verify 3.3V output with a multimeter if available.
Day 3 Wire ESP-01 to Arduino via SoftwareSerial. Send AT command manually. Verify OK response.
Day 4 Connect to Wi-Fi with AT+CWJAP. Verify with AT+CIFSR (shows IP address).
Day 5 Write an Arduino sketch that automatically sends AT commands to connect to Wi-Fi on startup.
Day 6 Perform an HTTP GET request to a simple public URL (httpbin.org/get). See response in Serial Monitor.
Day 7 Mini project + GitHub commit. Use YOUR_SSID_HERE placeholder in the committed sketch.

Common Mistakes

  • Powering ESP-01 from Arduino's built-in 3.3V pin (max ~50mA) — ESP-01 draws up to 250mA during Wi-Fi transmission, causing resets and potential damage.
  • Swapping TX and RX — Arduino TX connects to ESP RX and vice versa; they cross.
  • Sending 5V logic levels on ESP-01 RX pin — the ESP-01 is not 5V tolerant; use a 1kΩ/2kΩ voltage divider on Arduino TX → ESP RX.

How to Avoid Them

  • Always power ESP-01 from an AMS1117 3.3V regulator sourced from the Arduino 5V pin — not the Arduino 3.3V output.
  • Remember: TX connects to RX and vice versa. Draw the connection on paper before wiring.
  • Add a 1kΩ/2kΩ voltage divider on the Arduino TX (5V) → ESP-01 RX (3.3V) signal line.

Topics & Subtopics

  1. ThingSpeak platform — free IoT analytics platform by MathWorks; accepts HTTP and MQTT; free tier up to 3M messages/year.
  2. Creating a channel and fields — up to 8 data fields per channel; name each to match your sensor parameter.
  3. Write API key — unique string authenticating your device's writes; keep it private.
  4. HTTP Write URL — GET /update?api_key=WRITE_KEY&field1=VALUE HTTP/1.1 sent to api.thingspeak.com.
  5. Rate limit — ThingSpeak free tier: minimum 15 seconds between updates; use 60 seconds to be safe.
  6. Separate millis() timer for cloud — independent of the sensor read interval (5 s) and display update interval.
  7. ThingSpeak response codes — 200 = success, 0 = rate limit hit.

Week 14 — EnviroNode Full Integration & Stability Testing

Integrate all subsystems into one clean, stable, fully working EnviroNode sketch and run extended stability tests.

Topics & Subtopics

  1. State machine design — Normal, Warning, Alert, Wi-Fi Connecting, SD Error states; code each explicitly.
  2. Code modularization — multiple functions, clear naming, one responsibility per function.
  3. Avoiding Arduino String class — use C-style char arrays and sprintf() instead to prevent memory fragmentation over long runtimes.
  4. PROGMEM — store constant strings in Flash instead of the Uno's 2KB SRAM to prevent RAM exhaustion.
  5. 24-hour stability test — run the complete system without reset; check for SD write errors, Wi-Fi disconnects, DHT22 NaN errors, buzzer false triggers, LCD corruption.
  6. Wiring cleanup — organize jumper cables, label connections, strain-relieve wires on the breadboard.

Why This Week Matters: Code that works for 5 minutes in a demo can fail after 6 hours due to memory leaks, SD buffer overflows, or Wi-Fi drops. Stability testing is what separates a real IoT device from a prototype that only works at demo time.

Daily Study Plan

Day Focus

Day 2 Merge all subsystem sketches into one. Resolve pin conflicts.
Day 3 Replace String objects with char arrays. Check RAM usage (shown in IDE upload output).

Day 5 4-hour stability test. Fix any errors that appear.

Common Mistakes

  • Testing only for 10 minutes and declaring the system "stable" — most long-run failures appear after 2–6 hours.
  • Using Arduino String class for repeated string operations — causes heap fragmentation and eventual crashes.
  • Leaving SPI pin conflicts unresolved — causes SD writes to fail when another peripheral pulls the same pin.

How to Avoid Them

  • Run a minimum 12-hour test before the final week — non-negotiable.
  • Replace every String object with char buf[32]; and sprintf(buf, "format", value);.
  • Double-check all SPI and I2C pin assignments against the full combined wiring table.
