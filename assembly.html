Assembly x86-64: Beginner to Master

Complete 16-Week Curriculum — From Hello World to SIMD, Concurrency & Low-Level Systems Programming

About This Curriculum

This is a 16-week, project-driven Assembly curriculum designed to take a complete beginner to a professional-level systems programmer. The roadmap moves from CPU mental models, registers, arithmetic, and bitwise operations through conditional jumps, loops, procedures, and arrays, into structs, dynamic memory, error handling, floating-point, and C library interop, all the way to recursion, SIMD vectorization, OS-level concurrency, and three complete capstone projects.

The curriculum is organized into four phases: Phase 1 covers absolute basics and the CPU mental model, Phase 2 covers control flow, procedures, and data structures, Phase 3 covers intermediate and advanced core topics including structs and dynamic memory, and Phase 4 covers master-level SIMD, concurrency, and full systems-programming projects.

By Week 16, you will have built three complete, portfolio-ready low-level applications and be confident writing production-grade Assembly from scratch.

Level: Beginner → Master  |  Duration: 16 weeks  |  Stack: x86-64, NASM, Linux (System V ABI), GDB  |  Prerequisites: Basic programming logic (no Assembly experience needed)

Phase 1 — Absolute Basics

Weeks 1–4: Registers, data sections, arithmetic, and bitwise operations.

Week 1 — Environment Setup & The CPU Mental Model

Understand what Assembly actually is, how a CPU executes instructions, and assemble your first program.

Core Focus

Understanding what Assembly actually is, how a CPU executes instructions, and assembling your first working binary.

Topics & Subtopics

  1. What is Assembly Language? — Assembly is a low-level, human-readable representation of machine code where each instruction maps almost directly to a single CPU operation. Unlike high-level languages, there is no garbage collector, no automatic memory management, and no hidden abstractions — you control the CPU directly.
  2. Installing NASM & Assembling Your First Binary — NASM (Netwide Assembler) converts .asm source files into machine code object files, which the linker then turns into an executable binary. This two-step process — assemble, then link — is fundamental to how all Assembly programs are built.
  3. The CPU Mental Model: Registers, Memory & the Instruction Pointer — A CPU executes one instruction at a time, fetched from the address held in the Instruction Pointer (RIP). Registers are tiny, extremely fast storage slots built directly into the CPU, used for nearly all computation since accessing RAM is comparatively slow.
  4. General-Purpose Registers (RAX, RBX, RCX, RDX, etc.) — x86-64 provides 16 general-purpose 64-bit registers (RAX through R15), each usable for arithmetic, data storage, or addressing. Certain registers have conventional roles — RAX typically holds return values, and RDI/RSI/RDX/RCX/R8/R9 pass function arguments under the System V calling convention.

Why This Week Matters: The CPU mental model — registers, the instruction pointer, and memory — is the foundation every Assembly concept builds on. Getting the toolchain working from Day 1 means you spend time learning Assembly, not fighting your environment.

Code Example

; hello.asm — A minimal x86-64 NASM program
section .text
    global _start          ; Entry point must be marked global for the linker
_start:
    mov rax, 60             ; syscall number for exit (Linux x86-64)
    mov rdi, 0              ; exit code 0
    syscall                 ; trigger the kernel system call

; Assemble and link:
; nasm -f elf64 hello.asm -o hello.o
; ld hello.o -o hello
; ./hello && echo $?

Week 2 — Data Sections, Sizes & Moving Data

Understand how Assembly programs declare and organize data in memory and the rules of moving values between locations.

Core Focus

Understanding how Assembly programs declare and organize data in memory, and the rules of moving values between locations.

Topics & Subtopics

  1. The .data, .bss, and .text Sections — Assembly programs are organized into sections: .text holds executable instructions, .data holds initialized variables, and .bss reserves space for uninitialized variables. This separation lets the OS apply different memory permissions — .text is execute-only, .data is read-write.
  2. Defining Data with db, dw, dd, dq — NASM's data definition directives declare variables of specific sizes: db (byte, 1 byte), dw (word, 2 bytes), dd (doubleword, 4 bytes), and dq (quadword, 8 bytes). Choosing the right size matters for both correctness and memory efficiency.
  3. Register Sizes: 64-bit, 32-bit, 16-bit, 8-bit — Every 64-bit register has smaller "views" you can access directly: RAX (64-bit), EAX (lower 32 bits), AX (lower 16 bits), and AL/AH (lower/upper 8 bits). Writing to EAX automatically zeroes the upper 32 bits of RAX — a critical quirk worth memorizing.
  4. The mov Instruction & Addressing Modes — mov copies data between registers, between a register and memory, or an immediate value into a register — but never directly between two memory locations. Square brackets [ ] dereference an address, meaning "the value stored at this memory location."

Why This Week Matters: Understanding sections and data sizes is the foundation of every Assembly program structure. The EAX zero-extension behavior is one of the most common sources of subtle bugs in x86-64 code — knowing it from Week 2 prevents hours of debugging later.

Code Example

section .data
    message db "Hello, Assembly!", 0   ; Null-terminated string
    flag    db 1                        ; 1 byte
    score   dw 1000                     ; 2 bytes
    bigNum  dd 100000                   ; 4 bytes
    address dq 0x7fffffffffff           ; 8 bytes — full 64-bit

section .bss
    buffer resb 64                      ; Reserve 64 uninitialized bytes

section .text
    global _start
_start:
    mov rax, 0xFFFFFFFFFFFFFFFF  ; Fill all 64 bits
    mov eax, 0x12345678           ; Writing EAX ZEROES upper 32 bits of RAX!
    mov rbx, [score]              ; RBX = VALUE at 'score' address
    mov rax, 60
    xor rdi, rdi
    syscall

Week 3 — Arithmetic & The FLAGS Register

Perform arithmetic operations and understand how the CPU tracks results via status flags.

Core Focus

Performing arithmetic operations and understanding how the CPU tracks the results of those operations via status flags.

Topics & Subtopics

  1. Basic Arithmetic: add, sub, inc, dec — add and sub perform addition and subtraction between two operands, while inc/dec provide shorthand for adding or subtracting exactly 1. All of these modify the FLAGS register as a side effect.
  2. Multiplication & Division (mul, imul, div, idiv) — Multiplication and division in x86-64 have unusual register conventions: mul/div work on unsigned numbers using implicit RAX/RDX pairs, while imul/idiv handle signed numbers. Division requires the dividend split across RDX:RAX before executing.
  3. The FLAGS Register — The FLAGS register holds single-bit indicators set automatically by arithmetic and comparison instructions — the Zero Flag (ZF), Carry Flag (CF), Sign Flag (SF), and Overflow Flag (OF) are the most important. Conditional jumps read these flags to make decisions.
  4. The cmp Instruction — cmp performs a subtraction internally to set the FLAGS register but discards the result, leaving both operands unchanged. It exists purely to prepare the FLAGS for a subsequent conditional jump instruction.

Why This Week Matters: The FLAGS register is the bridge between arithmetic and control flow — without understanding it, conditional jumps next week will feel like magic. The RDX clearing requirement before division is one of the most common Assembly bugs — knowing it now prevents crashes later.

Code Example

section .text
    global _start
_start:
    mov rax, 50
    add rax, 25       ; RAX = 75
    sub rax, 10       ; RAX = 65
    inc rax           ; RAX = 66
    dec rax           ; RAX = 65

    ; Division: CRITICAL — clear RDX before div
    mov rax, 50
    xor rdx, rdx      ; RDX MUST be zero before unsigned division!
    mov rbx, 7
    div rbx           ; RAX = 7 (quotient), RDX = 1 (remainder)

    ; cmp sets FLAGS without modifying operands
    mov rax, 10
    mov rbx, 10
    cmp rax, rbx      ; Internally: 10 - 10 = 0 → ZF = 1

    mov rax, 60
    xor rdi, rdi
    syscall

Carefully handle the RDX clearing requirement before division. Exit with the quotient as your exit code and verify by hand-calculating the expected result.

Week 4 — Bitwise Operations & Number Representation

Manipulate individual bits and understand how Assembly represents signed numbers, hex, and binary.

Core Focus

Manipulating individual bits and understanding how Assembly represents signed numbers, hexadecimal, and binary.

Topics & Subtopics

  1. Bitwise Logical Operators: and, or, xor, not — Bitwise operators work on individual bits rather than the number as a whole — and is used for masking specific bits, or for setting bits, xor for toggling bits (and the classic zero-a-register trick), and not for inverting all bits.
  2. Bit Shifting: shl, shr, sar — shl (shift left) and shr (shift right) move bits and fill vacated positions with zero — equivalent to multiplying or dividing by powers of two. sar (arithmetic shift right) preserves the sign bit, making it correct for signed division by powers of two.
  3. Signed vs Unsigned Numbers & Two's Complement — The CPU does not inherently know if a value is signed or unsigned — the same bit pattern can be interpreted either way depending on which instructions you use. Negative numbers are stored using two's complement: invert all bits and add 1.
  4. Hexadecimal & Binary Literals in NASM — NASM accepts numeric literals in decimal, hexadecimal (0x prefix), and binary (0b prefix). Hex is the standard for representing memory addresses and bit masks since it maps cleanly to 4-bit nibbles.

Why This Week Matters: Bitwise operations are used everywhere in systems programming — setting hardware flags, packing/unpacking data, efficient multiplication/division by powers of two. The xor rcx, rcx zero pattern appears in virtually every Assembly program and is faster than mov rcx, 0 on most CPUs.

Code Example

section .text
    global _start
_start:
    ; Bitwise operations
    mov rax, 0b1100       ; RAX = 12 in binary
    and rax, 0b1010        ; AND → 0b1000 = 8 (keeps bits set in BOTH)
    mov rbx, 0b0001
    or  rax, rbx           ; OR  → 0b1001 = 9 (sets bits in EITHER)
    xor rcx, rcx           ; XOR with itself = fastest way to zero a register

    ; Bit shifting (powers of two math)
    mov rax, 5
    shl rax, 1             ; RAX = 10 (5 * 2)
    shl rax, 2             ; RAX = 40 (10 * 4)
    shr rax, 3             ; RAX = 5  (40 / 8)

    ; Two's complement: -1 in 64-bit = all bits set
    mov rax, -1            ; 0xFFFFFFFFFFFFFFFF

    mov rax, 60
    xor rdi, rdi
    syscall

Phase 2 — Control Flow & Logic

Weeks 5–8: Jumps, loops, procedures, calling conventions, arrays, and strings.

Week 5 — Conditional Jumps & Branching

Control program flow with labels and conditional jump instructions — Assembly's equivalent of if/else.

Core Focus

Controlling program flow with labels and conditional jump instructions — Assembly's equivalent of if/else.

Topics & Subtopics

  1. Unconditional Jumps (jmp) & Labels — A label marks a named location in your code, and jmp unconditionally transfers execution there — Assembly's equivalent of goto. Labels are how you build every higher-level control structure (loops, functions, if/else) since Assembly has no native syntax for them.
  2. Conditional Jumps (je, jne, jg, jl, etc.) — Conditional jumps only branch if specific FLAGS bits are set — je/jz (jump if equal/zero), jne/jnz (not equal), jg/jl (greater/less, signed), and ja/jb (above/below, unsigned). They always follow a cmp or arithmetic instruction that sets the relevant flags.
  3. Implementing if/else Logic — There is no if keyword in Assembly — you build conditional logic manually using cmp followed by a conditional jump that skips over or jumps to a block of code. The pattern inverts the condition: "jump AWAY if the condition is true" or vice versa, depending on your structure.
  4. The test Instruction — test performs a bitwise AND between two operands purely to set FLAGS, without storing the result — most commonly used as test rax, rax to efficiently check if a register is zero. It is faster than cmp rax, 0 on most CPUs.

Why This Week Matters: Every non-trivial Assembly program branches. Conditional jumps are the only decision-making mechanism in Assembly — understanding the signed vs unsigned jump variants (jg vs ja) prevents subtle comparison bugs when dealing with negative numbers or large addresses.

Code Example

section .text
    global _start
_start:
    ; Unconditional jump
    mov rax, 1
    jmp skip_this
    mov rax, 999       ; Never executed
skip_this:

    ; if/else: if (rax == 100) rbx = 1 else rbx = 0
    mov rax, 100
    cmp rax, 100
    jne else_branch    ; If NOT equal, jump to else
        mov rbx, 1
        jmp end_if
else_branch:
        mov rbx, 0
end_if:

    ; test: fast zero-check
    xor rax, rax
    test rax, rax      ; ZF = 1 because RAX is zero
    jz is_zero
is_zero:

    mov rax, 60
    xor rdi, rdi
    syscall

Week 6 — Loops

Build repeating control structures using conditional jumps and the dedicated loop instruction.

Core Focus

Building repeating control structures using conditional jumps and the dedicated loop instruction.

Topics & Subtopics

  1. Manual Loops with Labels & Jumps — Every loop in Assembly is fundamentally a label, a body of instructions, a counter update, and a conditional jump back to the label. Understanding this manual construction is essential before relying on the loop shorthand instruction.
  2. The loop Instruction — The dedicated loop instruction automatically decrements RCX and jumps to the given label if RCX is not zero, combining two manual steps into one. It is a convenient shorthand, though many modern compilers avoid it in favor of explicit jumps for performance reasons.
  3. While-Style & Do-While-Style Loop Patterns — A "while" loop checks its condition before the first iteration (may run zero times), while a "do-while" loop checks after the first iteration (always runs at least once). Both patterns are built from the same ingredients — labels, comparisons, and jumps — just arranged differently.
  4. Nested Loops — Nested loops require separate counter registers for each loop level, since a single register cannot track two independent counts simultaneously. Always double-check which register belongs to which loop level and reset the inner counter on every outer iteration.

Why This Week Matters: Loops power all algorithms — sorting, searching, array processing. Understanding the manual label/jump pattern first means you truly understand what the loop instruction does under the hood. Nested loops with mismatched counters are a classic Assembly bug — knowing it now prevents hours of debugging.

Code Example

section .text
    global _start
_start:
    ; Manual loop: sum 5+4+3+2+1 = 15
    mov rcx, 5
    mov rax, 0
sum_loop:
    add rax, rcx
    dec rcx
    jnz sum_loop       ; Jump back if RCX != 0

    ; loop instruction (RCX is always the counter)
    mov rcx, 10
    mov rbx, 0
count_loop:
    inc rbx
    loop count_loop    ; Decrements RCX, jumps if RCX != 0

    ; while-style: while (rax < 10)
    mov rax, 0
while_top:
    cmp rax, 10
    jge while_end      ; Exit if RAX >= 10
    inc rax
    jmp while_top
while_end:

    ; Nested loops: outer * inner = 3 * 4 = 12 iterations
    mov rcx, 3
outer_loop:
    mov rdx, 4         ; RESET inner counter each outer pass!
inner_loop:
    dec rdx
    jnz inner_loop
    dec rcx
    jnz outer_loop

    mov rax, 60
    xor rdi, rdi
    syscall

Week 7 — Procedures, the Stack & Calling Conventions

Organize code into reusable procedures and understand the stack-based mechanics of function calls.

Core Focus

Organizing code into reusable procedures and understanding the stack-based mechanics of function calls.

Topics & Subtopics

  1. The Stack: push and pop — The stack is a region of memory that grows downward, accessed via the RSP (stack pointer) register. push decrements RSP and stores a value; pop retrieves the top value and increments RSP — it is strictly Last-In-First-Out (LIFO).
  2. call and ret — Procedure Invocation — call pushes the return address (the instruction after the call) onto the stack and jumps to the target label; ret pops that address back into RIP, resuming execution right after the original call. This pair is how Assembly implements function calls.
  3. The System V AMD64 Calling Convention — On Linux x86-64, the first six integer/pointer function arguments are passed in RDI, RSI, RDX, RCX, R8, R9 (in that order), with the return value placed in RAX. Following this convention is essential for interoperating with C libraries and the OS.
  4. Caller-Saved vs Callee-Saved Registers & Stack Frames — Caller-saved registers (RAX, RCX, RDX, RSI, RDI, R8–R11) may be overwritten by any function call. Callee-saved registers (RBX, RBP, R12–R15) must be preserved by the called function using push/pop at the start and end of the procedure.

Why This Week Matters: The stack and calling convention are the backbone of every function call in any language. Getting caller-saved vs callee-saved registers wrong produces the hardest Assembly bugs to debug — values mysteriously change across function calls. This week's knowledge directly enables interop with C libraries in Week 12.

Code Example

section .text
    global _start

; add_function(a, b): RDI=a, RSI=b → return in RAX
add_function:
    push rbx               ; Callee-saved — preserve before using
    mov rax, rdi
    add rax, rsi
    pop rbx                ; Restore callee-saved register
    ret

_start:
    mov rdi, 15            ; First argument
    mov rsi, 27            ; Second argument
    call add_function      ; RAX = 42 after return

    ; Stack: push/pop LIFO demo
    mov rax, 10
    mov rbx, 20
    push rax               ; RSP -= 8, [RSP] = 10
    push rbx               ; RSP -= 8, [RSP] = 20
    pop rcx                ; RCX = 20 (last in, first out)
    pop rdx                ; RDX = 10

    mov rax, 60
    xor rdi, rdi
    syscall

Week 8 — Arrays, Strings & Indexed Addressing

Work with sequences of data in memory using indexed addressing modes and build string-processing routines.

Core Focus

Working with sequences of data in memory using indexed addressing modes and building string-processing routines.

Topics & Subtopics

  1. Declaring & Accessing Arrays — Arrays in Assembly are simply contiguous blocks of memory, declared with repeated dq/dd values or the times directive. You access an element by calculating its byte offset: base_address + (index * element_size).
  2. Indexed & Scaled Addressing Modes — x86-64 supports a powerful addressing mode [base + index*scale + displacement] that computes an array element's address in a single instruction. The scale factor (1, 2, 4, or 8) automatically matches your element size, eliminating manual multiplication.
  3. Null-Terminated Strings & String Length — C-style strings in Assembly are simply byte arrays terminated by a 0 (null) byte, with no separate length field. Computing a string's length requires manually scanning forward until you encounter that null terminator.
  4. Linux Syscalls: Writing to Standard Output — The write syscall (number 1 on Linux x86-64) outputs bytes to a file descriptor — 1 for stdout. It requires the file descriptor in RDI, a pointer to the data in RSI, and the byte count in RDX.

Why This Week Matters: Arrays are the most common data structure in any program, and the scaled addressing mode [rsi + rcx*8] is how you access them efficiently. The write syscall is your first tool for producing visible output — essential for every project from this point forward.

Code Example

section .data
    numbers dq 10, 20, 30, 40, 50    ; Array of 5 quadwords
    msg     db "Hello, syscalls!", 10  ; 10 = newline
    len     equ $ - msg               ; Assemble-time length calculation

section .text
    global _start
_start:
    ; Scaled indexed addressing: arr[2]
    mov rsi, numbers
    mov rcx, 2
    mov rax, [rsi + rcx*8]   ; RAX = 30 (index 2, scale 8)

    ; Null-terminated string length
    mov rsi, msg
    xor rcx, rcx
strlen_loop:
    cmp byte [rsi + rcx], 0
    je strlen_done
    inc rcx
    jmp strlen_loop
strlen_done:               ; RCX = string length

    ; write syscall: print to stdout
    mov rax, 1             ; syscall: write
    mov rdi, 1             ; fd = stdout
    mov rsi, msg
    mov rdx, len
    syscall

    mov rax, 60
    xor rdi, rdi
    syscall

Phase 3 — Intermediate & Advanced Core

Weeks 9–12: Structs, dynamic memory, error handling, macros, floating-point, and C library interop.

Week 9 — Structures, the Stack Frame & Local Variables

Model structured records in memory and manage local variables within procedures using proper stack frames.

Core Focus

Modeling structured records in memory and properly managing local variables within procedures using stack frames.

Topics & Subtopics

  1. Struct-Like Layouts with struc — NASM's struc/endstruc directives let you define a named memory layout with labeled field offsets, similar to a C struct. This makes structured data (like a "Point" with x/y fields) far more readable than tracking raw byte offsets manually.
  2. Accessing Struct Fields in Memory — Once a struct layout is defined, you access individual fields by adding the field's offset (provided automatically by struc) to the struct's base address. This pattern is identical to how compiled C struct access works under the hood.
  3. Stack Frames with push rbp / mov rbp, rsp — A stack frame is a dedicated region of stack memory for a single function call, anchored by RBP (the base pointer). The classic prologue saves the caller's frame and establishes a stable reference point for accessing local variables.
  4. Local Variables via Stack Offsets — Local variables that do not fit in registers are stored on the stack, accessed as negative offsets from RBP (e.g., [rbp-8] for the first local). This convention keeps local data isolated per function call, safely supporting recursion.

Why This Week Matters: Stack frames with RBP are what GDB uses to display a backtrace — understanding them lets you debug complex programs. NASM's struc is how you model any real-world data (packets, file headers, game entities) without losing track of byte offsets.

Code Example

struc Point
    .x: resq 1          ; Field x: 8 bytes (offset 0)
    .y: resq 1          ; Field y: 8 bytes (offset 8)
endstruc

section .bss
    myPoint resb Point_size

section .text
    global _start

calculate:
    push rbp
    mov rbp, rsp
    sub rsp, 16             ; Allocate 2 local variables (8 bytes each)

    mov qword [rbp-8],  100 ; local_var1 = 100
    mov qword [rbp-16], 200 ; local_var2 = 200
    mov rax, [rbp-8]
    add rax, [rbp-16]       ; RAX = 300

    mov rsp, rbp
    pop rbp
    ret

_start:
    ; Access struct fields
    mov rax, myPoint
    mov qword [rax + Point.x], 10
    mov qword [rax + Point.y], 20

    call calculate          ; RAX = 300

    mov rax, 60
    xor rdi, rdi
    syscall

Week 10 — Dynamic Memory & Linked Data Structures

Manage memory dynamically at runtime using syscalls and build pointer-based linked lists.

Core Focus

Managing memory dynamically at runtime using syscalls and building pointer-based data structures like linked lists.

Topics & Subtopics

  1. Dynamic Memory with the mmap Syscall — Since Assembly has no malloc() built in, you request memory directly from the kernel using the mmap syscall (number 9 on Linux), which maps a new region of virtual memory into your process. This is the low-level mechanism that malloc() itself is built on.
  2. Pointers & Pointer Arithmetic — A pointer is simply a register or memory location holding a memory address. Pointer arithmetic means adding or subtracting byte offsets to navigate between related memory locations — exactly how you traverse an array or follow a linked structure.
  3. Building a Singly Linked List — A linked list node contains a data field and a pointer ("next") to the following node, with the list's end marked by a null pointer (0). Each node can be allocated dynamically via mmap, then linked together by storing addresses in the "next" field.
  4. Traversing a Linked List — Traversal means starting at the head pointer and repeatedly following the "next" field until you reach a null pointer, processing each node's data along the way. This loop-with-pointer pattern is foundational to nearly all dynamic data structures.

Why This Week Matters: Dynamic memory is what separates toy programs from real applications — anything that grows at runtime needs it. Understanding mmap directly shows you what every OS-level allocator does. Linked lists built in Assembly demonstrate that every high-level data structure is ultimately just pointers and memory layouts.

Code Example

struc Node
    .data: resq 1          ; Value stored in this node
    .next: resq 1          ; Pointer to next Node (0 = end of list)
endstruc

section .text
    global _start
_start:
    ; mmap: request 4096 bytes from kernel
    mov rax, 9             ; syscall: mmap
    xor rdi, rdi           ; addr = NULL (kernel chooses)
    mov rsi, 4096          ; length = 4096 bytes
    mov rdx, 3             ; PROT_READ | PROT_WRITE
    mov r10, 0x22          ; MAP_PRIVATE | MAP_ANONYMOUS
    mov r8, -1             ; fd = -1
    xor r9, r9             ; offset = 0
    syscall                ; RAX = pointer to new memory

    ; Pointer arithmetic
    mov rsi, rax           ; RSI = base pointer
    add rsi, 8             ; Advance 8 bytes (one element forward)

    ; Linked list traversal
    ; RSI = head pointer
traverse_loop:
    test rsi, rsi          ; NULL check
    jz traverse_done
    mov rax, [rsi + Node.data]  ; Process node data
    mov rsi, [rsi + Node.next]  ; Follow the next pointer
    jmp traverse_loop
traverse_done:

    mov rax, 60
    xor rdi, rdi
    syscall

Week 11 — Error Handling, Syscall Return Values & Macros

Detect and handle syscall failures, and write reusable code with NASM macros.

Core Focus

Detecting and handling failures from system calls, and writing reusable code with NASM macros.

Topics & Subtopics

  1. Syscall Error Conventions (Negative Return Values) — Linux syscalls signal errors by returning a negative value in RAX corresponding to a negated errno code, rather than throwing an exception. Checking RAX after every syscall that can fail — file I/O, memory allocation — is essential defensive programming.
  2. Building an Error-Handling Routine — Without exceptions, error handling in Assembly means manually checking return codes after every risky operation and branching to dedicated cleanup code. A common pattern centralizes all error paths into a single labeled block that prints a message and exits cleanly.
  3. NASM Macros with %macro — A macro is a named, reusable block of Assembly code with parameters, expanded inline at assemble-time wherever it is invoked — similar to a C preprocessor macro but more powerful. Macros eliminate repetitive boilerplate like syscall setups.
  4. Conditional Assembly with %if/%ifdef — NASM supports compile-time conditional assembly, letting you include or exclude code blocks based on defined constants — useful for debug builds, platform-specific code, or feature flags. %ifdef checks if a symbol was defined via the -d command-line flag.

Why This Week Matters: Every production Assembly program checks syscall return values — ignoring them causes silent data corruption or crashes with no error message. NASM macros transform verbose, repetitive Assembly into readable, maintainable code, and are used in every serious Assembly codebase.

Code Example

; Macro: reduces 3-line syscall exit to 1 line
%macro EXIT 1
    mov rax, 60
    mov rdi, %1
    syscall
%endmacro

%macro SYS_WRITE 3
    mov rax, 1
    mov rdi, %1
    mov rsi, %2
    mov rdx, %3
    syscall
%endmacro

section .data
    filename db "data.txt", 0
    errmsg   db "File not found", 10
    errlen   equ $ - errmsg

section .text
    global _start
_start:
    ; open syscall with error check
    mov rax, 2             ; syscall: open
    mov rdi, filename
    xor rsi, rsi           ; O_RDONLY
    syscall
    cmp rax, 0
    jl open_failed         ; Negative RAX = error
    jmp open_ok

open_failed:
    SYS_WRITE 1, errmsg, errlen
    EXIT 1

open_ok:
    EXIT 0

Week 12 — Floating-Point (SSE/AVX) & Calling C Library Functions

Perform floating-point arithmetic using SSE registers and interoperate with the C standard library.

Core Focus

Performing floating-point arithmetic using SSE registers and interoperating with the C standard library.

Topics & Subtopics

  1. The XMM Registers & SSE Instructions — Floating-point math on modern x86-64 uses the dedicated XMM registers (XMM0–XMM15) rather than the general-purpose integer registers, accessed via SSE instructions. movsd moves a double-precision float; addsd/mulsd perform floating-point arithmetic.
  2. Converting Between Integers & Floats — Integer and floating-point values use entirely different binary representations, so converting between them requires dedicated conversion instructions. cvtsi2sd converts integer-to-double, while cvttsd2si converts double-to-integer (truncating).
  3. Calling C Library Functions (External Linking) — Assembly programs can call functions from the C standard library (like printf) by declaring them extern and linking against libc. This requires following the System V calling convention precisely and aligning the stack to 16 bytes before the call.
  4. Assembling & Linking with GCC for Libc Programs — When calling C library functions, you link using gcc instead of ld directly so that libc and the C runtime startup code are linked in automatically. This is the standard approach for Assembly programs that need C library features.

Why This Week Matters: Floating-point is used in game engines, scientific computing, graphics, and machine learning. Calling printf from Assembly is the gateway to using any C library (OpenGL, SDL, POSIX) from Assembly — dramatically expanding what you can build without reimplementing the world.

Code Example

extern printf
extern exit

section .data
    valA dq 3.14
    valB dq 2.0
    fmt  db "Result: %f", 10, 0

section .text
    global main
main:
    ; Floating-point with XMM registers
    movsd xmm0, [valA]    ; Load 3.14 into XMM0
    movsd xmm1, [valB]    ; Load 2.0 into XMM1
    addsd xmm0, xmm1      ; XMM0 = 5.14
    mulsd xmm0, xmm1      ; XMM0 = 10.28

    ; Integer to float conversion
    mov rax, 42
    cvtsi2sd xmm1, rax    ; XMM1 = 42.0

    ; Call printf (variadic: RAX = # of XMM args used)
    mov rdi, fmt
    ; xmm0 already holds the result to print
    mov rax, 1            ; 1 XMM register used
    call printf

    mov rdi, 0
    call exit

; Build: nasm -f elf64 program.asm -o program.o
;        gcc program.o -o program -no-pie

Week 13 — Recursion & Advanced Stack Management

Implement recursive algorithms in Assembly and deeply understand the stack's role in supporting them.

Core Focus

Implementing recursive algorithms in Assembly and deeply understanding the stack's role in supporting them.

Topics & Subtopics

  1. Recursive Procedures — A recursive procedure calls itself with modified arguments, relying on the stack to automatically maintain a separate frame (and separate local variables) for each recursive call. Every recursive call pushes a new return address and frame onto the stack, unwinding in reverse order as each call returns.
  2. Stack Depth & Stack Overflow — Each recursive call consumes additional stack space for its frame; with no base case or excessively deep recursion, the stack grows until it exceeds its allocated limit, causing a segmentation fault. Understanding this is critical for choosing between recursion and iteration for performance-sensitive problems.
  3. Tail Call Patterns (Manual Optimization) — A tail call is a recursive call that is the very last operation in a function, allowing it to be manually rewritten as a loop (avoiding stack growth entirely) — a transformation compilers do automatically in high-level languages but you must do by hand in Assembly. This converts O(n) stack usage into O(1).
  4. Recursive Data Structure Traversal (Binary Tree) — Recursive procedures map naturally onto recursive data structures like binary trees, where each node's traversal logic calls the same procedure on its left and right children. This mirrors how recursive algorithms are written in high-level languages, just with explicit register and stack management.

Why This Week Matters: Recursion in Assembly is the clearest demonstration of what the stack actually does — you can watch frames pile up in GDB in real time. Understanding tail call optimization by hand is a master-level skill — it is the basis of how functional language compilers eliminate stack overflows.

Code Example

section .text
    global _start

; Recursive factorial(n): RDI = n → result in RAX
factorial:
    cmp rdi, 1
    jle .base_case        ; Base case: 0! = 1! = 1
    push rdi              ; Save n (caller-saved — we need it after the recursive call)
    dec rdi
    call factorial        ; Recursive call: factorial(n-1)
    pop rdi               ; Restore original n
    imul rax, rdi         ; RAX = factorial(n-1) * n
    ret
.base_case:
    mov rax, 1
    ret

; Tail call optimization: sum(n) rewritten as loop
sum_iterative:
    xor rax, rax          ; Accumulator = 0
.loop:
    cmp rdi, 0
    je .done
    add rax, rdi          ; acc += n
    dec rdi
    jmp .loop             ; No recursion — constant stack usage!
.done:
    ret

_start:
    mov rdi, 10
    call factorial        ; RAX = 3628800
    mov rax, 60
    xor rdi, rdi
    syscall

Week 14 — SIMD & Performance Optimization

Process multiple data elements simultaneously with SIMD instructions and understand CPU-level performance.

Core Focus

Processing multiple data elements simultaneously with SIMD instructions and understanding CPU-level performance considerations.

Topics & Subtopics

  1. SIMD Fundamentals: Processing Multiple Values at Once — SIMD (Single Instruction, Multiple Data) lets one instruction operate on several values packed into a single wide register simultaneously — a 128-bit XMM register can hold four 32-bit integers and operate on all of them in one instruction. This is how high-performance code achieves massive speedups in video processing, machine learning, and graphics.
  2. Memory Alignment for SIMD — SIMD instructions like movdqa (aligned move) require data to start at a memory address divisible by 16 (or 32 for AVX), and will crash with a fault if given misaligned data. The align directive ensures your data section meets this requirement; the unaligned variant movdqu is slightly slower but works with any address.
  3. Loop Unrolling — Loop unrolling manually duplicates a loop's body multiple times per iteration, reducing the overhead of branch instructions and loop-counter updates relative to actual work performed. This is a classic manual optimization technique, trading code size for execution speed.
  4. Branch Prediction & Avoiding Pipeline Stalls — Modern CPUs use deep pipelines and predict which way a conditional branch will go, paying a significant performance penalty (a "pipeline flush") when wrong. Writing branch-light code using conditional moves like cmovg instead of jumps can meaningfully improve hot-loop performance.

Why This Week Matters: SIMD is the difference between a program that processes 1 million elements per second and one that processes 4 or 8 million with zero extra hardware. Understanding branch prediction and pipeline stalls is what separates a programmer who writes fast code from one who writes code and hopes it is fast.

Code Example

section .data
    align 16
    vecA dd 1,  2,  3,  4      ; Four packed 32-bit integers (16 bytes)
    vecB dd 10, 20, 30, 40

section .text
    global _start
_start:
    ; SIMD: add all 4 pairs simultaneously
    movdqa xmm0, [vecA]        ; Load 4 ints at once (aligned)
    paddd  xmm0, [vecB]        ; xmm0 = [11, 22, 33, 44] in ONE instruction

    ; Loop unrolling: 4 elements per iteration instead of 1
    mov rsi, vecA
    mov rcx, 0
unrolled_loop:
    mov rax, [rsi + rcx*4 + 0]   ; Element 0
    mov rbx, [rsi + rcx*4 + 4]   ; Element 1
    mov rdx, [rsi + rcx*4 + 8]   ; Element 2
    mov r8,  [rsi + rcx*4 + 12]  ; Element 3
    add rcx, 4
    cmp rcx, 100
    jl unrolled_loop

    ; Branchless max using cmov (no misprediction risk)
    mov rcx, rbx
    cmp rax, rbx
    cmovg rcx, rax               ; rcx = rax ONLY if rax > rbx

    mov rax, 60
    xor rdi, rdi
    syscall

Week 15 — Interfacing with the OS: Files, Threads & Concurrency

Perform real file I/O and explore low-level concurrency through Linux threads and atomic synchronization.

Core Focus

Performing real file I/O and exploring low-level concurrency through Linux threads and synchronization primitives.

Topics & Subtopics

  1. File I/O: open, read, write, close — Real file operations in Assembly use four core syscalls in sequence: open (returns a file descriptor), read/write (transfer data using that descriptor), and close (releases it). Always check RAX for errors after open, since a missing file or permissions issue returns a negative value.
  2. Linux Threads with clone — The raw clone syscall is the low-level primitive underlying pthread_create() — it spawns a new thread of execution sharing the same address space as the parent process. Working with clone directly requires manually setting up a new stack for the child thread.
  3. Atomic Operations with lock Prefix — The lock prefix makes certain instructions (like xadd, cmpxchg, inc) execute atomically across multiple CPU cores, preventing race conditions when multiple threads access shared memory simultaneously. This is the hardware-level foundation that mutexes and atomic variables in high-level languages are built upon.
  4. Compare-and-Swap (cmpxchg) for Lock-Free Programming — cmpxchg atomically compares a memory location against an expected value and, only if they match, swaps in a new value — the fundamental building block of lock-free data structures. Combined with the lock prefix, it enables thread-safe updates without traditional mutexes.

Why This Week Matters: Every file operation, network call, and thread ever created on Linux ultimately calls these syscalls. Understanding clone and the lock prefix at the hardware level means you truly understand what pthreads, Go goroutines, and Rust's Mutex do under the hood — a rare and valuable perspective.

Code Example

section .data
    align 8
    sharedCounter dq 0
    filename db "data.txt", 0

section .bss
    buffer resb 256

section .text
    global _start

; Atomic increment — safe across CPU cores
increment_counter:
    lock inc qword [sharedCounter]   ; Indivisible read-modify-write
    ret

_start:
    ; File I/O: open → read → close
    mov rax, 2             ; syscall: open
    mov rdi, filename
    xor rsi, rsi           ; O_RDONLY
    syscall
    mov r12, rax           ; Save file descriptor

    mov rax, 0             ; syscall: read
    mov rdi, r12
    mov rsi, buffer
    mov rdx, 256
    syscall

    mov rax, 3             ; syscall: close
    mov rdi, r12
    syscall

    ; Atomic CAS: if [sharedCounter] == 0, set to 1
    mov rax, 0             ; Expected value
    mov rbx, 1             ; New value
    lock cmpxchg [sharedCounter], rbx
    jz swap_ok             ; ZF=1: swap succeeded
swap_ok:
    mov rax, 60
    xor rdi, rdi
    syscall
